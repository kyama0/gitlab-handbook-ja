## エージェントの仕組み

AI モデルは単一の呼び出しです - プロンプトを送ると応答が返ってきます。エージェントは、同じモデルをループで実行するものです: リクエストを受け取り、アクション（ファイルを読む、コマンドを実行する、コードを編集する）を決定し、結果を観察し、タスクが完了するまで繰り返します。

より深く知るには、Anthropic の [building effective agents](https://www.anthropic.com/research/building-effective-agents) を参照してください。

## MCP の仕組み

MCP（Model Context Protocol）サーバーは、Snowflake、GitLab、dbt、Slack などの外部ツールやデータソースに接続することで、エージェントができることを拡張します。アクティブな各 MCP はエージェントのコンテキストウィンドウに加算されるため、目の前のタスクに実際に必要なものだけを有効にしてください。

## 設定

OpenCode は実行時にマージされる 2 つの設定ファイルを使用します - グローバル設定（`~/.opencode/config.json`）と、プロジェクトレベルの設定（リポジトリルートにある `.opencode/config.json`）です。両者が競合する場合、プロジェクト設定がグローバルを上書きします。

グローバル設定は最小限に保ちましょう。そこに置く価値があるのは、何に取り組んでいるかに関係なく、すべてのセッションで必要な MCP です。それ以外は、特に `dbt-mcp` のようなリポジトリ固有の MCP は、プロジェクト設定に属します。たとえば、`dbt-mcp` は analytics リポジトリの設定にのみ存在し、そこでしか関係ないからです。

ルールオブサム: どのプロジェクトの外で OpenCode を開いたときでも MCP をアクティブにしておきたいなら、グローバルへ。そうでなければ、ローカルに保ちます。

## プロンプトのベストプラクティス

コンテキストを軽量に保ちましょう。フォーカスを切り替えるたびに、新しい会話を始めてください - 新機能、別のバグ、関係のないレビューなど。迷ったら、新しいウィンドウへ。OpenCode では、`/new` コマンドでこれを行えます。これは応答の品質を向上させ、コストも下げます

取り組んでいる内容に対するスキルが存在する場合は、それを使ってください - スキルはエージェントに、それ以外の方法では得られないドメイン固有のコンテキストを与えてくれます。

## Plan モードと Build モードの使い分け

OpenCode には 2 つの主要モードがあります。

- **Plan** — リクエストと関連コードをレビューし、変更を加える *前* に詳細なアプローチを提案します
- **Build** — 変更を直接実行します

簡単ではないことには必ず Plan を先に実行しましょう。実装に手をつける前に、意外にも設計上の問題を捉えるのが得意です。

`plan` をデフォルトエージェントとして設定している場合（[セットアップガイド](agent-setup.md#step-6-set-plan-as-your-default-agent) で推奨）、デフォルトで Plan モードになります。実行する準備ができたら明示的に Build モードに切り替えてください。

## 利用可能なスキルとエージェント

スキルは、再利用可能なチーム固有の知識 - 規約、ワークフロー、ベストプラクティスをエンコードします。それらは本来、セッションごとに改めて説明する必要があるものです。ゼロからプロンプトを書く代わりに、スキルを呼び出すことで、エージェントに正しいコンテキストを即座に与えられます。

データチームが開発した利用可能なスキル一覧については、Agentic Tool Development ガイドの [利用可能なスキル](agentic-tool-development.md#available-skills) セクションを参照してください。
