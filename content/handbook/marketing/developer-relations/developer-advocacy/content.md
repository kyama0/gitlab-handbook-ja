Developer Advocacy チームは、キャンペーン、フィールドイネーブルメント、製品ローンチ支援、ユースケース採用、顧客向けデモ、一般的な学習やオンボーディングに使えるコンテンツを作成します。チームが参加するすべてのコンテンツと活動は、[チームのワークフロー](/handbook/marketing/developer-relations/developer-advocacy/workflow/)に従って Issue で追跡されます。

## コンテンツのリクエスト

**注**: 更新済みまたは新規作成のコンテンツアセットが必要な場合は、[コンテンツリクエストワークフロー](/handbook/marketing/developer-relations/content-request/#how-to-work-with-the-developer-advocate-team)に従ってください。

### ワークショップ

Developer Advocacy チームは、さまざまな製品・フォーカス領域でワークショップをメンテナンスしています。ワークショップにはいくつかのタイプがあります。受講者がエクササイズにライブで参加するハンズオン（トレーニングコース）、または受講者が質問しながら参加するショーケース型のライブデモ／プログラミングセッションです。ワークショップは [Solution Architect](/handbook/solutions-architects/) チームと [Customer Success](/handbook/customer-success/) チームと協力して作成し、ビジネス目標に沿うようにすべきです。

> 注: 特定の顧客に焦点を当てたワークショップは、[顧客コンサルティングハンドブック](/handbook/marketing/developer-relations/developer-advocacy/consultancy/)に従ってリクエストできます。

### 動画ガイドライン {#video-guidelines}

動画の録画について:

1. 動画機材
   - マイク: [Logitech Blue Yeti](https://www.logitechg.com/en-us/shop/p/yeti-premium-usb-microphone)
   - カメラ: [Logitech MX Brio Ultra HD 4k](https://www.logitech.com/en-us/shop/p/mx-brio-4k-webcam)
   - 照明: [Elgato Key Light](https://www.elgato.com/us/en/p/key-light)
1. ソフトウェア:
   - 画面録画ソフト: Quicktime、[OBS](#recording-with-obs)、または [Screen Studio](#recording-with-screen-studio) をおすすめします。
   - 後編集ソフト: Adobe Premiere Pro（動画のカット、音声ゲインのブーストなど）
   - アクセスリクエスト: @johncoghlan にツールへのアクセスを依頼し、[アクセスリクエスト](/handbook/security/corporate/end-user-services/access-requests/)を作成してください。
1. 従うべき動画設定:
   - アスペクト比／解像度: 録画ウィンドウのサイズを 16x9 のアスペクト比に設定します。
       - 一般的な解像度は 1920×1080（16:9 のアスペクト比）または、画面録画では 2880×1750 です。
       - 4K モニターに接続している場合は、ラップトップからキャプチャする方が簡単な場合があります。
       - ソーシャル向けに録画する場合は、マルチプラットフォーム互換のために正方形（1:1）または縦型（9:16）を検討してください。
   - Brand チームが編集する動画で使われる動画を Screen Studio で録画する場合は、次のガイダンスに従ってください:
       - Screen Studio ではすべての Zoom（ズーム）を削除してください。Brand チームが必要に応じて Zoom を追加します。
       - ウェブカメラ／ユーザービューは録画しないでください。
       - 画面サイズを最大化するため、角丸とパディングを 0% に設定してください。
       - `Always use pointer cursor` をオンにしてください。
       - カーソルサイズを 1.2x に設定し、最も左の `Cursor style` オプションを使ってください。
       - 次のようにエクスポートします: Format - MP4、Resolution - 1080p、Compression - Web。
   - その他のコンテンツタイプ（ソーシャル、YouTube への直接公開、ウェビナーなど）の場合:
       - Zoom の使用については、そのコンテンツの DRI と相談してください。ただし、Screen Studio がフォーカスすべき適切なコンテンツを正しく特定しているか確認するため、その使用をレビューすべきです。
       - ウェブカメラの使用が推奨されます。実際のユーザーが見えることで、オーディエンスとの信頼やつながりを築きやすくなります。注: 録画後にウェブカメラ映像を隠したい場合は、**Camera** アイコンをクリックして **Hide camera** スライダーを切り替えることで、ウェブカメラなしのバージョンをエクスポートできます。
       - ボーダーやフレームの使用は役立つ場合がありますが、それによってコンテンツの読みやすさが損なわれないよう注意してください。
       - `Always use pointer cursor` をオンにしてください。
       - カーソルサイズを 1.2x に設定し、最も左の `Cursor style` オプションを使ってください。
       - 次のようにエクスポートします: Format - MP4、Resolution - 1080p、Compression - Web。

デモ環境のセットアップ:

1. ユーザーアカウント
   - ブランドデモには [demo-tanuki アカウント](https://gitlab.com/demo-tanuki) を優先してください。（ログイン情報は 1Password の Marketing vault にあります）

1. [Dev Environments](/handbook/marketing/developer-relations/developer-advocacy/dev-environments/) での IDE

#### Screen Studio での録画 {#recording-with-screen-studio}

Screen Studio は画面領域、ウィンドウのキャプチャを提供し、ウェブカメラビューと組み合わせられます。シンプルで直感的な録画・編集 UI を備えており、タイムラインで編集できるインテリジェントな自動[「シーンへのズーム」](https://screen.studio/guide/adding-editing-zooms)を使います。

Screen Studio は[録画を開始する](https://screen.studio/guide/new-recording)ためのミニマルなオーバーレイ UI を提供します。

![Screen Studio のオーバーレイメニュー](/images/handbook/marketing/developer-relations/developer-advocacy/content/screenstudio-overlay-menu.png)

1. Display、Window、または Area を選択します
   - Screen Studio はどんな解像度でも録画できますが、[仕様](#screen-studio-recording-specifications)に応じて 16:9 または 4:3 を使うことが推奨されます。
   - Area: AR を選択し、録画領域をドラッグして指定します。

   ![Screen Studio のエリア選択](/images/handbook/marketing/developer-relations/developer-advocacy/content/screenstudio-recording-area-selector.png)

1. record をクリックして録画を開始します。

録画が終わったら、まず[プロジェクトを保存](https://screen.studio/guide/saving-your-project)してから、エディターを開いてください。

![Screen Studio のエディター](/images/handbook/marketing/developer-relations/developer-advocacy/content/screenstudio-editor.png)

1. エディターでは、最終的な動画を保存する前に、すべての設定、動画の配置、Zoom のタイムラインなどを確認できます。
1. このページの[動画ガイドラインセクション](/handbook/marketing/developer-relations/developer-advocacy/content/#video-guidelines)に詳述されている仕様に従っていることを確認してください。
1. これらの調整を行った後、プロジェクトを保存します。
1. 次に `export` と `Export to file` を選択します。
1. [エクスポートダイアログ](https://screen.studio/guide/exporting-the-video)で、`.mp4` ドロップダウンから `mp4` を選択します。

Screen Studio はプロジェクトを `~/Screen Studio Projects` フォルダにディスク保存します。プロジェクトファイルは名前と `.screenstudio` 拡張子を持ち、生の動画ファイルと設定が入ったコンテナです。

!![ディスク上の Screen Studio プロジェクト](/images/handbook/marketing/developer-relations/developer-advocacy/content/screenstudio-projects-on-disk.png)

ターミナルで `ls` を使って確認したり、`Screen Studio` のプロジェクトリストから再度開いたりできます。

![Screen Studio - 以前のプロジェクトを開く](/images/handbook/marketing/developer-relations/developer-advocacy/content/screenstudio-open-previous-projects.png)

コンテンツの納品が終わったら、ローカルストレージを節約するため、すべての動画ファイル（エクスポートしたアーティファクトと Screen Studio プロジェクト）をチームの GDrive にアップロードすることが推奨されます。

#### OBS での録画 {#recording-with-obs}

OBS は画面録画に対してより高度なコントロールと柔軟性を提供しますが、その代わりにセットアップや編集のプロセスがより複雑になります。

覚えておくべきいくつかのヒントを紹介します:

1. Scene - シーンは、同時に録画するソースの集まりです。
1. Sources - ソースには音声、画面録画、ウェブカメラなどがあります。
   - ソースを追加するには、`+` アイコンをクリックします。よく使う 3 つのソースは次のとおりです:
      - macOS Screen Capture: 画面録画
      - Audio Input Capture: 任意。直接の音声録音が必要な場合
      - Video Capture Input: 任意。録画にスピーカー／ウェブカメラを入れる場合
   - Macbook の画面を録画するには、`macOS Screen Capture` を使い、ソースに名前を付け、右クリックしてソースのプロパティを編集します。
   - Properties - これらは変更できるソースの要素です。異なる画面（ラップトップ vs モニター）を選んだり、画面全体・ウィンドウ・特定のアプリケーションのどれをキャプチャするかを選んだりできます。
1. Preferences - プリファレンスはシーンがどのように録画されるかをコントロールします。次のプリファレンスをおすすめします:
   - OBS > Preferences > Video
      - Base (Canvas) Resolution 1920x1080、Aspect ratio 16:9
      - Output (Scaled) Resolution 1920x1080、Aspect ratio 16:9
   - OBS > Preferences > Output > Dropdown: Advanced。
      - OBS > Preferences > Advanced: 「Recording > Automatically remux to mp4」を有効化します（これは mkv コンテナをインポートできない Adobe Premiere Pro にとって重要です）。

追加のガイダンスは以下にあります:

- [OBS Recording Studio Guide](https://docs.google.com/presentation/d/16b58bVgJfM2nIKFyVRurNsNJoK2yqS_1QKHRp0F7CNo/edit?slide=id.g3357bbe63e5_0_183#slide=id.g3357bbe63e5_0_183)
- [Developer Advocacy showcase notes](https://docs.google.com/document/d/1co-DKoR4xn4JNIcynWPjoJsWo3vYzwj8WDFzX3uKrH0/edit?tab=t.0#heading=h.yrmatlea9dpq) (内部) と[録画](https://www.youtube.com/watch?v=qNH8NDShGJA) (公開)

#### プロ用放送マイクでの音声ゲインブースト

プロ用の放送マイク（例: Shure SM7B）を使っていて音声ゲインが低すぎる場合は、いくつかのオプションがあります:

1. ハードウェアによる解決策（オーディオインターフェース + Cloudlifter）は初期の結果を大幅に改善します。SM7B マイクはコンピューターに届く前にその追加ゲインを本当に必要とするためです。
1. OBS 録画
   - 録画中に OBS で音声ゲインを上げます。
   - `Source > Audio Capture Device` を見つけます。
   - 右クリックして `Filters` を選択します。
   - Gain を +10 または +20dB に設定します。
   - Noise Suppression: RNNoise または Speex
   - Compressor: Ratio 3:1、Threshold -18dB、Attack 6ms、Release 60ms
   - Limiter: Threshold -6dB、Release 60ms
   - ヒント: ベストプラクティスの設定は [Claude](/handbook/tools-and-tips/ai/claude/) に尋ねてください（プロンプト: `How to boost audio in OBS for a SureSM7B microphone as input?`）。
1. Screen Studio の後編集
   - Screen Studio でエクスポートした録画を Adobe Premiere Pro で編集し、音声を調整します。
   - 新しいプロジェクトを作成し、メディアブラウザに動画ソースを追加します。
   - 動画を右側のエディタースコープにドラッグ＆ドロップします。
   - 音声トラックを右クリックし、`Audio Gain` を選択します。
   - `Set Gain to:` を選んで `25 dB` を入力します。

### プロダクトツアーガイドライン {#product-tours-guidelines}

#### Navattic で GitLab ツアーを作成するためのガイドラインとヒント {#guideline-and-tips-for-creating-gitlab-tours-with-navattic}

1. **簡潔に保つ** – プロダクトツアーは理想的には **5 〜 12 ステップ** にし、各ステップが明確な価値を届けることに集中すべきです。説明的になりがちなチュートリアルとは異なり、プロダクトツアーは手早く魅力的であるべきです。
2. **コンテキストを提供する** – ポップアップダイアログにブログ記事やドキュメントへのリンクを追加し、メインコンテンツを散らかさずに追加リソースを提供します。
3. **GitLab ブランディングを適用する** – テーマ設定で **「GitLab Branding [Navattic Team]」** を選択し、一貫したデザインを確保します。
4. **適切なキャプチャタイプを使う** – Navattic は 2 種類の画面キャプチャをサポートしています:
   - **Web captures**（推奨）は製品内体験を作り出します。
   - **Screenshots** は非ウェブベースのアプリケーション（例: IDE）に使うべきです。
5. **正しい解像度を設定する** – Chrome でキャプチャする際は、Navattic 推奨の解像度である **1440x900** にブラウザを設定します。Navattic の録画ツールバーにある組み込みのリサイズツールを使って解像度を調整できます。
6. **コールトゥアクションを含める** – 最後のステップには **CTA**、通常は **Free Trial** リンクを掲載すべきです。
7. **関連する要素をハイライトするために Beacon の使用を検討する** – 主要な要素を強調するために **Highlight** の外観設定を適用します。
8. **キーボードナビゲーションを有効にする** – ツアー設定のこの機能により、ユーザーは矢印キーを使ってツアーをより滑らかに操作できます。
9. **Presenter Mode を使う** – この機能は必要に応じてポップアップを隠し、よりクリーンなビジュアル表現を確保します。
10. **ポップアップを適切に設定する**:
    - 最初と最後のステップには、**heavy backdrop** 付きの **Modal** を使います。
    - 中間のステップには **Tooltip** を使い、適切な場合は **Backdrop 設定** を有効にします。
11. ダイアログでは、「Next」の代わりに「Commit」「Install」「Submit」のようなアクション固有のラベルを使うようにしてください。ユーザーが製品に能動的に関わっていると感じられるようにします。
12. **利用可能な例を参照する** – [既存のプロダクトツアー](#product-tours)を確認し、推奨フォーマットを理解してください。
13. 公開向けのプロダクトツアーをリリースする前に、Developer Advocacy チームにレビューを依頼してください。

#### Navattic へのアクセスリクエスト {#requesting-access-to-navattic}

チームにツアーに関する特別な要件があり、Navattic で自分自身でツアーを開発したい場合は、私たちが支援できます。

- アクセスをリクエストし、Navattic を使うための **正当なビジネス上の理由** を提示したチームメンバーには、**3 か月の試用ライセンス** を提供できます。
- ユーザーがツールを積極的に活用している場合は、必要に応じて **アクセスを延長** したり、追加ライセンスの購入を検討したりすることがあります。
- ユーザーがツールを積極的に使っていない場合は、そのライセンスが別のチームメンバーに **再割り当て** される場合があります。
- アクセスをリクエストするには、Lumos（Okta ダッシュボードで利用可能）でリクエストを開いてください。

質問やサポートについては、チームチャンネル [#dev-advocacy-team](https://app.slack.com/client/T02592416/CMELFQS4B) で連絡してください。

#### 新しい Navattic ユーザーの追加（Developer Advocacy チームのプロセス）

このセクションでは、Developer Advocacy チームが新しいチームメンバーを Navattic にオンボーディングするための手順を説明します。

チームメンバーが Navattic へのアクセスをリクエストしたら、次のプロセスに従ってください:

1. Navattic 管理者は Slack またはメール経由で Lumos リクエストを受け取ります。
2. **リクエストを検証する** – リクエスト者が Navattic を使う正当なビジネス上の理由を提示していることを確認します。
3. **ライセンスの空きを確認する** – [Navattic ユーザー追跡スプレッドシート](https://docs.google.com/spreadsheets/d/1x5T3rXhsqUZYHdMAz1BWt5gExNA_MLO9AtFpTVbzDwc/edit?gid=0#gid=0)を確認し、私たちのライセンスプールに追加ユーザー分の空きがあることを確認します。
4. **Lumos でリクエストを承認する** – 検証が済んだら、Okta アクセスのために Lumos でアクセスリクエストを承認します。
5. **Navattic の招待を送る** – Navattic で、GitLab チームの下に Builder ロールでチームメンバーを招待します。
6. **ガイドラインを送る** – [Navattic ガイドライン](#guideline-and-tips-for-creating-gitlab-tours-with-navattic)をチームメンバーと共有します。
7. **試用期間を伝える** – [Navattic へのアクセスリクエスト](#requesting-access-to-navattic)のプロセスに基づき、アクセスが **デフォルトで 3 か月間** 付与されることをチームメンバーに知らせます。
8. **追跡スプレッドシートを更新する** – [Navattic ユーザー追跡スプレッドシート](https://docs.google.com/spreadsheets/d/1x5T3rXhsqUZYHdMAz1BWt5gExNA_MLO9AtFpTVbzDwc/edit?gid=0#gid=0)に、チームメンバーの名前、メール、開始日、終了予定日を追加します。
9. **Slack チャンネルに追加する** – 継続的なコミュニケーションと更新のため、チームメンバーを [#navattic-updates](https://gitlab.slack.com/archives/navattic-updates) Slack チャンネルに招待します。

#### プロダクトツアーの公開

プロダクトツアーは以下の場所で公開されます:

- [Developer Advocacy コンテンツライブラリ](#product-tours)
- [Highspot](#highspot)
- [GitLab ドキュメント](https://docs.gitlab.com/)（デモの[リンク](https://docs.gitlab.com/ee/development/documentation/styleguide/#link-to-click-through-demos)については GitLab ドキュメントのスタイルガイドに従ってください）

#### プロダクトツアーへの貢献

Developer Advocacy チームは、**Advanced SAST**、**GitLab Duo**、**GitLab Duo with Amazon Q**、[その他](#product-tours)など、主要なワークフローを紹介し新機能を強調するプロダクトツアーを作成します。チームは、ユーザーや潜在顧客に一貫した高品質な体験を提供するため、プロダクトツアーの構築・レビュー・公開に取り組んでいます。追加のツアーを提案し、コンテンツ、台本、環境を提供することで、あなたも貢献できます。Developer Advocacy チームと協力して新しいプロダクトツアーを作成する方法は次のとおりです:

1. [プロダクトツアー提出テンプレート](https://gitlab.com/gitlab-com/marketing/developer-relations/developer-advocacy/developer-advocacy-meta/-/issues/new?issuable_template=product-tour-submission-template)を使ってアイデアを提案します。
1. テンプレートのすべての質問に答えます。これにより、Developer Advocacy チームがアイデアを効果的に評価できます。
1. Developer Advocacy チームがピッチをレビューし、次のいずれかを行います: a) ツアーを承認し、チームの他の優先事項に基づいて作成のタイムラインを提案する、b) 改善案を提示する、c) そのアイデアがツアーに合わない理由を説明し、メッセージを発信するための別のアイデアを提示する。
1. ツアーが承認されたら、著者は **プロダクトツアーテンプレート**（Google ドキュメント）を使って台本を作成します。テンプレートへのリンクは[プロダクトツアー提出テンプレート](https://gitlab.com/gitlab-com/marketing/developer-relations/developer-advocacy/developer-advocacy-meta/-/issues/new?issuable_template=product-tour-submission-template)にあります。
1. 提出テンプレートの Issue のコメントに、テンプレートへのリンクを追加します。
1. Developer Advocacy チームは著者と協力して台本を作り、Navattic を使ってツアーを作成し、該当する場合は公式のターゲット（例: ウェブサイト、ドキュメント、HighSpot、Developer Advocacy ハンドブック）で公開します。

## コンテンツ配信 {#content-distribution}

### URL タグ付けと追跡のための UTM

Developer Advocacy チームは、より大きな [Marketing UTM 戦略](/handbook/marketing/utm-strategy/)に基づく Developer Relations チームの [UTM 戦略](/handbook/marketing/developer-relations/utm-strategy/)に従って活動しています。Developer Evangelism チームの `utm_content` プレフィックスは `de_` で、これにより Sisense でチームのデータを簡単にフィルタリングできます。

Community UTM ページの [UTM Generator](/handbook/marketing/developer-relations/utm-strategy/#utm-generator) を使うと、キャンペーン用の UTM コードを簡単に生成できます。

私たちは以下のキャンペーンを使っています:

1. [Blog posts (general)](https://campaign-manager.gitlab.com/campaigns/view/17) - 私たちが著者ではないすべての GitLab ブログ。ソーシャルメディア向けのコンテンツ追跡。
1. [Blog posts (authors)](https://campaign-manager.gitlab.com/campaigns/view/18) - 私たちが著者である GitLab ブログ記事。ソーシャルメディア向けのコンテンツ追跡。
1. [Talk resources (general)](https://campaign-manager.gitlab.com/campaigns/view/13) - トークスライド用の QR コードと短縮 URL。
1. [Community newsletter](https://campaign-manager.gitlab.com/campaigns/view/27) - コミュニティニュースレターの短縮 URL。
1. [Demos (TMM, DE)](https://campaign-manager.gitlab.com/campaigns/view/50) - スタンドアロン URL を持つ技術デモ。
1. [Release Evangelism](https://campaign-manager.gitlab.com/campaigns/view/3) - [リリースエバンジェリズム活動](/handbook/marketing/developer-relations/developer-advocacy/social-media/#release-evangelism)。
1. [Podcast resources](https://campaign-manager.gitlab.com/campaigns/view/38) - ポッドキャストのホストと共有する URL。
1. [Contributed articles](https://campaign-manager.gitlab.com/campaigns/view/47) - ソーシャルメディアで共有する外部記事。

イベント・コンテンツ固有の追跡例は [KubeCon EU 2023](https://campaign-manager.gitlab.com/campaigns/view/36) です。

### コンテンツ配信ワークフロー

コンテンツを制作・公開した後、次のステップは配信です。プロセスを支援するいくつかのステップを紹介します:

1. [UTM Generator](/handbook/marketing/developer-relations/utm-strategy/#utm-generator) を使って、コンテンツ用の UTM コードと短縮 URL を作成します。これについての詳細は Developer Advocacy の [UTM 戦略](/handbook/marketing/developer-relations/utm-strategy/)にあります。

ドキュメントとコミュニティ向け:

1. 誰にとっても役立つチュートリアルブログ記事やデモなどは、[GitLab ドキュメント](https://docs.gitlab.com/)に追加すべきです。貢献ドキュメントに従い、まだ存在しない場合は[関連トピック](https://docs.gitlab.com/ee/development/documentation/topic_types/#related-topics)の見出しを作成します。MR を作成し、コンテンツの短縮 URL を使い、[担当のテクニカルライター](/handbook/product/ux/technical-writing/#assignments)にレビューを依頼します。MR の例: [Add tutorial blog posts to workspaces docs](https://gitlab.com/gitlab-org/gitlab/-/merge_requests/127500)
1. [GitLab Forum](https://forum.gitlab.com/) や [r/gitlab](https://www.reddit.com/r/gitlab) サブレディットのような、私たちのコミュニティプラットフォームのいずれかでコンテンツを共有することを検討してください。GitLab より広いトピックを扱っている場合は、他のサブレディットや、dev.to のようなブログサイトへのクロスポストも検討できます。

ソーシャルメディア向け:

1. ソーシャルメディアでコンテンツを共有するためのコピーを作成します。ヒントやさまざまなプラットフォームの概要については[コンテンツ共有](/handbook/marketing/developer-relations/developer-advocacy/social-media/#content)を参照してください。絵文字など追加のエンゲージメントのアイデアについては[メッセージのヒント](/handbook/marketing/developer-relations/developer-advocacy/social-media/#message-tips)を確認してください。
1. 他の GitLab チームメンバーも共有できるよう、EveryoneSocial でストーリーとしてコンテンツを提案します。[EveryoneSocial でコンテンツを提案する方法の詳細](/handbook/marketing/integrated-marketing/digital-strategy/social-marketing/team-member-social-advocacy/#suggesting-content-for-team-members-to-share-on-everyonesocial)。
1. ソーシャルメディアにコンテンツを投稿した後、ソーシャルメディアチームにプロモーションを依頼するため、[`#social-media-action`](https://gitlab.slack.com/archives/C01AZ9C8Z4G) Slack チャンネルにリンクを共有します。

GitLab チーム向け:

1. Field チームに役立つコンテンツは、[Highspot](https://gitlab.highspot.com/) にも投稿すべきです。
1. 次のメッセージテンプレートを使って、[#dev-advocacy-team](https://app.slack.com/client/T02592416/CMELFQS4B) Slack チャンネルで更新を共有します:

```text
:results-tanuki: <Content type> published: <title>

Social short UTM URLs:

1. LinkedIn:
2. Twitter:
3. Mastodon:

Content epic: <URL>

Thanks/cc @teammembers
```

例:

```text
:results-tanuki:  Blog published: Set up your infrastructure for on-demand, cloud-based development environments in GitLab

Social short UTM URLs:

1. LinkedIn: https://go.gitlab.com/EHIjRt
2. Twitter: https://go.gitlab.com/uz7OSE
3. Mastodon: https://go.gitlab.com/pFxdKa

Content epic: <URL>

Thanks a lot @HelpfulCoworker for editing this long read :handshake: :purple_heart:
```
