GitLab ブログ ハンドブックへようこそ！[GitLab ブログ](https://about.gitlab.com/blog/) はコンテンツマーケティングチームが管理しています。ブログのマネージング・エディターは Sandra Gittlen (@sgittlen) です。

## ブログ投稿とは何ですか？

GitLab では、ブログ投稿は主にオーディエンス（[DevSecOps プロフェッショナル](/handbook/product/personas/)）と有用な情報を共有することに焦点を当てています。ブログ投稿を提案または執筆する際は、読者に何を提供するかを常に考えてください。投稿が内部向けまたは個人的なエッセイのような内容であれば、ブログには適さない可能性があります（ただし、個人の LinkedIn ページには掲載できます）。

ブログは以下のカテゴリに分類されます:

- テクニカルチュートリアル／ハウツー
- 視点／ソートリーダーシップ
- 機能および能力の紹介
- オープンソースコミュニティ
- 顧客事例研究／インタビュー
- 会社のアナウンス（Executive comms チームとのパートナーシップで実施）
- 機能／変更／その他のアナウンス
- パートナーシップ／アライアンスをハイライトするゲストブログ

## 誰が GitLab ブログにコンテンツを公開できますか？

[誰もが GitLab に貢献できます](/handbook/company/mission/#mission)。ブログについては、これはあなたのブログ提案、アイデア、ドラフトを歓迎することを意味します。ただし、[メインブログ](https://about.gitlab.com/blog/) は GitLab の多くの公式の声の一つであり、つまりそこに公開されるコンテンツは、私たちが会社と製品の両方の GitLab を正確に代表していることを確実にするために、慎重に審査される必要があります。Blog Managing Editor および Director of Global Content Marketing は[公式 GitLab ブログの直接責任者 (DRI)](/handbook/people-group/directly-responsible-individuals/) であり、この責任が課されています。

GitLab ブログについて質問がある場合は、[sgittlen@gitlab.com](mailto:sgittlen@gitlab.com) までご連絡ください。

## ブログアイデアの提案方法 - 新プロセス {#how-to-suggest-a-blog-post}

ブログアイデアを提案するには、[ブログ提出テンプレート](https://gitlab.com/gitlab-com/marketing/brand-product-marketing/content-strategy-and-ops/blog/-/issues/new?issuable_template=blog-post-submission) を使用して Issue を作成してください。

アイデアを提出する際:

- テンプレートのすべての質問に答えてください。これらは、あなたのアイデアに関するフィードバックを提供するのに役立ちます。
- 目標公開日の少なくとも 2 週間前にアイデアを提出してください。
- コーポレート／comms リクエストやタイトな納期のブログについては、Issue を提出し、@sgittlen に直接連絡してください。

**注:** [ブログ提出テンプレート](https://gitlab.com/gitlab-com/marketing/brand-product-marketing/content-strategy-and-ops/blog/-/issues/new?issuable_template=blog-post-submission) を介して、Blog チームにタイプミスの通知や変更のリクエストを行うこともできます。

ブログ編集者はあなたのピッチをレビューし、a) 投稿を承認するか、b) 改善のための提案を提供するか、c) アイデアがブログに適さない可能性がある理由を説明し、メッセージを発信するための他のアイデアを提供します。

**ニュースレターコンテンツリクエストの場合:** リクエストを[この Issue](https://gitlab.com/gitlab-com/marketing/brand-product-marketing/content-strategy-and-ops/content-ops/-/issues/902) に追加してください。

### 外部からの貢献

**注:** GitLab は、依頼されていないブログ提出を受け付けません。あなたが GitLab パートナーで、ブログ投稿のアイデアを検討のためにピッチしたい場合は、[sgittlen@gitlab.com にメール](mailto:sgittlen@gitlab.com) してください。あなたが GitLab コミュニティメンバーで、ブログ投稿のアイデアを検討のためにピッチしたい場合は、[contributors@gitlab.com にメール](mailto:contributors@gitlab.com) してください。ピッチと共にブログのドラフトを送信しないでください。

ピッチアイデアは GitLab ブログと読者に関連性があり、以下の情報を含む必要があります:

- ブログは何についてですか？ できる限り具体的にしてください。
- あなたのブログを最もよく表現するのは、以下のどれですか？（少なくとも 1 つチェックしてください。）

- [ ] 機能／能力をどのように最適に使用するかについてのチュートリアル／ハウツー
- [ ] GitLab プラットフォームの使用方法のベストプラクティス
- [ ] オープンソースコミュニティと GitLab
- [ ] パートナー／アライアンスの統合
- [ ] その他 - 説明してください

**すべてのブログピッチアイデアと提出物は、GitLab チームメンバーによって審査・レビューされます。**

ブログ編集者は、貢献された記事の冒頭に以下の 2 文を追加してください:

***編集者注: 時折、私たちはコミュニティのメンバーを GitLab ブログに貢献するよう招待します。[エンティティ名] が私たちと共同制作してくれたことに感謝します。***

外部からの貢献は[External Blog Submissions Terms](https://about.gitlab.com/externalblogsubmissionsterms/) の対象となります。GitLab に資料を提出する前に、このドキュメントを読み、利用規約に同意してください。

## アイデアが承認されたら、ブログのドラフトを提出する方法

**ブログピッチが承認されたら、** 著者は以下のオプションのいずれかを使用してドラフトを提出します:

- GitLab でマージリクエストを介して提出します。これを行うには、[リポジトリから行うためのこれらの手順に従ってください](/handbook/marketing/digital-experience/decap-cms/#creating-a-blog-post-in-the-aboutgitlabcom-repository) または [Web IDE の場合](/handbook/marketing/digital-experience/decap-cms/#creating-a-blog-post-using-the-web-ide)。これらに注意して従ってください！ 質問は @sgittlen までお問い合わせください。

- 私たちの基本的なコンテンツマネジメントシステムである Decap を介して、[これらの手順を使用](/handbook/marketing/digital-experience/decap-cms/#creating-a-blog-post-using-decap-cms) して提出します。これらに注意して従ってください！ 質問は @sgittlen までお問い合わせください。

- Google ドキュメントを介して、[ブログ提出テンプレート](https://docs.google.com/document/d/1iWk0a5T14dG5fvIxvG4doqqRxx0pyv3ej_aMDqYXjgA/template/preview) (Google Doc) を使用してブログを執筆し、Google Doc が Issue にリンクされていることを確認します。**Blog チームが提出を受け入れるには、テンプレートのすべてのフィールドが完了している必要があります。** 著者は Issue 内で @sgittlen をタグ付けし、ドキュメントのリンクをコメントに入れ、編集準備が整ったら（必要な承認とレビューがすべて完了した後）Google Doc を共有します。**注:** すべての画像は Google Doc にインラインで含める必要があります。

**注:** すべてのブログ提出には、**Call to Action**（CTA）が必要であり、ブログドラフトテンプレートで提供するよう求められます。CTA は、読者がブログを読んだ後に何をしてほしいかです。別のページに行って詳細を学んでほしいですか、トライアルにサインアップしてほしいですか、ウェビナーに登録してほしいですか、デモを見てほしいですか？ 私たちは、ブログの全体的なメトリクスの一部として CTA を追跡できるでしょう。

ブログをどのように提出する方法を選んでも、@sgittlen がタグ付けされていることを確認してください。

**ブログチームのレビュー／承認なしにブログを公開することはできません。**

## MR のフロントマターガイド

MR には以下のフロントマターが含まれます。各セクションの後のコメントを参照してください

```text
seo:
  title: the blog post title
  description: the blog post description
  <!-- Blog team will fill this area in -->
config:
  slug: blog-post-slug
  featured: false
  template: BlogPost
  <!-- blog team will fill this in -->
content:
  title: the blog post title
  <!-- ideal length - 55-60 characters -->
  description: the blog post description
  <!-- ideal length - no more than 155 characters -->
  authors:
    - Blog post author
  <!-- Format: Sandra Gittlen (if this is your first time contributing, leave blank, and if there are multiple authors, add a comma no space between)-->
  heroImage: images/blog/hero-images/logoforblogpost.jpg
  <!-- blog team will fill this in -->
  date: '2021-03-31'
  category: engineering
  <!-- blog team will fill this in -->
  tags:
    - community
  <!-- blog team will fill this in -->
  body: |
    add the blog post body text in markdown
```

## ブログ編集プロセス

ブログチームは、Issue と Google Doc を使用して、著者に最初の編集／質問を伝えます。次にブログチームはブログを CMS に投入し、必要に応じて著者／DRI とプレビューリンクを共有します。注: ブログは Contentful から公開されます。

公開後に変更が必要な場合、著者は Slack 経由でブログチームに連絡して変更を説明するか、公開されたページの下部に移動して「Edit this page」をクリックして MR を提出します。これにより MR が開始されます。

ブログチームは、公開後にブログの URL を著者と共有します。

ブログ投稿の公開は編集カレンダーに従って行われます。ただし、緊急のブログに基づいて変更される可能性があります。ブログチームは、予想される公開日について著者を更新します。

### 法務レビュープロセス

一部のブログ投稿は、[Materials Legal Review Process/SAFE program](/handbook/legal/materials-legal-review-process/) に従って法務によるレビューを受ける必要があります。著者は、SAFE ガイドラインをレビューし、Google Doc をブログチームと共有する前に法務の承認を得る責任があります。このプロセスには時間がかかる可能性があるため、目標公開日に応じて計画してください。

GitLab は行動へのバイアスを持っており、Blog チームも同様です。ただし、GitLab ブログは公開向けのアセットであり、会社を代表します。ブログチームがブログ投稿に含まれる情報について懸念や質問を持っている場合、ブログチームは、会社の潜在的なリスクを軽減するために、Legal、Corporate Communications、Partner Marketing、CMO などがブログ投稿をレビューできるまで、ブログ投稿を保留する権限があります。

[ハンドブックページを読んで SAFE ガイドライン](/handbook/legal/safe-framework/) について詳しく学び、[Materials Legal Review Process](/handbook/legal/materials-legal-review-process/) に従ってください。

## 公開されたブログへの変更を提案する方法

- GitLab の内部に所属しており、公開された GitLab ブログへの変更を提案したい場合は、必要な変更を URL とともに詳細に `#content` Slack チャンネルに投稿し、@sgittlen をタグ付けするか、Slack で @sgittlen に直接 ping するか、公開されたページの下部に移動して「Edit this page」をクリックして必要な変更を含む MR を作成してください。これにより MR が開始されます。

- GitLab の外部の場合、Sandra Gittlen に `sgittlen@gitlab.com` のメールで提案された変更の詳細を送信するか、公開されたページの下部に移動して「Edit this page」をクリックして必要な変更を含む MR を作成してください。これにより MR が開始されます。

## ブログを起草する際の考慮事項

### ブログライター向け Diversity、Inclusion、Belonging (DIB) チェックリスト

私たちのブログコンテンツが、私たちの会社の Diversity、Inclusion、Belonging のバリューを代表することは重要です。これらのポイントすべてがブログ投稿に関連しているわけではありませんが、ライティングプロセス全体を通じて留意すべき重要なバリューとプラクティスです。ブログ編集チームはこれらを確認しようとしますが、すべてのコンテンツがこれらのバリューとプラクティスを念頭に置いて作成されていた方がよいです。質問がある場合は、私たちまたは [DIB チーム](/handbook/company/culture/inclusion/) のメンバーをタグ付けしてください！

#### 包括的なライティング

- 短く簡潔な文を書いてください。短い文での明確なライティングは、読者が追いかけるのを容易にします。
- 業界用語の使用を制限し、業界用語の用語を使う必要がある場合は、最初のインスタンスでそれを定義してください。
- GitLab はグローバルチームを持つグローバルコミュニティであるため、グローバルオーディエンス向けに書くことが望ましいです。これは、地域的なメタファーの使用を制限し、米国中心の方法で書かないことを意味します。
- 投稿は[包括的な言葉](/handbook/company/culture/inclusion/#inclusive-language) を使用していますか？
- ブログ投稿で引用されているすべての個人は、[望ましい代名詞](/handbook/people-group/orientation-identity/#how-do-i-know-which-pronouns-to-use) を使用していますか？ ヒント: 誰かの望ましい代名詞を知らない場合は、ただ尋ねてください。彼らは[チームページプロフィール](https://gitlab.com/close-modal) と Slack プロフィールにも含まれているはずです。

#### より多くの DIB ライティングのヒント

- あなたのブログ投稿はバイアスがありますか？ 私たちには全員、無意識のバイアスがあります。[DIB ハンドブックページの無意識のバイアスのリスト](/handbook/company/culture/inclusion/unconscious-bias) を確認し、自分のものを認識し管理するための戦略を実践してください。
- まだ質問がある場合は、編集チームのリード @sgittlen、ブログ Issue 内の適切な[チームメンバーリソースグループ (TMRG)](/handbook/company/culture/inclusion/tmrg-tmag/) のリードをタグ付けするか、質問を diversityinclusionandbelonging@gitlab.com にメールしてください。

### ブログカテゴリとタグ

#### カテゴリ

投稿ごとに、以下のカテゴリのうち **1 つだけ** を使用してください。
大文字小文字、スペル、その他を変更**しないで**ください。
そうしないと、別のカテゴリが作成され、私たちが誤って行いたくないことになります。

投稿がどのカテゴリに属するかわからない場合は、MR にプレースホルダーを入れて、レビュアーに対するコメントとして残してください。

- `agile planning` - Agile プランニングに関する投稿
- `ai-ml` – プラットフォーム内または業界全体の AI/ML に直接焦点を当てた投稿
- `customer stories` - 私たちの顧客が GitLab DevSecOps プラットフォームをどう使用しているかについての投稿
- `bulletin board` - 短いブログ／アナウンスが行く場所
- `DevSecOps` - DevSecOps についてのより一般的な投稿
- `engineering` – 技術的な、実行可能なコンテンツ。何かを行う方法、何かを使用する方法、または問題を解決する方法をカバーするものはすべてこのカテゴリに該当する必要があります
- `open source` – 私たちのコミュニティ、ユーザー、またはより広いオープンソースコミュニティからのまたはについてのストーリー
- `product` - 機能、ロードマップ、戦略についての詳細
- `news` – 会社または製品のアナウンス（ポリシー変更、運用アナウンス、ブレイキングチェンジを含む）、ニュース、またはイベント
- `security` – セキュリティ関連の投稿
- `releases` - リリースポスト、セキュリティおよびパッチリリース。`releases` カテゴリの投稿は、`sites/uncategorized/source/blog/blog-posts` ではなく `sites/uncategorized/source/releases/posts` ディレクトリにある必要があります。詳細は[リリースポストハンドブック](/handbook/marketing/blog/release-posts/) を参照してください。

### コードブロックの追加

[コードブロックを追加するための手順](/handbook/marketing/digital-experience/decap-cms/#creating-a-blog-post-using-the-web-ide)。

#### Mermaid チャート

[MR への mermaid チャートの埋め込み方法](https://gitlab.com/gitlab-com/marketing/digital-experience/gitlab-blog/-/merge_requests/211#usage-notes)。チャートが正しくレンダリングされない場合の細かい点があるため、必ずお読みください。

#### 画像の準備

- オリジナルのカバー画像を作成する場合、すべてのディスプレイで最適な品質を得るには、寸法は 1800px x 945px であるべきです。
- すべての画像は 1MB 未満を目指すべきです。JPEG は PNG よりも小さくなる傾向があるため、可能な場合は JPEG を使用してください。
- Preview でリサイズするには、`Tools`、`Adjust size` に移動し、`Resolution` フィールドのエントリを調整します。Preview は、`OK` をクリックして確認する前に、結果の画像サイズがどうなるかを見積もります。
- すべての画像の幅を同じに保ってください。

#### スクリーンショット

技術的／チュートリアルの投稿については、例を[コードブロック](#adding-code-blocks) またはスクリーンショットで説明してください。例には一貫性を持たせてください。たとえば、手順を例示するために一般的な URL `domain.com` を使用している場合、投稿全体を通して `domain.com` のままに保ってください。

- 静的な画像は、概念を説明したり、図を提供したり、UI の要素を示したり、読者を方向づけたりするために使用すべきです。
- コピー＆ペーストできなくなるため、画像をコマンドや設定のレンダリングに使用すべきではありません。
- アニメーション GIF は、プロセスや時間や複数のアクションの過程で起こる何らかのイベントを示す必要がある場合に控えめに使用できますが、テキストの説明や手順に取って代わるべきではありません。
- 画面の特定の部分を識別し、ローカライズするためにスクリーンショットを使用してください。これを行うための優れたツールがあります。たとえば、Nimbus Screenshot（ブラウザ拡張機能）、Mac スクリーンショット、Windows 用 Snipping Tool、Ubuntu 用 Screenshot Tool です。

**重要なセキュリティポイント:** 実際のトークンやセキュリティ認証情報を使用して個人情報を公開しないでください。代わりに `[project's CI token]` スタブのようなプレースホルダーを使用してください。または、スクリーンショットに表示される場合はぼかしてください。

#### ツイートや Instagram 投稿の埋め込み

#### GIF の作成

アニメーション GIF は、この種のリソースで理解しやすいかもしれない短いダイナミックなプロセスを説明するのに非常に役立ちます。
アニメーション GIF を作成する方法はいくつかあり、その 1 つは Mac 向けの軽量アプリ [Giphy Capture] を使用することです。

ファイルサイズが大きい GIF は避けてください。インターネット接続が悪いユーザーは読み込みが困難になります。そのような場合は、GIF を小さな部分にカットするか、動画を録画するか、シーケンシャル画像を使用できます。

### CMS での `Author` エントリの作成

CMS (Decap) で `Author` エントリを **作成する** 際の一連の推奨事項を以下に示します。

- `Name` フィールド
    これは必須フィールドです。また、ユニークなフィールドでもあります。

    1. 著者の名前は、複合名を含めて、姓と名のみの組み合わせである必要があります。このフィールドに役職の説明（このタスクには `Role` フィールドを使用）や英数字の組み合わせを追加しないでください。
    1. 二重著者（例: `/authors/<author1>-<author2>`）にしないでください。ブログ投稿には複数の単一著者を追加できます。個々の著者を作成してから、それらすべてを 1 対多の関係として他のコンテンツタイプに追加する必要があります。
    1. 上記と同じく、単一のブログ投稿に複数の著者は不可（例: `/authors/<author1>-<author2>-<author3>-and-<author4>`）。

- `Role` フィールド

    現在の作業ポジションまたは職務記述。

- `Bio` フィールド

    著者の経歴。

- `GitLab handle` フィールド

    小文字形式の著者の GitLab ユーザー名に対応します。このフィールドはユニークである必要があります。

- `Social media handles` フィールド

    ソーシャルメディアハンドルのオプションフィールド。

## CMS でのブログ投稿のローカライズ

英語以外の言語でブログ投稿を公開できます。現在、以下の言語をサポートしています:

| 言語 | URL 構造|
|---------|----------|
| フランス語 |`/fr-fr/blog/YEAR/MONTH/DAY/Title/` |
| ドイツ語 |`/de-de/blog/YEAR/MONTH/DAY/Title/` |
| 日本語 |`/ja-jp/blog/YEAR/MONTH/DAY/Title/` |

詳細については、GitLab ローカライゼーションチームにお問い合わせください。

### 翻訳の開始方法

ブログ投稿の翻訳プロセスを開始するのは簡単です。次の手順に従ってください:

1. Localization プロジェクトに翻訳リクエスト Issue を作成します。プロのヒント - ブログ投稿の URL と Contentful エントリへの直接リンクを追加してください。プロジェクトを追跡する際に非常に役立ちます。
2. Argo は英語のブログ投稿を JSON としてエクスポートし、翻訳された JSON を Contentful にインポートし直します。レビュー準備ができたら、Issue で ping されます。
3. ブログ投稿のエントリにジャンプして、翻訳されたコンテンツをレビューします。
4. 翻訳された投稿の公開について @sgittlen と調整します。
5. 公開されたら、about.gitlab.com/blog/ で期待通りに動作していることを必ず確認してください。
6. Slack に投稿することで、翻訳されたブログ投稿を祝います。

以下は、翻訳リクエストの作成から本番への公開までの完全なプロセスのビデオウォークスルーです。

- 以下を示すビデオ録画が近日公開予定
- Issue 作成
- Argo ラウンドトリップ
- Contentful での翻訳の表示
- 翻訳のレビュー
- ライブプレビュー／言語の変更
- 公開

### EN ブログ投稿をローカライズする際の注意点

EN ブログ投稿をローカライズする際に注意すべきこと。

- スラッグを変更しないでください！

### 整理するためのタグ

- 投稿が翻訳中の場合は、「translation-in-progress」ラベルを付けてください。これにより、チームは Contentful で翻訳およびレビュー中のすべてのブログ投稿を簡単に見つけることができます。
- 投稿が翻訳された言語のローカルラベルを削除して適用してください。
| タグ | 定義 | 使い方 |
|----|----|----|
| translation-in-progress | ブログ投稿が現在翻訳およびレビュー中であることを示す | 翻訳リクエストが開かれたときに適用。投稿が公開されたら削除 |
| language_de-DE | ドイツ語のエントリをマークする | ドイツ語に翻訳されたブログ投稿に適用 |
| language_fr-FR | フランス語のエントリをマークする | フランス語に翻訳されたブログ投稿に適用 |
| language_ja-JP | 日本語のエントリをマークする | 日本語に翻訳されたブログ投稿に適用 |
