GitLab のセールス開発組織へようこそ！私たちは、インバウンドとアウトバウンドの両方の戦略を通じて[顧客のための結果](/handbook/values/#results)を生み出すために設計されたチームです。私たちの構造は、アウトリーチ活動における効率性、応答性、創造性を最大化するように設計されています。

私たちの組織のクイック Roadshow プレゼンテーションは[こちらのスライド](https://docs.google.com/presentation/d/1SHal9oaX1C_eaVjIMACWjkK9RZZNtnTXTZVuIKR4TgA/edit)で確認できます。

**1. Sales Development Representatives (SDR) — インバウンドフォーカス**

主要属性: 迅速な応答時間 · グローバルカバレッジ · SLA とキャンペーンフィードバックに関するマーケティングとのアライメント · 定義された規定的なインバウンドプロセス · ラウンドロビン割り当てルール · BDR チームのタレントインキュベーター。

**2. Business Development Representatives (BDR) — アウトバウンドフォーカス**

主要属性: SAE/AE およびセールスリーダーシップとのアライメント · フィールドマーケティングおよび ABM とのコラボレーション · 戦略的なアカウントプランニングとリサーチ · ターゲットを絞った創造的なメッセージング · セールスチームのタレントインキュベーター。

## インバウンドおよびアウトバウンドプロセスの How-Tos

インバウンドおよびアウトバウンドのプロセスを説明する前に、私たちの SLA と KPI を認識しておくことが重要です。

### SLA と KPI

#### インバウンド SLA

| 指標 | しきい値 |
| ------ | --------- |
| 応答時間 — 完全新規 MQL | MQL 日付から 2 業務時間 |
| 応答時間 — インバウンドの返信 | 8 業務時間 |
| High Touch シーケンスの使用率 | インバウンドリードの 70% 超を High Touch シーケンスに登録する必要がある |
| 1 日あたりの期限超過タスク | 保留は 10% 以下。90% は適切に完了（スキップではない）する必要がある |
| 双方向の会話 | 週 50 件以上 |

#### アウトバウンド SLA

| 指標 | しきい値 |
| ------ | --------- |
| 四半期あたりの AWA アカウント — Enterprise | BDR あたり 75 アカウント（`Actively Working Start Date` で測定） |
| 四半期あたりの AWA アカウント — Hybrid Ent & Mid-Market、First Order | BDR あたり 125 アカウント（`Actively Working Start Date` で測定） |
| 四半期あたりの 6QA アカウント | BDR あたり 25 件 |
| リサーチの質 | `BDR Prospecting Status = Actively Working` のアカウントの 80% 超が、インテントデータまたは高い購入傾向を示す必要がある |
| アカウントあたりのプロスペクト | AWA アカウントあたり 5 〜 10 件、私たちの ICP ペルソナに合致 |
| 6QA アカウントレビュー SLA | 自動 6QA されたすべてのアカウントは、48 時間以内に `6QA Acceptance Status = Accepted or Disputed` にする必要がある |

> **注:** Ultimate Parent Account や重複したアカウントを Actively Working に設定する必要はありません。なぜなら、子会社が親アカウントと同じ名前を共有していれば、Traction と SDR は、あなたがその子会社を AW に設定していることを確認できるからです。

### リード、コンタクト、アカウントの見つけ方

インバウンドリードは、Sales and Marketing Operations チームによって自動的に Salesforce にインポートされます。アウトバウンドリードは、BDR チームによって直接手動でインポートされます。

国内 First Order RoE: 自分の地域のリードのみを扱います。既存のアカウントがないリードを変換する前に、住所が完全であること（国と郵便番号を含む）を確認してください。これにより、リードのローカル住所で新しいアカウントが作成されます。

#### リードへのアクセス

1. SFDC にログインします。
2. 以下の 3 つの方法のいずれかでリードにアクセスします:
   - **Views** — あなたの割り当てとリードステータスにスコープを絞った、あらかじめフィルタリングされたリスト。日々の優先順位付けに最適です。
   - **Reports** — 標準ビューでは表示されないデータが必要な場合に使用します（例: 特定の日付範囲のソース別リード、30 日間アクティビティのないアカウント）。
   - **Dashboards** — パイプラインの健全性、アクティビティ、SLA/KPI 遵守状況の視覚的な要約。[ダッシュボードセクション](/handbook/marketing/sales-development/#our-dashboards)を参照してください。

### リードビュー

リードビューは、パイプラインを効率的に優先順位付けし管理できるよう、ユースケースごとにあらかじめフィルタリングされています。アクセスするには、トップナビゲーションバーの `Leads` タブをクリックします。

> **SLA:** インバウンドリードを管理するための KPI と応答時間の要件は、[SLA ページ](/handbook/marketing/sales-development/#inbound-slas)で定義されています。

## リードデータベース管理の方法

SDR の責務の一部として、受け取った各リードに対してデューデリジェンスを実施することが期待されます。これには、重複レコードがないかデータベースを確認すること、および私たちの [RoE](https://internal.gitlab.com/handbook/sales/go-to-market/rules-of-engagement/) を参照することが含まれます。

> **このプロセスの主要フィールド:** `Lead Status · Matched Account Info · BDR Prospecting Status · Account Type · BDR Assigned · Initial Source · Admin Company Override · Matched Opportunities Info`

**ステップ 1 — 重複を見つける**

リードレコードを開き、ページ上部の `Find Duplicates` を押します。これを行うには SFDC Classic にいる必要があります。これは Leads、Contacts、Accounts 全体にわたってクエリを実行します。

**ステップ 2 — 結果を評価する**

関連するすべての一致を表示する画面に移動します。何も表示されない場合や、データが不完全だと感じる場合は、チェックボックスから **Email Domain** のみを選択して `Search` を新たにクリックし、優先的に一致させます。利用可能な他のデータポイントについても繰り返します。

次に、以下のルーティングロジックに従って作業を進めます:

- 同じ人物または企業からの類似のリードがすでに作業されていますか？その場合、その `Lead Status` と `Created Date` は何ですか？
- 一致したアカウントは存在しますか？その場合、その `Type`（Customer か Prospect か？）と `BDR Prospecting Status`（`Actively Working` か `Worked in FY` か）は何ですか？
- 現在オープンなオポチュニティ、または 90 日未満前にクローズしたオポチュニティはありますか？

**ステップ 3 — RoE に基づいてアクションする**

上記の状況を把握したら、以下の正しいアクションを適用します:

- `Account Type = Customer` かつリードが SMB セグメント → **Convert** を押します。リードを既存のアカウントに添付します。新しいアカウントを作成しないでください。注: ストレージのみが購入された場合でもアカウントは `Customer` とマークされることがあります — 真の Customer レコードとして扱う前に、まず `Total_CARR__c > 0` を確認してください。

- `Account Type` が Customer でなく、一致したアカウントにオープンなオポチュニティがある → `Lead Status` を `Recycle` に、`Recycle Reason` を `Evaluating` に設定します。Lead Lifecycle のルールに従ってリサイクルキューに残します。

- リードが大学または Education アカウントにあり、US Public Sector ではなく、コンタクトが技術系である → 通常のリードとして扱います。標準的なクオリフィケーションとシーケンシングを進めます。

- `Account BDR Prospecting Status = Actively Working` → まず、過去 30 日間にアカウントレコードまたは関連するリードレコードにアクティビティがないか確認します。アクティビティが存在する場合、リードを `BDR Assigned` フィールドのユーザーに再ルーティングします。これは必須です — SDR はこれらのリードを保持してはなりません。アクティビティが存在しない場合、リードは SDR に残すことができます。

- アカウントが Public Sector とフラグ付けされている（ウェブサイトで `.gov` パートナーシップや政府契約を確認）→ PubSec マネージャーにルーティングします。自分でリードを作業しないでください。

- アカウントが `BDR Prospecting Status = Restricted` の Major テリトリーアカウントである → リードを Contact に変換し、AE に Chatter します。リードを作業しないでください。

- アカウントが Actively Working ではない → そのアカウント上に重複したリードやコンタクトがないか確認します（ステップ 1 の結果から）。重複が存在する場合、それらをマージし、残存するレコードに最も古い `Initial Source` の値を保持します。

- リードの `Company Address` が外部リサーチと一致しない → `Lead/Contact Review Admin` セクションの `Admin Company Override` フィールドを入力します。これはリード変換の必須の前提条件です。

### 会社住所ベースのルーティング情報

誤った本社情報を発見した場合、Company Address ウォーターフォールフィールドは、アカウントルーティングとオポチュニティ割り当てに使用される **読み取り専用の数式フィールド** です。リードを新しいアカウントに変換する前に検証する必要があります。ウォーターフォールは優先順位順に解決されます — 各レベルは、その上のレベルが空白の場合にのみ使用されます:

1. **Account Demographic Fields** — リードが既存のアカウントと一致した場合に LeanData によって自動入力されます。
2. **Admin Company Override Fields** — ZoomInfo データが誤っているか欠落している場合に、Lead/Contact Review Admin セクションで手動で設定されます。
3. **ZoomInfo Company Address Fields** — 自動エンリッチされたベースライン。

**Admin Override フィールドを使用するタイミング:** ZoomInfo の住所が正しくない場合、または ZoomInfo の一致がなくウォーターフォールフィールドが空白の場合。

更新するには: Lead/Contact Review Admin に移動し、`Street`、`City`、`Country`、`Postal Code` を入力し、**Admin Company Address Source** フィールドを設定します — このフィールドは保存時に必須です。優先順位順の許容ソース: `ZoomInfo` → `Cognism` → `Company Website / SEC Filing`（URL を追加）→ `Customer confirmation` → `LinkedIn`（最終手段 — 理由を含める必要がある）。

> **注:** リード変換後、アカウント所有権の割り当ては約 4 時間間隔で実行されます — アカウント所有者がすぐに更新されない場合、それは想定どおりです。

完全なウォークスルーについては、**[こちらの動画](https://www.youtube.com/watch?v=QT-oOceFU6k&ab_channel=GitLabUnfiltered)** を参照してください。

### リードのマージ

同じ人物に対して重複レコードが存在する場合、マージが必要です。リードレコードで `Find Duplicates` を使用して一致を表示します。

**2 つのリードのマージ — [動画ウォークスルー](https://www.youtube.com/watch?v=Q_EyDQdaLZw&ab_channel=GitLabUnfiltered)**

2 つのリードをマージする際、一方のレコードがマスター（残存）レコードになります。どの値を保持するかを決定するために、以下のルールを使用します:

| フィールド | ルール |
| ----- | ---- |
| マスターレコード | 最も新しい `Last Interesting Moment Date` を持つリード。タイブレーク: `Last Activity Date`。 |
| Initial Source | **最も古い** リード（最も早い Created Date）の値を保持します。 |
| First Engagement Date | **最も早い** 値を保持します。 |
| Last Login Date | **最も新しい** 値を保持します。 |
| Company / Address フィールド | リサーチに基づいて最も正確で完全なセットを保持します。 |
| Email | 両方のリードが異なるメールアドレスを持つ場合、マージ前にセカンダリメールを保存し、マージしたレコードの **Supplemental Email** として追加します。 |
| Owner | 意図的でない限り再割り当てしないでください — アクティブな担当者からリードを誤って奪っていないか確認します。 |

**リードとコンタクトのマージ — [動画ウォークスルー](https://www.youtube.com/watch?v=qHrCyKiNwDQ&ab_channel=GitLabUnfiltered)**

これは、すでに SFDC にリードとして存在する人物に対して、ウェブ直接購入が新しいコンタクトとオポチュニティを作成した場合に適用されます。システムは自動的にはマージしません — 手動で行う必要があります。

1. リードレコードを開く → **Convert** を押します。
2. "Create Opportunity" ボックスの **チェックを外します** — これを怠ると重複したオポチュニティが自動作成されます。
3. リードを既存のアカウントに添付します。
4. 新規作成するのではなく、マージ先となる既存のコンタクトを選択します。
5. **Initial Source** を、Created Date が早いほうのレコード（通常はリード）の値に設定します。
6. アカウントが SMB または Mid-Market で Customer の場合 → Contact のみに変換し、アカウントの AE に Chatter して、リードの Last Interesting Moment とトライアルまたはアウトリーチのアクティビティを知らせます。

### 質の低いリードのクリーニング

時には、リードが GitLab にとって価値がなく、データベースから永久に削除すべき場合があります。このプロセスは **不可逆** です — 開始する前に注意を払ってください。

**削除の基準:**

- 一貫性のない、または明らかに偽造された個人情報（名前、会社、役職）
- 一時的または自己消滅するドメインからのメールアドレス

**削除プロセス:**

1. リードが上記の基準を満たし、プロスペクティングの価値がないことを確認します。
2. リードを **SPAM DELETION** Salesforce キャンペーンに追加します。重要なのはキャンペーンのメンバーシップのみです — 選択するステータス値は削除に影響しません。
3. 削除は Marketo を介して **毎日午前 12:05 PDT に** 自動的に実行されます。緊急の削除については、`#mktgops` で支援を依頼してください — 控えめに使用してください。

> ⚠️ このプロセスを通じて削除されたリードは **復元できません**。100% 確実でない限り、リードをこのキャンペーンに追加しないでください。

**ドメインブロッキング:** 複数のリードが同じ低品質ドメインを共有している場合、そのドメインをブロックリストに追加するよう Marketing Ops に Issue を作成します。

## リードのアクション方法

### High Touch リードと Low Touch リード

デフォルトは **High Touch** です。以下の 3 つのルックアップソースすべてを尽くしても電話番号が本当にソースできない場合にのみ Low Touch を使用します。

**ステップ 1 — 電話番号を確認する:**

- リードレベルで SFDC の `phone` データを確認します。
- 見つからない場合は、エンリッチメントツール（ZoomInfo、Cognism、SalesNav）で同じコンタクトを `email` を使って検索します。
- それでも見つからない場合は、手動リサーチ（会社のウェブサイトなど）を試みます。

**ステップ 2 — 見つけた内容に基づいてシーケンスタイプを選択する:**

→ いずれかの段階で電話番号が見つかった → **High Touch** シーケンスを使用します: マルチチャネル（メール + 電話 + LinkedIn）、約 2 週間、10 ステップ以上。これはほとんどのインバウンドおよびアウトバウンドリードの標準です。

→ 3 つすべての確認後に電話番号がない → **Low Touch** シーケンスを使用します（最終手段）: 最大 4 つの自動メールステップ、より短い期間。イベント招待や、本当にコンタクトデータが不完全なリードに適しています。

上記のシーケンスタイプに基づいて、Outreach の対応するコレクションを選択します。

### Do Not Call および Do Not Email オートメーション

#### Do Not Call

通話をかける前に、SFDC のリードまたはコンタクトレコードの **Do Not Call** フィールドを確認します。

- Do Not Call = true → 電話しないでください。これを回避しようとしないでください。
- Do Not Call = false → 通話は許可されますが、現地の法律と Marketo オートメーションに従います。

**プロスペクトが電話しないでほしいと求めた場合:** Do Not Call を true に設定します。電話番号を削除しないでください — どの番号がそれ以上の通話を求めなかったかの記録を保持する必要があります。

**Do Not Call のチェックを外す** には、以下の両方の条件が真である必要があります:

- 個人がメールにオプトインしている（Do Not Email フラグが false）、かつ
- 電話を希望することを明示的に述べている（通話中、または SFDC に記録されたメールで）。

フィールドのチェックを外す前に、オプトインのメールが証拠として SFDC に記録されていることを確認します。

> ⚠️ オプトインしていないコンタクトに、電話を希望するかどうかを尋ねるメールを送ることはできません。

#### Email、Unsubscribe、Do Not Contact オートメーション

BDR が送信するすべてのアウトバウンドメール（返信を含む）には、配信停止リンクが含まれている必要があります。正しい法的文言については、`#privacy-team-help` を介して Privacy Team に連絡してください。

リードが作成された直後、Marketo は国、リードソース、ドメインタイプ、コンプライアンス基準（例: 特定の Cognism/ZoomInfo リード、カナダの個人ドメイン、競合またはスパムドメイン）に基づいて、Do Not Call および/または Do Not Email を自動的に true に設定することがあります。

- Marketing Ops または Privacy チームから文書化された例外を介して明示的に指示されない限り、これらのフラグを上書きしないでください。
- これらのフラグによりリードがコンタクト不可に見える場合は、データエラーではなく意図的なコンプライアンス動作として扱います。

完全な基準: [internal.gitlab.com/handbook/marketing/marketing-ops-and-analytics/marketing-operations/do-not-contact](https://internal.gitlab.com/handbook/marketing/marketing-ops-and-analytics/marketing-operations/do-not-contact)

### White Glove イベントフォローアップシーケンス

Executive Roundtable のようなハイタッチイベントでは、リードレコードの **Last Event Notes** フィールドに、プロスペクトと交流した GitLab チームメンバーの名前を挙げるなど、具体的な指示が含まれることがあります。これらの状況では **white glove** シーケンスが必要で、Outreach で Outreach Collection の `White Glove` でフィルタリングして見つけることができます。

目的は、迅速にフォローアップし、関連する SAE/AE を最初のタッチポイントから会話に引き入れることです。

1. プロスペクトを適切な white glove シーケンスに登録します。
2. Last Event Notes で名前が挙がっている個人を参照または CC するように、最初のメールステップをカスタマイズします。SAE/AE が入力に時間を必要とする場合は、アクティベートする前に最初のステップを適宜遅らせます。
3. カスタマイズした最初のメールのスクリーンショットを CC された SAE/AE に送り、プロスペクトが応答しない場合により個別化されたフォローアップを可能にするために含まれていることを説明します。
4. 12 日目に、組み込みのタスクが、プロスペクトのエンゲージメントが発生していない場合に SAE/AE にフォローアップするよう促します。このタッチポイントを使って、最新情報を共有し、次のステップについてアライメントを取ります。

### High Priority キャンペーンとリード

リードは、SFDC で `High Priority = true` のキャンペーンのメンバーであり、関連するアカウントが `BDR Prospecting Status = Actively Working` を持つ場合に `High Priority` とマークされます。この分類は、リードの現在のリードステータスに関係なく適用されます。

High Priority リードは、SFDC の **B1 priority view** に自動的に表示されます。

特定のリードがなぜフラグ付けされたかを理解するには、リードレコードの以下のフィールドを確認します:

- **High Priority Timestamp** — フラグが適用された時刻。
- **High Priority Reason** — 分類を引き起こしたキャンペーンまたはトリガー。

リードをコンタクトせずに High Priority から外すには、[このシーケンスを使用します](https://web.outreach.io/sequences/239/overview)。これは、リードを Accepted ステータスに移動し、High Priority フラグを削除する 1 つのタスクを作成します。

## First Order (FO) アウトバウンドプロセス

FO アウトバウンドは、親レベルでこれまで商業的関係がないアカウントをターゲットにします。中核となる目的は、BDR の主要なパフォーマンスおよび報酬指標である **FO SAO**（First Order Sales Accepted Opportunities）を生み出すことです。BDR は自分の SAE/AE と協力して、**Actively Working (AWA)** アカウントを選定し戦略を構築します。

### BDR Comments — SFDC フィールド

これらの SFDC のアカウントレベルフィールドは、FO アカウントのステータス、戦略、タイミングに関する運用上の信頼できる情報源です。

| フィールド | 有効な値 / 用途 |
| ----- | -------------------- |
| **BDR Prospecting Status** | `Actively Working` · `Queued` · `Restricted` · `SDR Hold` · `Worked in FY` |
| **Actively Working Start Date** | アカウントが Actively Working に入ると自動入力されます。Days in Actively Working Status と BDR Recycle Date の計算を駆動します。 |
| **BDR Account Research** | フリーテキスト。シグナル、タイミング、価値ドライバーを文書化します。[BDR Account Research テンプレート](https://docs.google.com/document/d/1EN2U8wrUdti4MTxCNceoOlltAmtUt4pqN1GxwRTCnK4/edit?tab=t.0)を使用します。 |
| **BDR Next Steps** | アカウントの現在のプロスペクティング状態に関するフリーテキストの作業メモ。 |
| **BDR Account Strategy** | モーションラベル: `6QA Showing Intent` · `UserGems` · `Event Follow-up` · `Free-to-Paid` · `ABM` · `FO General` · `Growth - Upgrade` · `Risk Of Churn` |
| **BDR Recycle Date** | Actively Working Start Date から 2 か月後に自動設定されます。手動で延長できます。 |
| **Worked in FY Reason** | Worked in FY への移行時に自動入力されます。早期に移動する場合は手動で設定する必要があります（例: `No FO potential`、`Territory change`）。 |
| **6QA Acceptance Status** | `Accepted` または `Disputed` — 6QA を介して AWA に自動移動されたアカウントについては 48 時間以内に設定する必要があります。 |
| **6QA Dispute Reason** | 6QA Acceptance Status = Disputed の場合に必須。SDR がアクティブなオポチュニティを持っている場合は `Account in open opportunity` を使用します。アカウントの価値ポテンシャルが低い場合は `Low LAM Dev Count` を使用します。 |
| **Days in Actively Working Status** | Worked in FY に移行する前の Actively Working における自動計算された期間。サイクルタイムと衛生状態のレポートに使用されます。 |

> **6QA 注:** 6QA アカウントが非 FO かつ Compensation Target Account ではない場合、BDR Prospecting Status はまず 14 日間 `SDR Hold` に設定されることがあります。オポチュニティが作成されない場合、BDR の所有権で `Actively Working / Showing Intent` に自動移行します。

### FO アウトバウンド フェーズ 1 — プランニング

非常に戦略的なモーションとして、成功する BDR 戦略の最初の目標は、私たちの KPI に紐づいた週次および月次の目標を設定することです: 実施したアクティビティ、設定したミーティング、作成した SAO、Stage 1 Net ARR。

自分のコンバージョン比率（AWA アカウント → シーケンスされたプロスペクト → 会話 → FO SAO）を把握し、[パイプラインダッシュボード](/handbook/marketing/sales-development/#salesforce-and-tableau-dashboards-for-leaders)を使って調整します。

**真の FO アカウントの検証 — 3 つの条件すべてが真である必要があります:**

**条件 1 — 既存の商業ライセンスがない**

Account オブジェクトで、`Ultimate Parent Account` がターゲットと一致するすべてのアカウントを確認します。それらのアカウントのいずれも、Customer Status が `Customer` または `Former Customer`、または `Total CARR` がゼロより大きくなってはなりません。

**条件 2 — 過去 180 日間に失った更新がない**

Opportunity オブジェクトで、同じ `Ultimate Parent Account` にリンクされたすべてのオポチュニティを確認します。次のすべてを同時に持つものがあってはなりません: `Order Type = Renewal`、オポチュニティがクローズされている、Won ではない、`Close Date` が過去 180 日以内である。

**条件 3 — オープンなオポチュニティがない**

Opportunity オブジェクトで、ステージやオーダータイプに関係なく、同じ Ultimate Parent Account にリンクされたオープンなレコードがないことを確認します。

→ 3 つすべての条件が合格 → アカウントは FO として適格です。

→ いずれかの条件が不合格 → アカウントは FO ではありません。First Order として追求しないでください。

審査後、アカウントをコホート（例: `FO: high 6Sense intent`、`FO: strategic logos`、`FO: Free users`）にグループ化し、週あたり AWA に移動するアカウント数、アカウントあたりシーケンスするプロスペクト数（推奨: 5 〜 10）、ペルソナ、価値ドライバー、またはその組み合わせで焦点を絞るかどうかを決定します。

### FO アウトバウンド フェーズ 2 — アカウントリサーチ

シーケンシングの前に、各 FO アカウントには **BDR Account Research** フィールドに記録された文書化されたリサーチが必要です。

**調査すべき内部 SFDC シグナル:**

| シグナル | 見つかる場所 | プロスペクティング価値 |

| 過去のオポチュニティ | Opportunity オブジェクト — `Stage`、`Close Reason`、`Qualification Notes`、`AE notes` を確認 | Closed-lost オポチュニティは異議を浮き彫りにします。6 〜 12 か月前に失ったオポチュニティは、更新されたメッセージングで再エンゲージする価値があるかもしれません。 |
| 過去のリードとコンタクト | Lead および Contact オブジェクト — `Last Interesting Moment`、`Last Activity Date`、`Marketo activity` を確認 | ターゲットを絞った再エンゲージメントのために、温かいコンタクトと以前は冷たかったリードを特定します。 |

| シグナル | 見つかる場所 | プロスペクティング価値 |
| ------ | ---------------- | ----------------- |

### FO アウトバウンド フェーズ 3 — アカウントアウトリーチ

プロスペクティングしたいアカウントをリサーチして特定した後、エンゲージメントプランの誰を、何を、どのようにを定義します。これにより、**BDR Account Strategy** と **BDR Next Steps** に入力する内容が決まります。

- **ターゲットペルソナ:** どの特定の役職または意思決定者をシーケンスし、なぜか？（例: VP Engineering、CISO、CTO）
- **メッセージング戦略:** どの価値ドライバー、トラップ設定の質問、プレイブックがアウトリーチのアンカーになるか？どのシーケンスタイプを使用するか？
- **マルチスレッディングプラン:** AWA アカウントあたり 5 〜 10 プロスペクト、少なくとも 2 つのペルソナにまたがる。
- **シーケンスと戦略ラベル:** どの Outreach シーケンスを使用するか記録し、対応する BDR Account Strategy の値を設定します。

> **例:** C レベルのみをターゲットにします。デリバリーのペインとボトムラインへの影響をリードにします。フックとして個別化されたキャンペーンアセットを使用します。BDR Account Strategy = `FO General` に ABM オーバーレイ。

### グロース戦略、ランキング、RoE

純粋な FO プロスペクティングに加えて、BDR は既存のアカウント内および隣接するグロースオポチュニティを特定し追求することが期待されます。

**既存の顧客アカウント内でのグロース:** SAE/ISR と戦略的にパートナーを組み、追加のビジネスユニットを特定します。更新日を会話を拡大する説得力のあるイベントとして活用します。

**親/子/子会社への拡大:**

子会社のアカウントが ZoomInfo を介して特定されたがまだ SFDC にない場合、ステップ 1 はそのアカウントの Lead を SFDC で作成すること、ステップ 2 は最初のリードをクオリファイしたときに Contact に変換して Account を作成することです。

**Free から Paid へのアップグレード:** 既存の Core/CE ユーザーは、GitLab の有料版にアップグレードするようターゲットにできます。商業ライセンスなしの高い使用は、利用可能な最も強力な FO トリガーの 1 つです — これらのアカウントを優先します。それらを特定するには、[Self-Managed Instances Dashboard](/handbook/marketing/sales-development/#salesforce-and-tableau-dashboards-for-team-members)を使用します。

**ランキングと優先順位付け:**

| ランク | プロファイル | 戦略 |
| ---- | ------- | -------- |
| **Rank 1 — EwP** | 高い 6Sense インテントスコア | SAE との緊密な戦略的パートナーシップで BDR が High Touch。 |
| **Rank 2** | 階層に Core/CE ユーザーを持つ ICP Ultimate Parent Account、Total CARR/LAM 利用可能、中または低の 6Sense インテント、当 FQ における小規模な更新 | 標準の BDR アウトバウンドケイデンス。 |
| **Rank 3** | Rank 2 と同様だが ICP ではない | ナーチャーベースのモーション。 |

**Professional Services オポチュニティ:** PS オポチュニティは、別途販売され別途請求される統合、コンサルティング、トレーニングをカバーします。PS オポチュニティはセールス開発に **クレジットされません** — アカウントの AE に直接渡します。

## エンゲージメントのルール (RoE)

これらのエンゲージメントのルールは、Traction ルーティングが割り当てた後、SDR、BDR、ISR のためにリードとオポチュニティを誰が所有し作業すべきかを説明します。FY27 のリード/コンタクトルーティングロジック（BDR vs SDR vs ISR vs AE vs キュー）は、[Lead & Contact Routing (Traction) ハンドブックページ](https://internal.gitlab.com/handbook/marketing/marketing-ops-and-analytics/marketing-operations/traction/#fy27-lead-routing)で定義されています。この RoE は、そのルーティングロジックと併せて読むべきです。

### SDR インバウンド RoE

> インバウンドリードを作業する前に、上記の [ISR 基準](/handbook/marketing/sales-development/#isr-inbound-roe)を満たすかどうかを確認します。満たす場合はアクションせず、ISR キューに保持します。

**SMB（0 〜 250 名）**

→ Customers → 作業しないでください。あなたにルーティングされた場合は、SMB チームに渡します。更新のニーズがある場合は、Contact に変換して更新ケースを作成します。

→ Prospects → SDR が作業できます。

**Mid-Market（250 〜 4,000 名）**

→ Customers → 作業しないでください。BDR Assigned（Actively Working の場合）、AE Account Owner（AWA でない場合）、Base BDR（Base テリトリーの場合）、または Account Owner（パートナーアカウントの場合）にルーティングします。

→ Prospects → アカウントが Actively Working ではなく、オープンなオポチュニティがなく、Base テリトリーの一部でない場合に作業できます。

**Large / Enterprise（4,000 名以上）**

→ Customers → アカウントが Actively Working、Base テリトリーの一部、またはオープンなオポチュニティを持つ場合は作業しないでください。AWA の場合は BDR Assigned に渡します。顧客を作業する場合は、まず AE とアライメントを取ります。

→ Prospects → アカウントが Actively Working ではなく、Base テリトリーの一部でない場合に作業できます。

### ISR インバウンド RoE

Inside Sales Representative (ISR) チームは、プロダクト主導のグロースと従来のセールスの交差点に位置します。ISR はクオリファイされたトライアルリードの最初のコンタクトポイントであり、トランザクション型の取引を直接クローズし、より複雑なオポチュニティについては Field Sales とパートナーを組むことで、トライアルユーザーを顧客に変換することに焦点を当てています。

**ISR ルーティングは他のすべての RoE ステップよりも優先されます。** SDR または BDR のルーティングロジックを適用する前に、リードが以下のすべての基準を満たすかどうかを確認します。満たす場合、それは ISR チームに属します。

リードは、以下の **すべて** が真である場合に ISR チームにルーティングされます:

- `Last Interesting Moment Desc` に "Auto-MQL from a valuable MM+ account signing up for a trial" を含む
- `Last Interesting Moment Date` が ISR チームのゴーライブ日以降である
- `Is First Order Person` = `True`
- `Account Demographics: Sales Segment` が `SMB` ではない
- `Matched Account BDR Prospecting Status` が `Actively Working` または `Queued` ではない
- `Person Address: Country` = `CA` または `US`
- `Matched Account Owner Role` が `AE_FO_AMER` で始まる、または `Matched Account Owner Role` が空白である（アカウント所有者が割り当てられていない）

→ すべての条件を満たす → ISR チームにルーティングします。まだ ISR が割り当てられていない場合は、**ISR キュー** に保持します。SDR または BDR にルーティングしないでください。ここで停止します。

→ いずれかの条件を満たさない → 該当する場合 [SDR インバウンド RoE](/handbook/marketing/sales-development/#sdr-inbound-roe) または [Standard BDR RoE](/handbook/marketing/sales-development/#standard-bdr-roe) に進みます。

> **注:** この RoE は積極的に開発中です。ISR チームが成熟するにつれて、クレジットの帰属、Field Sales への引き継ぎ、エッジケースに関する追加のガイダンスがここに追加されます。上記でカバーされていないシナリオに遭遇した場合は、Sales Dev Ops チームに連絡してください。

### Standard BDR RoE

すべてのインバウンドリードについて、以下のステップを **順番に** 進めます:

**ステップ 1 — リードは Financial Services (AMER) からですか？

→ はい → すべての標準 RoE をバイパスします。FinServ BDR にルーティングします。ここで停止します。

→ いいえ → ステップ 2 に進みます。

**ステップ 2 — リードは既存のアカウントからですか？**

→ いいえ → リードが [ISR 基準](/handbook/marketing/sales-development/#isr-inbound-roe)を満たすかどうかを確認します。はいの場合 → ISR キューに保持します。ここで停止します。いいえの場合 → SDR チームがこのリードを作業します。ここで停止します。

→ はい → ステップ 3 に進みます。

**ステップ 3 — アカウントは Customer ですか？**

→ いいえ → ステップ 4 に進みます。

→ はい → ステップ 4 に進みます（更新オポチュニティを確認）。

**ステップ 4 — 既存の更新オポチュニティはありますか？**

→ はい → [更新に関する Sales RoE](https://internal.gitlab.com/handbook/sales/go-to-market/rules-of-engagement/#crediting--splits)とアライメントを取ります。SDR/BDR は更新オポチュニティでクレジットを受け取りません。例外が妥当だと考える場合は Sales Ops に連絡します。ここで停止します。

→ いいえ → ステップ 5 に進みます。

**ステップ 5 — アカウントは Actively Working ですか？**

→ いいえ → リードが [ISR 基準](/handbook/marketing/sales-development/#isr-inbound-roe)を満たすかどうかを確認します。はいの場合 → ISR キューに保持します。ここで停止します。いいえの場合 → SDR チームがこのリードを作業します。ここで停止します。

→ はい → ステップ 6 に進みます。

**ステップ 6 — BDR は過去 30 日間にアカウントにアクティビティを実施しましたか？**

SFDC のアカウントレベルで確認するか、[このレポート](https://gitlab.my.salesforce.com/00O8X000008RfvU)を使用します — 会社名とメールドメインでフィルタリングし、過去 30 日間のアクティビティを探します。

→ はい → BDR Assigned がこのリードを作業します。ここで停止します。

→ いいえ → SDR チームがこのリードを作業します。SDR は BDR Assigned とそのマネージャーに Chatter して知らせます。BDR マネージャーは BDR Prospecting Status を適宜更新します。ここで停止します。

**ステップ 7 — アカウントの BDR Prospecting Status は何ですか？**

→ Queued または Worked in FY → SDR チームがこのリードを作業します。ここで停止します。

→ Restricted → BDR Assigned にルーティングします。BDR は SAE に Chatter します: SAE は BDR にアウトリーチしてほしいですか、それとも SAE 自身が管理しますか？SAE は 48 時間以内に応答する必要があります。BDR が承認された場合 → リードレコードで進めます。AE が所有を希望する場合 → リードを Contact に変換します。ここで停止します。

**ステップ 8 — 新しい MQL は同じ会社の既存の MQL に関連していますか？**

会社名またはメールドメインで照合します。

→ いいえ → ラウンドロビンで受け取った SDR がこのリードを作業します。ここで停止します。

→ はい → ステップ 9 に進みます。

**ステップ 9 — 既存の MQL は Accepted または Qualifying ステータスで、過去 30 日間にアクティビティがある、または将来のタスクがスケジュールされていますか？**

→ はい → 既存の MQL のオーナーに割り当てます。ここで停止します。

→ いいえ → ラウンドロビンで受け取った SDR がこのリードを作業します。不明な場合は関連する SDR と再確認します。ここで停止します。

**ステップ 10 — リードは UserGems によって自動生成されましたか？**

**Initial Source** フィールドを確認します — `UserGems` を含んでいますか？

→ はい + SMB リード → インバウンド SDR チーム。ここで停止します。

→ はい + MM または Enterprise リード → 他のすべての RoE ステップに関係なく、テリトリーに割り当てられた BDR にルーティングします。ここで停止します。

### テリトリー移動 RoE

テリトリーが 1 人の BDR から別の BDR に移動し、退任する BDR が同じチームに残る場合、退任する BDR は特定のアカウントの所有権を一時的に保持できます — マネージャーの承認が条件です。引き継ぎの時点で Issue が作成され、30 日後にレビューされます。

**双方向エンゲージメントアカウント:** 退任する BDR は、IQM と新しいオポチュニティにつながる可能性のある双方向エンゲージメントを示せるアカウントを保留できます。30 日後に IQM が設定されておらず、かつ Stage 0 オポチュニティが存在しない場合 → アカウントは新しい BDR に移動します。新しいオポチュニティが作成された場合 → アカウントは、オポチュニティがクオリファイされるか非クオリファイされるまで退任する BDR に残ります。

**既存の Stage 0 オポチュニティアカウント:** 退任する BDR はこれらを 30 日間保持できます。30 日後にオポチュニティがクオリファイされない場合 → アカウントは新しい BDR に移動します。

**自発的に戻ってきたプロスペクト:** アカウントが再割り当てされ、プロスペクトが新しい BDR からの事前のアクティビティなしに戻ってきた場合 → マネージャーのチームが SAO クレジットを受け取ります。

異議 → BDR のリーダーおよびシニアリーダーにエスカレーションします。

#### クオリフィケーション中のテリトリー発見

クオリフィケーションの途中で、アカウントが（従業員数、本社の所在地、または親アカウントの関係により）別のセグメントに属することを発見した場合、SAO クレジットはケースバイケースで決定されます:

→ デューデリジェンスが実施され、かつその情報が早期に発見できなかった可能性があり、かつアカウントが AWA ではなく、かつ過去 30 日間に関連する SDR アクティビティがない → SDR/BDR とそのチームが SAO クレジットを受け取ります。

→ アカウントが AWA である、または過去 30 日間に SDR アクティビティがある → クレジットは AWA の BDR またはリードを作業している SDR に行きます。

→ 情報が早期に発見されるべきだった → クレジットは正しいテリトリーのマネージャーに行きます。

→ テリトリー情報が矛盾している → 潜在的な各テリトリーの AE に Chatter し、進める前に所有権を解決させます。

## オポチュニティの作成と SAO クレジットの取得方法

> **このプロセスの主要フィールド:** `Initial Engagement Channel · Qualification Questions · Next Steps · Next Steps Date · Stage Name · BDR/SDR field · Sales Qualified Source`

クオリフィケーション基準は、リードが Sales Accepted Opportunity (SAO) としてセールスに渡される前に満たすべき最低限のしきい値を定義します。完全な基準: [インバウンドおよびアウトバウンド](/handbook/sales/field-operations/gtm-resources/#opportunities)。

SFDC オポチュニティを作成するための完全なウォークスルーは[こちら](/handbook/sales/field-operations/gtm-resources/#how-to-create-an-opportunity)です。すべての Sales Dev メンバーは、オポチュニティを作成する前に、リードの Lead/Contact Review Admin セクションにある **Initial Engagement Channel** フィールドも入力する必要があります。

### IQM のスケジューリング

> **このプロセスの主要フィールド:** `Company Address · Company Address Checked · Related To · Next Steps Date`

**オポチュニティが SAO になるタイミング**

AE/SAL は IQM 後、Next Steps Date フィールドに反映されたミーティング日から 48 時間以内に、オポチュニティを Stage 1 Discovery に移動します。オポチュニティの Close Date はデフォルトで四半期の最終日になります — ベストプラクティスは、作成日から少なくとも 90 日後に設定することです。

オポチュニティが Stage 1 に移動した後にデータが不正確な場合 → マネージャーに Chatter → 彼らが Sales Dev Operations（Ramona、Panos、または Ed）の誰かに Chatter します。Sales Dev の他の誰も Stage 1 以降のオポチュニティを編集できません。

オポチュニティが、既存の中央オポチュニティを持つ Large アカウント内の新しいユーザーグループに対するものである場合 → あなたのオポチュニティはそれにマージされるべきです。[Stage 8 ガイダンス](/handbook/sales/field-operations/gtm-resources/#opportunity-stages)を参照してください。

### オポチュニティを作成するタイミング

**シナリオ A — AE/SAE との IQM がスケジュールされている:**

→ オポチュニティを Stage 0 (Pending Acceptance) で作成します。

→ すべての Qualification Questions フィールドを完成させます。

→ Next Steps Date と Next Steps の詳細を設定します。

→ すべての関係者からカレンダーの確認を受け取ったら → Ownership を AE/SAE に移し、自分を BDR/SDR フィールドに追加します。オポチュニティは Stage 0 に留まります。

**シナリオ B — プロスペクトは会話の継続にコミットしたが、完全にはクオリファイされていない:**

→ パイプライントラッキングとチームコラボレーションのために、自分の名前でオポチュニティを作成します。

→ 一部の Qualification Questions フィールドは未完成のままにできます。

→ Next Steps Date と Next Steps の詳細は必須です。

→ 注: 自分を Opportunity Owner と BDR/SDR Representative の両方として記載することはできません — 検証ルールがこれをブロックします。

**シナリオ C — 新しいアカウントからのリード:**

→ プロスペクトに会社の従業員数と本社の所在地を確認します — メールまたは記録された通話メモを介して。

→ ZoomInfo、Cognism、または Sales Navigator と相互参照します。

→ 従業員数が 250 に近い場合は特に慎重になってください。SMB/MM の境界は 250 名だからです。

### 引き継ぎミーティングと AE へのオポチュニティの受け渡し

BDR-AE 引き継ぎプロセスには 3 つの目標があります: プロスペクトに可能な限り最高の体験を提供すること、BDR に AE チームと協力する構造化された方法を提供すること、AE に初日から正しい情報で成功への準備を整えること。

**すべての引き継ぎの最低要件:**

- IQM は少なくとも 48 業務時間前に予約します。
- アウトバウンド SAO のクオリフィケーション基準を完全に満たすか、AE と協力して相互のコマンドプランを構築します。
- AE は IQM から 8 業務時間以内に SAO を受け入れ、BDR と AE マネージャーをタグ付けして、クオリフィケーションと引き継ぎのフィードバックを含む Chatter メモを残す必要があります。
- AE は引き継ぎ後、プロスペクトとの関係を所有します — 再スケジュールと競合の管理を含みます。

#### 1. BDR Qualified Meeting

BDR Qualified Meeting は、Discovery コールで BDR によって完全にクオリファイされたリードです。CoM の原則が適用され、Before/After シナリオ、PBO、要件、メトリクスが特定され合意されています。明確なニーズと意思決定者への明確なパスがあります。

**Discovery コール後の BDR ステップ:**

1. コール中に明らかになった CoM の原則を要約します。
2. コール中にまだ Outreach を通じて次のステップをスケジュールします — 45 分のコール、プロスペクトのニーズに合わせて調整した招待本文。
3. AE 紹介メールを送ります。要求の厳しい引き継ぎについては、顧客向けのアジェンダを添付します。
4. 必要な SFDC フィールドを記録し、Notes フィールドを入力します。
5. スケジュールの競合がない限り、Evaluation Orchestration Call に出席します: クオリフィケーションの会話を要約し、プロスペクトと Before/After シナリオを検証し、プロスペクトが状況が変わっていないことを確認したら AE に引き継ぎます。

#### 2. Joint IQM

Joint IQM は、事前に合意された Actively Working アカウントから予約され、BDR と AE が共同で出席しリードするミーティングです。

**コール前の BDR ステップ:**

1. Outreach を通じて AE のカレンダーに直接スケジュールします — 15 分の Discovery Call 形式。
2. SFDC オポチュニティを作成し、事前に決定したリサーチを記録します。
3. AE とアライメントを取り、相互のコマンドプランを構築します。キックオフ時には、BDR のリサーチと説得力のあるイベントを要約します。プロスペクトが状況を認識したら、事前に合意した discovery 構造で続けます。

#### IQM のスケジューリング — チェックリスト

1. **コミュニケーションのログと関連付け** → SFDC で、関連する各アクティビティレコードを選択 → **Related To** を押す → オポチュニティにリンクします。

2. **Sales Organisation RoE の検証** → ZoomInfo または別の承認済みソースで親/子のセグメンテーションと本社を確認します。不一致が存在する場合は、進める前に関連するチームと連絡を取ります。

3. **必要に応じて不正確なアカウント割り当てを上書き** → SFDC の Lead/Contact Review Admin に移動 → Company Address フィールドを更新 → **Company Address Checked** チェックボックスにチェックを入れます。これが完了していないと、SFDC はリード変換をブロックします。

4. **IQM のスケジュール** → AE に最低 24 時間前に通知します。コール中にプロスペクトに招待を承認してもらいます。

5. **AE レビュー** → AE はコール前にオポチュニティをレビューすることが期待されます。具体的な内容をブリーフィングし、必要に応じてリマインダーを送ります。

6. **時間通りに出席** → AE と BDR/SDR の両方が 5 分前に、カメラオン、静かな屋内の場所で出席すべきです。

7. **24 時間以内のデブリーフ** → 目標: AE と BDR/SDR が、オポチュニティがフリップされたか非クオリファイされたかについてデブリーフします。Slack またはメールで書面によるフィードバックを行い、AE が SFDC に追加します。

8. **IQM メモのログ** → オポチュニティの Initiative セクションに追加します: 出席者、Raw Notes、Questions、Summary、Next Steps。

9. **ノーショーの再予約** → BDR/SDR は構造化された Outreach シーケンスを介して再予約を所有します。最大 2 週間アウトリーチします。IQM を再スケジュールできない場合、AE はオポチュニティを非クオリファイします。

#### リセラーとの連携

エンドユーザーアカウントが BDR/SDR のアライメントを決定します。あなたがサードパーティリードに割り当てられた SDR である場合、ステップ 1 の情報を収集し、正しく割り当てられた BDR（エンドユーザーアカウントにアライン）に渡します。BDR がステップ 2 〜 5 を完了します。

| ステップ | アクション |
| ---- | ------ |
| 1. 請求およびエンドユーザーの詳細を収集 | 請求会社名/住所、請求コンタクト/メール、エンドユーザー会社名/住所、エンドユーザーコンタクト/メール。 |
| 2. エンドユーザーの詳細で新しいリードレコードを作成 | 元のリードからすべてのメモをコピーします。これが変換されるレコードです。 |
| 3. 新しいリードを変換 | オポチュニティに `[End-user account] via [Reseller account]` と名前を付けます。元のリセラーリードをリセラーアカウントの Contact に変換します（アカウントが存在しない場合は作成）。同じアカウント所有者に割り当てます。このリードで新しいオポチュニティを作成しないでください。 |
| 4. アクティビティをオポチュニティに添付 | Activity History → 関連する各アクティビティを編集 → Related To ドロップダウンから `Opportunity` を選択 → 新しいオポチュニティを見つける → 保存。 |
| 5. オポチュニティを更新 | Business type = New Business、stage = Pending Acceptance。Contacts の下: リセラーコンタクトを `Reseller`（プライマリ）として追加。Partners の下: リセラーアカウントを `VAR/Reseller` として追加。 |

### RelevanceAI

RelevanceAI は私たちの AI ワークフォースプラットフォームで、SDR および BDR チームのためにリードリサーチ、優先順位付け、重複排除を自動化するために使用されます。現在 2 つのエージェントがアクティブです:

**Mona** は MQL リサーチアシスタントです。リードが Salesforce で MQL ステータスに達すると、Mona は企業リサーチ、本人確認、購入シグナルのコンテキストを自動的にまとめ、傾向評価と推奨される次のアクションをリードレコードに直接書き込みます。

**Doope** は重複排除アシスタントです。一致するリードおよびコンタクトレコードを確認し、リサーチフィールドの競合を重複レコードへのリンクとともにフラグ付けするため、担当者はシーケンシングの前にそれらを解決できます。

#### RelevanceAI エージェントの使用

**Relevance Priority** と **Relevance Next Action** フィールドは SDR の Z リードビューに表示され、個々のレコードを開かずにトリアージしアクションできます。リードをクリックすると、本人確認の信頼度、企業コンテキスト、採用シグナル、評価の背後にある理由を含む完全な **Relevance Research** フィールドが表示されます。**Relevance Summary** フィールドは、簡単に参照できるよう主要なフラグのコンパクトなバージョンを提供します。

Mona の出力は担当者の判断の代替ではありません — 出発点として扱ってください。リサーチが不完全に見えたり、評価がおかしいと思われる場合は、**Relevance Last Updated** タイムスタンプを確認し、パターンがあれば Slack の `#sales-development-relevance-ai` で提起してください。

### 6Sense

6Sense は私たちのインテントデータおよび ABM プラットフォームです。購入シグナルと ICP フィットに基づいて、市場にいるアカウントを表示します。新しいセグメントやアラートを要求するには、[MktgOps リクエストフォーム](https://form.asana.com/?k=1i4lL5h0RLzfTqNWBTH84Q&d=306855239930259)を使用します。

#### 6QA オートメーション

6Sense でアカウントが 6QA ステータスに達すると、以下が自動的に行われます:

→ **BDR Prospecting Status** が `Actively Working` に設定されます

→ **BDR Account Strategy** が `Showing Intent` に設定されます

→ アカウントがレビューのために BDR の 1:1 ダッシュボードにフラグ付けされます。

**その後、BDR は 48 時間以内にアクションする必要があります:**

→ **6QA Acceptance Status** を `Accepted` または `Disputed` に設定します。

→ Disputed の場合 → **6QA Dispute Reason** を入力します（例: `Account in open opportunity` または `Low LAM Dev Count`）。

→ Accepted の場合 → ZoomInfo ワークフローが自動的にトリガーされ、関連する意思決定者で B6 リードビューを入力します。

6QA アカウントが非 FO かつ Compensation Target Account ではない場合、BDR Prospecting Status はまず 14 日間 `SDR Hold` に設定されます。14 日以内にオポチュニティが作成されない場合、BDR の所有権で `Actively Working / Showing Intent` に自動移行します。

完全なオートメーションの内訳: [内部ハンドブック](https://internal.gitlab.com/handbook/marketing/marketing-ops-and-analytics/marketing-operations/6sense/#workflow-breakdown)。

### UserGems

UserGems は、私たちが重視するアカウントでの転職や新規採用を追跡し、温かいリードを自動的に表示します。

**ユースケース 1 — コンタクトトラッキング（転職）**

追跡対象の人物が転職すると、更新された詳細で新しい Lead が SFDC に作成されます。**Initial Source** フィールドには `UserGems Contact Tracking` と表示されます。

ルーティング: アカウントは MM かつ FO ですか、または Focus Account ですか、または Actively Working ですか、または PubSec/FINS/Telco/Base/APJ ですか？はいの場合 → テリトリーに割り当てられた BDR にルーティングします。いいえの場合 → SDR ラウンドロビン。

**ユースケース 2 — 新規採用と昇進**

追跡対象の企業に関連する新規採用または昇進があると、**Initial Source** = `UserGems - New Hires and Promotions` で新しい Lead が作成され、専用の AI 駆動の Outreach シーケンスに自動登録されます。これらはインテントシグナルとしてダッシュボードにもフラグ付けされます。

→ 無関係な役職が作成されているのを見た場合 → `#usergems-feedback` でフラグ付けします。

→ または Lead Status を `Disqualified` に、Disqualified Reason を `No Authority` にマークします。

**ユースケース 3 — オープンなオポチュニティ上のコンタクト**

オープンなオポチュニティを持つアカウントに誰かが参加または離脱すると、Stage 3+ のオポチュニティ → セールスチームのみ、Stage 0 〜 2 のオポチュニティ → セールスおよびセールス開発チームに通知が送られます。

これらのリードをレビューするには[このレポート](https://gitlab.my.salesforce.com/00OPL000007z6XV)を使用します。特定のシーケンスは存在しません — シナリオに基づいてテンプレート化されたメッセージを Outreach で検索します。

### 月次監査プロセス

毎月、Sales Dev メンバーにクレジットされたすべてのオポチュニティについて完全な監査が実施されます。参加は必須です。SDR は SDR を監査し、BDR は BDR を監査します。ランピング四半期中のチームメンバーは監査者になることを免除されます。

**必須 SLA:** 監査と XDR の応答は翌月の初日までに完了します。判定は 2 日目の終わりまでに完了します。

**確認すべき内容:**

| チェック | 検証すべきこと | 根拠 |
| ----- | -------------- | --------- |
| オポチュニティ作成者 | XDR であるべきです。**Sales Qualified Source** = Sales Dev Generated で AE が作成した場合、それ以前の意味のあるエンゲージメントを証拠として示す必要があります。 | 私たちの標準は、XDR がオポチュニティを作成することです。 |
| Web Direct — タイムスタンプとエンゲージメント | エンゲージメントは購入よりも前である必要があります。 | クレジットには、購入決定への影響の実証が必要です。 |
| クオリフィケーションフィールドの入力 | すべての Stage 0 Qualification Questions フィールドを完成させる必要があります。 | 不完全なフィールドは、パイプライン進行プロセスから説明責任を取り除きます。 |
| オポチュニティ作成日 | アクティブなエンゲージメントと一致するべきです。IQM 後に作成された子/関連オポチュニティは、IQM アクティビティを関連タスクとして持つべきです。 | オポチュニティがどのようにソースされたかの明確な監査証跡を作成します。 |
| 意味のあるエンゲージメントを伴うコンタクト | オポチュニティが作成される前に、コンタクトが添付され、それ以前のエンゲージメントを示している必要があります。 | SAO クレジットに必要です。 |
| アクティビティが Outreach と一致 | SFDC アクティビティを Outreach レコードと相互参照します。Outreach は反論できない信頼できる情報源です — SFDC アクティビティは自由に編集できます。 | Outreach は常に最後のバックアップとして使用されます。 |
| AWA アカウントの RoE 遵守 | アクティビティは、チームメンバーの割り当てられたテリトリー内に収まる必要があります。 | パイプラインが正しい地域に対してカウントされることを保証します。 |

**典型的な危険信号:**

- 文書化されたソーシングケースなしに Sales Qualified Source が `SDR Generated` ではない
- 説明なしに AE が作成したオポチュニティ
- それ以前のアクティビティなしに同じ日に設定され完了した IQM
- 重複したオポチュニティ
- Net ARR が欠落またはゼロ

**SAO クレジットを守るためのベストプラクティス:**

1. アウトリーチしているすべてのリードを Outreach シーケンスに追加します。
2. 意味のあるエンゲージメントをどのように促進したかを示すすべてのアクティビティを記録します。
3. すべての Qualification Questions フィールドを入力します。
4. コールメモは後ではなく、コール時に記録します。
5. ミーティングが行われる前に Stage 0 オポチュニティを作成します。
6. LinkedIn または WhatsApp のエンゲージメントについては、スクリーンショットを撮って Chatter またはオポチュニティに添付します。
7. イベントソースのミーティングについては、イベント直後に Stage 0 オポチュニティを作成し、会話を要約したフォローアップメールを送ります。

### フィールドマーケティングと BDR のコラボレーションプロセス

私たちの FM/BDR コラボレーションプロセスは、クロスファンクショナルなコラボレーションを最大化する精神に沿って従う手法です。私たちはフィールドマーケターが入力する Issue テンプレートを作成しており、それは[こちらの Kanban ボード](https://gitlab.com/gitlab-com/marketing/sales-development/-/boards/5665082)から管理されます。[Issue テンプレート](https://gitlab.com/gitlab-com/marketing/sales-development/-/blob/main/.gitlab/issue_templates/FM_BDR_Collaboration_Template.md)はすべての次のステップを明確に示しながら、各特定のイベントが必要とするだけのアドホックなコラボレーションのための余地を残しています。

BDR Director として、あなたはまず主要 DRI としてタグ付けされます。BDR Manager として、あなたは地域の Director によってケースバイケースで関与します。すべての次のステップはテンプレートに明確に記載されています — 各ステップを順番に従ってください。Sales Dev Operations チームもタグ付けされ、進捗を監視し、要求されれば支援を提供します。

### Sales Dev President's Club の基準

President's Club の資格と順位は、各チームメンバーの報酬プランと Quota コンポーネントの重み付けに対するパフォーマンスを使用して決定されます。_**President's Club の検討において、Hybrid/Growth Pubsec BDR（およびそのリーダー）の Stage 1 Net ARR 達成度は、チームメンバーが他の Quota コンポーネントでも期待値を満たしていることを保証するために 200% で上限が設けられます。_

### Sales Dev パフォーマンスマネジメントプロセス

ランプ済みのチームメンバーが 2 か月連続で 80% を下回る達成度の場合、非公式のパフォーマンスマネジメントが開始されます。_**パフォーマンスマネジメントの検討において、Hybrid/Growth PubSec BDR の Stage 1 Net ARR 達成度は、チームメンバーが他の Quota コンポーネントでも期待値を満たしていることを保証するために 200% で上限が設けられます。_

チームメンバーに通知されると、期待値は以下のとおりです:

→ 1 か月目: 80% 達成 + [Our Three Pillars](/handbook/marketing/sales-development/#our-three-pillars)の遵守

→ 2 か月目: 90% 達成 + [Our Three Pillars](/handbook/marketing/sales-development/#our-three-pillars)の遵守

→ 3 か月目以降: 100% 達成 + [Our Three Pillars](/handbook/marketing/sales-development/#our-three-pillars)の遵守

継続的な不足 → 正式な警告 → 解雇を含む懲戒処分の可能性。

### Our Three Pillars

| 柱 | 主要な期待値 |
| ------ | ---------------- |
| **1. 日々のアクティビティ指標を維持する** | 2 時間 MQL SLA を遵守 · 1 日最低 50 のオムニチャネルアクティビティ（Golden Call 時間中は週 200 コール；個別化された LinkedIn/メール）· 週最低 2 件の discovery ミーティング · シーケンスステップを期限当日にアクション · アウトバウンドワークフロー全体を通じて SFDC データの整合性を維持 · 各 IQM に AE/SAE と出席 · SFDC に正確で最新のメモを文書化 |
| **2. ビジネスおよびセールスの手腕を発揮する** | CoM のメール作成原則を使用してインバウンド/アウトバウンドリードをパーソナライズ · CoM のコールドコール原則を使用してコールドコールに備える · 地域チームの期待値に沿って週次ケイデンスでアウトバウンドアカウントを追加 |
| **3. クロスファンクショナルな関係を維持する** | アカウントプランニングでセールスチームと協力 · イベントアウトリーチでフィールドマーケティングと協力 |

### PTO とフレキシブルワーキング

私たちの [Time Off Policy](/handbook/people-group/time-off-and-absence/time-off-types/) に従って、予定している休暇についてマネージャーに早めに通知してください。PTO Territory Planning リクエストをログするには、[SDR GitLab プロジェクト](https://gitlab.com/gitlab-com/marketing/sdr/-/issues/new#)に移動し、`PTO_Coverage_Template` を選択します。

セールス開発のプロスペクト対応の役割では、プロスペクトに電話やメールをする際に以下を念頭に置いてください: 電話に最適な時間帯は業務日の早朝と夕方です；Outreach では早朝と早い夕方の配信のためにメールをスケジュールできます；昼休みはアウトリーチに適しています；そして、あなたの地域のプロスペクトの通常の業務時間にスケジュールを合わせるべきです。

## よくある質問 (FAQ)

### 一般的な Sales Dev のトラブルシューティング

**Q:** 特定の Outreach コレクションが見えません。
**A:** Outreach で誤った Sales Dev Team に追加された可能性が高いです。Sales Dev Operations に連絡してください。

**Q:** 私の Salesforce のバージョンが非常に基本的なものに見えます。
**A:** Sales バージョンを使用していることを確認してください。右上の青いドロップダウンボタン → "Sales" を選択します。

**Q:** すべてのツールへのアクセスを受け取っておらず、1 週間が経過しました。
**A:** マネージャーに通知し、あなたの Onboarding Role Entitlements Issue にコメントしてもらいます。

**Q:** ZoomInfo から、SFDC ZoomInfo Lead View に表示されているよりも多くのリードをアップロードしました。
**A:** これらのリードはおそらくすでに SFDC に Contacts として存在します。それらは Leads ビューではなく、ZoomInfo Contacts ビューに表示されます。

**Q:** この人物が MQL としてスコアされた理由がわかりません。
**A:** SFDC レコードの **Last Interesting Moment** を確認し、次にリードページの Marketo Sales Insight ウィジェットの Scoring タブを確認します。行動スコアなしにデモグラフィックポイントのみが割り当てられている場合は、新しいアクションが取られなかったことを Marketing Ops が認識できるよう Slack で連絡します。

**Q:** プロスペクトから個人データ主体リクエストを受け取りました。
**A:** [Personal Data Subject Request フォーム](https://support.gitlab.io/personal-data-request/)にリダイレクトします。

**Q:** なぜ BDR はもう SFDC の Account Owner ではないのですか？
**A:** すべてのプロスペクトとアカウント（PubSec を除く）にわたる Sales Dev とセールスの可視性を向上させるためです。自分のアカウントにフィルタリングするには **BDR Assigned** フィールドを使用します。
