## Request for Information / Details

顧客または潜在的な顧客が、Product、Marketing、Legal、または Environmental, Social and Governance (ESG) に関する質問 (更新時または有効期間中を含む) を持っている場合は、以下の [Request for Proposal (RFP) Process](#request-for-proposal-rfp-process) に従ってください。

## Request for Proposal (RFP) Process {#request-for-proposal-rfp-process}

Field Security チーム、Legal & Corporate Affairs、Field Ops、Sales は、Field チーム向けの Request for Proposal または Information プロセスをアシストし簡素化するためにプロセスを作成しました。RFP に関しては、Field チームは以下の手順に従う必要があります。

1. Request for Proposal (RFP) または Request for Information (RFI) を受領した時点で、Sales のチームメンバーは SFDC でアカウントとオポチュニティが作成されていることを確認する必要があります。
1. RFP に対して見積もりの作成以上のもの (例: さまざまなステークホルダーからの洞察と支援) が必要な場合、Sales のチームメンバーは SFDC オポチュニティ内で Chatter を介して ASM (またはそれ以上) の承認をリクエストし受領する必要があります。
1. Sales のチームメンバーは、Security、Corporate、Privacy、Commercial Legal、ESG に関する質問について、GitLab の Trust Center [Knowledgebase](https://trust.gitlab.com) のコンテンツをレビューします。*注意:* すべてのチームメンバーにベースラインのエンタイトルメントがあります。
1. Sales のチームメンバーは、価格設定や見積もりに関する質問について、標準プロセスに従って [Deal Desk](/handbook/sales/field-operations/sales-operations/deal-desk/#communicating-with-the-deal-desk-team) に関与してもらいます。
1. Sales のチームメンバーは、製品機能に関する質問について、GitLab の[製品ドキュメント](https://docs.gitlab.com)のコンテンツをレビューする必要があります。その他の役立つリソースには[ブログ投稿](https://about.gitlab.com/blog/)や[リリース投稿](https://about.gitlab.com/releases/categories/releases/)があります。
1. チームには、質問に答えるために [Glean](/handbook/eta/ai/tools/glean/) と [GitLab Duo](https://docs.gitlab.com/user/duo_agent_platform/) を活用することを推奨します。外部に配布する前に必ず回答をレビューし検証してください。
1. 上記の手順を踏んでも回答されなかった Product、Security、Corporate、Privacy、Commercial Legal、ESG に関する質問については、Sales のチームメンバーは[こちら](https://gitlab.com/gitlab-com/legal-and-compliance/-/issues/new?issuable_template=RFP_Intake%20Form)で Issue をオープンできます: **必ず RFP Intake Issue Template を選択してください**
1. **RFP Intake Issue Template** に記入する際は、次のことを行ってください:
    - 該当するコンテンツ領域のチェックボックスを選択する。
    - 不要な通知を防ぐため、**該当しないコンテンツ領域の行を削除**する。関連する DRI のみが Issue でタグ付けされるべきです。
1. RFP Intake Issue は該当する DRI にアサインされます。

## Field Security ができないこと

Request for Proposal または Request for Information プロセスの一部に、Field Security のスコープ外のものがあります。*私たちは提案書を書くことや、以下を提供することはできません: 製品やサービスの仕様、実装スケジュールおよびプラン、トレーニングやサポートの仕様および条件、GitLab が適切なリソースを提供できるかに関する情報、他の GitLab 顧客のエクスペリエンス、コスト、価値、価格設定、支払いなどに関連する事項。*

私たちが*できること*は、セキュリティに関連するリクエストに回答し、People/HR、Finance、Privacy、Commercial Legal などの領域に関する他の SME の回答をファシリテートすることです。

## 役立つリソース

私たちのチームはプロセスをファシリテートするために最善を尽くしますが、特定のドキュメントを完了するためのセルフサービスリソースもあります。

`Vendor Set Up Form` を受け取った場合は、[こちら](/handbook/sales/field-operations/order-processing/#how-to-process-customer-requested-vendor-setup-forms)で詳細を確認できます。

`NDA` が必要な場合は、入手方法に関する詳細を[こちら](/handbook/legal/nda/)で確認できます。

各タイプの Customer Assurance Activity のタイムラインに関しては、[私たちの SLA](/handbook/security/security-assurance/field-security/customer-security-assessment-process/#service-level-agreements) をご確認ください。

**財務情報**については、すべての GitLab エンティティの銀行情報を含むあらゆる関連情報がある [Sales Team New Customer Form Resources](https://gitlab.com/gitlab-com/Finance-Division/finance/-/wikis/Sales%20Team%20New%20Customer%20Form%20Resources) を使用してください。また、提出書類および財務結果データについては [investor relations](https://ir.gitlab.com/) ページもご覧ください。
