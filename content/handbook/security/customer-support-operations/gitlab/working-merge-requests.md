このガイドでは、Customer Support Operations プロジェクトのマージリクエスト（MR）を作成、レビュー、マージする方法を説明します。これらのプラクティスに従うことで、コードの品質を確保し、セキュリティ基準を維持し、チーム全体での協働的な開発を可能にします。

## マージリクエストを理解する

### マージリクエストを使う理由

マージリクエストはいくつかの重要なメリットをもたらします。

- **ピアレビュー**: コードの品質を確保し、デプロイ前に潜在的な問題を見つけます。
- **コラボレーション**: チームメンバー間での議論と知識共有を可能にします。
- **監査証跡**: 何が、なぜ変更され、誰が承認したかについて、ドキュメント化された履歴を作成します。
- **品質ゲート**: 問題のあるコードが本番環境に到達するのを防ぎます。
- **バージョン管理**: 変更が問題を引き起こした場合に簡単にロールバックできます。

## マージリクエストを作成する

### 前提条件

マージリクエストを作成する前に、以下が揃っていることを確認してください。

- 些細なドキュメントの修正でない限り、対応する Issue（Feature Request、Administrative、Bug など）
- `USERNAME-PROJECT-IID` の形式に従った適切な名前のブランチ
  - 例: `jcolyer-support-ops-project-1963`
  - 詳細は[コーディング規約 - ブランチ](/handbook/security/customer-support-operations/resources/coding-standards#version-control)を参照してください。
- 説明的なコミットメッセージとともにコミットされた変更
- ローカルでテストされた変更（該当する場合）

### マージリクエストの作成

マージリクエストを作成するには:

1. GitLab.com 上で自分のプロジェクトに移動します。
1. 左サイドバーの **Merge requests** に移動します。
1. **New merge request** ボタンをクリックします。
1. ソースブランチ（作業ブランチ）とターゲットブランチ（通常は `master` または `main`）を選択します。
1. **Compare branches and continue** をクリックします。
1. マージリクエストの詳細を入力します。
   - **Title**: 変更内容を要約した、明確で説明的なタイトルを使用します。
     - 良い例: 「Add new trigger for premium support routing」
     - 悪い例: 「Update triggers」や「Fix stuff」
   - **Description**: 以下を含むコンテキストを記載します。
     - 関連する Issue へのリンク: `Relates to ISSUE_URL`
     - 何がなぜ変更されたかの要約
     - 特別なテストやデプロイ上の考慮事項
     - 破壊的変更や依存関係（該当する場合）
   - **Assignee**: 自分自身を割り当てます。
   - **Reviewer**: レビューのためにチームメンバーを割り当てます（[レビューを依頼する](#requesting-a-review)を参照）。
1. **Create merge request** をクリックします。

### 同期リポジトリの場合

同期リポジトリ（トリガー、ビュー、自動化など）でマージリクエストを作成すると、MR は自動的に比較パイプラインを実行し、以下を行います。

- YAML 構文を検証する
- 必須フィールドをチェックする
- 提案された変更を現在の Zendesk 構成と比較する
- 比較レポートを生成する

レビューを依頼する前に、パイプラインの結果を確認して変更が正しいことを確認してください。

## レビューを依頼する

### 誰にレビューを依頼するか

- **標準的な変更の場合**: その領域の知識を持つ、対応可能な任意のチームメンバーにレビューを依頼します。
- **複雑またはリスクのある変更の場合**: 複数のチームメンバーまたはシニアエンジニアにレビューを依頼します。
- **セキュリティ上重要な変更の場合**: セキュリティに精通したレビュアーを含めます。
- **不明な場合**: チームチャンネルでレビュアーの推薦を尋ねます。

### レビュアーが確認すべきこと

ピアレビューを実施する際、レビュアーは以下を検証すべきです。

- **機能性**: 変更が記載された目標を達成していること
- **コードの品質**: [コーディング規約](/handbook/security/customer-support-operations/resources/coding-standards)に従っていること
- **セキュリティ**: ハードコードされたシークレットがなく、適切な入力検証、セキュアなプラクティスがあること
- **テスト**: 変更がテストされていること（該当する場合）
- **ドキュメント**: 複雑なロジックにはコメントが付けられ、必要に応じて README が更新されていること
- **副作用**: 変更が既存の機能を壊さないこと
- **デプロイへの影響**: デプロイ時に何が起こるかを理解していること

## マージリクエストをレビューする

### レビューの方法

レビュアーとして割り当てられた場合:

1. コンテキストを理解するために、関連する Issue を読みます。
1. **Changes** タブで変更をレビューします。
   - 各ファイルの差分を確認します。
   - [レビュアーが確認すべきこと](#what-reviewers-should-check)に記載された問題をチェックします。
1. パイプラインの結果を確認します（特に同期リポジトリの場合）。
   - パイプラインがパスすることを確認します。
   - 比較レポートをレビューします。
1. 必要に応じて、ローカルで変更をテストします（コード変更の場合）。
1. 以下のいずれかの方法でフィードバックを残します。
   - **Approve**: 変更が良さそうであれば、マージリクエストで **Approve** をクリックします。
   - **変更を依頼する**: 特定の行へのコメント、または懸念を説明する一般的なコメントを追加します。
   - **質問する**: 不明な点を明確にするためにコメントを使用します。

### フィードバックを提供する

フィードバックを残す際は:

- **具体的に**: 正確な行を指し示し、問題を説明します。
- **建設的に**: 批判するだけでなく、改善案を提案します。
- **適時に**: 可能な限り 1〜2 営業日以内にレビューします。
- **質問する**: 何かが不明な場合は、推測せずに質問します。
- **良い仕事を認める**: 巧妙な解決策や改善を称賛します。

### マージリクエストを承認する

マージリクエストを承認するには:

1. すべての懸念が対処されていることを確認します。
1. 右サイドバーの **Approve** をクリックします。
1. 任意で「Looks good! ✅」や検証した内容を記したコメントを追加します。

## マージリクエストをマージする

### いつマージするか

マージリクエストは、以下の場合にマージできます。

- ✅ 少なくとも 1 人のピアレビュアーによって承認されている
- ✅ すべてのパイプラインチェックがパスしている（緑のチェックマーク）
- ✅ すべての議論が解決されている（または明示的にブロックしないものとしてマークされている）
- ✅ 関連する Issue がデプロイ準備完了であることを確認している
- ✅ Standard デプロイの場合: 次のデプロイ日にデプロイされる準備ができている

### マージの方法

承認されたマージリクエストをマージするには:

1. すべての[マージ基準](#when-to-merge)が満たされていることを確認します。
1. **Merge** をクリックします。

### マージ後

MR がマージされた後:

- **Ad-hoc デプロイの場合**: 変更はすぐに、または次回のスクリプト実行時にデプロイされます。
- **Standard デプロイの場合**: 変更は次回のスケジュールされたデプロイ（毎月 1 日）でデプロイされます。
- **関連する Issue にコメントを追加する**: MR がマージされたことを記します。
- **問題を監視する**: デプロイ後に予期しない動作がないか注意します。
- **緊急の場合は例外デプロイを実行する**: 例外デプロイの手順については、各ドキュメントページを参照してください。

## よくあるマージリクエストのシナリオ

### レビュー後に変更を加える

レビュアーが変更を依頼した場合:

1. ローカルブランチで依頼された変更を加えます。
1. 変更をコミットしてプッシュします。
1. MR は新しいコミットで自動的に更新されます。
1. レビュアーのコメントに返信し、何を変更したかを説明します。
1. 必要に応じて再レビューを依頼します。

### マージコンフリクトの処理

MR にマージコンフリクトが表示される場合は、次の方法で CLI 経由で解決できます。

1. コンピューター上のプロジェクトリポジトリに移動します。
1. ソースブランチをチェックアウトします（例として `jcolyer-source-branch` を使用します）。

   ```bash
   git checkout jcolyer-source-branch
   ```

1. ターゲットブランチから自分のブランチをリベースします（例として `jcolyer-target-branch` を使用します）。

   ```bash
   git rebase origin/jcolyer-target-branch
   ```

1. 出力を確認して、コンフリクトを含むファイルを特定します。
1. 該当するファイルを編集してコンフリクトを解決します。
1. 変更したファイルを追加します（例として `public/index.html` を使用します）。

   ```bash
   git add public/index.html
   ```

1. リベースを続行します。

   ```bash
   git rebase --continue
   ```

1. 必要に応じてステップ 4 から繰り返します（出力でリベースが完了したと表示されるまで）。
1. `--force-with-lease` フラグを使って、変更をソースブランチにプッシュします。

   ```bash
   git push origin jcolyer-source-branch --force-with-lease
   ```

1. MR は自動的に更新されます。

### マージせずにマージリクエストを閉じる

変更を進めないことに決めた場合:

1. MR を閉じる理由を説明するコメントを追加します。
1. 関連する Issue を閉じます（適切な場合）。
1. MR の下部にある **Close merge request** をクリックします。
1. 任意でソースブランチを削除します。

## ベストプラクティス

- **MR を焦点を絞ったものに保つ**: 1 つの MR は 1 つの Issue または機能に対処すべきです。無関係な変更を混ぜないようにします。
- **MR を小さく保つ**: 小さい MR はレビューしやすく、マージのリスクも低くなります。可能な限り、変更行数を 500 行未満に抑えることを目指します。
- **明確な説明を書く**: 将来のあなた（とチームメイト）は、なぜ変更が行われたかについてのコンテキストに感謝するでしょう。
- **レビューに迅速に対応する**: 作業を進め続けるために、1〜2 日以内にフィードバックに対処します。
- **レビューを依頼する前にテストする**: レビュアーを QA として使わないでください。まず自分で変更をテストします。
- **ドキュメントを更新する**: 変更がプロセスや使用方法に影響する場合は、関連ドキュメントを更新します。
- **関連する Issue にリンクする**: トレーサビリティのために、MR を必ず対処する Issue に関連付けます。

## よくある問題とトラブルシューティング

### 同期リポジトリでのパイプライン失敗

比較パイプラインが失敗する場合:

- パイプラインのログを確認して問題を特定します。
- よくある原因:
  - YAML 構文エラー（インデント、コロン、引用符を確認）
  - 必須フィールドの欠落（title、position など）
  - 同期リポジトリ内の重複したタイトル
  - 管理対象コンテンツファイルの欠落（contains_managed_content: true のトリガー／ビューの場合）
- 問題を修正し、新しいコミットをプッシュします。

### ブランチにプッシュできない

プッシュ時に権限エラーが出る場合:

- プロジェクトに対して少なくとも **Developer** アクセス権があることを確認します。
- 保護ブランチではなく、自分のブランチにプッシュしていることを確認します。
- ブランチ名が期待される形式と一致していることを確認します。

### レビュアーが応答しない

レビュアーが 2〜3 営業日以内に応答しない場合:

- MR のコメントで丁寧なリマインドを送ります: 「@reviewer - gentle reminder for review when you have a moment」
- 緊急の場合はチームの Slack チャンネルで連絡します。
- 元のレビュアーが対応できない場合は、別のレビュアーを依頼します。

### MR が長期間オープンのまま

MR がマージされずに長期間放置されている場合:

- オープンな議論をすべて解決します。
- 最終承認のためにレビュアーにリマインドします。
- コンフリクトが発生した場合は、最新の master でリベースします。
- 変更がまだ必要・関連しているかを検討します。
- もはや該当しない場合は閉じます。
