このガイドは、Customer Support Operations プロジェクトのマージリクエスト（MR）を作成、レビュー、マージする方法を説明します。これらのプラクティスに従うことで、コードの品質を確保し、セキュリティ基準を維持し、チーム全体での協力的な開発を可能にします。

## マージリクエストを理解する

### なぜ私たちはマージリクエストを使用するのか

マージリクエストはいくつかの重要なメリットを提供します:

- **ピアレビュー**: コードの品質を確保し、デプロイ前に潜在的な問題を捉えます
- **コラボレーション**: チームメンバー間の議論と知識共有を可能にします
- **監査証跡**: 何が、なぜ変更され、誰が承認したかについて文書化された履歴を作成します
- **品質ゲート**: 問題のあるコードが本番環境に到達するのを防ぎます
- **バージョン管理**: 変更が問題を引き起こした場合の簡単なロールバックを可能にします

## マージリクエストを作成する

### 前提条件

マージリクエストを作成する前に、以下を確認してください:

- 些細なドキュメント修正でない限り、対応する issue（機能リクエスト、管理用、バグなど）があること
- `USERNAME-PROJECT-IID` の形式に従って適切に命名されたブランチ
  - 例: `jcolyer-support-ops-project-1963`
  - 詳細については [コーディング規約 - ブランチ](/handbook/security/customer-support-operations/resources/coding-standards#version-control) を参照してください
- 説明的なコミットメッセージで変更をコミットしていること
- 変更をローカルでテストしたこと（該当する場合）

### マージリクエストの作成

マージリクエストを作成するには:

1. GitLab.com 上のあなたのプロジェクトに移動します
1. 左サイドバーの **Merge requests** に移動します
1. **New merge request** ボタンをクリックします
1. ソースブランチ（あなたの作業ブランチ）とターゲットブランチ（通常は `master` または `main`）を選択します
1. **Compare branches and continue** をクリックします
1. マージリクエストの詳細を記入します:
   - **タイトル**: 変更を要約した明確で説明的なタイトルを使用します
     - 良い例: 「Add new trigger for premium support routing」
     - 悪い例: 「Update triggers」や「Fix stuff」
   - **説明**: 以下を含むコンテキストを提供します:
     - 関連する issue へのリンク: `Relates to ISSUE_URL`
     - 何が、なぜ変更されたかの要約
     - 特別なテストやデプロイに関する考慮事項
     - 破壊的変更や依存関係（該当する場合）
   - **担当者**: 自分自身を割り当てます
   - **レビュアー**: レビューのためにチームメンバーを割り当てます（[レビューの依頼](#requesting-a-review) を参照）
1. **Create merge request** をクリックします

### 同期リポジトリの場合

同期リポジトリ（トリガー、ビュー、オートメーションなど）でマージリクエストを作成すると、MR は自動的に以下を行う比較パイプラインを実行します:

- YAML 構文を検証する
- 必須フィールドをチェックする
- 提案された変更を現在の Zendesk 設定と比較する
- 比較レポートを生成する

レビューを依頼する前に、パイプラインの結果を確認して変更が正しいことを確認してください。

## レビューを依頼する

### 誰にレビューを依頼するか

- **標準的な変更の場合**: その分野の知識を持つ、対応可能なチームメンバーの誰にでもレビューを依頼します
- **複雑またはリスクの高い変更の場合**: 複数のチームメンバーまたはシニアエンジニアにレビューを依頼します
- **セキュリティに敏感な変更の場合**: セキュリティに精通したレビュアーを含めます
- **不明な場合**: チームチャンネルでレビュアーの推薦を尋ねます

### レビュアーがチェックすべきこと

ピアレビューを実施する際、レビュアーは以下を検証するべきです:

- **機能**: 変更が記載された目標を達成していること
- **コードの品質**: [コーディング規約](/handbook/security/customer-support-operations/resources/coding-standards) に従っていること
- **セキュリティ**: ハードコードされたシークレットがなく、適切な入力検証と安全なプラクティスがあること
- **テスト**: 変更がテストされていること（該当する場合）
- **ドキュメント**: 複雑なロジックにコメントが付けられ、必要に応じて README が更新されていること
- **副作用**: 変更が既存の機能を壊さないこと
- **デプロイへの影響**: デプロイされたときに何が起こるかを理解していること

## マージリクエストをレビューする

### レビューの方法

レビュアーとして割り当てられた場合:

1. 関連する issue を読んでコンテキストを理解します
1. **Changes** タブで変更をレビューします
   - 各ファイルの差分を確認します
   - [レビュアーがチェックすべきこと](#what-reviewers-should-check) に記載された問題をチェックします
1. パイプラインの結果をチェックします（特に同期リポジトリの場合）
   - パイプラインが通過することを確認します
   - 比較レポートをレビューします
1. 必要に応じて変更をローカルでテストします（コード変更の場合）
1. 以下のいずれかの方法でフィードバックを残します:
   - **承認**: 変更が良さそうであれば、マージリクエストで **Approve** をクリックします
   - **変更を依頼**: 特定の行へのコメントや、懸念事項を説明する一般的なコメントを追加します
   - **質問する**: 不明確な点を明確にするためにコメントを使用します

### フィードバックの提供

フィードバックを残す際は:

- **具体的にする**: 正確な行を指し示し、問題を説明します
- **建設的にする**: 改善を提案し、単に批判するだけにしないようにします
- **タイムリーにする**: 可能な場合は1〜2営業日以内にレビューします
- **質問する**: 何かが不明確な場合は、推測するのではなく質問します
- **良い仕事を認める**: 賢い解決策や改善を取り上げます

### マージリクエストを承認する

マージリクエストを承認するには:

1. あなたのすべての懸念事項が対処されていることを確認します
1. 右サイドバーの **Approve** をクリックします
1. 任意で、「Looks good! ✅」やあなたが検証した内容を記したコメントを追加します

## マージリクエストをマージする

### いつマージするか

マージリクエストは以下の場合にマージできます:

- ✅ 少なくとも1人のピアレビュアーによって承認されている
- ✅ すべてのパイプラインのチェックが通過している（緑のチェックマーク）
- ✅ すべての議論が解決されている（または明示的にブロッキングでないとマークされている）
- ✅ 関連する issue でデプロイの準備ができていることが確認されている
- ✅ 標準デプロイの場合: 次のデプロイ日にデプロイされる準備ができている

### マージの方法

承認されたマージリクエストをマージするには:

1. すべての [マージ基準](#when-to-merge) が満たされていることを検証します
1. **Merge** をクリックします

### マージ後

MR がマージされた後:

- **アドホックデプロイの場合**: 変更はただちに、または次のスクリプト実行時にデプロイされます
- **標準デプロイの場合**: 変更は次の予定されたデプロイ（毎月1日）にデプロイされます
- **関連する issue にコメントを追加する**: MR がマージされたことを記載します
- **問題を監視する**: デプロイ後の予期しない動作がないか注意します
- **緊急の場合は例外デプロイを実行する**: 例外デプロイの手順については個別のドキュメントページを参照してください

## よくあるマージリクエストのシナリオ

### レビュー後に変更を行う

レビュアーが変更を依頼した場合:

1. ローカルブランチで依頼された変更を行います
1. 変更をコミットしてプッシュします
1. MR は新しいコミットで自動的に更新されます
1. 何を変更したかを説明するためにレビュアーのコメントに返信します
1. 必要に応じて再レビューを依頼します

### マージ競合の処理

MR にマージ競合が表示される場合、以下の方法を使用して CLI 経由で解決できます:

1. コンピューター上のプロジェクトリポジトリに移動します
1. ソースブランチをチェックアウトします（例として `jcolyer-source-branch` を使用します）

   ```bash
   git checkout jcolyer-source-branch
   ```

1. ターゲットブランチからブランチをリベースします（例として `jcolyer-target-branch` を使用します）:

   ```bash
   git rebase origin/jcolyer-target-branch
   ```

1. 出力を確認して、競合を含むファイルを特定します
1. 該当するファイルを編集して競合を解決します
1. 変更したファイルを追加します（例として `public/index.html` を使用します）:

   ```bash
   git add public/index.html
   ```

1. リベースを続行します:

   ```bash
   git rebase --continue
   ```

1. 必要に応じてステップ4から繰り返します（出力にリベースが完了したと表示されるまで）
1. `--force-with-lease` フラグを使用して変更をソースブランチにプッシュします:

   ```bash
   git push origin jcolyer-source-branch --force-with-lease
   ```

1. MR は自動的に更新されます

### マージせずにマージリクエストをクローズする

変更を進めないことに決めた場合:

1. MR がクローズされる理由を説明するコメントを追加します
1. 関連する issue をクローズします（適切な場合）
1. MR の下部にある **Close merge request** をクリックします
1. 任意でソースブランチを削除します

## ベストプラクティス

- **MR の焦点を絞る**: 1つの MR は1つの issue または機能に対処するべきです。無関係な変更を混ぜないようにします。
- **MR を小さく保つ**: 小さい MR はレビューしやすく、マージのリスクが低くなります。可能な場合は変更500行未満を目指します。
- **明確な説明を書く**: 将来のあなた（およびチームメイト）は、なぜ変更が行われたかについてのコンテキストに感謝するでしょう。
- **レビューに迅速に対応する**: 作業を進めるために、1〜2日以内にフィードバックに対処します。
- **レビューを依頼する前にテストする**: レビュアーを QA として使用しないでください。まず自分の変更をテストします。
- **ドキュメントを更新する**: 変更がプロセスや使用方法に影響する場合は、関連するドキュメントを更新します。
- **関連する issue にリンクする**: トレーサビリティのため、常に MR を対処する issue に接続します。

## よくある問題とトラブルシューティング

### 同期リポジトリでのパイプラインの失敗

比較パイプラインが失敗する場合:

- パイプラインのログを確認して問題を特定します
- よくある原因:
  - YAML 構文エラー（インデント、コロン、引用符をチェックする）
  - 必須フィールドの欠落（タイトル、位置など）
  - 同期リポジトリ内でのタイトルの重複
  - 管理対象コンテンツファイルの欠落（contains_managed_content: true のトリガー／ビューの場合）
- 問題を修正して新しいコミットをプッシュします

### ブランチにプッシュできない

プッシュ時に権限エラーが発生する場合:

- プロジェクトに対して少なくとも **Developer** アクセス権があることを検証します
- 保護されたブランチではなく、自分のブランチにプッシュしていることを確認します
- ブランチ名が期待される形式に一致していることをチェックします

### レビュアーが応答しない

レビュアーが2〜3営業日以内に応答しない場合:

- MR のコメントで丁寧にリマインドします: 「@reviewer - お手すきの際にレビューをいただけると助かります（やさしいリマインダーです）」
- 緊急の場合はチームの Slack チャンネルで連絡します
- 元のレビュアーが対応できない場合は別のレビュアーを依頼します

### MR が長く開いたままになっている

MR が長期間マージされないままになっている場合:

- 未解決の議論を解決します
- 最終承認のためにレビュアーにリマインドします
- 競合が発生した場合は最新の master にリベースします
- 変更がまだ必要／関連しているかどうかを検討します
- 該当しなくなった場合はクローズします
