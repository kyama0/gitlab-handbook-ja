UX テーマは、[Upstream Studios のビジョン](/handbook/upstream-studios/vision-strategy/)である戦略的パートナーシップと整合し、単なる戦術的実行ではなく、ビジョン優先のデザインワークを生み出します。UX テーマは、個別の機能ではなく、ユーザーの問題、ニーズ、成果を中心にして業務を組織することで、プロダクトデザイナーがプラットフォーム全体で思考し、複数のマイルストーンにまたがる全体的なソリューションを提供できるようにします。

## UX テーマとは？

UX テーマとは、以下のような包括的な業務の束です:

- 個別の機能に解決策を断片化するのではなく、ユーザーの問題を全体的に解決することに焦点を当てる
- 複数のマイルストーン（通常は 3 つ）にわたり、探索、検証、イテレーションのための時間を提供する
- [Jobs to Be Done（JTBD）](/handbook/product/ux/jobs-to-be-done/)手法を中心に組織する
- ユーザーワークフローがプロダクト領域にまたがる場合、ステージ横断のコラボレーションを可能にする
- 漸進的な提供を導く北極星のビジョンを作り出す

**テーマは戦略的なデザインイニシアチブと考えてください**。ステージグループの主要な JTBD という全体的な目標を達成するために必要であり、他の Issue（バグ、Deferred UX、ユーザビリティ改善）は私たちの基準どおりにエクスペリエンスを維持するためのものです。

## なぜ UX テーマを使うのか？

### 戦略的フォーカス

テーマは、プロダクトデザイン、UX リサーチ、プロダクトマネジメントを、個別の機能ではなく重要な問題領域に整合させます。これによりコンテキストスイッチを減らし、デザイナーが包括的なソリューションに深く取り組むことができます。

### ビジョン優先のデザイン

Upstream Studios で強調されているように、優れたデザインはビジョンから始まります。テーマは、業務をイテレーションに分割する前に北極星を確立し、目的のない磨き込みではなく、変革的な成果に向けた意図的な進歩を確実にします。

### プラットフォーム全体の思考

テーマは、ユーザーニーズが複数のプロダクト領域にまたがる場合に、ステージ横断のコラボレーションの機会を自然に浮かび上がらせ、戦略的業務を定義するプラットフォーム思考を可能にします。

### コラボレーションと整合

テーマは、カテゴリビジョンを実行するための包括的なアプローチを中心に安定したカウンターパートを統合し、測定可能なビジネス成果を通じてチームがデザインの価値を理解するのを助けます。

## UX テーマの構成要素

### テーマステートメント

受益者、そのジョブ、期待される成果を組み合わせた簡潔なステートメント:

**構造:** （成果 + 受益者 + Small Job）

**例:** 「セキュリティチームが資産内のビジネスクリティカルなリスクを優先する際の労力を削減する」

### 主要な JTBD

ユーザーが実行しているタスクを含む主要な Job-to-Be-Done。これにより、テーマの作業中に焦点が主要な JTBD に留まるようにします。

### ビジネス目標

ビジネスの観点からなぜこのテーマに取り組んでいるのか。仮説として機能します:「{ビジネス成果}が達成されれば、ニーズを解決したことがわかる」

### 信頼度レベル

問題とソリューション空間に対するチームの理解の評価:

- **高い信頼度**: チームがジョブを理解しており、問題はリサーチを通じて検証済み
- **中程度の信頼度**: 部分的な理解があり、検証が必要かもしれない
- **低い信頼度**: 信頼度を高めスコープを定義するためにリサーチが必要

### タイムフレーム

業務がいつ提供されるか:

- **Now（1〜3 マイルストーン）**: 高い信頼度、デザインの準備完了
- **Next（4〜7 マイルストーン）**: 問題検証のリサーチが必要かもしれない
- **Future（7〜13 マイルストーン）**: 信頼度を高めるためにリサーチが必要
- **Future+（13 以上のマイルストーン）**: さらなる需要を待っている

## UX テーマでの業務

### プロダクトデザイナー向け

ほとんどの業務は「Now」バケットのテーマで行われます。これは、デザインソリューションの準備ができた高信頼度のテーマです。

**主要な実践:**

1. **個別の Issue ではなくテーマから始める**: 断片化された機能ではなく、ワークフロー全体を全体的にデザインします
2. **最初にビジョンを確立する**: 詳細なデザインの前に、すべての要件に対応する低忠実度のフローを作成します
3. **早期に検証する**: 変更コストが低いうちにワークフローをテストします
4. **継続的に協働する**: PM、エンジニアリング、仲間のデザイナーと協力してビジョンを洗練します
5. **高忠実度に移行する**: 検証されたら、Pajamas コンポーネントを使って詳細なモックアップを開発します
6. **提供のために分解する**: カウンターパートと協力して、段階的な価値を提供する MVC を計画します

デザインアセットを UX テーマ Issue にアップロードして、SSOT として維持します。

### UX リサーチャー向け

「Next」または「Future」バケットのテーマで作業し、デザインの 2〜3 マイルストーン先を進みます。これにより、時間的プレッシャーなしに、調査の設計、参加者のリクルート、リサーチの実行、結果のまとめを行うための時間が確保されます。

リサーチはテーマの信頼度を高め、Future → Next → Now バケットへと移動させます。

### プロダクトデザインマネージャー向け

PM と協働して以下を行います:

- テーマレベルの業務を保証する戦略的 UX 機会を特定する
- テーマがカテゴリビジョンとビジネス目標に整合していることを確認する
- 戦略的テーマ業務とメンテナンス（バグ、Deferred UX）のバランスを取る
- テーマが複数のグループにまたがる場合、ステージ横断のコラボレーションを促進する

## UX テーマの作成

UX テーマは以下の場合に、プロダクトマネージャー、プロダクトデザインマネージャー、プロダクトデザイナーが協働して作成します:

- チームがステージグループの戦略的 UX 方向性を定義する必要がある
- リサーチがカテゴリビジョンに整合する未充足ニーズを発見する
- プロダクトが全体的なソリューションを必要とする新しいユーザーニーズを特定する
- ステージ横断のユーザーワークフローが調整されたデザイン作業を必要とする

## テーマの整理

既存のプロダクトエピック、カテゴリ成熟度エピック、またはビジョンエピックを、UX テーマ Issue のコンテナとして使用します。GitLab ボードを使用してテーマを可視化し、ロードマップビューを作成します。

個々のテーマ Issue は戦略的イニシアチブを計画し作業するための SSOT として機能し、ボードは共有とコミュニケーションのためのハイレベルなロードマップビューを提供します。
