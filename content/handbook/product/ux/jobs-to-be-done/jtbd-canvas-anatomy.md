## **ドメイン:** どこでイノベーションを起こしたいか？

ステージグループ内の関連領域を特定することから始め、誰のためにイノベーションを起こすか、彼らが達成する必要があるものは何かを判断します。

## **ジョブパフォーマー:** 誰のためにイノベーションを起こしたいか？ {#job-performer-who-do-you-want-to-innovate-for}

ジョブパフォーマーとは、特定のジョブを実行する人で、職種とは区別されます。ドメインを特定した後、メインジョブと整合する 1 つのジョブパフォーマーに焦点を当てます。

主要な特徴:

- 個別の役割を記述し、職種ではない
- 複合的な記述を避ける（AND/OR なし）
- 高レベルにとどまり、詳細なペルソナではない
- シンプルで、ターゲットジョブに直接結びついている

| 良い例 | 悪い例 |
|--------------|--------------|
| 新しい住宅購入者 | 億万長者の不動産投資家（ジョブではなく状況を記述している） |
| 著者 | Steven King（具体的すぎる） |
| コードレビュアー | ソフトウェアエンジニア（職種） |

### [オプション] 関連ジョブパフォーマー: 誰が私たちのジョブパフォーマーに影響を与えるか？

ドメインに関係する他のアクターを特定することは、貴重なコンテキストを提供し、プライマリージョブパフォーマーの選択を助けることができます。ジョブパフォーマーが選択されたら、これらの関連ジョブパフォーマーは、必要に応じて後で自分のキャンバスで文書化し探索できます。

### ジョブパフォーマー対ペルソナ

[ユーザーペルソナ](/handbook/product/personas/#user-personas) が職種（「Software Developer」のような）を表すのに対し、ジョブパフォーマーは役割内の特定のタスクに焦点を当てます。1 つのペルソナは複数のメインジョブ（コーディング、レビュー、インフラのメンテナンス）を処理することがあり、異なるペルソナが同じジョブパフォーマー役割を共有することがあります。たとえば、さまざまな役割がコードをレビューする可能性がありますが、その特定のタスクを理解するために単一の「コードレビュアー」ジョブパフォーマーを使用します。

## **メインジョブ:** ジョブパフォーマーは何を達成しようとしているか？

メインジョブは、イノベーションの焦点として機能する時代を超えた、機能的な目標であり、特定の解決策とは独立して、ジョブパフォーマーが達成したいものを表します。

主要な特徴:

- 動詞で始まる
- 明確な終了状態がある
- 複合的な記述を避ける（AND/OR なし）
- 解決策および技術に依存しない
- 形容詞を除外する（それらは Outcomes に属する）
- イノベーションを許可するのに十分広く保つ

構造: [動詞] + [オブジェクト] + [オプションの修飾子]

- 動詞: 達成する行動
- オブジェクト: 行動の対象
- 修飾子: 追加のコンテキストや条件

| 良い例 | 悪い例 |
|--------------|--------------|
| 仕事場から 10 分以内の新しい家を買う | 123 Main Street の家を希望価格以下で購入する（具体的すぎる） |
| 本を書く | ベストセラー作家になる（願望的、ジョブではない） |
| コード変更が組織の標準を満たしていることを保証する | マージリクエストを効率的にレビューする（技術固有、形容詞を使用） |

## **関連ジョブ:** ジョブパフォーマーは他に何を達成しようとしているか？

ジョブパフォーマーがドメイン内で持つ他の 3〜6 つの目標を考慮します。それらは別個の目的であり、同様の詳細レベルで策定されるべきです。これらの関連ジョブは、必要に応じて後で自分のキャンバスで探索できます。

## **願望:** ジョブパフォーマーは何になりたいと願っているか？

願望は通常 1〜3 個の「be」目標で、メインジョブを完了することによるジョブパフォーマーの望ましい個人的成長を表します。これらはキャンバスの上部、メインジョブの階層的に上に位置します。

住宅所有のための階層の例:

- ジョブパフォーマー: 新しい住宅所有者
- 願望: 家庭生活で幸せである、地域コミュニティの一員である
- メインジョブ: 新しい家を取得する
- 関連ジョブ: 新しい家を融資する、家を移る、古い家を売る

## **ジョブステップ:** ジョブパフォーマーはどのようにジョブを完了するか？

ジョブステップは、ジョブパフォーマーがメインジョブを達成するために完了する必要のある連続的な目的です。それらはジョブマップを形成し、各ステップは個別のタスクではなく高レベルです。ステップは、ジョブのすべてのパフォーマーに適用するのに十分広く保ちます。

主要な特徴:

- 一人称で動詞で始まる
- 複合的な記述を避ける（AND/OR なし）
- 解決策に依存しない
- ステージを通じて左から右に流れる
  - サブステップは各ステージ内で垂直に積み重なる

| 良い例 | 悪い例 |
|--------------|--------------|
| 新しい家を探す場所を決める | Richard にどの近所が人気かを聞く（1 つのシナリオに具体的すぎる） |
| 選択基準を決定する |  |
| 新しい家を探す |  |
| 家の所有権を移転する |  |

## **成果:** ジョブパフォーマーはどのように成功を測定するか？

成果は、ジョブパフォーマーが成功を測るために使用する主観的な基準です。メインジョブには通常 50〜100 個の成果ステートメントがあります。

主要な特徴:

- 方向性のある動詞で始まる（minimize、increase など）
- 測定可能な単位（時間、努力、可能性）を含める
- ジョブ固有の修飾子で終わる
- 複合的な記述を避ける（AND/OR なし）
- 解決策に依存しない

| 良い例 | 悪い例 |
|--------------|--------------|
| 潜在的な新しい家を特定するのにかかる時間を最小化する | 最高の家を素早く見つける（方向性なし、「最高」は具体的でない） |
| 新しい家を決定する際に行う妥協の数を減らす | 通りで最も魅力的な家を持つ（方向性なし、主観的な測定） |
| 雇用場所までの距離を最小化する |  |

## **感情的／社会的側面:** ジョブパフォーマーはどのように感じ、どう認識されたいか？

感情的および社会的側面は、ジョブの体験的な側面を表し、ジョブパフォーマーの個人的なニーズを満たすために解決策をどう提供すべきかを形成するのに役立ちます。

感情的側面の主要な特徴:

- 「feel」または「avoid feeling」で始まる
- 特定の感情を示す
- 複合的な記述を避ける（AND/OR なし）
- 解決策に依存しない

社会的側面の主要な特徴:

- 「appear as」または「avoid appearing as」で始まる
- 社会的認識を示す
- 複合的な記述を避ける（AND/OR なし）
- 解決策に依存しない

| 側面タイプ | 例 |
|-------------|----------|
| 感情的 | 家の取得プロセスをコントロールしていると感じる |
|           | 新しい家の選択について不確実に感じることを避ける |
| 社会的    | 良い未来の隣人として見られる |
|           | 新しい家の取得プロセスについて知識がないように見えることを避ける |

これらの側面を理解することは、機能的なニーズと個人的なニーズの両方に対処する解決策をデザインするのに役立ちます。たとえば、効率的に見えることを気にするプログラマー向けに時間節約機能を追加するなどです。

## **ジョブ差別化要因:** ジョブの遂行方法に影響を与える要因は何か？

ジョブ差別化要因は、ジョブの実行方法に影響する状況で、通常、時間、方法、または場所を含みます。

主要な特徴:

- 「if」で始まる
- 「versus/vs」を使用してオプションの範囲を示す
- 複合的な記述を避ける（AND/OR なし）
- 解決策に依存しない

| 良い例 |
|--------------|
| ジョブパフォーマーが独身か既婚か |
| ジョブパフォーマーに幼い子供がいるか、子供が家を出ているか |
| 潜在的な新しい家が地元か遠方か |

これらの差別化要因は、メインジョブのスコープを絞り込むのに役立ちます（例: 「朝にエネルギーを得る」と「午後の職場でエネルギーを得る」）。
