## オーナーシップ

以下の RACI テーブルは、私たちのプロダクトの道案内体験を構築する際の [Growth: Engagement](/handbook/product/categories/#engagement-group) と [Foundations: Design System](/handbook/product/categories/#design-system-group) チームの連携を示しています。Engagement チームがナビゲーションの意思決定とユーザーフロー最適化の最終オーナーとして機能する一方、Design System チームはこれらの体験を可能にする構造的基盤の構築 — コンポーネントフレームワーク、レイアウトテンプレート、レスポンシブデザインパターン、一貫性基準を含む — に主たる責任を持ちます。Design System チームは足場と再利用可能なシステムを構築し、Engagement チームはユーザーがこれらの構造をどのようにナビゲートし、相互作用してビジネス目的を達成するかを決定します。

### 凡例

- **R** = Responsible（作業を実行する）
- **A** = Accountable（最終的な意思決定者）
- **C** = Consulted（インプットを提供する）
- **I** = Informed（情報を共有される）

### RACI テーブル

| タスク／役割 | Engagement Team | Design System Team | メモ＆例 |
|-----------|-------------|-------------------|------------------|
| **構造的フレームワークとテンプレートの定義** | C | R/A | コンテナ構造、グリッドシステム、ページテンプレート。*例: ナビゲーションクロームのスロット作成* |
| **コンポーネントの関係性と階層の確立** | C | R/A | コンポーネントのネストの仕方、視覚的階層ガイドライン。*例: パネル構造、境界、パディング* |
| **レイアウトパターンとレスポンシブデザインの作成** | C | R/A | グリッドシステム、ブレークポイント、レイアウトテンプレート。*例: 定義済み領域を持つメインページテンプレート* |
| **ナビゲーションアーキテクチャとユーザーフローの定義** | R/A | C | メニュー構造、情報アーキテクチャ、ユーザー経路。*例: 機能間のユーザー移動方法* |
| **コンテンツ配置と目立たせ方の決定** | R/A | I | 定義済みテンプレート内で何をどこに配置するか、CTA、機能発見性。*例: ナビゲーションスロットを埋めるコンテンツ* |
| **コンバージョンパスとユーザー体験の最適化** | R/A | I | ファネル最適化、機能ポジショニング。*例: AI パネルの配置、トライアルウィジェットの配置* |
| **既存のレイアウトフレームワークの拡張** | R/A | C | 新しい領域を追加したりテンプレートを変更したりすること。*例: ボトムツールバーの追加、パネル数を 3 から 4 に変更* |
| **デザインシステムの一貫性の維持** | C | R/A | チーム間での再利用性とポータビリティを確保すること。*例: 他のチームが使えるパネルコンポーネント* |
| **デザイン原則とガイドラインの確立** | C | R/A | 高レベルのデザイン基準とベストプラクティス。*例: 視覚的階層ルール、カラーモード* |
| **アプリケーション固有のナビゲーション決定** | R/A | C | フレームワーク内でのプロダクト固有のナビゲーション選択。*例: ロゴ配置* |
| **コンポーネント機能の実装** | R/C | R/C | レイアウトコンポーネントの技術的実装。*例: リサイズ可能なパネル、レスポンシブ動作* |
| **クロスファンクショナルな変更でのコラボレーション** | R/C | R/C | 構造とコンテンツの両方の考慮を必要とする変更。*例: 右から左への言語サポート* |

### コラボレーションの原則

1. **コンテンツのないテンプレート**: Design System が構造テンプレートと領域を定義し、Growth がそれらの領域内のコンテンツを定義する
1. **要件よりガイドライン**: Design System は厳格なルールよりも柔軟な原則を提供する
1. **拡張可能なフレームワーク**: Growth はシステムを拡張し、パターンを Design System に貢献し戻すことができる
1. **コンサルテーション優先**: 主たるオーナーシップ領域であっても、大きな変更前にチームは互いに相談する
1. **同意できなくてもコミット**: 整合が得られない場合、アカウンタブルな当事者が最終決定を行う

### エスカレーションプロセス

- **レイアウトとナビゲーションの衝突**: 両チームのリードにエスカレートする
- **機能とシステムの優先順位付け**: プロダクトリーダーシップにエスカレートする
- **新たに生じるオーナーシップの問題**: ブロッカーになる前に定期同期ミーティングで対処する

## ナビゲーション変更のリクエスト

> 注: ナビゲーションへの未承認の変更を防ぐために [Code Owners 承認ルール](https://docs.gitlab.com/user/project/codeowners/) が設定されています。このプロセスに従っていない場合、マージリクエストはブロックされます。

### ナビゲーションとは何ですか？ {#what-is-navigation}

ナビゲーションとは、ユーザーが GitLab 内を移動するのを助ける要素であり、整理および道案内の手がかりを含みます。ナビゲーション体験は、私たちの機能のユーザビリティと発見可能性に直接影響します。このドキュメントは、[目標を満たし続けながら](https://about.gitlab.com/direction/foundations/personal_productivity/#navigation)、私たちが集団的にナビゲーションを進化させる方法を説明します。

### なぜナビゲーションを変更する際に慎重である必要があるのですか？

過去、チームは新機能をハイライトするために項目を追加してきました。しかし、これにより、ユーザーが必要なものを見つけるのが難しくなる、圧倒的なナビゲーション構造が作られました。私たちの[四半期 GitLab SUS 調査](https://gitlab.com/groups/gitlab-org/-/epics/1455)は、これらの繰り返し起こるテーマを引き続きハイライトしています。

- ナビゲーションが複雑あるいは混乱を招く
- ナビゲーションが学習しにくい
- ナビゲーションが直感的でない

### どのようなナビゲーション変更に承認が必要ですか？

GitLab の UX はユーザーがプロダクト全体をさまざまな方法でナビゲートできるようにしていますが、承認プロセスの焦点は左サイドナビゲーションにあります。この領域は、ナビゲーションとしてだけでなく、ユーザーにとっての機能発見ポイントとしても機能します。ナビゲーションが進化するにつれて、コアワークフローへのフォーカスと機能の可視性のバランスを維持することが重要です。

このバランスを維持するために、左サイドナビゲーションへの変更を提案する際は、全員にこのプロセスを使用していただくようお願いします。

- 第 1、第 2、第 3 レベルのナビゲーション追加
- ナビゲーション項目の名前変更
- ナビゲーション項目の削除
- ナビゲーション項目のソート順変更
- ナビゲーション機能や特徴の変更
- [Experiment](https://docs.gitlab.com/policy/development_stages_support/#experiment) または [Beta](https://docs.gitlab.com/policy/development_stages_support/) 機能の起動
- ナビゲーション項目の閲覧範囲変更（例: デフォルト無効からデフォルト有効に移動する）

### いつナビゲーションを変更するのか

私たちは、ユーザーワークフローの最適化を目的とした意図的なプロセスを通じてのみ、GitLab ナビゲーション構造に新規追加を行います。この[ビデオ](https://www.nngroup.com/videos/number-items-navigation-menu/)は、ナビゲーションをイテレーションする上で重要な主要因子を要約しています。過去、チームは新機能をハイライトするために項目を追加してきました。しかし、すべての新機能を収容することは不可能になります。これにより、ユーザーが自分のタスクを完了するために必要なものを見つけるには難しすぎる圧倒的なナビゲーション構造が作られるためです。

そのため、私たちは次の目的では **新しい項目を追加しません**。

- 新機能の発見可能性を向上させるため。代わりに、プロダクト全体で[機能をハイライトする](https://design.gitlab.com/usability/feature-discovery/)他の機会を探してください。
- 潜在的な将来のために最適化する。私たちは過剰最適化なしで前向きであるべきです。機能が開発され追加されるにつれ、成長する機能をサポートするために起こる必要のある変更を検討できます。

### ナビゲーション変更をどのように評価しますか？

ナビゲーション変更について答える必要のある主な質問は 2 つあります。

1. このナビゲーション提案は、私たちの[プライマリ JTBD](https://gitlab.com/gitlab-com/www-gitlab-com/-/blob/master/data/jtbd.yml?ref_type=heads) の 1 つを促進しますか？どのジョブですか？
1. この変更は、そのジョブを完了しようとするユーザーのワークフローをどのように改善しますか？

ナビゲーションの機会領域について学ぶために実施できる多くの異なるタイプの [Problem Validation リサーチ](/handbook/product/ux/experience-research/problem-validation-and-methods/) があります。実施される具体的なリサーチタイプは、研究のリサーチクエスチョンと目標に基づくべきです。次のリサーチ手法とフレームワークは、既存の GitLab ナビゲーションへの新規追加を正当化するためにリサーチを使用できる例です。

1. [コンテキスチュアル・インクワイアリ](https://www.nngroup.com/articles/contextual-inquiry/): この方法は、ユーザーが役割に関連するタスクを実行する際に観察・インタビューし、その「なぜ」と「どのように」を理解することを含みます。
   - **例**: ユーザーが GitLab で複数のメニューやページをナビゲートしてダッシュボードを見つけている様子を示すかもしれません。このインサイトは、すべてのダッシュボードを見つけるための単一のページが GitLab に必要であることを明らかにします。
1. [Jobs to be Done (JTBD)](/handbook/product/ux/jobs-to-be-done/): このフレームワークは、お客様が役割の中で達成したいジョブを学ぶことを意図した[基礎リサーチ](/handbook/product/ux/experience-research/foundational-research/)の一種です。ジョブは、お客様の作業の状況、目標、結果を定義できるようにします。
   - **例**: ユーザーをインタビューすることで、アプリケーションが故障したときに特定できるよう、アプリケーションの健全性を示すためにメトリクスを追跡するというメインジョブがあることを学びます。このインサイトは、アプリケーションメトリクスを表示するページを追加することで GitLab で埋められるギャップを示唆します。
1. [Diary study](/handbook/product/ux/experience-research/diary-studies/): この方法は、一定期間にわたってユーザーからフィードバックを得るために使用され、日、週、または月にわたって発生する変化を明らかにできます。
   - **例**: GitLab のナビゲーションバーへの主要な変更にユーザーがどのように関与するかについてのフィードバックを収集する際、GitLab 管理者が設定を調整するための管理エリアの場所を見つけることに毎月から次の月まで継続的に苦労していることを観察します。このインサイトは、時間の経過とともに見つけにくいので、ユーザーインターフェースで管理エリアを表面化させる必要があることを示唆します。

ナビゲーションへの変更が必要かどうかを学ぶために尋ねる質問:

1. 現在の役割に関連する主なタスクは何ですか？
1. それらのタスクはなぜ重要なのですか？それらのタスクを完了できなかった場合、どうなりますか？
1. 既存のツールを使用して特定のタスクを実行する方法を見せてください。
1. もしあれば、そのタスクを完了するプロセスについて何を改善しますか？

ナビゲーションの問題に関するインサイトが得られたら、Product チームの DRI は Product Design および UX Research と協力して、その後の[ソリューション検証](/handbook/product/ux/experience-research/evaluating-navigation/#our-approach-to-evaluating-navigation-changes)研究を通じて理想的な解決策を評価する必要があります。

### ナビゲーション変更を提案する方法

> 主な目的が機能の発見可能性を向上させることであれば、まずプロダクト全体で[機能をハイライトする](https://design.gitlab.com/usability/feature-discovery/)他の機会を探してください。

1. Issue を開く前に、[Pajamas のナビゲーション要素とパターン](https://design.gitlab.com/usability/navigation-sidebar/)を確認してください。
1. [ナビゲーション変更のリスト](/handbook/product/ux/navigation/#what-is-navigation)と、それらが何であるかを確認して、変更が該当することを確認してください。
1. [Growth: Engagement](/handbook/product/categories/#engagement-group) のプロダクトマネージャーが、ナビゲーション変更の [DRI](/handbook/people-group/directly-responsible-individuals/#what-is-a-directly-responsible-individual) です。提案が[フル検証](#full-validation-path)を必要とするか[限定検証](#limited-validation-path)を必要とするかを判断するために、彼らに連絡してください。
1. [Navigation Proposal](https://gitlab.com/gitlab-org/gitlab/-/issues/new?issuable_template=Navigation%20Proposals) Issue テンプレートを使用してこのプロセスのレビューを開始できます。
1. [Growth: Engagement](/handbook/product/categories/#engagement-group) のデザイナーが提案をレビューしインプットを提供することで DRI を支援します。[Growth: Engagement](/handbook/product/categories/#engagement-group) チームからの典型的なターンアラウンド時間は 1 マイルストーンです。フィードバックを提供した後、提案を前進させ、必要に応じて追加のフィードバックを求めることは DRI の責任です。
1. 承認を得て実装を開始する準備ができたら、[ナビゲーションへの項目追加に関する GitLab Docs](https://docs.gitlab.com/development/navigation_sidebar/#adding-page-specific-vue-content) に従ってください。

#### フル検証パス {#full-validation-path}

このパスは、大多数の GitLab ユーザーに影響するナビゲーション変更や、新しいデザインパターンを導入する変更に適しています。フル検証を必要とする変更の例には以下が含まれます。

- Beta または Generally available 機能の起動
- ナビゲーション構造または機能の変更
- 既存のナビゲーション項目の削除
- 既存のナビゲーション項目の名前変更

[Growth: Engagement](/handbook/product/categories/#engagement-group) PM は、提案がフル検証パスに従うべきかを判断する DRI です。このパスでは、ナビゲーション提案 Issue の一部として、以下のステップが完了し文書化されることが必要です。

| ステップ | 要件 |
| --- | --- |
| ビジネスケース | すべての提案には、根本的な問題とナビゲーション変更の目標を特定するビジネスケースを含める必要があります。 |
| 問題検証 | ナビゲーションの機会領域を発見・検証するための問題検証リサーチが必要です。 |
| UX レビュー | 関連するステージグループのデザイン DRI と UX レビューが実施されています。 |
| カウンターパートのサポート | 提案は、関連するステージグループのプロダクト、デザイン、リサーチのカウンターパートによってサポートされています。 |
| ソリューション検証 | 他の潜在的な解決策と比較してナビゲーション変更を評価するためのソリューション検証リサーチが実施されます。 |

#### 限定検証パス {#limited-validation-path}

このパスは、少数のユーザーに影響するナビゲーション変更や、既存のデザインパターンに従う変更に適しています。限定検証を必要とする変更の例には以下が含まれます。

- フィーチャーフラグの背後で利用可能な実験的機能
- 既存の統合のパターンに従う新しいサードパーティ統合
- すでに不一致がある場所に一貫性をもたらす変更

[Growth: Engagement](/handbook/product/categories/#engagement-group) PM は、提案が限定検証パスに従えるかを判断する DRI です。このパスでは、ナビゲーション提案 Issue の一部として、以下のステップが完了し文書化されることが必要です。

| ステップ | 要件 |
| --- | --- |
| ビジネスケース | すべての提案には、根本的な問題とナビゲーション変更の目標を特定するビジネスケースを含める必要があります。 |
| 問題検証 | ナビゲーションの機会領域を特定するためにユーザーリサーチは不要です。 |
| UX レビュー | 関連するステージグループのデザイン DRI と UX レビューが実施されています。 |
| カウンターパートのサポート | 提案は、関連するステージグループのプロダクトとデザインのカウンターパートによってサポートされています。 |
| ソリューション検証 | 提案では、ユーザージャーニーの他のポイントで変更を表面化させる代替ソリューションを記述または視覚化する必要があります。次に、提案された解決策と代替解決策がビジネスケースで示された問題と目標にどれだけ対処しているかを批判的に評価します。 |

### 調整プロセス

ナビゲーション提案プロセスは、コアワークフローへのフォーカスと機能の可視性のバランスを取ろうとします。これは、提案作成者が時々 [Growth: Engagement](/handbook/product/categories/#engagement-group) チームの決定に同意しないことを意味します。バランスが間違っていると感じる場合、提案にコメントを追加し、以下を行ってください。

1. 提案が却下された理由と、なぜこの決定が誤っていると考えるかについてのあなたの視点を要約する
1. マネージャーと [Growth: Engagement](/handbook/product/categories/#engagement-group) PM マネージャーをメンションする
1. 決定に関する彼らのインプットをリクエストする
