## オーナーシップ

以下の RACI テーブルは、プロダクトの道案内体験を構築する際の [Growth: Engagement](/handbook/product/categories/#engagement-group) と [Foundations: Design System](/handbook/product/categories/#design-system-group) チームの連携を示しています。Engagement チームがナビゲーションに関する意思決定とユーザーフロー最適化の最終オーナーとして機能する一方、Design System チームはこれらの体験を可能にする構造的基盤の構築（コンポーネントフレームワーク、レイアウトテンプレート、レスポンシブデザインパターン、一貫性基準を含む）に主たる責任を持ちます。Design System チームは足場と再利用可能なシステムを構築し、Engagement チームはユーザーがこれらの構造をどのようにナビゲートし、相互作用してビジネス目的を達成するかを決定します。

### 凡例

- **R** = Responsible（実作業を行う）
- **A** = Accountable（最終意思決定者）
- **C** = Consulted（インプットを提供する）
- **I** = Informed（情報を共有される）

### RACI テーブル

| タスク／ロール | Engagement Team | Design System Team | メモ・例 |
|-----------|-------------|-------------------|------------------|
| **構造フレームワークおよびテンプレートの定義** | C | R/A | コンテナ構造、グリッドシステム、ページテンプレート。*例: ナビゲーションクロームのスロット作成* |
| **コンポーネントの関係性と階層の確立** | C | R/A | コンポーネントの入れ子構造、視覚階層のガイドライン。*例: パネル構造、ボーダー、パディング* |
| **レイアウトパターンとレスポンシブデザインの作成** | C | R/A | グリッドシステム、ブレークポイント、レイアウトテンプレート。*例: 定義された領域を持つメインページテンプレート* |
| **ナビゲーションアーキテクチャとユーザーフローの定義** | R/A | C | メニュー構造、情報アーキテクチャ、ユーザー経路。*例: ユーザーが機能間をどう移動するか* |
| **コンテンツ配置と目立たせ方の決定** | R/A | I | 何をどこに置くか、定義されたテンプレート内の CTA、機能の発見しやすさ。*例: ナビゲーションスロットに何のコンテンツを入れるか* |
| **コンバージョン経路とユーザー体験の最適化** | R/A | I | ファネル最適化、機能配置。*例: AI パネルの配置、トライアルウィジェットの配置* |
| **既存のレイアウトフレームワークの拡張** | R/A | C | 新しい領域の追加やテンプレートの変更。*例: ボトムツールバーの追加、パネル数を 3 から 4 に変更* |
| **デザインシステムの一貫性維持** | C | R/A | チーム横断での再利用性とポータビリティの確保。*例: 他チームから利用可能なパネルコンポーネント* |
| **デザイン原則およびガイドラインの確立** | C | R/A | 高レベルのデザイン基準とベストプラクティス。*例: 視覚階層のルール、カラーモード* |
| **アプリケーション固有のナビゲーション決定** | R/A | C | フレームワーク内でのプロダクト固有のナビゲーション選択。*例: ロゴの配置* |
| **コンポーネント機能の実装** | R/C | R/C | レイアウトコンポーネントの技術実装。*例: リサイズ可能なパネル、レスポンシブな振る舞い* |
| **クロスファンクショナルな変更での協業** | R/C | R/C | 構造とコンテンツの両方の検討を必要とする変更。*例: 右から左へ書く言語のサポート* |

### コラボレーション原則

1. **コンテンツのないテンプレート**: Design System が構造的テンプレートと領域を定義し、Growth がその領域内のコンテンツを定義する
1. **要件よりガイドライン**: Design System は厳格なルールではなく柔軟な原則を提供する
1. **拡張可能なフレームワーク**: Growth はシステムを拡張し、パターンを Design System に貢献できる
1. **コンサルテーション ファースト**: 主たるオーナーシップ領域であっても、チームは大きな変更の前に互いに相談する
1. **不同意かつコミット (Disagree and Commit)**: 整合が取れないときは Accountable な側が最終意思決定を行う

### エスカレーションプロセス

- **レイアウト対ナビゲーションの衝突**: 両チームのリードにエスカレーション
- **機能対システムの優先度付け**: プロダクトリーダーシップにエスカレーション
- **オーナーシップが新たに問題化したケース**: ブロッカーになる前に定期同期ミーティングで対応

## ナビゲーションの変更を依頼する

> 注: ナビゲーションへの未承認の変更を防ぐため、[Code Owners 承認ルール](https://docs.gitlab.com/user/project/codeowners/)が設定されています。このプロセスに従っていない場合、マージリクエストはブロックされます。

### ナビゲーションとは {#what-is-navigation}

ナビゲーションとは、ユーザーが GitLab 内を移動するのを助ける要素であり、整理と道案内の手がかりを含みます。ナビゲーション体験は、機能のユーザビリティと発見性に直接影響します。このドキュメントでは、私たちが[ゴールを満たし](https://about.gitlab.com/direction/foundations/personal_productivity/#navigation)つつ、ナビゲーションを協働的に進化させていく方法を説明します。

### なぜナビゲーション変更には注意が必要なのか

過去には、各チームが新機能を目立たせるために項目を追加してきました。しかし、その結果、過剰なナビゲーション構造が生まれ、ユーザーが必要なものを見つけにくくなりました。私たちの[四半期 GitLab SUS サーベイ](https://gitlab.com/groups/gitlab-org/-/epics/1455)では、繰り返し次のテーマが浮き彫りになっています:

- ナビゲーションが複雑または分かりにくい
- ナビゲーションを学習するのが難しい
- ナビゲーションが直感的でない

### 承認が必要なナビゲーション変更とは

GitLab の UX ではユーザーがプロダクト全体をさまざまな方法でナビゲートできますが、承認プロセスの焦点は左サイドナビゲーションにあります。この領域はユーザーにとってナビゲーションであると同時に機能発見の起点でもあります。ナビゲーションが進化するにつれ、コアワークフローへのフォーカスと機能の可視性のバランスを維持することが極めて重要です。

このバランスを維持するため、左サイドナビゲーションへの変更を提案する場合は、全員にこのプロセスを使うようお願いしています:

- 第 1、第 2、第 3 階層のナビゲーション項目の追加
- ナビゲーション項目のリネーム
- ナビゲーション項目の削除
- ナビゲーション項目のソート順変更
- ナビゲーションの機能や挙動の変更
- [Experiment](https://docs.gitlab.com/policy/development_stages_support/#experiment) または [Beta](https://docs.gitlab.com/policy/development_stages_support/) 機能の起動
- ナビゲーション項目の表示範囲の変更（例: デフォルト無効からデフォルト有効への変更）

### ナビゲーションをいつ変更するのか

GitLab のナビゲーション構造への新規追加は、ユーザーワークフローの最適化を意図した慎重なプロセスを通じてのみ行います。[こちらの動画](https://www.nngroup.com/videos/number-items-navigation-menu/)では、ナビゲーションをイテレートする際に考慮すべき主要な要素がまとめられています。過去には、各チームが新機能を目立たせるために項目を追加してきました。しかし、新機能が登場するたびに項目を追加するのは不可能で、過剰なナビゲーション構造を生み出し、ユーザーがタスクを完了するために必要なものを見つけるのが難しくなりすぎます。

そのため、私たちは次の目的で**新しい項目を追加しません**:

- 新機能の発見性を向上させるため。代わりに、プロダクト全体で[機能を目立たせる](https://design.gitlab.com/usability/feature-discovery/)別の機会を探してください。
- 将来の可能性に向けた最適化のため。過剰最適化せず先を見据えるべきです。機能が開発・追加されたら、機能の成長に応じて必要な変更を検討できます。

### ナビゲーション変更をどう評価すればよいか

ナビゲーション変更について答えるべき主な質問は 2 つあります:

1. このナビゲーション提案は、私たちの[主要な JTBD](https://gitlab.com/gitlab-com/www-gitlab-com/-/blob/master/data/jtbd.yml?ref_type=heads) のいずれかを促進しますか？どのジョブですか？
1. この変更は、そのジョブを完了しようとするユーザーのワークフローをどう改善しますか？

ナビゲーションの機会領域について学ぶために実施できる[Problem Validation リサーチ](/handbook/product/ux/experience-research/problem-validation-and-methods/)にはさまざまなタイプがあります。実施するリサーチの具体的タイプは、研究の質問とゴールに基づいて決めるべきです。以下のリサーチ手法・フレームワークは、既存の GitLab ナビゲーションへの新規追加を正当化するためにリサーチを使う方法の例です:

1. [コンテキスチュアル・インクワイアリー](https://www.nngroup.com/articles/contextual-inquiry/): この手法では、役割に関連するタスクを実行するユーザーを観察・インタビューし、行動の「なぜ」と「どうやって」を理解します。
   - **例**: ユーザーが GitLab で複数のメニューやページをナビゲートしてダッシュボードを見つける様子を見せてくれるかもしれません。このインサイトから、GitLab 内ですべてのダッシュボードを見つけられる単一ページが必要だと明らかになります。
1. [Jobs to be Done (JTBD)](/handbook/product/ux/jobs-to-be-done/): このフレームワークは[基礎リサーチ](/handbook/product/ux/experience-research/foundational-research/)の一種で、顧客が自分の役割の中で達成したいジョブを学ぶためのものです。ジョブによって、顧客の仕事の状況、ゴール、アウトカムを定義できます。
   - **例**: ユーザーへのインタビューにより、彼らの主要なジョブはアプリケーションのヘルスを示すメトリクスを追跡し、アプリケーションが失敗しているタイミングを特定することだと分かります。このインサイトから、アプリケーションメトリクスを表示するページを追加することで GitLab で埋めるべきギャップがあると示唆されます。
1. [ダイアリースタディ](/handbook/product/ux/experience-research/diary-studies/): この手法は、ユーザーからある期間にわたるフィードバックを得るために使われ、数日、数週間、数か月にわたる変化を明らかにできます。
   - **例**: GitLab のナビゲーションバーへの大きな変更にユーザーがどう関わるかについてフィードバックを集める際、GitLab 管理者が月をまたいで継続的に管理エリアを見つけて設定を調整するのに苦労している様子が観察されます。このインサイトから、長期的に見つけにくい管理エリアを UI 上で目立たせる必要があると示唆されます。

ナビゲーションへの変更が必要かどうかを学ぶために尋ねるべき質問:

1. 現在の役割に関連する主なタスクは何ですか？
1. なぜそれらのタスクが重要ですか？それらのタスクを完了できなかった場合、何が起こりますか？
1. 既存のツールを使ってその特定のタスクをどう実行するか見せてください。
1. そのタスクを完了するプロセスについて、もしあれば、何を改善したいですか？

ナビゲーションに関する問題の見通しが立ったら、Product チームの DRI は Product Design と UX Research と協力し、その後の[ソリューション・バリデーション](/handbook/product/ux/experience-research/evaluating-navigation/#our-approach-to-evaluating-navigation-changes)スタディを通じて理想的なソリューションを評価します。

### ナビゲーション変更を提案する方法

> もしあなたの主な目的が機能の発見性を改善することなら、まずプロダクト全体で[機能を目立たせる](https://design.gitlab.com/usability/feature-discovery/)別の機会を探してください。

1. Issue を開く前に、[Pajamas のナビゲーションの要素とパターン](https://design.gitlab.com/usability/navigation-sidebar/)を確認します。
1. [ナビゲーション変更のリスト](/handbook/product/ux/navigation/#what-is-navigation)を確認し、あなたの変更が該当するか確認します。
1. [Growth: Engagement](/handbook/product/categories/#engagement-group) の Product Manager がナビゲーション変更の [DRI](/handbook/people-group/directly-responsible-individuals/#what-is-a-directly-responsible-individual) です。提案が[フルバリデーション](#full-validation-path)と[リミテッドバリデーション](#limited-validation-path)のどちらを必要とするかを判断するため、彼らに連絡してください。
1. [Navigation Proposal](https://gitlab.com/gitlab-org/gitlab/-/issues/new?issuable_template=Navigation%20Proposals) の Issue テンプレートを使ってこのプロセスのレビューを開始できます。
1. [Growth: Engagement](/handbook/product/categories/#engagement-group) のデザイナーが、提案をレビューしインプットを提供することで DRI をアシストします。[Growth: Engagement](/handbook/product/categories/#engagement-group) チームからの通常の対応時間はマイルストーン 1 つです。フィードバックを提供した後は、提案を前に進め、必要に応じて追加のフィードバックを求めるのは DRI の責任です。
1. 承認を得て実装を開始する準備ができたら、[ナビゲーションへの項目追加に関する GitLab Docs](https://docs.gitlab.com/development/navigation_sidebar/#adding-page-specific-vue-content) に従ってください。

#### フルバリデーションパス {#full-validation-path}

このパスは、GitLab ユーザーの大多数に影響を与える、または新しいデザインパターンを導入するナビゲーション変更に適しています。フルバリデーションが必要となる変更の例:

- Beta または Generally available 機能の起動
- ナビゲーション構造や挙動の変更
- 既存のナビゲーション項目の削除
- 既存のナビゲーション項目のリネーム

[Growth: Engagement](/handbook/product/categories/#engagement-group) の PM が、提案がフルバリデーションパスに従うべきかを判断する DRI です。このパスでは、次のステップを完了し、ナビゲーション提案 Issue の一部として記録することを求めます。

| ステップ | 要件 |
| --- | --- |
| ビジネスケース | すべての提案は、ナビゲーション変更の根底にある問題とゴールを示すビジネスケースを含めなければなりません。 |
| 問題のバリデーション | ナビゲーションの機会領域を発見・検証するために、問題バリデーションのリサーチが必要です。 |
| UX レビュー | 関連するステージグループのデザイン DRI と UX レビューを実施済み。 |
| カウンターパートのサポート | 関連するステージグループのプロダクト、デザイン、リサーチのカウンターパートから提案が支持されている。 |
| ソリューションのバリデーション | 他の潜在ソリューションと比較してナビゲーション変更を評価するため、ソリューションバリデーションのリサーチを実施。 |

#### リミテッドバリデーションパス {#limited-validation-path}

このパスは、少数のユーザーにのみ影響を与える、または既存のデザインパターンに沿ったナビゲーション変更に適しています。リミテッドバリデーションが必要となる変更の例:

- フィーチャーフラグの背後にある実験的機能
- 既存の連携のパターンに沿った新しいサードパーティ連携
- 既に不整合がある領域に一貫性をもたらす変更

[Growth: Engagement](/handbook/product/categories/#engagement-group) の PM が、提案がリミテッドバリデーションパスに従えるかを判断する DRI です。このパスでは、次のステップを完了し、ナビゲーション提案 Issue の一部として記録することを求めます。

| ステップ | 要件 |
| --- | --- |
| ビジネスケース | すべての提案は、ナビゲーション変更の根底にある問題とゴールを示すビジネスケースを含めなければなりません。 |
| 問題のバリデーション | ナビゲーションの機会領域を特定するためのユーザーリサーチは必須ではありません。 |
| UX レビュー | 関連するステージグループのデザイン DRI と UX レビューを実施済み。 |
| カウンターパートのサポート | 関連するステージグループのプロダクトとデザインのカウンターパートから提案が支持されている。 |
| ソリューションのバリデーション | 提案では、ユーザー体験の他のポイントで変更を浮かび上がらせる代替案を記述または可視化する必要があります。そのうえで、提案案と代替案がビジネスケースで概説された問題とゴールにどの程度応えるかを批判的に評価します。 |

### 調整プロセス

ナビゲーション提案プロセスは、コアワークフローへのフォーカスと機能の可視性のバランスを取ろうとするものです。つまり、提案者が [Growth: Engagement](/handbook/product/categories/#engagement-group) チームの判断に同意しないことが時折あります。バランスを間違っていると感じる場合は、提案にコメントを追加し、次のことを行ってください:

1. 提案が却下された理由と、その判断が誤っていると考える理由をまとめる
1. 自分のマネージャーと [Growth: Engagement](/handbook/product/categories/#engagement-group) PM のマネージャーをメンションする
1. 判断についてのインプットを依頼する
