## 概要

Fleet メトリクスの一環として、ユーザーがパイプラインを監視し、パイプライン速度・一般的なジョブ失敗などについての知見を要約するための CI insights 専用セクションを設けることを検討しています。最終的には、ユーザーが CI/CD の Issue を最適化・修正できるよう、実行可能なアクションを提供することを目指します。

## 動機

CI/CD アナリティクスのページ（[CI/CD Analytics ページ](https://gitlab.com/gitlab-org/gitlab/-/pipelines/charts?chart=pipelines)）には、パイプラインに関する基本的な分析情報が含まれています。この情報のほとんどは、時系列での**合計**パイプライン数に関するもので、顧客にとって実質的な価値を提供していません。プロジェクトは常にパイプライン数が増加していくため、パイプラインの合計数はほとんど意味を持ちません。

![現在のページ](/images/engineering/architecture/design-documents/ci_builds_runner_fleet_metrics/current_page.png)

このページには実際の insights が不足しているため、パイプラインの遅延や失敗を把握・追跡することが困難であり、非常に手作業に頼った作業になっています。私たちは、データを得るために API をクエリして手動でデータを解析したり、UI を何十ものページにわたってナビゲートして必要な insights やアクションを見つけたりするといった手作業を排除し、一元化された場所でユーザーがワークフローを最適化できるようにしたいと考えています。

プロジェクトのパイプラインに関連する大量のデータを処理することになるため、将来的には AI ツールを使って知見を要約し、ジョブの失敗・パイプラインの遅延・フレーキースペックについての洞察を提供できる可能性があります。AI は私たちのプロダクトロードマップにおいて重要な要素となっており、Verify セクションにはこの分野で有望なリードがないため、このページが新しい機能追加の中心になれる可能性があります。

- 新しいパイプライン分析ダッシュボードページを提供します
- 優れたデータ可視化で情報を素早く理解できるようにします
- 柔軟なクエリ機能でユーザーが必要な情報を取得できるようにします

- ページに表示される情報に基づいた明確なアクションを示します
- ランディング時にパイプラインの所要時間推移や最も遅いジョブなどのデフォルト情報を表示します
- CI/CD アナリティクスをより身近で、好まれ、記憶されるものにします（つまり、より多くのページビュー）

### 非目標

GitLab プロジェクト自体のパイプライン速度を向上させることは目指していません。この機能が結果的にその実現に役立つ可能性はありますが、直接の目標ではありません。

また、最初のイテレーションに AI を含めることも目指しておらず、代わりにできるだけ多くの情報を利用可能かつ消化しやすい形にすることに注力します。

## 提案

[CI/CD アナリティクスページ](https://gitlab.com/gitlab-org/gitlab/-/pipelines/charts?chart=pipelines)をリニューアルして、ユーザーがパイプラインをスムーズにトラブルシューティングできるよう、より有意義なデータを含めます。主な改善点は以下のとおりです:

### 全体統計

現在の「全体統計」は、情報を利用可能な状態に保ちつつもビジュアルスペースをあまり取らないよう、小さいフォントの 1 行ヘッダーになります。パイプラインのグラフについては、各スタックがステータスを表し、各バーが単位（日の場合は 1 日、月の場合は 1 か月、年の場合は 1 年）を表す積み上げ棒グラフに変更します。これにより、特定の時間単位に実行されたパイプライン数と、それらのパイプラインのうち何パーセントが失敗または成功で終わったかを把握できます。

### パイプライン所要時間グラフ

現在の `過去 30 コミットのパイプライン所要時間` グラフを置き換える新しいパイプライン所要時間グラフで、種類（MR パイプライン、特定ブランチのパイプラインなど）・実行回数・ステータス（成功、失敗など）でカスタマイズできます。既存のグラフはフィルタリングなしにリポジトリの最新 30 コミットを確認するもので、提示される結果はあまり価値がありません。

また、過去 x 件のパイプライン（master の場合）において複数回失敗したジョブや最も遅いジョブも追加します。これはすべて、ユーザーがパイプラインデータをクエリして、改善すべき点や CI/CD 設定に問題があるかどうかを把握できるようにする取り組みをサポートするためです。

### 可視性

この機能の可視性を高めるために `pipelines` ページにリンクを追加します。`Run pipeline` プライマリボタンに新しいオプションを追加できます。

### マスターの状態

過去 x 件のメインブランチのパイプラインをスキャンして失敗したジョブを確認する「Is master broken?」クイックオプションを追加します。複数回失敗したすべてのジョブが、そのリストからインシデントを作成するオプションとともにテーブルに表示されます。

### カラースキーム

パイプラインステータスに関するデータ可視化での現在のカラースキームを見直します。現在はデフォルトの可視化色を使用していますが、ユーザーがパイプライン/ジョブのステータスに慣れ親しんでいる色とは実際には一致していません。より適切なカラースキームとより良い可視化によって、ユーザーがデータをより深く理解できるよう支援できる機会があります。

### ルーティング

`charts` はデータ可視化について話す際に非常に制限的な用語であるため、ルーティングを `pipelines/charts` から `pipelines/analytics` に変更します。また、このページが「nice なグラフを見るための場所」ではなく「情報を得るための場所」であることも表現できていません。そうすれば、タブのクエリパラメーターをなくして、代わりにファーストクラスのルーティングをサポートすることもできます。

## 設計と実装の詳細

### 集計データのための新しい API

この機能は、ジョブとパイプラインの insights を集計してクライアントが利用できるようにする新しいデータセットに依存しています。

まず MVC として、おそらく `gitlab.com` のみを対象に、ClickHouse からのデータ集計を開始します。バックエンドでオンザフライでデータを集計します。これまでのところ、ClickHouse はそのような処理を十分にこなせています。

集計されたデータはどこにも保存しません（ClickHouse にマテリアライズドビューを持つ可能性はありますが、それ以上複雑なものは使いません）。そして機能がトラクションを得れば、ClickHouse のない環境にこれらの機能を提供する方法を模索できます。

このようにして、迅速に進み、実際のユーザーでアイデアをテストし、フィードバックを得ることができます。

### フィーチャーフラグ

この新しいアナリティクスページを開発するにあたり、フィーチャーフラグ `ci_insights` の背後に新しいページをゲートして、古いまたは新しいアナリティクスページを条件付きでレンダリングします。フラグがオンのときに新しい `/analytic` ルートへ、オフのときに古い `/charts` ルートへとルートを選択するために、コントローラー上にフラグを追加することも検討できます。

### ページビューのアナリティクスを追加

このページがどの程度閲覧されているかについての情報を取得できるようにします。もし取得できない場合は、このページの視認性を把握するための実装を行いましょう。このセクションへの変更によりビュー数が増加するはずで、その数を成功の指標として追跡します。

### ルーティング

ページには新しいルートを設け、いくつかのリダイレクトを設定する計画です。ルーティング提案の詳細については、[関連する Issue](https://gitlab.com/gitlab-org/gitlab/-/issues/437556) を参照してください。

### パイプライン所要時間グラフ

さまざまな条件でパイプラインのデータをクエリできる方法が必要です。特に、`finished` スコープのパイプラインのみをクエリしたり、ステータス（`success` または `failed`）でフィルタリングしたりすることが重要です。ref によるスコープ指定も可能にしたいので、ユーザーはメインブランチをテストしたり、CI/CD の変更を導入したブランチをテストしたりすることができます。パイプライン速度のブランチ比較も必要です。

より正確なデータを得るために、要求するパイプラインの数を増やしたいと思います。GraphQL では 100 アイテムの制限があり、パフォーマンス低下がすぐに発生する可能性があります。より正確なデータ可視化のためにどのようにより多くのデータセットを取得できるかを定義する必要があります。

### ジョブ insights

現時点では、複数のパイプラインにわたって単一のジョブをクエリする方法がなく、次のようなクエリを実行できません:

```graphql
query getJob($projectPath: ID!, $jobName: String!){
  project(fullPath:$projectPath){
    job(name: $jobName, last: 100){
      nodes{
        id
        duration
      }
    }
  }
}
```

ジョブアナリティクスの情報をクエリする統一された方法が必要です。概ね、次のように表現できるでしょう:

```ruby
get_jobs(project_id:, job_name: nil, stage: nil, stage_index: nil, *etc)
# >
[{id: 1, duration: 134, status: 'failed'}, *etc]

get_jobs_statistics(project_id, job_name:, *etc)
# >
[{time_bucket: '2024-01-01:00:00:00', avg_duration: 234, count: 123, statuses_count: {success: 123, failed: 45, cancelled: 45}}]
```

### グラフのリニューアル

グラフの新しいカラースキームとよりスタイリッシュな外観を検討します。UX と協力して、これが私たちが念頭に置いていたものかどうかを確認し、記憶に残りやすいよりモダンなルックのグラフを実現するためのイニシアチブを支援します。

## 代替案

### 新しいページ

このセクションをそのままにして、まったく新しいページを作成することもできます。メリットとしては、`Build` の下のナビゲーションにより目立つ配置ができる可能性があります。デメリットとしては、このセクションと明確な重複が生じます。

### パイプライン単位のパイプライン分析

以前、パフォーマンス insights を**パイプラインごと**に追加する[実験](https://gitlab.com/gitlab-org/gitlab/-/issues/365902)がありました。この実験は削除され、実現困難と判断されました。主な知見は次のとおりです:

- ユーザーは想定ほどページを操作せず、insights を表示するボタンをクリックしませんでした
- ボタンをクリックしたユーザーは、ジョブの詳細な insights を得ようとはしませんでした
- ユーザーは Issue でフィードバックを残しませんでした

この実験から、パイプライングラフページ `pipelines/:id` を訪問するユーザーは、パイプラインのパフォーマンスを改善しようとしているのでは**ない**ことがほぼ明らかになりました。代わりに、このページはパイプラインの失敗をデバッグするために使用されている可能性が高く、つまりワークフロー改善を目指す DevOps エンジニアではなく、IC/開発者のペルソナのユーザーが利用していることを意味します。このセクションをより「広範な」エリアに配置することで、はるかに良いアダプションと有用なアクションが期待できます。

### 何もしない

このセクションをそのままにして、新しい形式のアナリティクスを追加しないこともできます。メリットとしては、リソースと時間の節約です。デメリットとしては、ドキュメントを読む以外に CI/CD 設定の速度改善を支援する方法が現在ない点です。このリニューアルされたセクションは、AI 機能の優れた入り口となり、ユーザーが設定を反復的に改善できるよう支援するものになるでしょう。
