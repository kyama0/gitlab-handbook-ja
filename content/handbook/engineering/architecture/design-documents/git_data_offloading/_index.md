## サマリー

Git データのストレージコストを管理することは、ビジネスの重要な部分です。Git データを低コストのストレージにオフロードすることで、ストレージコストを削減できます。

## 動機

GitLab では、データアクセスを高速に保つために、ほとんどの Git データを SSD に保存しています。これは頻繁にアクセスする必要があるデータには合理的です。しかし、ストレージコストはデータの増加とともにスケールするため、どの種類のデータを SSD に保存するか、どの種類のデータを低コストのストレージにオフロードできるかについてより賢くなれます。

例えば、大きなファイル（Git の用語では「blob」）は通常テキスト以外のファイル（画像、動画、バイナリなど）であるため、あまり変更されません。大きな blob を含むリポジトリでは、Git サーバーに大きなファイルをプッシュする必要がないように、[Git LFS](https://git-lfs.com/) がよく使われています。しかし、これはクライアント側のセットアップに依存します。

あるいは、プロジェクトが「古くなって」長期間アクセスされていない場合、そのプロジェクトの高速ストレージに対して費用を払い続ける必要はありません。

代わりに、その古くなったプロジェクトの **すべての** blob を低コストのストレージに保存することができます。こうすることで、アプリケーションはコミット履歴とツリーへのアクセスを維持するため、プロジェクトの閲覧エクスペリエンスは影響を受けませんが、すべてのファイルはほとんどアクセスされないため、低速ストレージに配置されます。

Git データを異なるカテゴリに分離する方法があれば、特定のデータを低コストの 2 次的な場所にオフロードできます。例えば、あまり頻繁にアクセスされない可能性のある大きなファイルを残りの Git データから分離して、SSD マウントではなく HDD に保存できます。

## 要件

特定のソリューションに対して与えられなければならない一連の要件と不変量があります。

### ストレージコストを削減する

最終的に、このソリューションはストレージコストを削減する必要があります。特定の Git データを低コストのストレージ用に分離することは、この削減に貢献できます。

ソリューションの追加コストを低コストのストレージへのデータオフロードから見込まれる節約と比較評価する必要があります。以下はいくつかの考慮すべき基準です。

- X より大きい大きな blob をすべて HDD に配置した場合、どれだけの費用が節約できるか？
- 古くなったすべてのプロジェクトが HDD に blob を持つ場合、どれだけの費用が節約できるか？
- オフロードメカニズムを実行するための追加の CPU/メモリコストという点での運用オーバーヘッドは？
- ネットワークオーバーヘッドは？例えば、大きな blob を取得するためにネットワーク経由で別のノードへの追加往復がある場合。
- アクセスコスト、例えば blob がオブジェクトストアに保存される場合。

### Gitaly のダウンストリームコンシューマーに対して透過的

この機能は純粋にストレージの最適化であり、潜在的なパフォーマンスの低下を除いて、Gitaly のダウンストリームコンシューマーに影響を与えるべきではありません。例えば、GitLab アプリケーションはこの機能をサポートするためにロジックを変更する必要はありません。

この機能は Gitaly のすべての呼び出し元に対して完全に見えないようにするべきです。Rails やコンシューマーはこれを知る必要も、いかなる方法でも管理する必要もありません。

### 運用上シンプル

Git データを扱う際、リポジトリの破損のリスクを最小化するために、できるだけシンプルに保ちたいです。シンプルに保ち、Git 自体の外部の動く部品を最小限にしてください。リポジトリデータを変更するロジックはすべて Git 自体にアップストリームすべきです。

## プロポーザル

[Git alternates メカニズム](https://git-scm.com/docs/gitrepository-layout#Documentation/gitrepository-layout.txt-objects)を通じて接続された各リポジトリに別個のオブジェクトデータベースを維持します。この 2 次的なオブジェクトデータベース（ODB）に対して特定の Git オブジェクトをフィルタリングして除外することを選択できます。

フィルターに基づいて Git データをこの 2 次的な ODB に配置します。[Git のフィルター](https://git-scm.com/docs/git-rev-list#Documentation/git-rev-list.txt---filterltfilter-specgt)に基づくオプションがあります。

何らかの制限に基づいて大きな blob を 2 次的な ODB に配置するか、すべての blob を 2 次的な ODB に配置するかを選択できます。

## 設計と実装の詳細

### Git

異なる種類の blob を異なるオブジェクトデータベースにセグメント化できるように `git-repack(1)` に機能を追加する必要があります。この取り組みは[この Issue](https://gitlab.com/gitlab-org/git/-/issues/159) で追跡しています。

### Gitaly

Gitaly のハウスキーピング中に、以下を行えます。

1. `git-repack(1)` を使用して、メインリポジトリのオブジェクトデータベースと 2 次的なオブジェクトデータベースの両方にパックファイルを書き込む。各リポジトリは何らかの基準に基づいて blob をオフロードするための独自の 2 次的なオブジェクトデータベースを持ちます。
1. `.git/objects/info/alternates` ファイルがステップ（1）の 2 次的なオブジェクトデータベースを指すようにする。

### 基準

オブジェクトが別のオブジェクトデータベースにオフロードされるかどうかは、以下の基準の 1 つまたは複数に基づいて決定できます。

#### Tier による

Free プロジェクトは多くの blob を低コストのストレージにオフロードする場合があり、Ultimate プロジェクトはすべてのオブジェクトが最速のストレージに配置される場合があります。

#### 履歴による

blob が長い前に追加されて、最近のコミットで参照されていない場合はオフロードされる場合があり、新しい blob はメイン ODB に残ります。

#### サイズによる

大きな blob は高価なストレージサイズを削減するための近道であり、低コストのストレージに移行するために優先される場合があります。

#### アクセス頻度による

頻繁に使用されるプロジェクトは高速ストレージに完全に残る場合があり、非アクティブなプロジェクトは blob がオフロードされる場合があります。

### オープンクエスチョン

#### オブジェクトをどのように削除するか？

到達不能なオブジェクトを削除したい場合、repack は両方の ODB を認識し、オブジェクトがメイン ODB にあるか 2 次的な ODB にあるかに関係なく、到達不能なオブジェクトを退去させることができる必要があります。メイン ODB にも[オブジェクトプール](https://gitlab.com/gitlab-org/gitaly/-/blob/master/doc/object_pools.md) ODB がある場合、この状況は複雑です。プールの ODB からオブジェクトを削除することは絶対にしたくないからです。

#### 潜在的なソリューション: alternates からオブジェクトを削除するよう Git を変更する

repack が alternate ODB の到達不能なオブジェクトを削除する機能を与えるために repack を変更する必要があります。alternate ディレクトリでどのように動作するかを指示する repack 設定 `repack.alternates.*` を追加できます。例えば、`repack.alternates.explodeUnreachable` を追加して、リンクされている alternate ODB でも `-A` のように動作することを示すことができます。

#### オブジェクトプールとどのように連携するか？

alternates を使用する場合、オブジェクトプールとどのように相互作用するか？オブジェクトプールも 2 次的なストレージにデータをオフロードするか？オブジェクトプールのメンバーは？最も複雑なケースでは、単一のリポジトリが 4 つの異なるオブジェクトデータベースを持つことを意味し、複雑さが増す可能性があります。

[--keep-pack](https://git-scm.com/docs/git-pack-objects#Documentation/git-pack-objects.txt---keep-packltpack-namegt) と [--honor-pack-keep](https://git-scm.com/docs/git-pack-objects#Documentation/git-pack-objects.txt---honor-pack-keep) オプションを使用して、一部のパックファイルを「keep」としてマークできる可能性があります。

#### 潜在的なソリューション: オブジェクトプールが blob をオフロードしないようにする

複雑さをあまり追加しないために、オブジェクトプールは blob をオフロードしないと決定できます。代わりに、オブジェクトプールで重複排除する前にリポジトリから blob をオフロードするようにハウスキーピングを設計できます。理論的には、オフロードされた blob はオブジェクトプールに入らないことを意味します。

#### Raft + WAL との連携は？

このメカニズムは Raft と先行書き込みログとどのように相互作用するか？

WAL はハードリンクとコピーフリーの移動を使用して、遅いコピー操作を避けます。しかし、これは異なるファイルシステムをまたいでは機能しません。ある時点では、repack などもログを通じて行われる可能性が高いです。ファイルシステム間でデータを転送すると、トランザクション処理に遅延が生じる可能性があります。

理想的には、alternate の使用をノード内部に留め、クラスターの残りにこの複雑さを漏らさないようにしたいです。配置決定を行う際に利用可能なスペースを考慮する必要があるため、これは課題です。2 つのストレージの低い方のキャパシティのみを表示することで内部に保つことは可能ですが、それは非効率なストレージ使用にもつながる可能性があります。

## 設計の問題点

### 追加された複雑さ

混合に別のオブジェクトプールを追加するという事実は、システムに複雑さを追加します。特に、レプリケートするデータのもう一つの場所を追加するため、リポジトリのレプリケーションに関して複雑さが増します。

### 時間とともにコストが変化する可能性

異なるストレージタイプのコストは時間とともに変化する可能性があります。これに備えるために、そのような変化に適応するのが容易であるべきです。

### 障害点の増加

大きな blob を別のストレージデバイスに持つことは、大きな blob をホストするデバイスが故障する可能性があるという 1 つの障害シナリオを追加します。

## 代替ソリューション

### プロジェクト全体を低コストのストレージに配置する

Git データを低コストのストレージに配置する代わりに、Rails アプリケーションがプロジェクト全体をマウントされた HDD ドライブに移動することを選択できます。

#### 可能な最適化

低コストのストレージを持つこれらのマシンに追加の RAM を提供することで、ページキャッシュの使用により低速な読み書き速度に対処するのに役立つかもしれません。ただし、これが全体的に低コストになるかどうかは確かではありません。
