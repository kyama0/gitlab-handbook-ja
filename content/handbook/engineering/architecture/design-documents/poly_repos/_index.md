このドキュメントは作成中であり、複数のリポジトリを含むワークフローをサポートするための計画を概説しています。これを略して「ポリリポジトリ（poly repos）」と呼びます。

## 概要

「ポリリポジトリ」アーキテクチャは GitLab に「チェンジセット」を導入し、複数のリポジトリのコードベースを持つ組織（Android Open Source Project など）が複数リポジトリにまたがる変更を単一のユニットとして管理できるようにします。これにより、チームが複数のリポジトリにまたがる関連する変更を同時にレビュー、テスト、マージし、必要に応じてこれらの変更を一緒に元に戻す機能を提供することで、重要なワークフローの改善がもたらされます。これらすべてが再現可能な CI ビルドを維持しながら実現されます。

この機能は、分散したコードベースで作業するエンタープライズが直面する複雑な統合課題に直接対処し、開発の摩擦を軽減してソフトウェア提供の信頼性を向上させます。

## 動機

多くの顧客は他のリポジトリに依存するソースコードリポジトリを持っています。たとえば、Android 開発者は一握りのリポジトリで作業するかもしれませんが、Android のビルドには千以上のリポジトリが関与しています。サンプル:

- [platform/manifest](https://android.googlesource.com/platform/manifest) - ビルドに必要な他のすべてのリポジトリを定義する中央リポジトリ
- [platform/frameworks/base](https://android.googlesource.com/platform/frameworks/base/) - コア Android フレームワーク
- [platform/system/core](https://android.googlesource.com/platform/system/core) - 低レベルのシステムコンポーネント
- [platform/bionic](https://android.googlesource.com/platform/bionic) - Android の C ライブラリ実装

現在、GitLab は単一プロジェクトのマージリクエストのみをサポートしていますが、ユーザーはプロジェクトをまたいで作業できるようにすべきです。ユーザーが以下を行えるようにする必要があります:

1. プロジェクトにまたがる変更をプッシュする。
1. これらの変更を 1 か所でまとめてレビューする。
1. これらの変更で CI が通ることを確認する。
1. すべてのテストが通った時にマージする。
1. 何かうまくいかない場合に変更を元に戻す。

## 用語の定義

- **チェンジセット** - 複数のプロジェクトにまたがる提案されたコード変更のグループ。
- **ターゲットプロダクト** - チェンジセットのターゲットプロダクトを表すプロジェクトのグループ。たとえば、Android ソースコードは数百の個別の Git リポジトリで構成されるプロダクトの一例です。
- **マニフェストファイル** - 通常、アプリのパッケージ名、コンポーネント、それらの間の依存関係と現在のリビジョンを宣言する XML 設定ファイル。

### ゴール

1. ターゲットプロジェクトを構成するプロジェクトを識別する方法を定義する。
1. 複数のプロジェクトにわたるマージと差し戻しがどのように機能するかを定義する。

1. ポリリポジトリの変更でコードレビューがどのように機能するかを議論する。

## 設計と実装の詳細

### チェンジセットとは何か？

チェンジセットを表現できる 2 つの主要な方法があります:

- パッチセット: 異なるプロジェクトに適用される個別の Git パッチのグループ。
- グループ化されたマージリクエスト: プロジェクトの境界を越えるマージリクエストのグループ。

#### グループ化されたマージリクエスト対パッチセット

GitLab モデルでは、マージリクエストを論理的なセットにグループ化し、一緒に適用する必要がある変更を表すことができます。これには既存のワークフロー（コードレビュー、マージリクエストパイプラインなど）と連携できるという利点があります。

ただし、いくつかの欠点があります:

- ターゲットブランチが変わるたびにサーバーで差分を再計算する必要があります。多くのリポジトリにわたる未解決のマージリクエストが多い場合、サーバーに大きな負荷がかかる可能性があります。この問題はモノリポジトリにも既に存在しています。
- プロジェクトへの各変更には、そのプロジェクトに新しいブランチを作成する必要があります。開発者は多くのリポジトリにわたる個別のブランチを命名・管理する必要があります。

パッチセットにはいくつかの利点があります:

- クライアントサイドで計算されます。
- 開発者に良いコミット衛生を強制し、関連する変更をまとめます。
- 個々のパッチのマージと差し戻しは、大きなマージコミットを差し戻すよりも簡単かもしれません。

主な欠点:

- パッチセットの使用はコードレビューの品質を低下させる可能性があります。たとえば、コードカバレッジの表示、セキュリティスキャンの実行、品質メトリクスのレポート、ファイルの全内容の表示などが機能しない可能性があります。
- CI ジョブが再現可能なビルドを作成できるかどうかが明確ではありません。CI ジョブは何らかのリビジョンをチェックアウトしてからパッチセットを適用する必要があります。ジョブはどのリビジョンを使用すべきでしょうか？
- パッチセットの更新は面倒で、ユーザーエクスペリエンスを損なう可能性があります。レビュアーが変更を要求した場合、ユーザーはパッチセットを再作成する必要があり、これは MR でブランチの履歴をスカッシュして上書きするのと同等です。
- ローカルでパッチセットを適用する良い方法がありません。[このディスカッション](https://groups.google.com/g/repo-discuss/c/43juvD1qGIQ)によると、すべてのプロジェクトにわたってすべてのパッチを適用できる `repo apply` コマンドが存在しません。

#### チェンジセットはどのように作成されるか？

チェンジセットを作成できる方法はいくつかあるべきです:

1. UI から
1. API を使用して
1. Git プッシュオプション

チェンジセットは変更をプッシュする直前、あるいは開発時間中でも作成できます。MR を追加/削除するためのバケットです。

変更をプッシュする前にチェンジセット ID が割り当てられていれば、プッシュオプションを使用してチェンジセット ID を渡すことができます:

```bash
glab cset new --product=10 #=> CSET ID: 1234

cd ../proj-1
git push -o cset=1234 origin HEAD
cd ../proj-2
git push -o cset=1234 origin HEAD
...
```

これは `glab` で自動的に行うことができます:

```bash
# プロダクト ID=10 を使用して複数のリポジトリに変更をプッシュし、存在しない場合は CSET ID を割り当て
glab push --polyrepo --product=10 #=> CSET created: https://gitlab.com/-/csets/1234

# または
glab push --polyrepo-manifest=../manifest.xml
```

#### チェンジセットの制約

1. チェンジセットに属するマージリクエストは、チェンジセット内の他の MR と一緒にのみマージできます。
1. マージリクエストは、ある時点で最大 1 つのチェンジセットにのみ存在できます。
1. チェンジセットは、特定の MR がマージされた時点の情報を表すため削除できません。マージリクエストを削除できないのと同様です。

既知の不明点:

- チェンジセットに属するマージリクエストをクローズすることを許可すべきでしょうか？チェンジセットから削除すべきでしょうか？
- チェンジセットの変更（マージリクエストの追加または削除）は CI パイプラインをトリガーすべきでしょうか？技術的には、チェンジセットの全体的な差分が変わり、ビルドが無効化されます。

### 「プロダクト」を構成するプロジェクトを識別する

[Epic](https://gitlab.com/groups/gitlab-org/-/epics/17278)

組織（インスタンスまたはグループとして大まかに定義される）は複数の「プロダクト」を持てます。特定の組織内に複数のポリリポジトリが存在することができます。

「プロダクト」はグループ/サブグループ/プロジェクト階層に対して直交しています。組織内の任意のプロジェクトのマップであり、必ずしも同じグループに属している必要はありません。

- 組織は多くの「プロダクト」を持つ
- プロダクトは多くのプロジェクト/リポジトリを持つ
- プロダクトは多くのチェンジセットを持つ
- チェンジセットは多くの MR を持つ（暗黙的にプロジェクト + ref）

組織はまだ一般的に利用可能ではありませんが、組織が利用可能になるまでトップレベルグループが一時的なコンテナと見なされる可能性があります。

通常、チェンジセットはターゲットプロジェクトに関与するプロジェクトのサブセットを含みます。顧客は現在、プロダクトを識別するためにいくつかのアプローチを使用しています。たとえば:

1. モノリポジトリ
1. Git サブモジュールを持つ中央リポジトリ
1. [Android マニフェストファイル](https://gerrit.googlesource.com/git-repo/+/master/docs/manifest-format.md)

ポリリポジトリに焦点を当てたこの議論のために、モノリポジトリは無視します。

#### Git サブモジュール

Git サブモジュールを使用すると、コミットを別々に保ちながら、他の Git リポジトリをリポジトリ内のサブディレクトリとして含めることができます。主な特性:

1. 外部リポジトリの特定のコミットをメインリポジトリに埋め込む。
1. サブモジュールへの変更は親リポジトリによって単一コミットポインターとして追跡される。
1. 明示的なコマンドによる更新が必要（git submodule update）。
1. ネイティブ Git 機能、追加ツール不要。
1. 各サブモジュールは独自の .git ディレクトリを維持。

利点:

1. 追加のツールが不要な直接 Git 機能。
1. シンプルな概念モデル - リポジトリ内のリポジトリ。
1. 変更頻度の低い安定した依存関係に適している。
1. 固定バージョンのライブラリやコンポーネントに適している。

課題:

1. ユーザビリティの Issue と学習曲線で悪名高い。
1. サブモジュールの更新や初期化を忘れやすい。
1. 複数のリポジトリを頻繁に変更する場合に扱いが難しい。
1. コマンドが冗長でエラーが発生しやすい。

#### Android マニフェストファイル

Android マニフェストアプローチは XML ファイル（通常 `default.xml`）を使用してリポジトリ構造を記述し、Google の「repo」ツールで管理されます。主な特性:

1. リポジトリ、ブランチ、パスを指定するマニフェスト XML ファイルを使用。
1. リポジトリはネストではなく並列にクローンされる。
1. 単一のマニフェストリポジトリで集中管理。
1. 多くのリポジトリを持つ大規模で複雑なプロジェクトを処理するために特別に設計。
1. Git のラッパーとして「repo」ツールを使用。

利点:

1. 多くのコンポーネントを持つ大規模プロジェクトに適している。
1. リポジトリ間のブランチトラッキング管理に柔軟。
1. 複数のリポジトリを一貫した状態に同期することを簡素化。
1. リポジトリをまたぐトピックブランチなど、より複雑なワークフローをサポート。
1. 複数のリポジトリを一度に処理する高レベルのコマンドを提供。

課題:

1. 「repo」ツールとそのコマンドを学ぶ必要がある。
1. repo ツールへの追加の依存。
1. Git ホスティングプラットフォームやツールでのサポートが少ない。

#### 欠点

Android マニフェストアプローチのいくつかの制限:

1. フラットな構造: マニフェストは主に特定のリビジョンまたはブランチでチェックアウトされる Git リポジトリのフラットなコレクションを定義します。どのリポジトリがどれに依存するかを本質的に追跡しません。
1. 依存関係の解決なし: 依存関係やサブ依存関係の自動解決がありません。マニフェストは単純に存在すべきすべてのリポジトリをリストするだけです。
1. 手動調整: プロジェクトのメンテナーはどのバージョンのリポジトリが一緒に機能するかを手動で決定し、それに応じてマニフェストを更新します。
1. 推移的な依存関係なし: リポジトリ A がリポジトリ B に依存し、リポジトリ B がリポジトリ C に依存する場合、マニフェストはこのチェーンを自動的に解決しません - 3 つすべてを明示的に含める必要があります。
1. グループ: マニフェストにはリポジトリを分類できる「グループ」の概念がありますが、これは組織目的のためであり、依存関係の関係を表すものではありません。

## イテレーション計画

思考実験として、ポリリポジトリのワークフローを完全にサポートし始めると仮定してみましょう。大まかな概要は以下の通りです:

### v0.1: チェンジセットオブジェクトの導入

1. チェンジセットを作成して MR をリンクできる。
1. チェンジセットは複数のマージリクエストからマージリクエストのステータスを集約する。
1. MR の差分へのリンク。
1. MR を取得するための API サポート

### v0.2: チェンジセットマージ V1

1. ベストエフォートのマージ、アトミック性の保証なし。
2. プロジェクトにまたがる差し戻しを集約。
3. シンプルな権限: マージはすべてのプロジェクトのマージを開始できるユーザーのみが利用可能？

### v0.3: チェンジセットでのコードレビュー V1

1. チェンジセットの差分を単一ビューで集約。
1. 特定のプロジェクトからの差分を表示可能。
1. [ラピッド差分](../rapid_diffs/index.md)のサポート。

### v0.4: チェンジセットマージ V2

1. GitLab はチェンジセットのマージ中に他のマージリクエストからのマージをブロックする。
1. マージコミットは各マージに対して追跡される。
1. チェンジセットの差し戻し: 差し戻しは今、一緒にマージできる差し戻しのセットになる。

### v0.5: チェンジセットパイプライン V1

1. チェンジセットをターゲットビルドプロジェクトに関連付ける。
1. デフォルトで、チェンジセットへのプッシュはターゲットビルドプロジェクトにパイプラインを作成する。
1. 影響を受けるチェンジセットのプロジェクト/ファイルが CI 変数に公開される。

### v0.6: チェンジセットでの CI ランナーの統合

1. 複数のリポジトリをチェックアウトするための CI ステップまたはネイティブ統合をサポートする（例: Android マニフェストファイル経由？）。

## 既知の不明点

これはまだ洗練されていない設計側面のリストです:

- 「プロダクト」は単純にリポジトリのコレクションですか、それとも追加のメタデータが必要ですか？プロダクトを作成するにはどのような権限が必要ですか？
- 制限を定義する。1 つのチェンジセットに関連付けられる MR の数は？グループまたは組織がホストできるプロダクトの数は？
- チェンジセットのライフサイクルは何ですか？
- システムは依存関係の処理方法を持つべきですか、それともユーザーに任せますか？
- 複数のリポジトリにわたるマージの競合はどのように検出、報告、解決されますか？
- リポジトリをまたぐマージで真のアトミック性を達成できますか、またはどのレベルの一貫性保証を提供できますか？
