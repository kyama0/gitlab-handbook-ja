## 概要

**Org Mover** は、Cell 間での GitLab.com 組織の移動をオーケストレーションする、[Runway](https://docs.runway.gitlab.com/) でデプロイされるコントロールプレーンサービスです。

| | |
|---|---|
| **課題** | レガシー Cell からターゲット Cell へ組織を移行するためのオートメーションが存在しない。 |
| **アプローチ** | あらゆるエンジニアが、各移動を順序付けられた一連のステップとして実行できる、Runway でデプロイされるコントロールプレーンとして Org Mover を構築する。Org Mover はデータを移動するシステムを調整するのであって、データ自体を移動するわけではない。Org Mover は REST API を公開する。 |
| **ステータス** | Proposed |
| **スコープ** | GitLab.com SaaS のみ: レガシー Cell からターゲット Cell へ。[スコープ](#scope)と[非目標](#non-goals)を参照。 |
| **必要な意思決定** | Tenant Scale、Geo、Siphon、Infrastructure のステークホルダーによる、コントロールプレーンの形状・スコープ・非目標の承認。 |
| **関連情報** | [ADR-002 ロールバック](002_rollback_strategy.md) · [ADR-010 読み取り専用モード](../../organization/decisions/010_organization_read_only_mode.md) · [Org データ移行ブループリント](../_index.md) |

### なぜ Cell 間で組織を移動する必要があるのか?

GitLab.com は単一のモノリシックなデプロイとして稼働しており、まずデータベースにおいて、厳しいスケーリングの上限に達しつつあります。[Cells](../../cells/) は、独立して水平方向にスケール可能なインスタンスに[組織](../../organization/)を分散させることで、これに対処します。

### なぜこれが難しいのか

次の 2 つの理由から、単に組織をオフラインにして移動することはできません。

1. 大規模な組織を移動できなければならない
2. お客様への影響を最小限に抑えてそれを行わなければならない。

#### お客様の規模

多くの小規模な組織は、データが十分に少ないため数分で転送でき、夜間や週末には非アクティブなものも多くあります。それらの組織は、比較的単純な手段と比較的単純なオーケストレーションで移動できます。

しかし、大規模な組織は[データベース負荷の大きな割合（社内リンク）](https://gitlab.com/gitlab-com/gl-infra/tenant-scale/staff/lstaff/-/work_items/3)を占めているため、レガシー Cell を完全に解放するには、最終的に小規模な組織だけでなく、それらの大規模な組織も移動する必要があります。それらはまた、[データ量の大きな割合（社内リンク）](https://gitlab.com/gitlab-com/gl-infra/tenant-scale/tenant-services/team/-/work_items/417)も占めています。

#### ダウンタイム

レガシー Cell を解放するには、それを空にする必要はありませんが、その負荷とデータのかなりの部分を移動する必要があります。わずかに削るのではなく、意味のある余裕を回復できるだけの量です。そして Cell は成長を続けており、その成長は加速しているため、解放は一度きりの取り組みではなく、持続的なものでなければなりません。その目標を満たすには、多くの移動を継続的かつ並行して実行する必要があります。移行のペースをお客様の非稼働時間の枠に依存させると、ペースを維持できなくなってしまいます。

最大規模の組織は、ダウンタイムの枠を完全に排除します。それらはあらゆるタイムゾーンにまたがっているため、読み取り専用期間をスケジュールする非稼働時間がありません。失われる開発時間のコストはユーザー数に比例してスケールするため、1 日の中断が数百万ドルに達します。GitLab.com のスケーリングは GitLab の責任であってお客様の責任ではないため、私たちはそのコストをお客様に押し付けることはできません。

したがって、組織の規模にかかわらず、組織のダウンタイムを最小化しなければなりません。ところが、組織は許容できるどの読み取り専用の枠の中でも転送しきれないほど多くのデータを保持しています。これが、組織を移動するための中心的な要件を強います。すなわち、読み取り専用の枠が開く前にデータの大部分がすでにターゲット上にあり、枠の間にはすべてを転送するのではなく小さな差分だけを排出すればよい状態にしなければなりません。

## 意思決定

私たちは、GitLab.com の Cell 間での組織の移動をオーケストレーションする、Runway でデプロイされるコントロールプレーンサービスとして Org Mover を構築します。

1. **Org Mover サービスはコントロールプレーンである。** 各移動を順序付けられた一連のステップとして実行し、各遷移の前に条件をチェックし、すべての移動に関する状態を保持し、リアルタイムのステータスを公開し、監査証跡を記録する。
2. **デプロイモデル: RunwayService と RunwayJobs。** Org Mover は、永続的な状態データベースに支えられた長寿命の [RunwayService](https://docs.runway.gitlab.com/welcome/introduction/) と、個々の移行フェーズを実行する RunwayJobs から構成される。ジョブのみのモデルでは、REST コントラクトを提供することも、リアルタイムのステータスをサポートすることもできない。
3. **移動は保存されたステップの連続として実行される。** 各移動は、プリフライトチェック、データレプリケーション、最終同期、ルーティング切り替えといった順序付けられたステップを経る。Org Mover は各ステップの後に進捗を保存するため、中断された移動は最初からやり直すのではなく、中断した箇所から再開する。
4. **不完全なデータにルーティングを切り替えてはならない。** Org Mover は、チェックサム・行数・スポットチェックを使ってターゲットのデータが完全でソースと一致していることを確認するまで、組織のトラフィックをターゲット Cell に送らない。途中で発生する日常的なレプリケーションの失敗は想定されており、リトライされる。それらは移動を止めない。そのチェックなしに切り替えると、お客様のデータを失うリスクがある。
5. **当初は go/no-go の判断を人が行う。** カットオーバーとは、Org Mover が組織のトラフィックを古い Cell から新しい Cell へ送る瞬間、すなわち移動が本番稼働する時点である。カットオーバーの前に、SRE がレプリケーションと検証のステータスをレビューし、切り替えを承認する。メトリクスと私たちの確信が高まるにつれて、この判断は自動化できる。
6. **ダウンタイムの枠の前に、各組織のデータを事前レプリケーションする。** あらゆるデータストアにまたがる組織のデータの大部分は、ソースが書き込み可能なままの状態でターゲットにレプリケーションされ、短い読み取り専用の枠の間に排出すべき小さな差分だけが残される。
7. **PostgreSQL レプリケーションは Siphon に委譲される。** Org Mover はストリームを設定してレプリケーションを開始する。Siphon が初期スナップショットを取得し、その後は自律的に CDC をストリーミングする。Org Mover はラグを監視し、すべてのテーブルが追いつくのを待ち、完了または中止時にレプリケーションスロットを削除する。
8. **非 PG レプリケーションはベースラインとして Geo を再利用する。** これには Git リポジトリ、オブジェクトストレージ、コンテナレジストリ、Secrets Manager が含まれる。Org Mover は、レプリケーションのために Geo を再利用できるよう、両方の Cell を必要に応じて設定する。将来的には、Geo はその全部または一部が置き換えられる可能性がある。コントロールプレーンのコントラクトは、それにかかわらず同一である。
9. **ルーティングは Topology Service に委譲される。** Org Mover は Topology Service の 2 フェーズクレーム更新を通じてカットオーバーを起動し、ルートを検証する。Topology Service がルーティングの権威である。
10. **ロールバックは ADR-002 に従う。** カットオーバー後のデータロールバックは存在しない。この戦略は、カットオーバー前の go/no-go ゲート、短い [ADR-002](002_rollback_strategy.md) のステージ 2 の切り戻しの枠、そしてその後のフィックスフォワードに依存する。
11. **1 つのコントロールプレーンが多数の同時並行の移動をオーケストレーションする。** 単一の Org Mover デプロイは、それぞれが独自の保存された状態を持つ多数の移動を同時に実行し、それらすべてにわたる可視性と制御を提供する。移動ごとに 1 つのサービスがあるわけではない。

### 検討した代替案

#### デュアルライト

データをソース Cell にコミットした後にレプリケーションする代わりに、アプリケーションが遷移中にすべての変更をソースとターゲットの両方に書き込み、ターゲットが追いついた時点でカットオーバーすることも考えられます。デュアルライトは、組織が使用するすべてのデータストア（PostgreSQL、Git リポジトリ、オブジェクトストレージ、コンテナレジストリ、Secrets Manager）にわたる書き込みパスの変更を必要とします。それは、各ストアについてモノリスに対する大規模で侵襲的な変更です。Cells の緊急性とあわせて、私たちは今のところこれを却下します。

## スコープ {#scope}

この ADR は、**GitLab.com SaaS** のユースケースを特に対象とします。すなわち、GitLab.com の Cell 間で組織を移動すること、**レガシー Cell からターゲット Cell へ**です。この設計は、意図的により広範なユースケースを視野に入れていますが（[非目標](#non-goals)を参照）、GitLab.com のレガシーからターゲットへのケースを超えるものはここでは約束されていません。Org Mover が扱わなければならない具体的な事項は以下のとおりです。

- レガシー Cell は複数の物理 PostgreSQL データベースを持つのに対し、ターゲット Cell は複数の論理データベースを持つ 1 つの物理 PostgreSQL を使用する。
- レガシー Cell のデプロイインフラはカスタムであり、ターゲット Cell は標準化された Cell デプロイを使用する。Org Mover 自体は標準化された Runway デプロイを使用するが、これはまだセルフマネージドでは利用できない。Org Mover のソース Cell とのやり取りは、レガシーインフラを反映する。
- レガシー Cell はターゲット Cell よりもはるかに多くの Gitaly ストレージを持つ。Org Mover はそれらの間を変換しなければならない。

## 非目標 {#non-goals}

これらのユースケースはスコープ外です。それぞれ後で再検討される可能性があります。

- **セルフマネージドおよび GitLab Dedicated の移行オーケストレーション。** Org Mover は GitLab.com の Cell を対象とする。
- **クロスプラットフォーム移行（セルフマネージドから SaaS へ、またはその逆）。**
- **任意の Cell から任意の Cell へ。** これは将来の方向性であり、この設計では提供されない。GitLab.com のレガシーからターゲットが稼働した後の後続ワークストリームである。
- **派生データストア（ClickHouse、Elasticsearch）の移行。** これらはレプリケーションされるのではなく、移行後にターゲット Cell 上で再構築される。

## エンジニアリングの原則

これらの原則は、すべての実装上の選択とレビューを導きます。これらはデータプレーンに依存しません。チームがどのデータプレーンアーキテクチャを選んでも成り立ちます。いずれかに違反する変更には、明示的で記録された正当化が必要です。

1. **Org Mover サービスはコントロールプレーンであり、データはそれやそのジョブを通過しない。** Org Mover はコマンドを発行し、進捗を読み取り、判断する。それが調整するシステムがバイトを移動する。
2. **各ステップをリトライしても安全にする。** 中断または再実行されたステップは、データを二重適用したり破損したりするのではなく、同じ結果に到達すべきである。insert よりも upsert、ゼロからの再起動よりもチェックポイントからの再開といった、安全に繰り返せる操作を優先する。
3. **読み取り専用の前に、コストの高いものをすべて事前レプリケーションする。** ソースが書き込み可能なうちにバルクデータを移動し、その後、読み取り専用の枠の間に小さな差分だけを排出する。
4. **コンポーネントを直交させる。** Org Mover は、小さく固定されたコントラクト（コマンドの発行、進捗の読み取り、検証、ゲート）を通じて各サブシステムを調整する。正しくゲートするためにサブシステムのセマンティクスが漏れ出さざるを得ない場合は、それを隠すのではなく、コントラクトの中で結合を明示する。

## 境界コントラクト

Org Mover は複数のシステムを調整します。高レベルのコントラクトは以下のとおりです。

移動ごとに、またフリート全体にわたって、以下に責任を負います。

- エンジニアが移動を管理するための UI と API を提供する
- 移動の状態を追跡する
- デプロイフリーズやアクティブなバッチバックグラウンドマイグレーションのチェックなど、移動の遷移を事前検証する
- Organization Read-Only Mode を切り替える
- Siphon を設定・管理する
- Geo を設定・管理する
- 監査のために Org Mover で SRE のアクションを追跡する
- 一時停止・再開・中止を調整する

### Siphon（PostgreSQL レプリケーション）

以下に責任を負います。

- PostgreSQL のチェンジデータキャプチャ: 初期スナップショットと CDC ストリーミング
- 起動時にレプリケーションスロットを作成する（Org Mover がそれを削除する）

コントロールプレーンが考慮しなければならない制約:

- ターゲットスキーマはソーススキーマと後方互換でなければならない
- Siphon は起動時にレプリケーションスロットを作成するが、削除はしない。消費されないスロットはソースプライマリのディスクを埋めてしまうため、Org Mover がそれを削除しなければならない

### Geo（非 PG データレプリケーション）

以下に責任を負います。

- Git リポジトリ、オブジェクトストレージ、コンテナレジストリ、Secrets Manager のデータをターゲット Cell にコピーまたはレプリケーションする
- 非 PG データのレプリケーションを検証する

コントロールプレーンが考慮しなければならない制約:

- 非同期レプリケーションは、アクティブな組織については決して 100% に達しない

### モノリス（アプリケーションと組織の状態）

以下に責任を負います。

- アプリケーションの読み取り専用モード（正しさはアプリ側が所有する、[ADR-010 読み取り専用モード](../../organization/decisions/010_organization_read_only_mode.md)を参照）
- ターゲット Cell モード
- スキーマと DDL

### Topology Service（ルーティング）

以下に責任を負います。

- ルーティング権威としての、ルーティング
- Org Mover がカットオーバー（ポイントオブノーリターン）のために起動する、2 フェーズクレーム更新
