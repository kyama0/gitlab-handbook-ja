## 概要

**Org Mover** は、GitLab.com の組織を Cell 間で移動することをオーケストレーションする、[Runway](https://docs.runway.gitlab.com/) でデプロイされるコントロールプレーンサービスです。

| | |
|---|---|
| **問題** | 組織を Legacy Cell から移行先 Cell へ移行する自動化が存在しない。 |
| **アプローチ** | Org Mover を、どのエンジニアでも各移動を順序付けられた一連のステップとして実行できる、Runway でデプロイされるコントロールプレーンとして構築する。Org Mover はデータを移動するシステムを連携させるものであり、自身でデータを移動するわけではない。Org Mover は REST API を公開する。 |
| **ステータス** | Proposed |
| **スコープ** | GitLab.com SaaS のみ: Legacy Cell から移行先 Cell へ。[スコープ](#scope)と[非目標](#non-goals)を参照。 |
| **必要な決定** | Tenant Scale、Geo、Siphon、Infrastructure のステークホルダーによる、コントロールプレーンの形・スコープ・非目標の承認。 |
| **関連** | [ADR-002 ロールバック](002_rollback_strategy.md) · [ADR-010 読み取り専用モード](../../organization/decisions/010_organization_read_only_mode.md) · [組織データ移行ブループリント](../_index.md) |

### なぜ Cell 間で組織を移動する必要があるのか?

GitLab.com は、まずデータベースから始まる厳しいスケーリングの上限に達しつつある、単一のモノリシックなデプロイメントとして稼働しています。[Cells](../../cells/) は、独立した水平方向にスケール可能なインスタンス全体に[組織](../../organization/)を分散させることで、これに対処します。

### なぜこれが難しいのか

組織を単純にオフラインにして移動できない理由は 2 つあります:

1. 大規模な組織を移動できなければならない
2. 顧客への影響を最小限にして行わなければならない

#### 顧客のサイズ

多くの小規模な組織はデータが十分に少なく、数分で転送できるほどで、その多くは夜間や週末には非アクティブです。これらの組織は、比較的単純な手段と比較的単純なオーケストレーションで移動できます。

しかし、大規模な組織は[データベース負荷の大きな割合（内部リンク）](https://gitlab.com/gitlab-com/gl-infra/tenant-scale/staff/lstaff/-/work_items/3)を占めるため、Legacy Cell を完全に軽減するには、小規模な組織だけでなく、最終的にそれらの大規模な組織も移動する必要があります。それらはまた、[データ量の大きな割合（内部リンク）](https://gitlab.com/gitlab-com/gl-infra/tenant-scale/tenant-services/team/-/work_items/417)も占めています。

#### ダウンタイム

Legacy Cell を軽減するために空にする必要はありませんが、その負荷とデータのかなりの割合を移動する必要はあり、わずかな削減ではなく意味のあるヘッドルームを回復できる程度の量です。そして Cell は成長を続けており、その成長は加速しているため、軽減は一度きりの取り組みではなく持続的でなければなりません。その目標を達成するには、多くの移動を継続的かつ並行して実行する必要があります。顧客のオフピーク時間帯のウィンドウに移行レートを依存させると、ペースを保てなくなります。

最大規模の組織は、ダウンタイムウィンドウを完全に排除します。それらはあらゆるタイムゾーンにまたがるため、読み取り専用期間をスケジュールするオフピーク時間帯がありません。失われる開発時間のコストはユーザー数に比例してスケールするため、1 日の中断が数百万ドルに達します。GitLab.com をスケールすることは顧客ではなく GitLab の責任であるため、そのコストを顧客に押し付けることはできません。

したがって、組織のサイズにかかわらず、組織のダウンタイムを最小化しなければなりません。しかし、組織は許容可能な読み取り専用ウィンドウ内で転送するには多すぎるデータを保持しています。これが、組織を移動するための中心的な要件を生みます: データの大部分が、読み取り専用ウィンドウが開く前にすでに移行先に存在していなければならず、ウィンドウではすべてを転送するのではなく小さなデルタのみをドレインするようにします。

## 決定

私たちは Org Mover を、GitLab.com の Cell 間の組織の移動をオーケストレーションする、Runway でデプロイされるコントロールプレーンサービスとして構築します。

1. **Org Mover サービスはコントロールプレーンである。** 各移動を順序付けられた一連のステップとして実行し、各遷移の前に条件をチェックし、すべての移動に関する状態を維持し、リアルタイムのステータスを公開し、監査証跡を記録する。
2. **デプロイメントモデル: RunwayService と RunwayJobs。** Org Mover は、永続的な状態データベースを背後に持つ長命の [RunwayService](https://docs.runway.gitlab.com/welcome/introduction/) と、個々の移行フェーズを実行する RunwayJobs から構成される。ジョブのみのモデルでは REST 契約を提供できず、リアルタイムのステータスもサポートできない。
3. **移動は保存されたステップのシーケンスとして実行される。** 各移動は、事前チェック、データレプリケーション、最終同期、ルーティング切り替えといった順序付けられたステップを経る。Org Mover は各ステップの後に進捗を保存するため、中断された移動は最初からやり直すのではなく、中断したところから再開する。
4. **不完全なデータへ決してルーティングを切り替えない。** Org Mover は、チェックサム、行数カウント、スポットチェックを使って移行先のデータが完全で移行元と一致していることをチェックするまで、組織のトラフィックを移行先 Cell へ送らない。途中で発生する通常のレプリケーション失敗は想定内であり、再試行される。それらは移動を停止させない。そのチェックなしで切り替えると、顧客データを失うリスクがある。
5. **当初は人が go/no-go の判断を行う。** カットオーバーとは、Org Mover が組織のトラフィックを古い Cell から新しい Cell へ送る瞬間、すなわち移動がライブになる地点である。カットオーバーの前に、SRE がレプリケーションと検証のステータスをレビューし、切り替えを承認する。メトリクスと私たちの自信が高まるにつれて、この判断は自動化できる。
6. **ダウンタイムウィンドウの前に各組織のデータを事前レプリケートする。** すべてのデータストアにわたる組織のデータの大部分は、移行元が書き込み可能なまま移行先へレプリケートされ、短い読み取り専用ウィンドウ中にドレインする小さなデルタのみを残す。
7. **PostgreSQL のレプリケーションは Siphon に委譲する。** Org Mover はストリームを構成しレプリケーションを開始する。Siphon は初期スナップショットを取得し、その後 CDC を自身でストリーミングする。Org Mover はラグを監視し、すべてのテーブルが追いつくのを待ち、完了または中止時にレプリケーションスロットを削除する。
8. **非 PG レプリケーションはベースラインとして Geo を再利用する。** これには Git リポジトリ、オブジェクトストレージ、Container Registry、Secrets Manager が含まれる。Org Mover は、レプリケーションのために Geo を再利用するべく、両方の Cell を必要に応じて構成する。将来、Geo は全体または一部が置き換えられる可能性がある。コントロールプレーンの契約はそれに関係なく同一である。
9. **ルーティングは Topology Service に委譲する。** Org Mover は、Topology Service の 2 フェーズのクレーム更新を通じてカットオーバーを呼び出し、ルートを検証する。Topology Service がルーティングの権限を持つ。
10. **ロールバックは ADR-002 に従う。** カットオーバー後のデータロールバックはない。この戦略は、カットオーバー前の go/no-go ゲート、短い [ADR-002](002_rollback_strategy.md) の Stage 2 スイッチバックウィンドウ、そしてその後の fix-forward に依拠する。
11. **1 つのコントロールプレーンが多数の同時移動をオーケストレーションする。** 単一の Org Mover デプロイメントが、それぞれ独自の保存された状態を持つ多数の移動を同時に実行し、それらすべてにわたる可視性と制御を提供する。移動ごとに 1 サービスではない。

### 検討された代替案

#### Dual-write

データを移行元 Cell にコミットした後にレプリケートする代わりに、アプリケーションが遷移中にすべての変更を移行元と移行先の両方に書き込み、移行先が追いついた時点でカットオーバーすることもできる。Dual-write は、組織が使用するすべてのデータストア（PostgreSQL、Git リポジトリ、オブジェクトストレージ、Container Registry、Secrets Manager）にわたる書き込みパスの変更を必要とする。これは各ストアについてモノリスへの大規模で侵襲的な変更である。Cells の緊急性も相まって、現時点ではこれを却下する。

## スコープ {#scope}

この ADR は、**GitLab.com SaaS** のユースケース、すなわち GitLab.com の Cell 間で組織を移動する、**Legacy Cell から移行先 Cell へ** のケースを特に対象とする。この設計は、より広範なユースケースを意図的に視野に入れている（[非目標](#non-goals)を参照）が、GitLab.com の Legacy から移行先へのケースを超えるものはここでは約束されていない。Org Mover が扱わなければならない具体的な事項:

- Legacy Cell は複数の物理 PostgreSQL データベースを持つのに対し、移行先 Cell は複数の論理データベースを持つ 1 つの物理 PostgreSQL を使用する。
- Legacy Cell のデプロイメントインフラはカスタムであり、移行先 Cell は標準化された Cell デプロイメントを使用する。Org Mover 自身は標準化された Runway デプロイメントを使用するが、これはまだセルフマネージドでは利用できない。Org Mover の移行元 Cell とのやり取りは Legacy インフラを反映する。
- Legacy Cell は移行先 Cell よりもはるかに多くの Gitaly ストレージを持つ。Org Mover はそれらの間で変換しなければならない。

## 非目標 {#non-goals}

これらのユースケースはスコープ外である。それぞれ後で再検討される可能性がある:

- **セルフマネージドおよび GitLab Dedicated の移行オーケストレーション。** Org Mover は GitLab.com の Cell を対象とする。
- **クロスプラットフォーム移行（セルフマネージドから SaaS、またはその逆）。**
- **任意の Cell から任意の Cell へ。** 将来の方向性であり、この設計では提供されない。GitLab.com の Legacy から移行先への移行が運用可能になった後の後続ワークストリームである。
- **派生データストア（ClickHouse、Elasticsearch）の移行。** これらはレプリケートされるのではなく、移行後に移行先 Cell で再構築される。

## エンジニアリング原則

これらの原則は、すべての実装上の選択とレビューを導く。それらはデータプレーンに依存しない: チームが選択するどのデータプレーンアーキテクチャでも成り立つ。いずれかに違反する変更には、明示的に記録された正当化が必要である。

1. **Org Mover サービスはコントロールプレーンであり、データはそれやそのジョブを通過しない。** Org Mover はコマンドを発行し、進捗を読み取り、判断する。それが連携させるシステムがバイトを移動する。
2. **各ステップを再試行安全にする。** 中断または再実行されたステップは、二重適用やデータ破損ではなく、同じ結果に到達するべきである。insert より upsert、ゼロからの再起動よりチェックポイントからの再開のように、安全に繰り返せる操作を優先する。
3. **コストの高いものはすべて読み取り専用の前に事前レプリケートする。** 移行元が書き込み可能な間にバルクデータを移動し、その後、読み取り専用ウィンドウ中に小さなデルタのみをドレインする。
4. **コンポーネントを直交させる。** Org Mover は、小さく固定された契約を通じて各サブシステムを連携させる: コマンドの発行、進捗の読み取り、検証、ゲート。サブシステムのセマンティクスが正しくゲートするために漏れ出さなければならない場合は、それを隠すのではなく、結合を契約の中で明示的にする。

## 境界契約

Org Mover はいくつかのシステムを連携させる。ハイレベルな契約は次のとおりである:

移動ごと、およびフリート全体について、次に責任を持つ:

- エンジニアが移動を管理するための UI と API の提供
- 移動状態の追跡
- デプロイフリーズやアクティブなバッチ化バックグラウンドマイグレーションのチェックなど、移動遷移の事前検証
- Organization 読み取り専用モードの切り替え
- Siphon の構成と管理
- Geo の構成と管理
- 監査のための Org Mover での SRE アクションの追跡
- 一時停止、再開、中止の連携

### Siphon（PostgreSQL レプリケーション）

次に責任を持つ:

- PostgreSQL の変更データキャプチャ: 初期スナップショットと CDC ストリーミング
- 起動時のレプリケーションスロットの作成（Org Mover が削除する）

コントロールプレーンが考慮しなければならない制約:

- 移行先スキーマは移行元スキーマと後方互換でなければならない
- Siphon は起動時にレプリケーションスロットを作成するが、削除はしない。消費されないスロットは移行元プライマリのディスクを埋めるため、Org Mover が削除しなければならない

### Geo（非 PG データレプリケーション）

次に責任を持つ:

- Git リポジトリ、オブジェクトストレージ、Container Registry、Secrets Manager のデータの移行先 Cell へのコピーまたはレプリケーション
- 非 PG データのレプリケーションの検証

コントロールプレーンが考慮しなければならない制約:

- 非同期レプリケーションは、アクティブな組織では決して 100% に到達しない

### モノリス（アプリケーションと組織の状態）

次に責任を持つ:

- アプリケーション読み取り専用モード（正しさはアプリが所有する、[ADR-010 読み取り専用モード](../../organization/decisions/010_organization_read_only_mode.md)を参照）
- 移行先 Cell モード
- スキーマと DDL

### Topology Service（ルーティング）

次に責任を持つ:

- ルーティングの権限としてのルーティング
- カットオーバー（後戻りできない地点）のために Org Mover が呼び出す 2 フェーズのクレーム更新
