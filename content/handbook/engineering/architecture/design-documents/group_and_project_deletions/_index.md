## 概要

このアーキテクチャブループリントは、GitLab のグループとプロジェクトの削除フローの包括的な再設計を概説します。現在のシステムには、誤った永続削除、復元リクエストのサポート負担の増大、ロックされたネームスペースパスなど、重大な課題があります。

私たちの提案は、「削除保留」状態、回復オプション、自動ネームスペースリネームを備えた、すべての価格ティアにわたって一貫したユーザーフレンドリーな削除体験を導入します。このビジョンを 3 つのイテレーションで実装することを推奨します。価格ティアにかかわらず、すべての GitLab ユーザーのデータ損失に対する保護を強化しつつユーザーコントロールを維持します。これらの変更により、サポートリクエストが大幅に減少し、誤った削除が軽減され、すべての GitLab ユーザーの全体的な体験が向上します。

## モチベーション

現在の GitLab 削除フローには、ユーザーと内部チームの両方に悪影響を与える重大な欠点がいくつかあります。個人ネームスペース内のプロジェクトは削除保護がまったくなく、グループベースのプロジェクトの保護は価格ティア間で一貫していません。ユーザーが誤ってコンテンツを削除した場合、限られた手段しかなく、不必要なサポートリクエストや手動復旧のためのインフラチームの関与につながります。さらに、一部のネームスペースパスは削除保留中にロックされたままになり、ユーザーはパスをすぐに再利用できません。

### 目標

1. すべての価格ティアとネームスペースタイプ（グループ、プロジェクト、個人ネームスペース）にわたって一貫した削除体験を作成する
2. すべてのエンティティに「削除保留」状態を実装することで誤った削除を減らす
3. セルフサービス回復オプションを有効化してサポートおよびインフラチームの負担を軽減する
4. 自動リネームにより削除時にネームスペースパスをすぐに解放する
5. 削除プロセスに対する明確な可視性とコントロールを提供することでユーザー体験を向上させる
6. 適切な UI を用いて「ゴミ箱に移動」（取り消し可能）と「永続削除」（取り消し不可）の概念を分離する

### 非目標

- プロジェクトがグループまたはユーザーネームスペースと関連付けられる方法の変更

## 提案

すべての GitLab 環境と価格ティアにわたって統一削除フローを実装することを提案します。3 つのイテレーションからなる段階的なアプローチを採用します：

1. **[イテレーション 1: 基本回復フレームワーク](https://gitlab.com/groups/gitlab-org/-/epics/17367)**
   - すべての価格ティアで永続削除前の一貫した削除保留期間を実装
   - 削除されたすべてのアイテムのセルフサービス回復オプションを追加
   - ユーザーおよびグループネームスペース全体で削除ステータスの UI を標準化
   - パスをすぐに解放するための一貫した自動リネームを実装

2. **[イテレーション 2: 削除保護の拡張](https://gitlab.com/groups/gitlab-org/-/epics/17366)**
   - SaaS での削除保留期間を 7 日から 30 日に延長
   - 削除イベントおよび今後の永続削除に関するメール通知を実装
   - より長い保持期間をサポートするためにスケジューリングシステムを更新

3. **[イテレーション 3: 削除インターフェースの強化](https://gitlab.com/groups/gitlab-org/-/epics/17368)**
   - 削除されたアイテムを管理するための専用「ゴミ箱」セクションを UI に導入
   - 「ゴミ箱に移動」と「永続的に削除」アクションの明確な分離を作成
   - ゴミ箱管理のための一括アクションを実装（複数復元、ゴミ箱を空にする）
   - 「ゴミ箱」UI にフィルタリングとソート機能を追加

この提案のビジュアルリファレンスは [Figma](https://www.figma.com/board/AKGxnlizU5pr8r8zRled7z/Group-and-project-deletion-flow?node-id=0-1&p=f&t=9BDJWEeUGiTugJAW-0) で確認できます。

## 設計と実装の詳細

### 現在の状態

GitLab の削除フローは現在、以下に基づいて異なる動作をします：

- 価格ティア（無料 vs. プレミアム/アルティメット）
- エンティティタイプ（プロジェクト vs. グループ）
- ネームスペースタイプ（ユーザー vs. グループ）

プレミアムおよびアルティメットティアでは、プロジェクトとグループは 7 日間の「削除保留」状態になり、その間ユーザーは回復または永続的な削除を選択できます。無料ティアでは、削除されたアイテムはユーザーからすぐに非表示になり、管理者のみがアクセスできます。ユーザーネームスペース内のプロジェクトは、回復期間なしに永続的に削除されます。

### 新しい統一削除フロー

![image](/images/handbook/engineering/architecture/design-documents/group_and_project_deletion/group_and_project_deletion_iteration3.png)

#### コアコンポーネント

1. **削除の開始**
   - ユーザーが削除を開始すると、アイテムとその子アイテムすべて（存在する場合）が「削除保留」状態に移行する
   - システムはネームスペースを自動的にリネームして元のパスを解放する
   - アイテムは通常の操作からはアクセス不可になるが、新しい「ゴミ箱」UI には表示される
   - すべての子アイテムは削除された親アイテムと同じように扱われる

2. **削除保留状態**
   - アイテムはこの状態で設定可能な期間（最初は 7 日間、イテレーション 2 では 30 日間）留まる
   - 削除ステータスと永続削除までの残り時間を示す明確な UI インジケーターが表示される
   - 主要なマイルストーン（最初の削除、永続削除の 7 日前）でメール通知が送信される
   - 削除保留中のアイテムは使用割り当てのカウントに含まれる

3. **回復メカニズム**
   - 適切な権限を持つユーザーがセルフサービス回復オプションを利用可能
   - 回復アクションにより、子アイテムを含む元のコンテンツを持つアクティブ状態にアイテムを復元する
   - システムは回復中の潜在的なネームスペースの競合を処理する

4. **永続削除**
   - イテレーション 1 では：適切な権限を持つユーザーが利用可能
   - イテレーション 3 では：ゴミ箱の「永続的に削除」アクションから利用可能

5. **ゴミ箱インターフェース**
   - ユーザーがアクセスできる削除されたすべてのアイテムを表示する専用セクション
   - フィルタリング、ソート、一括アクション機能
   - 一時削除と永続削除の明確な区別

#### 技術的実装の考慮事項

1. **データベースの変更**
   - 削除状態、スケジュールされた永続削除日、元のパスを追跡するフィールドを追加
   - すべての関連エンティティにわたるソフト削除パターンの実装

2. **API の拡張**
   - 新しい削除状態と操作をサポートするために API を拡張
   - 回復、スケジュールされた削除、ゴミ箱管理のためのエンドポイントを実装

3. **ネームスペースの処理**
   - ネームスペースパスを解放するための自動サフィックス追加を実装
   - 回復シナリオのための競合解決戦略を作成
   - すべてのネームスペース操作に対する適切なバリデーションを確保

4. **権限モデル**
   - アイテムの表示、回復、永続削除のための権限要件を定義
   - UI と API 全体で権限チェックを一貫して実装
   - 削除操作のためのロールベースアクセス制御をサポート

5. **通知システム**
   - 削除関連の通知テンプレートを作成
   - 時間ベースの通知トリガーのスケジューリングを実装
   - 通知メールからの明確なアクションパスを提供

### ユーザー体験フロー

#### プロジェクト/グループをゴミ箱に移動するユーザー

1. ユーザーがプロジェクト/グループの「ゴミ箱に移動」オプションを選択する
2. システムがプロセスを説明する確認ダイアログを表示する
3. 確認後、システムが：
   - アイテムを削除保留状態に移動する
   - 一意の識別子を追加してネームスペースをリネームする
   - ゴミ箱へのリンクを含む成功メッセージを表示する
   - 回復情報を含む確認メールを送信する

#### プロジェクト/グループを回復するユーザー

1. ユーザーがゴミ箱にアクセスする
2. 検索/フィルターを使用して目的のアイテムを見つける
3. 「回復」オプションを選択する
4. システムが回復と潜在的なネームスペースの競合を処理する
5. 回復されたアイテムへのリンクを含む成功メッセージを表示する

#### 永続削除

1. 設定された期間（7 日/30 日）後に自動的に発生する
2. 管理者は管理インターフェースを使用して早期に実行可能（すべてのイテレーション）
3. ユーザーは非アクティブタブの「永続的に削除」を使用して早期に実行可能（イテレーション 1）
4. ユーザーはゴミ箱の「永続的に削除」を使用して早期に実行可能（イテレーション 3）
5. システムは永続削除の確認を要求する
6. 永続削除完了後に通知が送信される

## 代替ソリューション

### 永続削除の制限

管理者ユーザーのみがスケジュールされた削除日前にコンテンツを永続的に削除できるようにするアプローチです。このアプローチはユーザーが削除プロセスとどのように対話するかを大幅に変え、重要なトレードオフをもたらします。

![image](/images/handbook/engineering/architecture/design-documents/group_and_project_deletion/group_and_project_deletion_alternative.png)

#### メリット

- データ保護の強化：ユーザーが即時の永続削除を実行する機能を削除することで、誤った永続的なデータ損失の事例を劇的に減少させる
- サポート負担の軽減：誤った削除チケットが減り、サポートリソースを他の問題に充てられる
- シンプルなユーザーインターフェース：ユーザーはソフト削除と永続削除のどちらかを選択する必要がなく、1 つの削除アクションのみに直面する
- 一貫した回復体験：すべてのユーザーがティアやネームスペースタイプに関わらず、同じ 30 日間の回復ウィンドウを得られる

#### デメリット

- ストレージコストの増加：削除保留期間を 7 日から 30 日に延長すると、ストレージ要件が約 4 倍になる
- ユーザーの自律性の喪失：ユーザーがコンテンツのライフサイクルを直接制御できなくなり、不満を引き起こす可能性がある
- 管理者の作業負荷の増加：多くのユーザーが早期の永続削除を要求する場合、管理者がボトルネックになる可能性がある
- プライバシーへの懸念：管理者の関与なしに機密データをすぐに永続的に削除できなくなる可能性がある
- 実装の複雑さ：削除リクエストを管理するための追加の権限コントロールと管理インターフェースが必要
