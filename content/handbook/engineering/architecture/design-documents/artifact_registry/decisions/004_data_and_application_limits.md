## コンテキスト {#context}

Artifact Registry は、システムの安定性を確保し、不正利用を防ぎ、予測可能なリソース消費を可能にするために、明示的な上限を定義する必要があります。これらの上限は、データの保持期間を管理するデータ保持ポリシー（[ADR-010](010_data_retention.md) を参照）とは別物です。ここでの上限は、任意の時点で許可されるデータ量と操作数を管理します。

上限が定義されていないと、少数のテナントが不均衡なリソースを消費し、すべての顧客に対するサービス品質を低下させる可能性があります。CI/CD のワークロードは非常にスパイクの多いトラフィックを生み出すため（[ADR-003](003_system_requirements.md) を参照）、バースト制御が特に重要になります。

この ADR では、3 種類の制約を区別します。

- **Limits（上限）** はアプリケーションの安定性とセキュリティを保護します。リクエストごと、または時間ウィンドウごとに強制され、課金には紐付きません。例: アーティファクトの最大アップロードサイズ、API レート上限、エンティティ数の上限。
- **Quotas（クォータ）** は、課金サイクルまたはプランに紐付いた集約的な消費の制約です。名前空間が全体として消費できるリソース量を管理します。例: 名前空間ごとのストレージクォータ。
- **Policies（ポリシー）** は、Artifact Registry へのアクセスを制御する、アプリケーションコードから分離されたビジネスロジックを表します。ポリシーは上限とクォータよりも先に評価されます。
  - _例:_ 組織の IP 許可リストにない IP からのアクセスを拒否する
  - _例:_ ブロックまたは禁止されたユーザーのアクセスを拒否する

この ADR は、**上限**（アーティファクトサイズ、API レート、エンティティ数）と **クォータ**（ストレージ）を扱います。

Artifact Registry は GitLab.com 規模を対象としています。定常状態で毎秒数千の API リクエスト、合計数十ペタバイトの格納データ、大規模な組織あたり数百万のアーティファクトです（[ADR-003](003_system_requirements.md) を参照）。上限は、正当なエンタープライズのワークロードをサポートできるほど高く、かつ単一の名前空間がサービスを不安定化させることを防げるほど低くする必要があります。

該当する場合、上限はパッケージレジストリおよびコンテナレジストリですでに確立されているものと整合または拡張し、GitLab のレジストリ全体で一貫した体験を提供します。

## 決定 {#decision}

**Artifact Registry は、Artifact Registry のテナンシーおよびパーティションの境界である名前空間レベルで、ストレージ（クォータ）、アーティファクトサイズ（上限）、API レート（上限）、エンティティ数（上限）の 4 つの次元にわたってクォータと上限を強制します。レート上限とストレージクォータのプランティアごとのデフォルトは課金アンカー（組織）から継承されますが、すべての上限は名前空間ごとに強制および測定されます。複数の名前空間を所有する組織は、その名前空間の使用量の合計に対して課金されます。名前空間をまたいだ重複排除やプールはありません。**

アーティファクトサイズの上限とエンティティ数の上限は、すべてのインストールタイプで同じデフォルトを使用します。レート上限は GitLab.com ではプラン依存です。各プランティアが独自のデフォルトレート上限を定義し、個々の名前空間は正当な高負荷利用のためにカスタムのオーバーライドを受け取ることができます。Self-Managed では、管理者がインスタンスレベルですべての上限を調整できます。

### ストレージクォータ {#storage-quota}

ストレージクォータは名前空間レベルで強制されます。ストレージは、名前空間内のすべての一意な blob の重複排除後の合計として測定されます。重複排除は [ADR-002](002_storage_deduplication_scope.md) と整合して名前空間ごとにスコープされます。名前空間をまたいだ重複排除はありません。複数の名前空間を所有する組織は、各名前空間のストレージの合計を消費し、それに対して課金され、組織は引き続き課金アンカーとなります（[ADR-001](001_organizations_as_anchor_point.md)）。

| クォータ | デフォルト | 備考 |
| ----- | ------- | ----- |
| 名前空間あたりの最大ストレージ | プラン依存（SKU ベース） | Artifact Registry は新しいプレミアム SKU です（[概要](../_index.md) を参照）。ストレージクォータは購入したアドオンによって定義されます |
| リポジトリあたりの最大ストレージ | デフォルトでハード上限なし | 名前空間レベルで構成可能なオプションの警告しきい値。超過すると名前空間のオーナーに通知されますが、アップロードはブロックされません |

警告しきい値（例: クォータの 80%）を名前空間ごとに構成できます。使用量がこのしきい値を超えると名前空間のオーナーに通知されますが、クォータに完全に達するまではアップロードはブロックされません。

新しいアップロードセッションを受け入れる前に、システムは名前空間がストレージクォータに達しているか超過しているかをチェックします。名前空間にまだ余裕がある場合、アップロードは受け入れられ、正常に完了した後に使用量カウンターが更新されます。名前空間がすでにクォータに達しているか超過している場合、アップロードセッションは拒否されます。

使用量カウンターはアップロードが正常に完了するたびに更新されるため、並行するアップロードセッションはそれぞれ独立してクォータチェックを通過し、合計してクォータをわずかに超過する可能性があります。これは許容されます。厳格な強制にはすべてのアップロードを直列化する必要があり、これは [ADR-003](003_system_requirements.md) の並行性の目標と矛盾します。超過分は並行アップロードの数によって制限され、次のクォータチェックで修正されます。

### アーティファクトサイズの上限 {#artifact-size-limits}

アーティファクトサイズの上限はアーティファクトタイプによって異なります。以下のデフォルト値はすべてのインストールタイプに適用されます。Self-Managed では、管理者がインスタンスレベルでこれらのデフォルトをオーバーライドできます。

| アーティファクトタイプ | 最大アーティファクトサイズ | 備考 |
| ------------- | ----------------- | ----- |
| Maven | 5 GB | 既存のパッケージレジストリの上限と一致 |
| npm | 5 GB | 既存のパッケージレジストリの上限と一致 |
| コンテナイメージ | 50 GB | blob あたりの最大サイズ。 |

さらに、コンテナイメージには次の上限が適用されます。

| 上限 | デフォルト | 備考 |
| ----- | ------- | ----- |
| 最大マニフェストペイロード | 250 KB | Docker および OCI マニフェストに適用。 |
| 最大マニフェスト参照数 | 200 | マニフェストが参照できる blob またはマニフェストの最大数 |

アップロードセッションには非アクティブタイムアウトが適用されます。タイムアウトウィンドウ内にデータが受信されない場合、セッションは終了され、部分的にアップロードされたデータは破棄されます。これにより、低速なサービス拒否攻撃（Slowloris など）を緩和し、アイドル状態のセッションがリソースを無期限に保持することを防ぎます。

### レート制限 {#rate-limits}

レート上限は API を不正利用から保護し、単一のユーザーや名前空間が共有インフラストラクチャを枯渇させることを防ぎます。レート上限の違反は、`Retry-After` ヘッダーとともに HTTP 429（Too Many Requests）を返します。

Self-Managed では、管理者がインスタンスレベルでレート上限のデフォルトをオーバーライドできます。

GitLab.com では、レート上限は最初はロギング／警告モードでデプロイされ、トラフィック分布が分析された後にのみ強制されます（[ロールアウト戦略](#implementation-notes) を参照）。この段階的なロールアウトは、適切なデフォルトを確立するための一度限りのプロセスです。Self-Managed では、管理者がレート上限を直接構成および強制できます。

#### IP ベースの上限 {#ip-based-limits}

IP ベースのレート上限は、不正利用に対する最も外側の防御層を提供し、ロールアウト中に最初に導入される上限です（[ロールアウト戦略](#implementation-notes) を参照）。これらは [コンテナレジストリの GCRA ベースのミドルウェア](https://gitlab.com/gitlab-org/container-registry/-/issues/1225) を再利用します。初期のデフォルト値は、本番トラフィックの分析に基づいて Observe フェーズ中に決定されます。

#### ユーザー単位の上限 {#per-user-limits}

ユーザーごとのレート上限はプラン依存です。組織内のユーザーは、組織のプランティアからデフォルトを継承します。プランごとのデフォルト値は、[ロールアウト戦略](#implementation-notes) の Observe フェーズ中に、ユーザーごとのトラフィック分布に基づいて決定されます。既存の [パッケージレジストリのレート上限](https://docs.gitlab.com/administration/settings/package_registry_rate_limits/) がベースラインの参考になります。

#### namespace 単位の上限 {#per-namespace-limits}

名前空間ごとのレート上限はプラン依存です。各プランティア（Premium、Ultimate）が独自のデフォルトレート上限を定義し、名前空間の課金アンカーから継承します。個々の名前空間は、構成の第一級プロパティとして独自のレート上限を設定でき、例外ベースで管理されることはありません。これにより、同じプランのすべての名前空間に対するデフォルトを引き上げることなく、正当な高負荷利用に対応できます。プランごとのデフォルト値は、[ロールアウト戦略](#implementation-notes) の Observe フェーズ中に、名前空間ごとのトラフィック分布に基づいて決定されます。

このティア化されたアプローチは、単一のグローバルなデフォルトの問題を回避します。正当な高負荷ユーザーに十分なほど高い値を 1 つ設定すると、それ以外のすべての人に対してレート制限が事実上無効化されてしまいます。名前空間ごとのオーバーライドを備えたプランベースのデフォルトにより、プラットフォームは低いティアでの不正利用から保護しつつ、高いティアで大量利用の顧客をサポートできます。

名前空間レベルの上限は、プラットフォームの主要な保護メカニズムです。ユーザーごとの上限は、単一のユーザーが名前空間のクォータ全体を消費することを防ぐ二次的な層を提供します。

#### 同時アップロードセッション {#concurrent-upload-sessions}

並行アップロードセッションの上限は、ユーザーごとおよび名前空間ごとに進行中のアップロード数を制限します。これにより、単一のアクターがアップロードインフラストラクチャを独占することを防ぎます。レート上限と同様に、並行アップロードセッションの上限はプラン依存です。プランごとのデフォルト値は、[ロールアウト戦略](#implementation-notes) の Observe フェーズ中に決定されます。

### エンティティ数の上限 {#entity-count-limits}

エンティティ数の上限は、API の動作、プロキシチェーンの複雑さ、ポリシー評価のコストに影響する特定の運用パラメータを制限します。リポジトリ数は、自動化のバグによる暴走的な作成を防ぐため、名前空間ごとアーティファクトタイプごとに制限されます。

| エンティティ | 上限 | 備考 |
| ------ | ----- | ----- |
| 名前空間ごとアーティファクトタイプごとのリポジトリ数 | 1,000 | インスタンスレベルで構成可能 |
| アーティファクト／パッケージバージョンあたりのタグ数 | 1,000 | 不正利用と誤用を防止 |
| パッケージあたりのバージョン数 | 25,000 | 不正利用と誤用を防止 |
| 仮想リポジトリあたりのアップストリームソース数 | 20 | プロキシチェーンの複雑さを制限 |
| リポジトリあたりのライフサイクルポリシールール数 | 50 | ポリシー評価の複雑さを制限 |

アップストリームソースとライフサイクルポリシールールの上限は、書き込み時に同期的に強制されます。これらの上限を超えようとすると HTTP 422（Unprocessable Entity）を返します。

## 影響 {#consequences}

### ポジティブな影響 {#positive}

1. **システムの安定性**: 明示的な上限により、暴走的なワークロードが共有インフラストラクチャを不安定化させることを防ぎます
2. **予測可能なパフォーマンス**: ポリシー評価とプロキシチェーンの複雑さが制限されることで、メタデータクエリが大規模でも高パフォーマンスを維持します
3. **明確な顧客の期待**: 文書化された上限により、顧客は使用量とキャパシティを計画できます
4. **公正なマルチテナンシー**: 名前空間スコープのレート上限、ストレージクォータ、エンティティ上限により、テナント間でリソースが分配されます
5. **強制可能な SLA**: 定義された上限は、API レスポンスタイムに意味のある SLO を設定するための前提条件です

### ネガティブな影響 {#negative}

1. **顧客の摩擦**: 正当な大量利用のワークロードがデフォルトの上限に達し、手動でのクォータ引き上げが必要になる場合があります
2. **運用上のオーバーヘッド**: クォータ管理により、使用状況の監視と上限引き上げリクエストの処理という継続的な作業が増えます
3. **結果整合性のあるストレージ会計**: 使用量カウンターは非同期で更新されます。大きなアップロードの直後の短い期間は、古い使用量が表示される場合があります
4. **上限のチューニングが必要**: デフォルト値は初期の見積もりであり、観測された本番トラフィックのパターンに基づいて調整が必要になります

## 検討した代替案 {#alternatives-considered}

### 代替案 1: 明示的な上限なし（フェアユースを信頼） {#alternative-1-no-explicit-limits-trust-fair-use}

プロアクティブな上限ではなく、監視とリアクティブな介入に依存します。

#### ポジティブな影響 {#positive-1}

- 上限に達することによる顧客の摩擦がない
- 上限チューニングのオーバーヘッドがない

#### ネガティブな影響 {#negative-1}

- 単一の不適切なテナントがすべての顧客に対するサービスを低下させる可能性がある
- リアクティブな介入は自動的な強制よりも遅い
- インシデント中の顧客とのコミュニケーションに明確な根拠がない

**却下理由:** GitLab.com 規模では、信頼性の高いマルチテナンシーのためにプロアクティブな上限が必要です。コンテナレジストリに組織ごとの上限がなかったことが [運用上の複雑さ](https://gitlab.com/gitlab-org/container-registry/-/issues/1242) の一因となりました。

### 代替案 2: インスタンスレベルの上限のみ {#alternative-2-instance-level-limits-only}

名前空間ごとではなく、インスタンスレベルでグローバルに上限を適用します。

#### ポジティブな影響 {#positive-2}

- 実装と理解がよりシンプル

#### ネガティブな影響 {#negative-2}

- 単一の大きな名前空間がインスタンスのキャパシティ全体を消費することを防げない
- 課金とキャパシティプランニングのために、リソース使用量を特定の名前空間に帰属させられない

**却下理由:** GitLab.com での公正なマルチテナンシーには、名前空間ごとの上限が必要です。組織は引き続きアンカーおよび課金ポイントであり（[ADR-001](001_organizations_as_anchor_point.md)）、その名前空間の使用量を集約します。

### 代替案 3: CDN／ロードバランサーでのみ制限 {#alternative-3-limit-at-cdn-load-balancer-only}

すべてのレート上限を、アプリケーションではなく CDN またはロードバランサー層（Cloudflare）で強制します。

#### ポジティブな影響 {#positive-3}

- レート上限の計算をアプリケーションからオフロードする
- スロットルされたリクエストの拒否のレイテンシが低い

#### ネガティブな影響 {#negative-3}

- CDN 層の上限は GitLab.com にのみ適用され、Self-Managed のインストールには同等の保護がない
- 複雑な CDN 構成なしでは名前空間ごとの上限を強制できない

**却下理由:** アプリケーション層のレート制限は、すべての GitLab インストールタイプにわたって一貫した強制を提供します。CDN 層の上限は IP ベースのスロットリングでアプリケーションの上限を補完できますが、置き換えることはできません。

## 実装メモ {#implementation-notes}

Artifact Registry はスタンドアロンのサービスです。**未解決の問題:** 上限とクォータの構成をどこに格納・管理するか（例: Rails モノリスに格納してレジストリが API 経由で取得する、またはレジストリ内で直接管理する）はまだ決定されていません。

1. ストレージクォータ: 各アップロードと GC サイクルの後に更新される Redis カウンターを使用し、アップロードセッション開始時に非同期チェックを行う
2. アーティファクトサイズ: リクエストボディから読み取るバイト数をカウントしてアップロード中に強制する（クライアントが省略または誤報告しうる `Content-Length` ヘッダーを信頼しない）。バイト数が上限を超えると読み取りを中止し、リクエストを拒否する。これは Workhorse がアップロードサイズの強制に使用しているものと同じアプローチである
3. レート上限: GCRA アルゴリズムを使用した Redis カウンター。違反時には `Retry-After` 付きの HTTP 429。レート制限の実装は、[コンテナレジストリ](https://gitlab.com/gitlab-org/container-registry/-/issues/1225) で確立された GCRA ベースのミドルウェアと Redis カウンターのパターンを再利用し、構成スキーマを Artifact Registry 固有の操作に適応させる
4. エンティティ数（アップストリームソース、ライフサイクルポリシールール）: 挿入前のデータベースクエリ。違反時には HTTP 422

GitLab.com では、Runway を介したインフラストラクチャレベルのレート上限が、IP ベースのスロットリングに対する追加の防御層を提供します。これらはアプリケーションレベルの上限を補完しますが、置き換えるものではありません。アプリケーションレベルの上限は、すべてのインストールタイプにわたる名前空間ごとおよびユーザーごとの強制に必要です。

すべての上限違反には、機械可読のエラーコード（例: `artifact_registry/quota_exceeded`）と、超過した上限を示す人間可読のメッセージが含まれます。

**ロールアウト戦略:** レート上限と名前空間ごとのクォータは、3 つのフェーズで導入されます。

1. **Observe** — すべての上限（IP ベース、名前空間ごと、ユーザーごと）が構成されますが、メトリクスを出力して警告をログに記録するだけです（ブロックなし）。IP ベースの上限は、コンテナレジストリの既存の GCRA ミドルウェアを再利用します。このフェーズは、すべての次元にわたってトラフィック分布を収集するために、少なくとも 1 つのマイルストーンの間実行されます。
1. **Warn** — 名前空間ごとまたはユーザーごとの上限を超過するリクエストは、警告ヘッダー（例: `X-RateLimit-Remaining`）を返しますが、依然として処理されます。上限に近づいている名前空間には通知されます。
1. **Enforce** — 名前空間ごととユーザーごとの上限が HTTP 429 レスポンスで積極的に強制されます。デフォルト値は、フェーズ 1 と 2 で収集されたデータに基づいて調整されます。
