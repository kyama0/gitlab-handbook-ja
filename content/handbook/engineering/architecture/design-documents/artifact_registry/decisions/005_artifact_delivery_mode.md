## コンテキスト

クライアントがアーティファクト（コンテナレイヤー、Maven JAR、npm tarball）をダウンロードする際、Artifact Registry はオブジェクトストレージからどのようにコンテンツを配信するかを決定しなければなりません。業界および GitLab 内部で、確立されたパターンが 2 つあります:

1. **リダイレクトモード**: サービスは事前署名されたオブジェクトストレージ URL への HTTP リダイレクト（302/307）で応答します。クライアントはストレージバックエンドから直接コンテンツを取得します。サービスがダウンロードパスでアーティファクトのバイトを扱うことはありません。
2. **プロキシモード**: サービスがオブジェクトストレージからコンテンツを取得し、それをクライアントへストリーミングします。すべてのアーティファクトのバイトはサービスを経由します。

GitLab モノリスと Container Registry はいずれも、現在両方のモードをサポートしています:

- **Container Registry**: `storage.redirect.disable`（boolean、デフォルト `false`）。リダイレクトを無効化すると、ストレージバックエンドへリダイレクトする代わりに、レジストリがコンテンツをクライアントに直接ストリーミングします。
- **GitLab モノリス**: すべてのオブジェクトストレージバックエンド機能（artifacts、LFS、packages、uploads）における `proxy_download` 設定（デフォルト `false`）。

### プロキシモードが必要な理由

GitLab の[ドキュメント](https://docs.gitlab.com/administration/object_storage/)には、リダイレクトが機能せずプロキシが必須となる実際のデプロイシナリオが列挙されています:

- **ファイアウォール制限**: クライアントネットワークがオブジェクトストレージエンドポイントへの直接アクセスをブロックしている。これは規制環境やエアギャップ／半隔離ネットワークでよく見られます。
- **信頼されない CA**: オブジェクトストレージがクライアントの信頼していない CA の証明書を使用している（NetApp アプライアンスや Ceph Object Gateway などのサードパーティ S3 互換バックエンドでよく発生します）。
- **非公開ストレージバックエンド**: オブジェクトストレージのエンドポイントがクライアントネットワークから到達不可能（プライベート VPC、内部限定エンドポイント）。
- **HTTPS のダウングレード**: オブジェクトストレージのエンドポイントが平文 HTTP で応答するため、クライアントが HTTPS でレジストリにアクセスしている場合に mixed-content エラーが発生する。
- **CORS 制限**: ブラウザベースのクライアントは、ストレージ側に適切な CORS ヘッダーが無いとオブジェクトストレージへのクロスオリジンリダイレクトを追跡できません。
- **事前署名 URL のセキュリティ**: 事前署名 URL には有効期限がありますが、特定のユーザーやセッションに紐づいているわけではありません。高セキュリティ環境では、アーティファクトのバイトがサービスのネットワーク境界を出ないことを組織が望むケースがあります。

これらのシナリオの一部は、現時点で大規模に GitLab の顧客に影響しています。サードパーティ S3 互換ソリューション（NetApp、Ceph Object Gateway）を利用しており、CA／ファイアウォール／ルーティングの制約が現実問題になっている顧客などです。

### デュアルパーソナリティ・デプロイモデル {#the-dual-personality-deployment-model}

Artifact Registry は、GitLab.com（SaaS）、GitLab Dedicated、Self-Managed という複数のデプロイモデルに対応できる、モジュラーなサービスとして設計されています。リモート／デュアルパーソナリティモデルでは、Self-Managed もしくは Dedicated のインスタンスが、リモートバックエンドとして GitLab.com の Artifact Registry に接続します。

このモデルでは、GitLab.com の Artifact Registry は、企業のファイアウォールの背後にあり、かつ基盤ストレージプロバイダー（GCS、Cloud CDN）の IP レンジを許可リストに追加できないクライアントへダウンロードを配信することになります。サービスは、ストレージへ直接到達できる namespace についてはリダイレクトを継続しつつ、それ以外の namespace についてはアーティファクトのコンテンツをプロキシできなければなりません。

このため、グローバル（インスタンスレベル）のリダイレクト／プロキシ切り替えだけでは不十分です。namespace 単位の設定（[ADR-022](022_namespace_decoupling.md)）は、インスタンスのデフォルトが何であっても利用可能でなければなりません。なぜなら、同一インスタンスや同一組織内の namespace でも、それぞれ異なるネットワーク制約を抱えうるからです。

## 決定

**Artifact Registry は 2 つのアーティファクト配信パターン（リダイレクトとプロキシ）をサポートし、これを 2 軸モデルとして選択します。すなわち、インスタンスレベルのデフォルトと、常に利用可能な namespace 単位の上書きです。**

### 軸 1: インスタンスのデフォルト

インスタンスは、次の 2 つの配信パターンのいずれかをデフォルトとして設定します:

- **リダイレクト**（推奨されるデフォルト）。すべてのアーティファクトダウンロードは、事前署名されたオブジェクトストレージ URL へクライアントをリダイレクトすることで配信されます。クライアントはストレージバックエンドから直接コンテンツを取得します。サービスがダウンロードパスでアーティファクトのバイトを扱うことはありません。これにより、サービスへの負荷とエグレス帯域を最小化できます。サービスからの応答はすべて小さいため、タイトなタイムアウトと小さなリソースフットプリントを実現できます。すべてのクライアントがストレージバックエンドへ直接到達できるデプロイに適しています。
- **プロキシ**。すべてのアーティファクトダウンロードは、オブジェクトストレージからのコンテンツをサービス経由でストリーミングすることによって配信されます。すべてのアーティファクトのバイトはサービスを経由します。これは、ファイアウォールの背後にある Self-Managed インスタンスのように、クライアントがストレージバックエンドへ直接到達できないデプロイ向けです。プロキシでは、サービスが大きなレスポンスボディ（コンテナレイヤーで数 GB に及ぶ可能性あり）を扱う必要があるため、書き込みタイムアウト、メモリ確保、帯域容量に影響が出ます。

インスタンスのデフォルトはサービス設定で指定し、変更は再起動時にのみ反映されます。

### 軸 2: namespace 単位の上書き

NULL を許容する namespace 単位の上書き（namespace レコードとともに保存されます。[ADR-022](022_namespace_decoupling.md) および [ADR-007](007_database_schema.md) を参照）は、インスタンスのデフォルトに関係なく、すべてのインスタンスで利用可能です。その 3 値のセマンティクスは次のとおりです:

- `NULL` — インスタンスのデフォルトを継承する。
- `redirect` — この namespace でリダイレクトを強制する。
- `proxy` — この namespace でプロキシを強制する。

任意のダウンロードリクエストにおける実効配信パターンは次のとおりです:

```text
effective = namespace.delivery_mode_override ?? instance.delivery_mode
```

namespace 単位の設定は、GitLab の組織オーナーが namespace 管理 API（実装仕様では S17 が所有）を通じて管理します。粒度は namespace 単位であり、組織単位、リポジトリ単位、アーティファクト単位ではありません。

この 2 軸の形式が、サービスが公開する唯一のノブの形です。独立した「ハイブリッド」インスタンスモードは存在しません。上書きはすべてのインスタンスで同じカラムであり、運用者は必要な組み合わせを選択します。

### この形式が表現するシナリオ

Artifact Registry がサポートする必要のあるすべてのデプロイシナリオは、2 軸の組み合わせに帰着します:

- **シングルテナント、すべてのクライアントがストレージに到達できる。** インスタンスのデフォルト = `redirect`、namespace 単位の上書きなし。
- **ファイアウォールの背後／信頼されない CA を持つシングルテナント。** インスタンスのデフォルト = `proxy`、namespace 単位の上書きなし。
- **マルチテナント SaaS／デュアルパーソナリティ**（GitLab.com）: インスタンスのデフォルト = `redirect`、クライアントがストレージに到達できない個々の namespace は上書きを `proxy` に設定する（またはその逆）。
- **1 つだけ例外的な namespace を持つシングルテナント。** インスタンスのデフォルトは一般的なケースに合わせ、例外となる namespace は上書きを反対の値に設定する。

### システムへの制約

両方の配信パターンをサポートすることは、サービス設計に次のような影響を及ぼします:

- **タイムアウト**: リダイレクト応答は小さく高速です。プロキシ応答は数 GB になり、数分かかる場合もあります。サービスは、リダイレクト性能に影響を与えずにプロキシダウンロード向けの長いタイムアウトをサポートする必要があります。
- **リソースフットプリント**: プロキシダウンロードはサービスを経由して大きなペイロードをストリーミングするため、リクエストあたりのメモリと帯域要件が増加します。キャパシティプランニングでは、プロキシとして配信するトラフィックの割合を考慮しなければなりません。
- **ストレージバックエンドアクセス**: ストレージバックエンドは、事前署名 URL の生成（リダイレクト用）とストリーミング読み取り（プロキシ用）の両方をサポートする必要があります。
- **CDN との相互作用**: リダイレクトはクライアントが CDN にバックされたストレージ URL から直接取得するため、CDN キャッシュの恩恵を受けます。プロキシは CDN をバイパスするためキャッシュ効果を失います。実装仕様（S06）は、実効配信パターンがプロキシの場合に、CDN／URL キャッシュのミドルウェアスタックをエンドツーエンドでバイパスします。
- **モニタリング**: サービスは、メトリクスとログでリダイレクトトラフィックとプロキシトラフィックを区別し、運用者がパターンごとの帯域・レイテンシ・キャパシティへの影響を追跡できるようにする必要があります。
- **namespace 単位のルックアップ**: 上書きは、リクエストハンドラが認可とルーティングのためにすでに結合している同じ namespace 行から読み取られます。別個のルックアップや二重の結合は不要です。コストは、ハンドラがすでに実行しているクエリに NULL 許容カラムが 1 つ増えるだけです。

## 影響

### ポジティブな影響

1. **ネットワーク制限による顧客への影響なし**: ファイアウォールの背後、信頼されない CA の使用、ストレージへの直接アクセス不可といったデプロイでも、回避策なしにプロキシ経由で Artifact Registry を利用できます。
2. **既存 GitLab 機能との同等性**: Container Registry も GitLab モノリスもプロキシダウンロードをサポートしています。Artifact Registry でこれをサポートしないことは、これらのシステムから移行する顧客にとって機能後退となります。
3. **デュアルパーソナリティのサポート**: 上書きにより、すべてのトラフィックをサービスに通すことなく、ネットワーク制約の異なる namespace を GitLab.com が配信するリモートモデルが可能になります。
4. **運用上一貫した操作面**: 運用者はどのデプロイでも、1 つのインスタンスレベルのノブと 1 つの namespace 単位のカラムを目にします。namespace 単位の設定を暗黙のうちに無効化するインスタンスモードのゲートは存在しません。
5. **上書きは namespace 単位のオプトイン**: namespace 単位の制御を必要としないデプロイには何のコストもかかりません。上書きは `NULL` であり、実効パターンは単にインスタンスのデフォルトです。

### ネガティブな影響

1. **サービスの複雑性の増加**: ダウンロードパスは、根本的に異なる 2 種類の応答パターン（小さなリダイレクト vs 大きなストリーミングボディ）を、それぞれ異なるタイムアウト、メモリ、エラー処理特性で扱う必要があります。
2. **プロキシトラフィックでのリソース要件の増加**: プロキシとして配信される namespace やインスタンスは、リクエストあたりの帯域とメモリを大幅に消費します。少数の高トラフィックなプロキシ namespace でも、サービスの容量に過大な影響を与える可能性があります。
3. **プロキシトラフィックでの CDN 効果の低下**: プロキシは CDN をバイパスするため、ストレージバックエンドの負荷を減らしダウンロードレイテンシを改善するキャッシュの恩恵を失います。
4. **両パターンのテストと保守**: 各パターンはダウンロードパスで異なる挙動を示します。テストは、「上書きなし」と「上書きが反対を強制する」の両方の組み合わせで、両パターンをカバーする必要があります。

## 代替案

### 代替案 1: リダイレクトのみ

サービスは常にリダイレクトで応答する。クライアントがストレージへ到達できない顧客は、サービスの外側で問題を解決しなければならない（VPN、プロキシのサイドカー、ファイアウォールルールなど）。

**長所:**

- もっともシンプルなサービス設計。すべての応答が小さい。タイトなタイムアウトと最小限のメモリ。
- CDN が常に有効。

**短所:**

- **実顧客のデプロイを破壊する**: ファイアウォール、信頼されない CA、非公開バックエンドは仮想的な話ではありません。これらは現実に GitLab の顧客に影響しています。
- **既存機能からの後退**: Container Registry も GitLab モノリスもプロキシダウンロードをサポートしています。この機能を取り除くのは後退です。
- **デュアルパーソナリティモデルを阻害する**: GitLab.com に接続するリモートの Self-Managed／Dedicated インスタンスは、クライアントが GCS／Cloud CDN に到達できない場合に機能できなくなります。

**却下理由:** 顧客への影響が具体的かつ十分に文書化されているため。

### 代替案 2: 3 つの名前付きインスタンスモード（`redirect` / `proxy` / `hybrid`）

インスタンスは 3 つのモードのいずれかに設定される: `redirect`（namespace 単位のルックアップなし。インスタンスは常にリダイレクト）、`proxy`（namespace 単位のルックアップなし。インスタンスは常にプロキシ）、または `hybrid`（リクエストごとに namespace 単位の上書きを評価し、インスタンスのデフォルト = redirect）。これは本 ADR に以前記録されていた決定です。

**長所:**

- `redirect` および `proxy` モードでは namespace 単位のカラムをまったく読まないため、ダウンロードパスでわずかにコストが安い。
- モード名が、設計議論における当初の 3 つのシナリオに直接対応する。

**短所:**

- **運用上一貫しない**: namespace 単位の上書きは `hybrid` モードでは利用可能で、他の 2 つでは利用不可です。運用者は、見ている namespace 設定が意味を持つかどうかを知るために、自分がどのインスタンスモードにいるかを知る必要があります。namespace オーナーも反対側から同じ曖昧さに直面します。自分の上書きが、通常は見ることのできないグローバル設定によって効いたり効かなかったりするのです。
- **表現力の追加なし**: 3 モード形式が表現するすべてのシナリオは `(インスタンスのデフォルト, 上書き)` で表現可能です。`redirect` モード ≡ インスタンス=`redirect` で上書きゼロ、`proxy` モード ≡ インスタンス=`proxy` で上書きゼロ、`hybrid` モード ≡ インスタンス=`redirect`（または `proxy`）で一部の namespace を反対の値に設定。
- **最適化が限界的**: `redirect` / `proxy` モードで上書きカラムの読み取りをスキップしても、測定可能な節約にはなりません。ハンドラはすでにリクエストごとに認可とルーティングのために namespace 行をロードしているので、上書きの読み取りは既存クエリの一部です。
- **例外が必要になった場合の一方通行**: 今日 `redirect` または `proxy` モードを選んだデプロイは、インスタンスレベルのモード変更（と暗黙の再起動）なしには、例外的な namespace を 1 つ上書きする能力を失います。

**却下理由:** 2 軸形式は、より小さく一貫した操作面で同じ範囲をカバーします。3 モード形式が節約する限界的なコストは、それが招く運用上の曖昧さに見合いません。この代替案は当初の決定であり、現在は置き換えられました。

### 代替案 3: リポジトリ単位の設定

namespace レベルではなくリポジトリレベルで配信の上書きを許容する。

**長所:**

- より細かい粒度での制御。namespace 配下のほとんどのリポジトリは redirect を使い、特定のリポジトリだけ proxy にする、といったことが可能。

**短所:**

- **問題に対するオーバーエンジニアリング**: ネットワーク制約は namespace のクライアント環境に対して適用されるものであり、個々のリポジトリに対するものではありません。クライアントがストレージに到達できないなら、その namespace のどのリポジトリにも到達できません。
- **管理オーバーヘッド**: 管理者はリポジトリごとに個別設定する必要があります。
- **一貫性のないユーザー体験**: 同じ namespace 内のリポジトリで挙動が異なるのは混乱を招きます。

**却下理由:** ネットワーク制約は通常、namespace／ネットワーク境界で適用されるもので、リポジトリレベルではありません。namespace 単位が適切な粒度です。具体的なユーザー要望が出てきた場合は、namespace 設定を継承しつつリポジトリ単位で個別に上書きできる仕組みを将来検討できますが、現時点ではスコープ外です。
