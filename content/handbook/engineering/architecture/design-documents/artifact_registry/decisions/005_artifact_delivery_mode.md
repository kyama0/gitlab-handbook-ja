## コンテキスト

クライアントがアーティファクト（コンテナレイヤー、Maven JAR、npm tarball）をダウンロードする際、Artifact Registry はオブジェクトストレージからどのようにコンテンツを配信するかを決定しなければなりません。業界および GitLab 内部で、確立されたパターンが 2 つあります:

1. **リダイレクトモード**: サービスは事前署名されたオブジェクトストレージ URL への HTTP リダイレクト（302/307）で応答します。クライアントはストレージバックエンドから直接コンテンツを取得します。サービスがダウンロードパスでアーティファクトのバイトを扱うことはありません。
2. **プロキシモード**: サービスがオブジェクトストレージからコンテンツを取得し、それをクライアントへストリーミングします。すべてのアーティファクトのバイトはサービスを経由します。

GitLab モノリスと Container Registry はいずれも、現在両方のモードをサポートしています:

- **Container Registry**: `storage.redirect.disable`（boolean、デフォルト `false`）。リダイレクトを無効化すると、ストレージバックエンドへリダイレクトする代わりに、レジストリがコンテンツをクライアントに直接ストリーミングします。
- **GitLab モノリス**: すべてのオブジェクトストレージバックエンド機能（artifacts、LFS、packages、uploads）における `proxy_download` 設定（デフォルト `false`）。

### プロキシモードが必要な理由

GitLab の[ドキュメント](https://docs.gitlab.com/administration/object_storage/)には、リダイレクトが機能せずプロキシが必須となる実際のデプロイシナリオが列挙されています:

- **ファイアウォール制限**: クライアントネットワークがオブジェクトストレージエンドポイントへの直接アクセスをブロックしている。これは規制環境やエアギャップ／半隔離ネットワークでよく見られます。
- **信頼されない CA**: オブジェクトストレージがクライアントの信頼していない CA の証明書を使用している（NetApp アプライアンスや Ceph Object Gateway などのサードパーティ S3 互換バックエンドでよく発生します）。
- **非公開ストレージバックエンド**: オブジェクトストレージのエンドポイントがクライアントネットワークから到達不可能（プライベート VPC、内部限定エンドポイント）。
- **HTTPS のダウングレード**: オブジェクトストレージのエンドポイントが平文 HTTP で応答するため、クライアントが HTTPS でレジストリにアクセスしている場合に mixed-content エラーが発生する。
- **CORS 制限**: ブラウザベースのクライアントは、ストレージ側に適切な CORS ヘッダーが無いとオブジェクトストレージへのクロスオリジンリダイレクトを追跡できません。
- **事前署名 URL のセキュリティ**: 事前署名 URL には有効期限がありますが、特定のユーザーやセッションに紐づいているわけではありません。高セキュリティ環境では、アーティファクトのバイトがサービスのネットワーク境界を出ないことを組織が望むケースがあります。

これらのシナリオの一部は、現時点で大規模に GitLab の顧客に影響しています。サードパーティ S3 互換ソリューション（NetApp、Ceph Object Gateway）を利用しており、CA／ファイアウォール／ルーティングの制約が現実問題になっている顧客などです。

### デュアルパーソナリティ・デプロイモデル {#the-dual-personality-deployment-model}

Artifact Registry は、GitLab.com（SaaS）、GitLab Dedicated、Self-Managed という複数のデプロイモデルに対応できる、モジュラーなサービスとして設計されています。リモート／デュアルパーソナリティモデルでは、Self-Managed もしくは Dedicated のインスタンスが、リモートバックエンドとして GitLab.com の Artifact Registry に接続します。

このモデルでは、GitLab.com の Artifact Registry は、企業のファイアウォールの背後にあり、かつ基盤ストレージプロバイダー（GCS、Cloud CDN）の IP レンジを許可リストに追加できないクライアントへダウンロードを配信することになります。サービスは、ストレージへ直接到達できる namespace についてはリダイレクトを継続しつつ、それ以外の namespace についてはアーティファクトのコンテンツをプロキシできなければなりません。

このため、グローバル（インスタンスレベル）のリダイレクト／プロキシ切り替えだけでは不十分です。決定は namespace 単位（[ADR-022](022_namespace_decoupling.md)）で行う必要があります。なぜなら、同一インスタンスや同一組織内の namespace でも、それぞれ異なるネットワーク制約を抱えうるからです。

## 決定

**Artifact Registry はインスタンスレベルで設定可能な 3 つのアーティファクト配信モードをサポートします: redirect（デフォルト）、proxy、hybrid。**

### Redirect モード（デフォルト）

すべてのアーティファクトダウンロードは、事前署名されたオブジェクトストレージ URL へクライアントをリダイレクトすることで配信されます。クライアントはストレージバックエンドから直接コンテンツを取得します。サービスがダウンロードパスでアーティファクトのバイトを扱うことはありません。サービスへの負荷とエグレス帯域を最小化できるため、これがデフォルトとなっています。サービスからの応答はすべて小さく、タイトなタイムアウトと小さなリソースフットプリントを実現できます。

このモードは、すべてのクライアントがストレージバックエンドへ直接到達できるデプロイに適しています。

### Proxy モード

すべてのアーティファクトダウンロードは、オブジェクトストレージからのコンテンツをサービス経由でストリーミングすることによって配信されます。すべてのアーティファクトのバイトはサービスを経由します。このモードは、ファイアウォールの背後にある Self-Managed インスタンスのように、クライアントがストレージバックエンドへ直接到達できないデプロイ向けです。

Proxy モードでは、サービスが大きなレスポンスボディ（コンテナレイヤーで数 GB に及ぶ可能性あり）を扱うため、書き込みタイムアウト、メモリ確保、帯域容量に影響が出ます。

### Hybrid モード

インスタンスのデフォルトはリダイレクトですが、namespace（[ADR-022](022_namespace_decoupling.md)）ごとに配信モードを proxy に上書き可能です。namespace 単位の設定が評価されるのは、このモードのみです。ダウンロードリクエストを配信する際、サービスは namespace に設定された配信モードを確認し、リダイレクトかプロキシかを決定します。

namespace 単位の設定は、GitLab の組織オーナーが管理します。粒度は namespace 単位であり、組織単位、リポジトリ単位、アーティファクト単位ではありません。

Hybrid モードはマルチテナントデプロイ向けに設計されており、特に上記の[コンテキスト](#the-dual-personality-deployment-model)で述べたデュアルパーソナリティモデルで運用される GitLab.com で、同一インスタンス上の namespace 間でネットワーク制約が異なる場合を想定しています。

### システムへの制約

proxy および hybrid モードのサポートは、サービス設計に次のような影響を及ぼします:

- **タイムアウト**: リダイレクト応答は小さく高速です。プロキシ応答は数 GB になり、数分かかる場合もあります。サービスは、リダイレクト性能に影響を与えずにプロキシダウンロード向けの長いタイムアウトをサポートする必要があります。
- **リソースフットプリント**: プロキシダウンロードはサービスを経由して大きなペイロードをストリーミングするため、リクエストあたりのメモリと帯域要件が増加します。キャパシティプランニングでは、プロキシモードで配信するトラフィックの割合を考慮しなければなりません。
- **ストレージバックエンドアクセス**: ストレージバックエンドは、事前署名 URL の生成（リダイレクト用）とストリーミング読み取り（プロキシ用）の両方をサポートする必要があります。
- **CDN との相互作用**: リダイレクトモードはクライアントが CDN にバックされたストレージ URL から直接取得するため、CDN キャッシュの恩恵を受けます。プロキシダウンロードは CDN をバイパスするためキャッシュ効果を失います。
- **モニタリング**: サービスは、メトリクスとログでリダイレクトトラフィックとプロキシトラフィックを区別し、運用者がモードごとの帯域・レイテンシ・キャパシティへの影響を追跡できるようにする必要があります。
- **namespace 単位のルックアップ（hybrid のみ）**: hybrid モードでは、サービスはダウンロードリクエストごとに namespace の配信モードを確認します。redirect および proxy モードでは、このルックアップはスキップされます。

## 影響

### ポジティブな影響

1. **ネットワーク制限による顧客への影響なし**: ファイアウォールの背後、信頼されない CA の使用、ストレージへの直接アクセス不可といったデプロイでも、回避策なしにプロキシモード経由で Artifact Registry を利用できます。
2. **既存 GitLab 機能との同等性**: Container Registry も GitLab モノリスもプロキシダウンロードをサポートしています。Artifact Registry でこれをサポートしないことは、これらのシステムから移行する顧客にとって機能後退となります。
3. **デュアルパーソナリティのサポート**: Hybrid モードにより、すべてのトラフィックをサービスに通すことなく、ネットワーク制約の異なる namespace を GitLab.com が配信するリモートモデルが可能になります。
4. **デフォルトのリダイレクトが効率を保つ**: redirect および hybrid モードでは、一般的なケースはリダイレクトであり、サービスのリソースフットプリントを低く保てます。
5. **シンプルなデプロイに不要なオーバーヘッドが乗らない**: redirect および proxy モードは、namespace 単位のルックアップを完全にスキップします。namespace 単位の設定を評価するコスト（軽微）は、hybrid モードのみが負担します。

### ネガティブな影響

1. **サービスの複雑性の増加**: ダウンロードパスは、根本的に異なる 2 種類の応答パターン（小さなリダイレクト vs 大きなストリーミングボディ）を、それぞれ異なるタイムアウト、メモリ、エラー処理特性で扱う必要があります。
2. **プロキシトラフィックでのリソース要件の増加**: proxy モードの namespace（または proxy モードのインスタンス）は、リクエストあたりの帯域とメモリを大幅に消費します。少数の高トラフィック proxy namespace でも、サービスの容量に過大な影響を与える可能性があります。
3. **プロキシトラフィックでの CDN 効果の低下**: プロキシダウンロードは CDN をバイパスするため、ストレージバックエンドの負荷を減らしダウンロードレイテンシを改善するキャッシュの恩恵を失います。
4. **3 モードのテストと保守**: 各モードはダウンロードパスで異なる挙動を示します。テストは 3 モードすべてをカバーする必要があります。

## 代替案

### 代替案 1: リダイレクトのみ

サービスは常にリダイレクトで応答する。クライアントがストレージへ到達できない顧客は、サービスの外側で問題を解決しなければならない（VPN、プロキシのサイドカー、ファイアウォールルールなど）。

**長所:**

- もっともシンプルなサービス設計。すべての応答が小さい。タイトなタイムアウトと最小限のメモリ。
- CDN が常に有効。

**短所:**

- **実顧客のデプロイを破壊する**: ファイアウォール、信頼されない CA、非公開バックエンドは仮想的な話ではありません。これらは現実に GitLab の顧客に影響しています。
- **既存機能からの後退**: Container Registry も GitLab モノリスもプロキシダウンロードをサポートしています。この機能を取り除くのは後退です。
- **デュアルパーソナリティモデルを阻害する**: GitLab.com に接続するリモートの Self-Managed／Dedicated インスタンスは、クライアントが GCS／Cloud CDN に到達できない場合に機能できなくなります。

**却下理由:** 顧客への影響が具体的かつ十分に文書化されているため。

### 代替案 2: redirect と proxy のみ（hybrid なし）

namespace ごとの混在を許容しない、インスタンスレベルの 2 モード（redirect または proxy）のみ。

**長所:**

- hybrid よりシンプル。ダウンロードパスでの namespace 単位ルックアップが不要。
- シングルテナントデプロイ（Dedicated、Self-Managed）には十分。

**短所:**

- **マルチテナント SaaS には不十分**: GitLab.com では、一部の namespace は proxy が必要で、別の namespace は redirect の方が有利です。インスタンスレベルの切り替えでは all-or-nothing になり、redirect で済む namespace の性能を低下させるか、proxy が必要な namespace を破壊するかのどちらかになります。
- **デュアルパーソナリティモデルをサポートできない**: 同じ GitLab.com インスタンスが、ダイレクトアクセスの namespace とファイアウォール制限のある namespace の両方を配信する必要があります。

**却下理由:** マルチテナントとデュアルパーソナリティの要件は、インスタンスレベルのモードだけでは提供できない、namespace 単位の柔軟性を要求するため。

### 代替案 3: hybrid モードでのリポジトリ単位設定

hybrid モードで、namespace レベルではなくリポジトリレベルの配信モード上書きを許容する。

**長所:**

- より細かい粒度での制御。namespace 配下のほとんどのリポジトリは redirect を使い、特定のリポジトリだけ proxy にする、といったことが可能。

**短所:**

- **問題に対するオーバーエンジニアリング**: ネットワーク制約は namespace のクライアント環境に対して適用されるものであり、個々のリポジトリに対するものではありません。クライアントがストレージに到達できないなら、その namespace のどのリポジトリにも到達できません。
- **管理オーバーヘッド**: 管理者はリポジトリごとに個別設定する必要があります。
- **一貫性のないユーザー体験**: 同じ namespace 内のリポジトリで挙動が異なるのは混乱を招きます。

**却下理由:** ネットワーク制約は通常、namespace／ネットワーク境界で適用されるもので、リポジトリレベルではありません。namespace 単位は hybrid モードに適切な粒度です。具体的なユーザー要望が出てきた場合は、namespace 設定を継承しつつリポジトリ単位で個別に上書きできる仕組みを将来検討できますが、現時点ではスコープ外です。
