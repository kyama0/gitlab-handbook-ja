## コンテキスト

ドメインモジュールおよびプラットフォームモジュールのスチュワードシップをどのように割り当てるか？コードのそのパートに対してビジョンと方向性を提供できる明示的なスチュワードがいない共有コードが大量に存在します。

## 決定事項

私たちは **オーナー（owners）** の代わりに **スチュワード（stewards）** という用語を使用することで、GitLab の原則である **誰でも貢献できる（everyone can contribute）** により沿うようにします。スチュワードはコードの管理者です。特定の機能がどのように設計され、なぜそうなっているかを理解しています。アーキテクチャの特性と制約を把握しています。ただし、変更を歓迎し、貢献者が成功できるよう導きます。

ドメインの境界付けられたコンテキストまたはプラットフォームモジュールのいずれであれ、モジュールには少なくとも1つのスチュワードグループが必要です。このグループはチーム名（または GitLab グループハンドル）にできます。オプションで、スチュワードのリストには個人の IC（Individual Contributor）エントリを含めることができます。

Packwerk パッケージを使用してモジュールを抽出する際は、`package.yml` でスチュワードシップを直接指定できます:

```yaml
metadata:
  stewards:
    - group::pipeline execution # チーム名
    - group::pipeline authoring # チーム名
    - @grzesiek  # IC
    - @ayufan    # IC
```

プラットフォームモジュール（例: `Gitlab::Redis`）については、すべてのプラットフォームコードが「共有」として分類されているため、専任スチュワードとしてチーム全体を持てない場合があります。ただし、チームメンバーは特定の機能のエキスパートとして自分自身を追加できます。

## 結果

コード上で定義されたスチュワードシップは非常に強力な効果をもたらします:

- CODEOWNERS のセクションをパッケージのメタデータから自動生成できます。
- Review Roulette や Suggested Reviews 機能がこのリストを第一優先として使用できます。
- エンジニアはスチュワードを簡単に特定し、早期に設計の議論ができます。
- モノリス内に存在する gem（`gems/`）は、Packwerk パッケージにラップされるべきであり、明示的なスチュワードを持つことで恩恵を受けられます。

## 代替案

Packwerk を採用する前のモジュール化の初期フェーズでは、オーナーシップの明示的な概念がありません。当初は各チームが自分たちが担当する境界付けられたコンテキストを把握していることに依存しています。プラットフォームモジュールの「共有コード」については、当初はメンテナーがスチュワードの役割を担うことを期待しています。

- メリット: トレーニー（trainee）メンテナーに明確な開発パスと目標を与えられます。現状では、成功したメンテナーになるために何を習得すべきかが不明確に感じられます。
- デメリット: 「共有」コードの量は非常に多く、特定の機能について誰が最もよく知っているかを理解するのが難しい状況が続いています。コードを gem に分離しても、明示的なオーナーシップの欠如は解決されません。
