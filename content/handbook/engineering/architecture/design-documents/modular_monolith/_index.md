## 概要

メインの [GitLab Rails](https://gitlab.com/gitlab-org/gitlab) プロジェクトは、[Ruby on Rails](https://rubyonrails.org/) フレームワークを使用して大規模なモノリシックアプリケーションとして実装されてきました。220 万行を超える Ruby コードを持ち、何百人ものエンジニアが毎日コントリビュートしています。

このアプリケーションは 10 年以上にわたって複雑さを増し続けてきました。モノリシックなアーキテクチャはこの間、私たちによく貢献してくれて、高い開発速度と優れたエンジニアリング生産性を維持することを可能にしてくれました。

私たちは[親しみやすいオープンコアアーキテクチャ](https://about.gitlab.com/blog/2022/07/14/open-core-is-worse-than-plugins/)を目指していますが、速度を維持し、開発の予測可能性を高めるためには、ドメイン間の境界を強化する必要があります。

フロントエンドについても同じことが言えます。Web UI は、単一の Webpack バンドルとしてビルド・出荷される、大規模で密結合した JavaScript と Vue のコードベースです。これはモノリスと運命を共にしています。すべてが一緒にコンパイル、デプロイ、リリースされ、UI の特定領域を他の部分から独立して進化させたり出荷したりする方法はありません。したがって分解には 2 つの相補的なトラックがあります。Rails アプリケーションをモジュール化する **バックエンド** トラックと、ビルドパイプラインと UI をモジュール化する **フロントエンド** トラックです。どちらも以下で説明する同じ目標と原則を共有しています。

### なぜ今なのか: エージェンティックの必然性

ソフトウェアの作り方は変化しつつあります。エージェンティック AI は、Issue が起票されるとすぐにエージェントが作業を引き受け、コードベースの明確に定義された領域内で独立して動作し、MR を開く前に自分自身の作業を検証するようになると、何が可能になるかを変えます。これを実現するには、モジュールに明確な境界、明示的な契約が必要であり、そのビルド方法や実行方法に暗黙知が組み込まれていてはなりません。

モノリスは、頭の中に何年分ものコンテキストを持つ人間の開発者向けに構築されました。エージェントにはそのようなコンテキストがありません。依存関係が暗黙的であり、安全に作業するために必要な知識がすべてリポジトリに存在するわけではないため、エージェントは全体について推論することなしに、ある領域を安全に変更することはできません。モジュール境界は、コードベースのある領域を、エージェント（そして人間）が単独で推論できるほど小さくします。

これはモジュール化の捉え方を再構築します。モジュール化はもはやエンジニアリング速度や認知負荷だけの話ではありません。それは大規模なエージェンティック開発のための構造的な前提条件です。明確な契約、自己完結したコンテキスト、高いテストカバレッジを持つモジュールは、エージェントが自律的に作業でき、修正がより早く顧客に届き、チームがモノリスの結合に足を引っ張られることなく自分のペースで動けるモジュールです。

### 私たちが目指しているもの

私たちは、サテライトサービスを伴う[モノリシックアーキテクチャ](https://en.wikipedia.org/wiki/Monolithic_application)を引き続き使用しながら、[モジュラーモノリス設計](https://en.wikipedia.org/wiki/Modular_programming)へと進化させたいと考えています。そして、それが見合うモジュールについては、GitLab.com、Dedicated、セルフマネージドインスタンスにおいて、単一の契約に対してモノリスと統合される独立してデプロイ可能なユニットへと進化させたいと考えています。

これにより、エンジニアリングの効率を高め、認知負荷を減らし、コードベースを AI エージェントにとって読みやすくし、最終的には、その投資に対するリターンが明確になったときに、内部コンポーネントを別々にデプロイ・実行できる程度まで分離できるようになるはずです。

フロントエンドでは、これは、共有シェルアプリケーションと明示的なモジュール契約を通じて統合される、独立してビルド・デプロイ可能な UI モジュールを意味します。これはバックエンドのバウンデッドコンテキストとパブリックインターフェースの視覚的な対応物です。

## モチベーション

大規模で密結合したモノリシックアプリケーションを扱うことには困難が伴います:

エンジニアリング:

- エンジニアのオンボーディングに時間がかかります。コンテキストの大きさと結合の量のため、エンジニアが生産的だと感じるまでに時間がかかります。
- いくつかのドメインで `CODEOWNERS` ファイル機能を使う必要がありますが、[これらのルールは複雑です](https://gitlab.com/gitlab-org/gitlab/-/blob/409228f064a950af8ff2cecdd138fc9da41c8e63/.gitlab/CODEOWNERS#L1396-1457)。
- アプリケーションの大きさのため、エンジニアがその全体像を頭の中に描くことは困難です。
  一見孤立した変更でさえ、モノリスの他の部分に[広範囲に及ぶ影響](../../../devops/#reducing-the-impact-of-far-reaching-work)を及ぼす可能性があります。
- エンジニアリング人材の離職・定着。生産性を阻む障害に絶えず対処することは、エンジニアにとって疲労を招き、士気を下げるものです。

アーキテクチャ:

- モノリス内部にはほとんど構造がありません。私たちは[いくつかのモジュール](https://gitlab.com/gitlab-org/gitlab/-/issues/212156)の作成を強制しようとしてきましたが、モノリスの機能的な構成要素がどうあるべきか、コードをどう整理すべきかについて、全社的な戦略を持っていません。
- 既存のモジュール間に隔離がありません。Ruby は境界を効果的に強制するためのツールをすぐに使える形では提供していません。すべてが同じメモリ空間の下に存在します。
- 効率を高められる抽象化を構築することはまれです。
- 境界はデータベース上で沈黙しています。一見孤立したドメインでさえ、テーブルを共有し、境界をまたぐ外部キーを保持し、境界線をまたぐ ORM のアソシエーションを保ち続けています。抽出を実際に阻んでいるのはデータの結合です。
- 結合度が高いため、アプリケーションの安定した部分を別々のサービスに移すことは不可能です。
- 特定のドメインへの変更を個別にデプロイしたり、その内部で発生している障害を隔離したりすることができません。

エージェンティック AI:

- エージェントには暗黙のコンテキストがありません。依存関係が暗黙的であり、変更の影響範囲が不明確であるため、エージェントは全体について推論することなしに、ある領域を安全に変更することはできません。
- ドメインで作業するために必要な知識（その境界、契約、不変条件、ランブック）は、すべてがリポジトリに存在するわけではありません。
- 明確な境界と自己完結したコンテキストがなければ、エージェントは Issue を引き受け、変更を加え、それを検証し、MR を開くことを、人間が暗黙知を提供することなしには行えません。

顧客価値:

- セルフマネージドおよび Dedicated の顧客は、アップグレードするまで修正を目にすることができず、それが数か月後になることもあります。私たちは、モノリス全体のリリースを出荷することなしに、単一の機能に対する的を絞った修正を展開することができません。
- 1 つのドメインの障害がアプリケーション全体を劣化させる可能性があり、ホットなドメインを他の部分から独立してスケールすることができません。独立したモジュールは別々にスケール・隔離でき、信頼性と耐障害性を向上させます。

生産性:

- 複雑な変更については本番反映までの中央値時間が長いです。
- より広いコミュニティのメンバーがコントリビュートすることは、圧倒されるような体験になりかねません。
- テスト時間を短縮するには、入念で粘り強い努力が必要です。

## 目標

- 関心の分離を通じて、開発速度と予測可能性を高めます。
- 結合を減らし、有用な抽象化を導入することで、コード品質を向上させます。
- モジュールをエージェントレディにします。明確な境界、明示的な契約、そしてエージェントが自律的に作業するために必要なすべてのコンテキストをリポジトリにコミットします。
- 依存関係を明示的にし、強制します。宣言されていない依存関係は存在しません。結合を裏口から再生成することはできません。
- 広くモジュール化し、選択的に抽出します。ほとんどのコードはモノリス内のその場でモジュール化され、モジュールが別々にデプロイされるサービスへと抽出されるのは、ROI が明確な場合に限られます。
- 抽出が正当化されるときにそれを安価にする抽象化（データ所有権、独立した運用性、独立したデプロイ）を構築し、修正がモノリス全体のリリースなしに顧客に届くようにします。

## バックエンドの分解

バックエンドトラックは、[GitLab Rails](https://gitlab.com/gitlab-org/gitlab) アプリケーションを、明示的で強制される境界を持つバウンデッドコンテキストへとモジュール化し、独立して運用可能であり、ROI が明確な場合には独立してデプロイ可能なドメインへと進化させていきます。

### どうやってそこへたどり着くのか?

モジュール化が重要な技術的取り組みであることは認識していますが、私たちは主な課題が技術的なものよりむしろ組織的なものだと考えています。私たちは、モジュールが実用的な方法で疎結合になるように分離を設計する必要があるだけでなく、GitLab で私たちが働きたいと考えている方法にモジュール化を一致させる必要があります。

モノリスのモジュール化を成功させるには、多くの側面と詳細が必要です。私たちは以下に挙げる側面に取り組み、それらを洗練させ、目標に向かって進む中でさらに重要な詳細を追加していきます:

1. [重要な洞察をもたらすモジュール化の概念実証を提供する](proof_of_concepts.md)。
1. [バウンデッドコンテキストを定義する](bounded_contexts.md)ことで、モジュール化計画を製品構造に一致させる。
1. [横断的なライブラリを gem に抽出し](library_extraction.md)、プラットフォームコードを `lib/`/`ee/lib` から引き出す。価値は抽出パターン、ツール、強制された隔離にあり、節約される CI 時間にあるのではありません。
1. [トランスポート層をアダプターに分解し](transport_layer.md)、Web、REST、GraphQL、Sidekiq の各トランスポートを隔離して、それぞれがドメイン層のみに依存するようにする。これにより、API のみや Sidekiq のみのノードといったランタイムプロファイルが可能になります。
1. [ドメイン層を隔離し](domain_layer.md)、各ドメインが自身のデータを所有し、そのパブリック API を通じてのみアクセスされるようにする。
1. 疎結合化されたドメインの扱い方についてのチームメンバー向けトレーニングプログラムを開始する (TODO)
1. 制御の反転を通じて疎結合化されたドメインを構築しやすくするツールを構築する (TODO)
1. [モノリス内にヘキサゴナルアーキテクチャを導入する](hexagonal_monolith/index.md)
1. 一方向の依存関係とホストアプリケーションを伴うクリーンアーキテクチャを導入する (TODO)
1. ドメインを別々に実行・デプロイできるようにする抽象化を構築する (TODO)

### 用語集

- `modules` は Ruby のモジュールであり、コードを階層的にネストするために使用できます。
- `namespaces` は Ruby の定数の一意な階層です。例えば、`Ci::` だけでなく `Ci::JobArtifacts::` や `Ci::Pipeline::Chain::` などです。
- `packages` は関連する機能をまとめるための Packwerk パッケージです。これらのパッケージは、設計とアーキテクチャに応じて大きくも小さくもなり得ます。パッケージ内では、すべての定数（クラスとモジュール）が同じ名前空間を持ちます。例えば:
  - パッケージ `ci` では、すべてのクラスが `Ci::` 名前空間の下にネストされます。`Ci::PipelineProcessing::` のようなネストされた名前空間も存在し得ます。
  - パッケージ `ci-pipeline_creation` では、すべてのクラスが `Ci::PipelineCreation` の下にネストされます。例えば `Ci::PipelineCreation::Chain::Command` です。
  - パッケージ `ci` では、`MergeRequests::UpdateHeadPipelineService` という名前のクラスは、パッケージの名前空間と一致しないため許可されません。
  - これは [Packwerk ベースの RuboCop Cop](https://github.com/rubyatscale/rubocop-packs/blob/main/lib/rubocop/cop/packs/root_namespace_is_pack_name.rb) で簡単に強制できます。
- `bounded context` は、ドメインのマクロな側面を表すトップレベルの Packwerk パッケージです。例えば: `Ci::`、`MergeRequests::`、`Packages::` などです。
  - バウンデッドコンテキストは単一の Ruby モジュール/名前空間で表されます。例えば、`Ci::JobArtifacts::` ではなく `Ci::` です。
  - バウンデッドコンテキストは 1 つまたは複数の Packwerk パッケージで構成できます。ドメインが非常に複雑で、すべての実装の詳細の間でプライバシーを強制したい場合には、ネストされたパッケージが推奨されます。例えば、`Ci::PipelineProcessing::` と `Ci::PipelineCreation::` は同じバウンデッドコンテキストの別々のパッケージとして、実装の詳細をプライベートに保ちながらパブリック API を公開できます。
  - `RemoteDevelopment::` のような新しいバウンデッドコンテキストは単一のパッケージで表せますが、`Ci::` のような大規模で複雑なバウンデッドコンテキストは、より小さく/ネストされたパッケージへと整理する必要があります。

## フロントエンドの分解

目標は、独立してビルド可能な UI モジュールが、共有シェルアプリケーションと明示的なモジュール契約を通じて統合される、モジュラーなフロントエンドです。これはバックエンドのバウンデッドコンテキストとパブリックインターフェースを反映したものです。

フロントエンドのワークストリームは以下のとおりです:

1. **ビルドパイプラインをモダナイズ** して、独立したフロントエンドモジュールのデプロイを可能にします。
1. **Webpack 4 から Webpack 5 / Vite へ移行** し、これによってブロックが解除される Vue 3 への移行を完了します。
1. モジュール境界をまたいで認可、ユーザーコンテキスト、フィーチャーフラグを管理する **モジュール契約を定義** します。
1. **サポートするインフラストラクチャを構築** します。シェルアプリケーション、モジュールレジストリ、Context プロバイダー API です。
1. **「Frontend LabKit」を作成** して、共有ライブラリを標準化し、モジュール間で一貫したユーザー体験を保証します。

詳細な設計については [フロントエンドの分解](frontend/_index.md) を参照してください。
