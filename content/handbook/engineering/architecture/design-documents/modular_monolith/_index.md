## 概要 {#summary}

メインの [GitLab Rails](https://gitlab.com/gitlab-org/gitlab) プロジェクトは、[Ruby on Rails](https://rubyonrails.org/) フレームワークを使った大規模なモノリシックアプリケーションとして実装されてきました。220 万行を超える Ruby コードを擁し、毎日数百名のエンジニアがこれにコントリビュートしています。

このアプリケーションは 10 年以上にわたって複雑さを増し続けてきました。この間、モノリシックアーキテクチャは私たちに大いに役立ち、高い開発速度と優れたエンジニアリング生産性を維持することを可能にしてきました。

私たちは [親しみやすいオープンコアアーキテクチャ](https://about.gitlab.com/blog/2022/07/14/open-core-is-worse-than-plugins/) を目指していますが、開発速度を維持し開発の予測可能性を高めるためには、ドメイン間の境界を強化する必要があります。

これはフロントエンドについても同様です。Web UI は大規模で密結合した JavaScript と Vue のコードベースであり、単一の Webpack バンドルとしてビルド・出荷されています。これはモノリスと運命を共にしています。すべてがまとめてコンパイル、デプロイ、リリースされ、UI のある領域を他の領域から独立して進化させたり出荷したりする方法がありません。したがって分解には 2 つの補完的なトラックがあります。Rails アプリケーションをモジュール化する **バックエンド** トラックと、ビルドパイプラインと UI をモジュール化する **フロントエンド** トラックです。どちらも以下で説明する同じゴールと原則を共有します。

### なぜ今なのか: エージェンティックの必然性 {#why-now-the-agentic-imperative}

ソフトウェアの作られ方が変化しています。エージェンティック AI は、Issue が起票されるとすぐにエージェントが作業を引き受け、コードベースの明確に定義された領域内で独立して動作し、MR を開く前に自身の作業を検証するようになると、何が可能になるかを変えます。これを実現するには、モジュールに明確な境界と明示的な契約が必要で、そのモジュールがどのように構築・実行されるかに暗黙知が組み込まれていてはいけません。

モノリスは、何年もの文脈を頭の中に持つ人間の開発者向けに作られました。エージェントにはそのような文脈がありません。依存関係が暗黙的であり、安全に作業するために必要な知識がすべてリポジトリ内にあるわけではないため、エージェントは全体について推論することなしにある領域を安全に変更することはできません。モジュラーな境界は、コードベースのある領域を、エージェント（そして人間）が単独で推論できるほど小さくします。

これはモジュール化を捉え直すものです。それはもはやエンジニアリング速度と認知負荷だけの問題ではありません。それは大規模なエージェンティック開発のための構造的な前提条件です。明確な契約、自己完結した文脈、高いテストカバレッジを備えたモジュールは、エージェントが自律的に作業でき、修正がより早く顧客に届き、チームがモノリスの結合に足を引っ張られることなく自分たちのペースで進められるモジュールです。

### 私たちが目指すもの {#what-we-are-building-toward}

私たちは [モジュラーモノリス設計](https://en.wikipedia.org/wiki/Modular_programming) へと進化させたいと考えています。その一方で、サテライトサービスを伴う [モノリシックアーキテクチャ](https://en.wikipedia.org/wiki/Monolithic_application) を引き続き使用しつつ、投資が報われるモジュールについては、GitLab.com 上でも、Dedicated 上でも、self-managed インスタンス上でも、単一の契約に対してモノリスと統合する独立デプロイ可能なユニットへと進化させたいと考えています。

これにより、エンジニアリング効率を高め、認知負荷を減らし、コードベースを AI エージェントにとって読み取り可能なものにし、最終的には内部コンポーネントを、その投資に対するリターンが明確なときに別々にデプロイ・実行できる程度まで疎結合化できるはずです。

フロントエンドでは、これは共有シェルアプリケーションと明示的なモジュール契約を通じて統合される、独立してビルド・デプロイ可能な UI モジュールを意味します。これはバックエンドの境界づけられたコンテキストと公開インターフェイスの、視覚的な対応物です。

## 動機 {#motivation}

大規模で密結合したモノリシックアプリケーションを扱うことには困難が伴います。

エンジニアリング:

- エンジニアのオンボーディングに時間がかかります。文脈の大きさと結合の量のため、エンジニアが生産的だと感じられるようになるまでにしばらくかかります。
- 私たちはいくつかのドメインで `CODEOWNERS` ファイル機能を使う必要がありますが、[これらのルールは複雑です](https://gitlab.com/gitlab-org/gitlab/-/blob/409228f064a950af8ff2cecdd138fc9da41c8e63/.gitlab/CODEOWNERS#L1396-1457)。
- そのサイズのため、エンジニアがアプリケーションのメンタルマップを構築するのは困難です。一見すると孤立した変更でさえ、モノリスの他の部分に [広範囲に及ぶ影響](../../../devops/#reducing-the-impact-of-far-reaching-work) を及ぼすことがあります。
- エンジニアリング人材の離職／定着。生産性に対する障害に絶えず対処することは、エンジニアにとって疲弊し士気を下げるものです。

アーキテクチャ:

- モノリス内部にはほとんど構造がありません。私たちは [いくつかのモジュール](https://gitlab.com/gitlab-org/gitlab/-/issues/212156) の作成を強制しようと試みてきましたが、モノリスの機能的な部分がどうあるべきか、コードをどう構成すべきかについての全社的な戦略がありません。
- 既存のモジュール間に分離がありません。Ruby は境界を効果的に強制するためのツールをすぐに使える形では提供していません。すべてが同じメモリ空間の下に存在します。
- 効率を高めうる抽象化を私たちはめったに構築しません。
- データベース上では境界が暗黙的です。一見すると孤立したドメインでさえ、テーブルを共有し、境界をまたぐ外部キーを持ち、線をまたぐ ORM の関連を保持しています。抽出を実際にブロックしているのはデータの結合です。
- 結合度が高いため、アプリケーションの安定した部分を別々のサービスに移すことは不可能です。
- 特定のドメインへの変更を別々にデプロイしたり、その内部で発生している障害を分離したりすることができません。

エージェンティック AI:

- エージェントには暗黙の文脈がありません。依存関係が暗黙的であり、変更の影響範囲が不明確であるため、エージェントは全体について推論することなしにある領域を安全に変更することができません。
- ドメイン内で作業するために必要な知識（その境界、契約、不変条件、ランブック）は、すべてがリポジトリ内にあるわけではありません。
- 明確な境界と自己完結した文脈がなければ、エージェントは人間が暗黙知を提供することなしに、Issue を引き受け、変更を加え、それを検証し、MR を開くことができません。

顧客価値:

- self-managed および Dedicated の顧客は、アップグレードするまで（ときには数ヶ月後まで）修正を目にすることができません。私たちはモノリス全体のリリースを出荷することなしに、単一の機能に的を絞った修正をロールアウトすることができません。
- 1 つのドメインの障害がアプリケーション全体を劣化させることがあり、ホットなドメインを他から独立してスケールさせることができません。独立したモジュールは別々にスケール・分離でき、信頼性とフォールトトレランスを向上させます。

生産性:

- 複雑な変更の本番投入までの中央値が長くなっています。
- 広いコミュニティのメンバーがコントリビュートするのは大変なことがあります。
- テスト時間の短縮には、入念で粘り強い努力が必要です。

## ゴール {#goals}

- 関心の分離を通じて開発速度と予測可能性を高めます。
- 結合を減らし有用な抽象化を導入することでコード品質を改善します。
- モジュールをエージェント対応にします。明確な境界、明示的な契約、そしてエージェントが自律的に作業するために必要なすべての文脈をリポジトリにコミットします。
- 依存関係を明示的かつ強制されたものにします。宣言されていない依存関係は存在せず、結合を裏口から再生成することはできません。
- 幅広くモジュール化し、選択的に抽出します。ほとんどのコードはモノリス内でその場でモジュール化され、モジュールが別々にデプロイされるサービスへ抽出されるのは ROI が明確なときだけです。
- 正当化されるときに抽出を安価にする抽象化を構築します。データの所有権、独立した運用性、独立したデプロイです。これにより、モノリス全体のリリースなしに修正を顧客に届けられます。

## バックエンドの分解 {#backend-decomposition}

バックエンドトラックは、[GitLab Rails](https://gitlab.com/gitlab-org/gitlab) アプリケーションを、明示的かつ強制された境界を持つ境界づけられたコンテキストへとモジュール化し、独立して運用可能なドメイン、そして ROI が明確な場合には独立してデプロイ可能なドメインへと進めていきます。

### どうやってそこに到達するか? {#how-do-we-get-there}

モジュール化が重要な技術的取り組みであることは認識していますが、私たちは主な課題が技術的というより組織的なものだと考えています。私たちはモジュールが実用的な形で疎結合になるように分離を設計する必要があるだけでなく、モジュール化を GitLab での働き方と整合させる必要があります。

モノリスのモジュール化を成功させるために必要な側面や詳細は数多くあります。私たちは以下に挙げる側面に取り組み、それらを洗練させ、ゴールに向かって進むにつれてさらに重要な詳細を追加していきます。

1. [重要な洞察をもたらすモジュール化の概念実証を提供する](proof_of_concepts.md)。
1. [境界づけられたコンテキストを定義する](bounded_contexts.md) ことで、モジュール化計画をプロダクト構造に整合させる。
1. [横断的なライブラリを gem へ抽出する](library_extraction.md)。プラットフォームコードを `lib/`／`ee/lib` から取り出します。価値は抽出のパターン、ツール、強制された分離にあり、節約される CI 時間ではありません。
1. [トランスポート層をアダプターへ分解する](transport_layer.md)。Web、REST、GraphQL、Sidekiq の各トランスポートを分離し、それぞれがドメイン層のみに依存するようにします。これにより、API 専用ノードや Sidekiq 専用ノードといったランタイムプロファイルが可能になります。
1. [ドメイン層を分離する](domain_layer.md)。各ドメインが自身のデータを所有し、その公開 API を通じてのみ到達されるようにします。
1. 疎結合なドメインを扱う方法についてチームメンバー向けのトレーニングプログラムを開始する（TODO）
1. 制御の反転を通じて疎結合なドメインを構築しやすくするツールを構築する（TODO）
1. [モノリス内にヘキサゴナルアーキテクチャを導入する](hexagonal_monolith/index.md)

### 用語集 {#glossary}

- `modules` は Ruby のモジュールであり、コードを階層的にネストするために使えます。
- `namespaces` は Ruby 定数の一意な階層です。例えば `Ci::` ですが、`Ci::JobArtifacts::` や `Ci::Pipeline::Chain::` もそうです。
- `packages` は関連する機能をまとめるための Packwerk パッケージです。これらのパッケージは、設計とアーキテクチャ次第で大きくも小さくもなります。1 つのパッケージ内では、すべての定数（クラスとモジュール）が同じ名前空間を持ちます。例えば:
  - `ci` パッケージでは、すべてのクラスが `Ci::` 名前空間の下にネストされます。`Ci::PipelineProcessing::` のようなネストされた名前空間もありえます。
  - `ci-pipeline_creation` パッケージでは、すべてのクラスが `Ci::PipelineCreation` の下にネストされ、`Ci::PipelineCreation::Chain::Command` のようになります。
  - `ci` パッケージでは、`MergeRequests::UpdateHeadPipelineService` という名前のクラスは、パッケージの名前空間に一致しないため許可されません。
  - これは [Packwerk ベースの RuboCop Cops](https://github.com/rubyatscale/rubocop-packs/blob/main/lib/rubocop/cop/packs/root_namespace_is_pack_name.rb) で簡単に強制できます。
- `bounded context` は、ドメインのマクロな側面を表すトップレベルの Packwerk パッケージです。例えば `Ci::`、`MergeRequests::`、`Packages::` などです。
  - 境界づけられたコンテキストは、単一の Ruby モジュール／名前空間によって表されます。例えば `Ci::JobArtifacts::` ではなく `Ci::` です。
  - 境界づけられたコンテキストは、1 つまたは複数の Packwerk パッケージで構成できます。ドメインがかなり複雑で、すべての実装詳細の間でプライバシーを強制したい場合は、ネストされたパッケージが推奨されます。例えば `Ci::PipelineProcessing::` と `Ci::PipelineCreation::` は、同じ境界づけられたコンテキストの別々のパッケージとして、公開 API を公開しつつ実装詳細を非公開に保つことができます。
  - `RemoteDevelopment::` のような新しい境界づけられたコンテキストは単一のパッケージで表現できますが、`Ci::` のような大規模で複雑な境界づけられたコンテキストは、より小さな／ネストされたパッケージへと整理する必要があります。

## フロントエンドの分解 {#frontend-decomposition}

ゴールは、独立してビルド可能な UI モジュールが共有シェルアプリケーションと明示的なモジュール契約を通じて統合される、モジュラーなフロントエンドです。これはバックエンドの境界づけられたコンテキストと公開インターフェイスを反映したものです。

フロントエンドのワークストリームは以下のとおりです。

1. 独立したフロントエンドモジュールのデプロイを可能にするために **ビルドパイプラインをモダナイズする**。
1. **Webpack 4 から Webpack 5 / Vite へ移行し**、これによってブロックが解除される Vue 3 への移行を完了する。
1. モジュール境界をまたいで認可、ユーザー文脈、フィーチャーフラグを管理する **モジュール契約を定義する**。
1. **サポートインフラストラクチャを構築する**。シェルアプリケーション、モジュールレジストリ、Context プロバイダー API です。
1. 共有ライブラリを標準化し、モジュール間で一貫したユーザー体験を保証するために **「Frontend LabKit」を作成する**。

詳細な設計については [フロントエンドの分解](frontend/_index.md) を参照してください。
