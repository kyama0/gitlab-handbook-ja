## 概要

GitLab のフロントエンドは、同時にビルドされる複数の Webpack バンドルとしてビルドされ、
出荷されます。このトラックは、UI モジュールが独立して開発、ビルド、テスト、デプロイでき、
共有のシェルアプリケーションと明示的なモジュールコントラクトを通じて統合されるように
フロントエンドをモジュール化します — バックエンドの
[境界づけられたコンテキスト](../bounded_contexts.md)に対応するフロントエンド版です。

このドキュメントは初期のスケルトンです。以下のワークストリームは、フロントエンドチームが
拡張するためのプレースホルダーです。これは、分解の取り組み全体の
[動機](../_index.md#motivation)、
[エージェンティックの必然性](../_index.md#why-now-the-agentic-imperative)、
[ゴール](../_index.md#goals)を共有します。

## 動機

GitLab のフロントエンドには、より速い開発イテレーションを妨げる一連の問題があります。

- **Rails への密結合。** 約 440 のファイルが Vue アプリをハイドレートするために
  `el.dataset` を読みます。77 のファイルが `window.gon` を読みます。フィーチャーフラグは
  完全に `window.gon.features` 経由で注入され、API による代替手段がありません。これにより、
  フロントエンドは Rails のレンダリングパイプラインから切り離せなくなっています。
- **CSS アーキテクチャの不在。** 223 のファイルにまたがる 36,000 行の SCSS には、分離も、
  共置（co-location）も、オーナーシップもありません。適切な CSS の書き方についての
  ガイドラインがありません。スタイルは、それが影響する HTML から遠く離れた場所に存在し、
  カスケードに大きく依存しています。いずれかの SCSS ファイルを変更すると、影響範囲が
  不明であるため、すべての統合テストを実行せざるを得ません。Tailwind はレガシー CSS の
  上に載せられており、複雑さとメンテナンスを増大させています。
- **CI のコストと時間。** フロントエンドの予測的テスト選択は存在しますが、ほとんど
  効果がありません。大半のフロントエンド MR は、フル E2E スイートの実行へフォールバック
  します。RSpec のシステムスペックには、フロントエンドの変更に対するカバレッジベースの
  マッピングがなく、静的なファイルパターンマッチングだけです。フィクスチャ生成（8 つの
  並列 RSpec ジョブ）は、予測的選択を完全に無視します。
- **アプリ間の調整。** アプリ間の通信は、形式的なコントラクトのない 3 つの場当たり的な
  パターン（mitt ベースの EventHub、DOM の CustomEvent、Pinia/Apollo Client キャッシュ/Vue
  リアクティブ変数を介した状態の共有）に依存しています。
- **外部サービス向けのバンドルサイズのリスク。** モジュールフェデレーションやランタイムの
  モジュール共有は存在しません。各外部サービスは、Vue、GitLab UI、Apollo の重複した
  コピーを出荷することになります。

## ゴール

1. **明示的な依存関係。** 2 つのコードの間の関係がツールで追跡できないなら、テスト、
   ビルド、ロードの目的上、それは存在しません。これは JS、CSS、Rails からのデータ、
   アプリ間通信に適用されます。
2. **Rails の注入よりも API コントラクト。** データは `el.dataset` や `window.gon` では
   なく、GraphQL/REST から来るべきです。フィーチャーフラグは API を通じて利用できるべき
   です。
3. **スコープ付きスタイル。** CSS は、それを使うコンポーネントと共置され、そこへ追跡
   できなければならず、ビルドグラフの一部になります。
4. **速くて信頼できる CI。** - フロントエンドのみの変更は、起こり得るリグレッションを
   検証するために、可能な限り最小のテストの部分集合を実行すべきです。
