## 概要

GitLab のフロントエンドは、同時にビルドされる複数の Webpack バンドルとして構築・出荷されています。このトラックは、
UI モジュールを独立して開発、ビルド、テスト、デプロイできるようにフロントエンドをモジュール化し、共有のシェルアプリケーションと
明示的なモジュール契約を通じて統合します。これは、バックエンドの [bounded context](../bounded_contexts.md) に対応する
フロントエンド版です。

このドキュメントは初期のスケルトンです。以下のワークストリームは、フロントエンドチームによって拡張される予定の
プレースホルダーです。分解の取り組み全体の [モチベーション](../_index.md#motivation)、
[エージェント駆動の必然性](../_index.md#why-now-the-agentic-imperative)、
[目標](../_index.md#goals) を共有しています。

## モチベーション

GitLab のフロントエンドには、より速い開発イテレーションを妨げる一連の問題があります。

- **Rails への密結合。** 約 440 個のファイルが Vue アプリをハイドレートするために `el.dataset` を読み込んでいます。77 個のファイルが `window.gon` を読み込んでいます。フィーチャーフラグは API の代替手段なしに、完全に `window.gon.features` を介して注入されています。これにより、フロントエンドが Rails のレンダリングパイプラインから切り離せなくなっています。
- **CSS アーキテクチャの欠如。** 223 個のファイルにまたがる 36,000 行の SCSS には、分離も、コロケーションも、所有権もありません。適切な CSS の書き方に関するガイドラインがありません。スタイルは、それが影響する HTML から遠く離れた場所に存在し、カスケードに大きく依存しています。影響範囲が不明なため、いずれかの SCSS ファイルを変更すると、すべての統合テストを実行することを強いられます。Tailwind はレガシー CSS の上に置かれており、複雑さとメンテナンスを増大させています。
- **CI のコストと時間。** フロントエンドの予測的テスト選択は存在しますが、ほとんど効果がありません。大半のフロントエンド MR は、フルの E2E スイートの実行にフォールバックします。RSpec のシステムスペックには、フロントエンド変更のためのカバレッジベースのマッピングがなく、静的なファイルパターンマッチングのみです。フィクスチャ生成（8 個の並列 RSpec ジョブ）は、予測的選択を完全に無視しています。
- **アプリ間の調整。** アプリ間通信は、正式な契約を持たない 3 つのその場しのぎのパターン（mitt ベースの EventHub、DOM の CustomEvent、Pinia/Apollo Client キャッシュ/Vue リアクティブ変数を介した状態の共有）に依存しています。
- **外部サービスのバンドルサイズリスク。** モジュールフェデレーションも、ランタイムでのモジュール共有も存在しません。各外部サービスは、Vue、GitLab UI、Apollo の重複したコピーを出荷することになります。

## 目標

1. **明示的な依存関係。** 2 つのコード片の関係がツールで追跡できない場合、それはテスト、ビルド、ロードの目的において存在しないものとみなされます。これは JS、CSS、Rails からのデータ、アプリ間通信に適用されます。
2. **Rails の注入よりも API 契約。** データは `el.dataset` や `window.gon` ではなく、GraphQL/REST から来るべきです。フィーチャーフラグは API を通じて利用可能であるべきです。
3. **スコープ化されたスタイル。** CSS は、それを使用するコンポーネントとコロケーションされ、追跡可能でなければならず、ビルドグラフの一部にする必要があります。
4. **速くて信頼できる CI。** - フロントエンドのみの変更では、起こりうるリグレッションを検証するために、可能な限り最小のテストのサブセットを実行すべきです。
