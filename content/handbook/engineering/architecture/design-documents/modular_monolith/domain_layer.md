このページは、**アプリケーションドメイン**（[ヘキサゴナルモノリス](hexagonal_monolith/index.md) の中核にあるビジネスロジック）を、境界づけられた独立所有のモジュールへと分離することについての唯一の情報源です。

このページはドメインコードのみを扱います。横断的な **プラットフォーム** コードの抽出については [横断的なライブラリを gem へ抽出する](library_extraction.md) で、**トランスポート層**（Web、REST、GraphQL、Sidekiq）の抽出については [トランスポート層をアダプターへ分解する](transport_layer.md) で扱います。ドメインがどのように識別され命名されるかについては [境界づけられたコンテキストを定義する](bounded_contexts.md) で扱います。

ドメインの分離はモジュール化の最も難しい部分です。プラットフォーム層とトランスポート層には比較的きれいな継ぎ目がありますが、結合が実際に存在するのはドメインです。

## ドメインの分離とは何を意味するか {#what-domain-isolation-means}

ドメインは、次の条件を満たすときに分離されています。

- **公開 API を通じる以外、その境界を何も越えない。** 他のドメインは、文書化され意図された 1 つのインターフェイスを呼び出します。決して内部や AR モデル、プライベートなサービスに手を伸ばすことはありません。
- **自身のデータを排他的に所有する。** ドメインは自身のデータベーステーブルと、それらをマッピングする ActiveRecord モデルを所有します。他のドメインがそれらのテーブルをクエリしたり、それらのモデルへの参照を保持したりすることはありません。

よく設計されたドメインモジュールはまた **深く**（その小さなインターフェイスの背後に大量の内部ロジック、状態、データをカプセル化し）、**凝集的** であり、それが記述する機能の唯一の情報源として機能します。それは [名前空間の命名に関するガイドライン](https://docs.gitlab.com/ee/development/software_design.html#use-namespaces-to-define-bounded-contexts) に従い、CRUD 用語ではなく [ユビキタス言語](https://docs.gitlab.com/ee/development/software_design.html#use-ubiquitous-language-instead-of-crud-terminology) を使います。ドメインがどのように識別され機能カテゴリーにマッピングされるかについては [境界づけられたコンテキストを定義する](bounded_contexts.md) で扱います。

以下のすべては、そこに到達するために必要な作業についてのものです。

## 難しい部分: ドメイン間の結合 {#the-hard-part-cross-domain-coupling}

私たちのモジュラーモノリスの研究は、これを難しくしている結合を浮き彫りにしました。現在のコードベースでは:

- **ポリシーが他のポリシーに依存している** — 例えば `PipelinePolicy` は `ProjectPolicy` に委譲しています。
- **サービスが共有コンポーネントに依存している** — イベントの発行、メールの送信、システムノート、監査などです。
- **ドメインが互いの AR モデルに依存している**。明示的（`Ci::Pipeline.find_by_id`）にも、関連を通じて暗黙的（`security_scan.pipeline`）にも依存しています。
- ドメイン間の **循環依存** がよく見られます。

これが、ドメインの分離が、ライブラリとトランスポートの抽出が大部分そうであるような機械的なファイル移動にはなりえない理由です。労力のほとんどは、コードの移動ではなく *結合を取り除くリファクタリング* です。

## データの所有 {#owning-the-data}

排他的なデータの所有は分離の核心です。各ドメインは自身のテーブルと、それらをマッピングする AR モデルを所有し、それらをクエリできる唯一のコードです。

これに伴って 2 つの強制上の問題が生じます。

**ドメイン間の AR 関連。** 今日、どのドメインも `project.ci_pipelines` を呼び出して、CI のデータへ直接入り込めます。分離するには、ドメイン間の関連を取り除き、代わりに所有するドメインの公開 API を経由する必要があります。`project.ci_pipelines` ではなく `Ci::Pipeline.all(project)` です。その前提条件は [全知クラスを手なずける](https://docs.gitlab.com/development/software_design/#taming-omniscient-classes) ことです。`Project` や `User` のようなクラスはあらゆるドメインから関連やメソッドを積み重ねており、スリムにする必要があります。

**モデルへの直接参照。** 関連がなくなったとしても、どんなコードでも `Ci::Pipeline` を直接参照できます。1 つの選択肢は、AR モデルをドメインに対してプライベートに保つこと（例えば `Ci::Internal::Pipeline`）で、外部から参照できないようにし、パイプラインのデータを `Ci::*` の公開インターフェイスを通じてのみ公開することです。

### リポジトリパターンの選択肢 {#repository-pattern-option}

より構造化されたバリエーションは、AR をドメインオブジェクトとは分離して、厳密に永続化／リポジトリ層としてのみ使うものです。

```ruby
# Persistence only — scopes and AR persistence, nothing else. Private to the domain.
Ci::Repository::Pipeline

# Domain object wrapping the record, exposing behaviour. The public type.
Ci::Pipeline   # #builds, #cancelable?, ...
```

AR モデルは、そのドメインオブジェクトの外で使われることは決してありません。これにより、テストでドメインを簡単にスタブ化でき、呼び出し元から AR への依存を取り除けます。コストは大きく、最終的に gem を抽出するかどうかにかかわらず、大規模で侵襲的なリファクタリングになります。

## ドメイン間のデータと共有モデル {#cross-domain-data-and-shared-models}

最も厄介な問題は、`Project` や `User` のような、ほぼすべてのドメインが触れる共有エンティティに何が起こるかです。

**共有モデルをアイデンティティオブジェクトに削ぎ落とす。** ドメイン間のアクセサが完全な `Project` AR オブジェクトを返すと、そのオブジェクトの表面全体を引きずり込み、ドメインを再び結合させてしまいます。共有データは、完全なモデルではなく、最小限のアイデンティティオブジェクトに削減すべきです。

**ドメイン固有の表現。** すべてのドメインが共有の `Project` を消費するのではなく、ドメインは自身が必要とするデータを自身のファサードの背後でラップできます。

```ruby
project = Ci::Internal::Project.new(project_id) # PORO or record local to the CI domain
project.variables       # internal to CI
project.public_builds?  # CI-specific project setting
project.pipelines       # pipelines queried via Ci::Project, not Project
```

これにより、`project.pipelines` をグローバルな `Project` 上ではなく、CI ドメイン内に保てます。関連するアイデアとして、ドメインごとの所有者オブジェクトがあります。例えば `user_id` をラップし、アイデンティティ固有の関心事についてのみ `User` に委譲する `Ci::JobOwner` です。

**モノリスのコールバックよりも直接的な依存関係を優先する。** ドメインが別のドメインからデータを必要とする場合、コールバックをモノリスへ配線し直すよりも、そのドメインの公開インターフェイスへの直接的で宣言された依存関係を優先します。直接的な依存関係は追跡可能です。私たちはドメイン間の依存グラフを描画でき、CI でそれを使って影響を受けるテストだけを実行できます。一方、コールバックは結合を隠してしまいます。

**後で gRPC に置き換え可能な、安定したクライアントインターフェイス。** ドメインは最初から外部のものとして扱えます。Ruby のクライアントインターフェイスを公開し、それ以外のすべてをプライベートに保ちます。そのインターフェイスの背後にあるバックエンドは、呼び出し元を変更することなく、後で gRPC へ移行できます。

**おしゃべりなドメインを避ける。** 境界が強制されると、互いに依存しすぎているコンポーネントが見つかるでしょう。Rails はすべてを常に利用可能にするため、単一のビジネストランザクションが多くのドメインにまたがって手を伸ばします。おしゃべりなドメイン間呼び出し（ドメインがサービスになると現実の問題になります）を避けるとは、データを非正規化または複製し、ローカルの表現を保ち、インターフェイス分離を適用することを意味します。例えば、完全な `User` ではなく、基本的なアイデンティティオブジェクトをラップするスリムな `Ci::Internal::User` です。

> すべてのドメインが必要とする共有の「足場」（`Project`／`User` のルックアップ、権限チェックなど）は、ドメインごとに注入されるのではなく、代わりにすべてのドメインが依存する粗粒度の `gitlab-platform`／`gitlab-core` gem に存在するかもしれません。そのトレードオフは [ライブラリ抽出ページ上のオープンな問い](library_extraction.md#open-questions) として追跡されています。

## 抽出の前に解決しなければならないこと {#what-must-be-resolved-before-extraction}

ドメインの分離は、ほとんどが未解決の設計作業です。ドメインを gem **または** パッケージへ抽出する前に、リファクタリングがエンジニアごと・エージェントごとの解釈に委ねられないよう、少なくとも以下について再現可能なガイドラインが必要です。

1. ドメイン間の **AR 関連** に何が起こるか。
1. どのロジックが **AR モデル** 上に存在し、どのロジックが **ドメインオブジェクト** 上に存在するか。
1. どのデータが **境界を越える** ことを許され、どのような形で越えるか。
1. 呼び出し元がデータオブジェクトを受け取ったものの、**ドメインオブジェクト** 上のメソッド（例えば `Ci::Pipeline#merge_request`）を必要とするときに何をするか。おしゃべりなドメインを作ったり、レスポンスを過剰にシリアライズしたり（REST における現在の悩みの種）することなく。
1. 誰が **データベースマイグレーションを所有** し、それらがどこに存在するか。
1. `Project` が **共有モデル** であるか、`Projects` **ドメイン** になるか。
1. **EE と JH** の拡張が、ドメインとともにどのように分離されるか。これはライブラリ層とトランスポート層が直面するのと同じ拡張の問題です。
1. **テスト** がどのように疎結合になるか。ファクトリーはグローバルであり、激しく結合しています。ドメインが疎結合になるにつれて、それぞれが spec／ファクトリー用の公開インターフェイスを公開するか、呼び出し元がドメイン間のレコードを構築するのではなくスタブ化すべきです。

  ```ruby
  RSpec.describe Ci::Pipeline do
    let(:user)    { Gitlab::Platform::Specs.create_user }
    let(:project) { Gitlab::Platform::Specs.create_project }
  end
  ```

移行期間中にドメイン間を橋渡しするグルーコードは、非常に大きくなる可能性があります。戦略にコミットする前に、それが実際に何を含むのかを学ぶための概念実証がおそらく必要です。

## ドメインコードはどこに存在するか: `domains/` か `gems/` か {#where-domain-code-lives-domains-vs-gems}

ドメインコードは、横断的なライブラリに使われる `gems/` ディレクトリとは別に、専用の `domains/` ディレクトリの下に存在すべきです。どちらも gem 形式を使えますが、それらを分離することで以下が得られます。

- 単一の `gems/` の下に、責務の混在した gem が平らに散らばることがなくなる。
- **ドメインコード** と **汎用コード** の明確な区別。
- 汎用ライブラリへの依存とは別に、**ドメイン間** の依存関係を追跡・可視化する能力。ドメイン間の関係をアーキテクチャレベルで示す図。
- 強制可能なルール。ライブラリは別のライブラリに依存してよいが、**決してドメインには依存しない**。静的解析が CI でこれを強制できます。

ドメインは [`config/bounded_contexts.yml`](https://gitlab.com/gitlab-org/gitlab/-/blob/master/config/bounded_contexts.yml) に従い、`Gitlab::` プレフィックスではなく、その境界づけられたコンテキストの名前空間を直接使います（`Ci::`、`Packages::`）。

オプションとして、大規模なドメインはネストされたプライベートなサブドメイン gem に分割できます。

```ruby
# domains/ci/ci.gemspec  (the public domain gem)
spec.add_dependency "ci-pipelines"
spec.add_dependency "ci-runners"
spec.add_dependency "ci-catalog"
```

## どうやってそこに到達するか: gem、Packwerk、またはその両方 {#how-we-get-there-gems-packwerk-or-both}

**これは未決定です。** 私たちは、ドメインの分解に Ruby の gem、[Packwerk](https://github.com/Shopify/packwerk) パッケージ、またはその組み合わせのどれを使うかについてまだ合意していません。なぜなら、上記の疎結合化の課題が、メカニズムの選択を支配するからです。

トレードオフ:

- **gem** は強固な分離と、明示的に宣言された依存関係を提供します。しかし gem の抽出は **アトミック** です。コードは単一のステップで完全に分離されなければなりません。これは、徐々にしかほどけない大規模で複雑かつ密結合なドメインにとっては苦痛です。
- **Packwerk（オプション）。** 静的解析は、徐々にガイドされた *その場での* 分離を支援できます。ファイルを移動せずにパッケージの境界を引き、プライバシーと依存関係の違反を浮き彫りにし、ドメインの所有者が時間をかけて違反を減らしていけるようにします。RBS 型シグネチャがあれば、`Ci::Pipeline` が許可リストに載った定数でない場合の `security_scan.pipeline` のような暗黙の依存関係さえ検出できます。この方法でドメインが十分に分離されれば、後の gem 抽出ははるかに簡単になります。

この 2 つは相互に排他的ではありません。抽出された gem は、何に依存してよいかを強制し続けるために、自身の Packwerk 設定を持てます。Packwerk は、徐々に進める道のための **補完的でオプションの** ツールであり、コミットされたアプローチではなく、gem の代替でもありません。

### Packwerk を使ったオプションの段階的な道 {#an-optional-gradual-path-with-packwerk}

段階的なルートを取る場合、以下のステップはそれがどのように機能しうるかを記述したものです。これらは決定ではなく、**1 つの選択肢** です。

1. **まず名前空間を。** ある [境界づけられたコンテキスト](bounded_contexts.md) のすべてのクラスとモジュールを、同じ名前空間の下に置きます。ドメインの大まかな理解がなければ計画は立てられません。一貫した名前空間付けが前提条件です。
1. **Rails を Packwerk 向けに準備する** — 一度限りのステップです。オートローダーを Packwerk のディレクトリレイアウトで機能させ（[この PoC](https://gitlab.com/gitlab-org/gitlab/-/merge_requests/129254/diffs#note_1512982957) のように）、CI で [Danger-Packwerk](https://github.com/rubyatscale/danger-packwerk) を実行し、オプションで pre-commit／pre-push チェックを実行します。
1. **ファイルをパッケージへ移動する** — パッケージを作成し、ファイルを反復的に移動します。定数は、パッケージ内の `app/` の下にあっても `lib/` の下にあってもオートロードされます。このフェーズではコードがパッケージと Rails ツリーの間に分割されるので、すばやく移動します。
1. **境界を強制する** — パッケージに [依存関係を明示的に宣言する](https://github.com/Shopify/packwerk/blob/main/USAGE.md#enforcing-dependency-boundary) ことと、パッケージの [公開インターフェイス](https://github.com/rubyatscale/packwerk-extensions#privacy-checker) のみに依存することを要求します。ファイルを移動した *後で* プライバシーを強制すると、定数とドメインの間の本当の結合が、記録された違反（RuboCop の TODO のように）として明らかになります。
1. **記録された違反に取り組む** — ドメインの DRI が育てる長期的なフェーズです。結合しすぎた定数をプライベートにし、定数をよりよく適合するパッケージへ移動し、または結合しすぎたパッケージをマージします。依存関係の図を使って設計を導きます。

ツールが存在すれば、新たに生まれるドメインは最初からパッケージとして実装でき、すぐに分離と明確なインターフェイスを得られます。
