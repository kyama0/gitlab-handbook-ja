このページは、[ヘキサゴナルモノリス](hexagonal_monolith/index.md)の中核にあるビジネスロジックである **アプリケーションドメイン** を、境界が定められ独立して所有されるモジュールへと隔離するための唯一の信頼できる情報源です。

ここで扱うのはドメインコードのみです。横断的な **プラットフォーム** コードの抽出は [横断的なライブラリを gem に抽出する](library_extraction.md) で扱い、**トランスポート層**（Web、REST、GraphQL、Sidekiq）の抽出は [トランスポート層をアダプターに分解する](transport_layer.md) で扱います。ドメインがどのように識別され命名されるかは [バウンデッドコンテキストの定義](bounded_contexts.md) で扱います。

ドメインの隔離はモジュール化の中で最も困難な部分です。プラットフォーム層とトランスポート層には比較的きれいな継ぎ目がありますが、ドメインは結合が実際に存在する場所です。

## ドメインの隔離とは何を意味するのか

ドメインは次の場合に隔離されています:

- **そのパブリック API を通じる以外に、境界をまたぐものが何もない。** 他のドメインは、文書化された意図的なインターフェースを呼び出します。内部実装、AR モデル、プライベートサービスに手を伸ばすことは決してありません。
- **自身のデータを排他的に所有している。** ドメインは自身のデータベーステーブルと、それらをマッピングする ActiveRecord モデルを所有します。他のどのドメインも、それらのテーブルをクエリしたり、それらのモデルへの参照を保持したりしません。

よく設計されたドメインモジュールは **深い** ものでもあります。すなわち、その小さなインターフェースの背後に、大量の内部ロジック、状態、データをカプセル化します。そして **凝集している** ものでもあり、それが記述する機能の唯一の信頼できる情報源として機能します。これは[名前空間の命名に関するガイドライン](https://docs.gitlab.com/ee/development/software_design.html#use-namespaces-to-define-bounded-contexts)に従い、CRUD 用語ではなく[ユビキタス言語](https://docs.gitlab.com/ee/development/software_design.html#use-ubiquitous-language-instead-of-crud-terminology)を使用します。ドメインがどのように識別され、フィーチャーカテゴリにマッピングされるかは [バウンデッドコンテキストの定義](bounded_contexts.md) で扱います。

以下のすべては、そこにたどり着くために必要な作業についての説明です。

## 難しい部分: ドメイン間の結合

私たちのモジュラーモノリスの調査は、これを困難にしている結合を浮き彫りにしました。現在のコードベースでは:

- **ポリシーが他のポリシーに依存している** — 例えば `PipelinePolicy` は `ProjectPolicy` に委譲します。
- **サービスが共有コンポーネントに依存している** — イベントの発行、メールの送信、システムノート、監査などです。
- **ドメインが互いの AR モデルに依存している** — 明示的に（`Ci::Pipeline.find_by_id`）も、アソシエーションを通じて暗黙的に（`security_scan.pipeline`）も、です。
- ドメイン間の **循環依存** が一般的です。

これが、ドメインの隔離が、ライブラリやトランスポートの抽出が大部分そうであるような機械的なファイル移動にはなり得ない理由です。ほとんどの労力は、コードの移動ではなく、*結合をリファクタリングして取り除くこと* に費やされます。

## データを所有する

排他的なデータ所有権は隔離の中核です。各ドメインは自身のテーブルとそれらをマッピングする AR モデルを所有し、それらをクエリすることを許される唯一のコードです。

これに続いて 2 つの強制の問題が生じます:

**ドメインをまたぐ AR アソシエーション。** 今日、どのドメインも `project.ci_pipelines` を呼び出して、CI のデータにまっすぐ入り込むことができます。隔離には、ドメインをまたぐアソシエーションを取り除き、代わりに所有するドメインのパブリック API を通すことが必要です。つまり `project.ci_pipelines` ではなく `Ci::Pipeline.all(project)` です。前提条件は [全知のクラスを飼いならす](https://docs.gitlab.com/development/software_design/#taming-omniscient-classes) ことです。`Project` や `User` のようなクラスは、あらゆるドメインからアソシエーションやメソッドを蓄積するため、スリム化されなければなりません。

**モデルへの直接参照。** アソシエーションが取り除かれても、どのコードも `Ci::Pipeline` を直接参照できます。1 つの選択肢は、AR モデルをドメインに対してプライベートに保つことです。例えば `Ci::Internal::Pipeline` のようにして、外部から参照できないようにし、パイプラインデータを `Ci::*` のパブリックインターフェースを通じてのみ公開します。

### リポジトリパターンという選択肢

より構造化されたバリエーションは、AR を厳密に永続化/リポジトリ層として、ドメインオブジェクトとは分離して使用します:

```ruby
# 永続化のみ — スコープと AR の永続化のみで、それ以外はなし。ドメインに対してプライベート。
Ci::Repository::Pipeline

# レコードをラップし、振る舞いを公開するドメインオブジェクト。パブリックな型。
Ci::Pipeline   # #builds, #cancelable?, ...
```

AR モデルは、自身のドメインオブジェクトの外では決して使用されません。これにより、テストでドメインを簡単にスタブ化でき、呼び出し元から AR への依存を取り除けます。コストは大きいです。最終的に gem を抽出するかどうかにかかわらず、これは大規模で侵襲的なリファクタリングです。

## ドメインをまたぐデータと共有モデル

最も厄介な問題は、ほぼすべてのドメインが触れる `Project` や `User` のような共有エンティティに何が起こるか、です。

**共有モデルを ID オブジェクトに切り詰める。** ドメインをまたぐアクセサが完全な `Project` AR オブジェクトを返す場合、そのオブジェクトの表面全体を引きずり込み、ドメインを再び結合させてしまいます。共有データは、完全なモデルではなく、最小限の ID オブジェクトに切り詰められるべきです。

**ドメイン固有の表現。** すべてのドメインが共有の `Project` を消費するのではなく、ドメインは必要とするデータを自身のファサードの背後にラップできます:

```ruby
project = Ci::Internal::Project.new(project_id) # CI ドメインにローカルな PORO またはレコード
project.variables       # CI に内部的
project.public_builds?  # CI 固有のプロジェクト設定
project.pipelines       # Project ではなく Ci::Project 経由でクエリされるパイプライン
```

これにより、`project.pipelines` がグローバルな `Project` ではなく CI ドメインの中に保たれます。関連するアイデアとして、ドメインごとのオーナーオブジェクトがあります。例えば `Ci::JobOwner` は `user_id` をラップし、ID 固有の関心事についてのみ `User` に委譲します。

**モノリスのコールバックよりも直接的な依存関係を優先する。** あるドメインが別のドメインからデータを必要とする場合、コールバックをモノリスに配線して戻すよりも、そのドメインのパブリックインターフェースへの直接的で宣言された依存関係を優先します。直接的な依存関係は追跡可能です。ドメイン間の依存関係グラフを描画し、それを CI で使用して影響を受けるテストだけを実行できます。一方、コールバックは結合を隠します。

**安定したクライアントインターフェース、後で gRPC に差し替え可能。** ドメインは最初から外部のものとして扱えます。すなわち、Ruby のクライアントインターフェースを公開し、それ以外のすべてをプライベートに保ちます。そのインターフェースの背後にあるバックエンドは、呼び出し元を変更することなく、後で gRPC に移行できます。

**おしゃべりなドメインを避ける。** 境界がいったん強制されると、互いに依存しすぎているコンポーネントが見つかるでしょう。Rails はすべてを常に利用可能にするため、単一のビジネストランザクションが多くのドメインにまたがって手を伸ばします。おしゃべりなドメイン間呼び出しを避けること（ドメインがサービスになると現実の問題になります）は、データを非正規化または複製し、ローカルな表現を保ち、インターフェース分離を適用することを意味します。例えば、完全な `User` の代わりに基本的な ID オブジェクトをラップするスリムな `Ci::Internal::User` です。

> すべてのドメインが必要とする共有の「足場」（`Project`/`User` のルックアップ、権限チェックなど）は、代わりに、すべてのドメインが依存する粗粒度の `gitlab-platform`/`gitlab-core` gem に存在するかもしれません。ドメインごとに注入されるのではなくです。そのトレードオフは、[ライブラリ抽出ページの未解決の問題](library_extraction.md#open-questions)として追跡されています。

## 抽出の前に解決しなければならないこと

ドメインの隔離は、大部分が未回答の設計作業です。ドメインを gem **または** パッケージに抽出する前に、再現可能なガイドラインが必要です。そうすれば、リファクタリングがエンジニアごと、あるいはエージェントごとの解釈に委ねられることがなくなります。少なくとも以下について必要です:

1. ドメインをまたぐ **AR アソシエーション** に何が起こるか。
1. どのロジックが **AR モデル** に存在し、どのロジックが **ドメインオブジェクト** に存在するか。
1. どのデータが **境界をまたぐ** ことを許され、どのような形で許されるか。
1. 呼び出し元がデータオブジェクトを受け取ったが **ドメインオブジェクト** のメソッド（例えば `Ci::Pipeline#merge_request`）を必要とするときにどうするか — おしゃべりなドメインを作ったり、レスポンスを過剰にシリアライズしたり（REST における現在の痛みの種）することなしに。
1. 誰が **データベースマイグレーションを所有** し、それらがどこに存在するか。
1. `Project` が **共有モデル** なのか、それとも `Projects` **ドメイン** になるのか。
1. **EE と JH** の拡張がドメインとともにどのように隔離されるか — ライブラリ層とトランスポート層が直面するのと同じ拡張の問題です。
1. **テスト** がどのように疎結合化されるか。ファクトリはグローバルで、大きく結合しています。ドメインが疎結合化されるにつれて、各ドメインは spec/ファクトリ向けのパブリックインターフェースを公開するべきであり、あるいは呼び出し元はドメインをまたぐレコードを構築するのではなくスタブ化するべきです:

  ```ruby
  RSpec.describe Ci::Pipeline do
    let(:user)    { Gitlab::Platform::Specs.create_user }
    let(:project) { Gitlab::Platform::Specs.create_project }
  end
  ```

移行中にドメイン間を橋渡しするグルーコードは非常に大きくなる可能性があります。戦略にコミットする前に、それが実際に何を含んでいるかを学ぶための概念実証がおそらく必要です。

## ドメインコードの置き場所: `domains/` か `gems/` か

ドメインコードは、横断的なライブラリに使われる `gems/` ディレクトリとは別の、専用の `domains/` ディレクトリの下に置かれるべきです。どちらも gem 形式を使用できますが、それらを分離することで次のものが得られます:

- 単一の `gems/` の下に、責任が混在した gem がフラットに散在することがない。
- **ドメインコード** と **汎用コード** の明確な区別。
- 汎用ライブラリへの依存関係とは別に、**ドメインをまたぐ** 依存関係を追跡・可視化する能力 — ドメイン関係のアーキテクチャレベルの図。
- 強制可能なルール: ライブラリは別のライブラリに依存してよいが、**決してドメインには依存してはならない**。静的解析でこれを CI で強制できます。

ドメインは [`config/bounded_contexts.yml`](https://gitlab.com/gitlab-org/gitlab/-/blob/master/config/bounded_contexts.yml) に従い、`Gitlab::` プレフィックスではなく、自身のバウンデッドコンテキストの名前空間（`Ci::`、`Packages::`）を直接使用します。

オプションで、大規模なドメインはネストされたプライベートなサブドメイン gem に分割できます:

```ruby
# domains/ci/ci.gemspec  (パブリックなドメイン gem)
spec.add_dependency "ci-pipelines"
spec.add_dependency "ci-runners"
spec.add_dependency "ci-catalog"
```

## どうやってそこへたどり着くのか: gem、Packwerk、またはその両方

**これは決定されていません。** ドメインの分解が Ruby の gem を使うのか、[Packwerk](https://github.com/Shopify/packwerk) パッケージを使うのか、あるいはその組み合わせを使うのか、私たちはまだ合意していません。上記の疎結合化の課題がメカニズムの選択を支配するからです。

トレードオフ:

- **Gem** は、ハードな隔離と、明示的で宣言された依存関係を提供します。しかし gem の抽出は **アトミック** です。すなわち、コードは単一のステップで完全に隔離されなければなりません。これは、徐々にしかほぐせない大規模で複雑な密結合したドメインにとっては苦痛です。
- **Packwerk（オプション）。** 静的解析は、その場での段階的で導かれた隔離を支援できます。すなわち、ファイルを移動することなくパッケージ境界を引き、プライバシーと依存関係の違反を浮き彫りにし、ドメインのオーナーが時間をかけて違反を減らしていけるようにします。RBS の型シグネチャがあれば、`Ci::Pipeline` が許可リストに含まれた定数でないときに `security_scan.pipeline` のような暗黙の依存関係を捕捉することさえできます。この方法でドメインが十分に隔離されれば、後の gem 抽出ははるかに簡単になります。

この 2 つは相互排他的ではありません。抽出された gem は、自身が何に依存してよいかを強制し続けるために、自身の Packwerk 設定を持つことができます。Packwerk は、段階的な道のための **相補的でオプションの** ツールです。コミットされたアプローチではなく、gem の代替でもありません。

### Packwerk を使ったオプションの段階的な道

段階的なルートを取る場合、以下のステップはそれがどのように機能し得るかを説明しています。これらは **1 つの選択肢** であり、決定ではありません:

1. **まず名前空間。** [バウンデッドコンテキスト](bounded_contexts.md)のすべてのクラスとモジュールを同じ名前空間の下に置きます。ドメインのおおまかな理解なしには計画は不可能です。一貫した名前空間化が前提条件です。
1. **Packwerk のために Rails を準備する** — 一度きりのステップ: オートローダーを Packwerk のディレクトリレイアウトで動作させ（[この PoC](https://gitlab.com/gitlab-org/gitlab/-/merge_requests/129254/diffs#note_1512982957) のように）、CI で [Danger-Packwerk](https://github.com/rubyatscale/danger-packwerk) を実行し、オプションで pre-commit/pre-push チェックを行います。
1. **ファイルをパッケージに移動する** — パッケージを作成し、ファイルを反復的に移動します。定数は、パッケージ内の `app/` の下にあっても `lib/` の下にあってもオートロードされます。このフェーズではコードがパッケージと Rails のツリーの間で分割されるため、素早く移動します。
1. **境界を強制する** — パッケージに[依存関係を明示的に宣言](https://github.com/Shopify/packwerk/blob/main/USAGE.md#enforcing-dependency-boundary)させ、パッケージの[パブリックインターフェース](https://github.com/rubyatscale/packwerk-extensions#privacy-checker)のみに依存させます。ファイルを移動した *後* にプライバシーを強制すると、定数とドメインの間の本当の結合が、記録された違反として（RuboCop の TODO のように）明らかになります。
1. **記録された違反に取り組む** — ドメインの DRI が育てていく長期的なフェーズです。結合しすぎた定数をプライベートにし、定数をより適合するパッケージに移動し、あるいは結合しすぎたパッケージをマージします。依存関係の図を設計の指針として使います。

ツールがいったん存在すれば、新たに生まれるドメインは最初からパッケージとして実装でき、隔離と明確なインターフェースをすぐに得られます。
