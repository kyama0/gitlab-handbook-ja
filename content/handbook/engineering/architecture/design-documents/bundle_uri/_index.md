## サマリー

[bundle-URI](https://git-scm.com/docs/bundle-uri) は Git のかなり新しい概念で、クライアントが 1 つ以上のバンドルをダウンロードして、リモートから残りのオブジェクトをフェッチする前にオブジェクトデータベースをブートストラップできます。クライアントがあらかじめシンプルな HTTP(S) サーバーから静的ファイルをダウンロードすることで、リモート側で実施する必要のある作業が減ります。

Git バンドルは、パックファイルを格納するファイルで、ref のセットや、(空の場合もある) 必要なコミットのセットなど、追加メタデータを伴います。ユーザーがリポジトリをクローンする際、サーバーはこれらのバンドルを提供する 1 つ以上の URI をアドバタイズできます。クライアントはこれらをダウンロードして Git オブジェクトデータベースを投入できます。これを行った後、サーバーとクライアント間でどのオブジェクトをフェッチする必要があるかを決定するためのネゴシエーションプロセスが始まります。クライアントがバンドルからのデータでデータベースをあらかじめ投入していると、サーバーからのオブジェクトのネゴシエーションと転送が削減され、サーバーの CPU 負荷が低下します。

## 動機

ユーザーが変更をプッシュすると、通常多くのジョブを伴う CI パイプラインが起動されます。CI ランナーが全員リポジトリをゼロからクローンするとき、[`git clone`](https://docs.gitlab.com/ee/ci/pipelines/settings.html#choose-the-default-git-strategy) を使用すると、全員がサーバーとクローンに必要なものをネゴシエーションし始めます。これはサーバーにとって CPU 集約的な処理です。

少し前に [pack-objects](https://docs.gitlab.com/ee/administration/gitaly/configure_gitaly.html#pack-objects-cache) を導入しましたが、これにはいくつかの落とし穴があります。ブランチの先端が変更されると、新しいパックファイルを計算する必要があり、キャッシュをリフレッシュする必要があります。

Git バンドルはより柔軟です。バンドルが最新のオブジェクトをすべて持っていなくても大きな問題ではありません。かなり最近の状態を含んでいるが最新の ref が欠けている場合、クライアント (つまり CI ランナー) はバンドルを適用した後に「キャッチアップ」を行い、追加のオブジェクトをフェッチします。Gitaly からフェッチする必要のあるオブジェクトのセットは大幅に小さくなります。

### ゴール

クライアントがリポジトリをクローンする際に Gitaly サーバー上で実施する必要のある作業を削減します。これは特に CI ビルドファームで有効で、サーバーへのプッシュごとに大量のトラフィックを生成します。

バンドルを使うことで、サーバーはバンドルが使われていない場合のすべてのオブジェクトを含むパックファイルと比較して、より小さなデルタパックファイルを作成すれば良くなります。これによりサーバーの CPU 負荷が削減されます。これはパックファイルキャッシュにもメリットがあります。パックファイルが小さく、生成も速くなるので、キャッシュミスの可能性が下がります。

### 非ゴール

bundle-URI を使うことで、ディスクに保存されているリポジトリのサイズが削減されることは **ありません**。この機能は Gitaly ノードからクラウドストレージへ、リポジトリを完全にも部分的にもオフロードするためには使われません。逆に、バンドルは別の場所に保存されるので、一部のデータが重複し、ストレージコストが増加します。

このフェーズでは、増分フェッチのパフォーマンスをブーストすることはゴールではありません。クライアントがすでにリポジトリをクローンしている場合、新規データのフェッチを最適化するためにバンドルは使われません。

現在 bundle-URI は浅いクローンと完全には互換性がないので、それはスコープ外とします。詳しくは [Git issue #170](https://gitlab.com/gitlab-org/git/-/issues/170) を参照してください。

## 提案

クライアントがリポジトリをクローンするとき、Gitaly はバンドル URI をアドバタイズします。この URI は、たとえばハウスキーピングなどの定期的な間隔でリフレッシュされるバンドルを指します。各リポジトリにつきバンドルは 1 つしか存在せず、新しいものが作成されると古いものは無効化されます。

バンドルはクラウドオブジェクトストレージに保存されます。バンドルを使用するには、管理者が Gitaly でこれを設定する必要があります。

## 設計および実装の詳細

クライアントが `git clone` を開始すると、サーバー側で Gitaly が `git upload-pack` プロセスを起動します。Gitaly は追加の Git 設定を渡せます。`git upload-pack` にバンドル URI をアドバタイズさせるには、以下の設定を渡す必要があります:

- `uploadpack.advertiseBundleURIs` :: アドバタイズされたバンドルの利用を有効にするには、これを `true` に設定する必要があります。
- `bundle.version` :: 現時点では `1` のみ受け付けられます。
- `bundle.mode` :: `any` または `all` のいずれかにできます。初回のクローンにのみバンドルを使いたいので、`any` が推奨されます。
- `bundle.<id>.uri` :: これは `<id>` で識別されるバンドルの実際の URI です。初期はリポジトリごとに 1 つのバンドルのみとします。

### クライアント側でアドバタイズされたバンドルの利用を有効化する

Git の現在のバージョンは、リモートからクローンまたはフェッチする際にデフォルトではアドバタイズされたバンドルを使いません。
幸いなことに、私たちはほとんどの CI ランナーを自分たちで管理しています。なので bundle-URI を使うために、ランナーで使われる Git の設定を変更して `transfer.bundleURI=true` を設定できます。

### アクセス制御

プライベートリポジトリのデータを公開 HTTP(S) ホスト経由でリークさせたくありません。これを克服するためのいくつかのオプションがあります:

- bundle-URI の利用をパブリックリポジトリでのみ有効化する。
- [signed-URL](https://cloud.google.com/cdn/docs/using-signed-urls) のようなソリューションを使う。

#### パブリックリポジトリのみ

Gitaly 自体はプロジェクトおよびそのリポジトリがパブリックかどうかを知りません。なのでバンドルが使えるかどうかを判断するには、GitLab Rails が Gitaly に伝える必要があります。この情報を Gitaly に渡すのは複雑であり、このアプローチを使うとこの機能はパブリックプロジェクトでのみ利用可能になるので、このソリューションには進みません。

#### Signed URL

[signed-URL](https://cloud.google.com/cdn/docs/using-signed-urls) の利用は、バンドルへのアクセスを制御するもう 1 つのオプションです。Google Cloud によって提供されるこの機能は、Gitaly が短い有効期限を持つ URI を作成できるようにします。

このアプローチの欠点は、クラウド固有の機能に依存することで、各クラウドプロバイダーがこの機能を提供する方法は少しずつ異なるか、提供しないかもしれません。しかし、私たちはまずこの機能を Google Cloud 上でホストされている GitLab.com にロールアウトしたいので、最初のイテレーションではこれを使います。

### バンドルの作成

#### サーバーサイドバックアップを使用

現在 Gitaly は、リポジトリをバンドルとしてクラウドストレージにバックアップする方法を知っています。[ドキュメント](https://gitlab.com/gitlab-org/gitaly/-/blob/master/doc/gitaly-backup.md#user-content-server-side-backups) はその使い方を説明しています。

bundle-URI の初期実装では、この機能に便乗できます。管理者は bundle-URI を使いたいリポジトリのバックアップを作成する必要があります。バックアップ用の既存設定により、Gitaly はクラウドストレージにアクセスできます。

#### ハウスキーピングの一部として

Gitaly には、最適化対象のリポジトリを毎日探すハウスキーピングワーカーがあります。理想的には、ハウスキーピング (つまりガベージコレクションと repacking) が完了した直後にバンドルを作成します。これにより、最適なバンドルファイルが保証されます。

バンドルを自動作成する際に念頭に置くべきことがいくつかあります:

- **バンドルを再作成する必要があるか?** リポジトリにあまり活動がなかった場合、新しいバンドルファイルを作成する必要はおそらくありません。クライアントは、欠けているオブジェクトを Gitaly から直接フェッチできるからです。ハウスキーピングタスクは、ハウスキーピングジョブの戦略を決定するためにさまざまなヒューリスティクスを使うので、このロジックの一部をバンドル作成で再利用できます。
- **そもそもバンドルを作成する必要があるか?** リポジトリの中には、非常に小さかったり活動がほとんどなかったりするものもあります。これらのバンドルを作成し、データをオブジェクトストレージに重複保存しても、大きな価値を提供せず、コストとメンテナンスを生むだけです。

#### GitLab Rails による制御

バンドルはストレージコストを増加させるので、最終的には GitLab 管理者にバンドル作成の完全な制御権を提供したいと考えています。これを実現するため、bundle-URI 設定は GitLab 管理インターフェースで利用できるようにします。ここで管理者は、bundle-URI を有効化するプロジェクトをプロジェクトごとに設定できます。

### 設定

この機能を使うには、Gitaly を設定する必要があります。このため、Gitaly の設定ファイルに以下の設定を追加します:

- `bundle_uri.strategy` :: bundle-URI を作成および提供するためにどの戦略を使うべきかを示します。現時点でサポートされる値は "backups" のみです。この設定をその値に設定すると、Gitaly はサーバーサイドバックアップが利用可能か確認し、それを使います。
- `bundle_uri.sign_urls` :: true に設定されると、クラウドストレージの URL はそのままクライアントに渡されず、署名付き URL に変換されます。この設定はオプションで、(現時点では) Google Cloud Storage のみサポートします。

クラウドストレージにアクセスするための認証情報は、Gitaly Backups ドキュメントに記載されているものを再利用します。

### メタデータの保存

現時点では、クラウドにバンドルを保存するために必要なすべてのメタデータは Gitaly のサーバーサイドバックアップによって管理されます。

### バンドルのクリーンアップ

ある時点で、管理者が 1 つ以上のリポジトリのバンドルをクリーンアップすることを決めるかもしれません。このための管理コマンドを追加すべきです。現在は `gitaly-backup` で作成されたバンドルのみ使用しているので、これはスコープ外とします。

### Gitaly Cluster との互換性

サーバーサイドバックアップの作成は現状 Praefect 経由では発生しません。バックアップを作成するノードに対処するのは管理者次第です。ノードを最新の状態に保てば、すべてのノードが最新のバンドルにアクセスでき、適切な bundle-URI パラメータをクライアントに渡せます。なので、サーバーサイドバックアップのバンドルを bundle-URI で再利用するのに追加作業は必要ありません。
