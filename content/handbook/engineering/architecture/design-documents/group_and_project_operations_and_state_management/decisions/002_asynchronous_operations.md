## コンテキスト

現在のグループとプロジェクトの操作は重大なパフォーマンス問題に悩まされています：

- グループ移転：P99.95 パフォーマンスが 51 秒
- プロジェクト移転：P99.95 パフォーマンスが 27 秒
- これらの長時間実行される同期操作はタイムアウトとユーザー体験の低下を引き起こす
- 子孫へのカスケード変更を伴う操作はリソース集約的になりえる
- 一括操作によるデータベースロックが同時操作に影響する

現在の同期アプローチは、ユーザー体験とシステムの安定性の両方に影響するボトルネックと信頼性の問題を生じさせています。

## 決定事項

以下の基準のいずれかを満たす場合、操作は非同期でなければなりません：

1. **パフォーマンス閾値**：P99.95 パフォーマンスが 10 秒を超える
2. **カスケード変更**：操作が子孫へのカスケード変更を伴う
3. **外部依存関係**：操作が外部サービスの呼び出しやインテグレーションを必要とする
4. **一括操作**：操作が一括データベースの書き込み、削除、または移行を伴う
5. **ロックリスク**：同時操作に影響するデータベースロックを生じさせるリスク
6. **リソース集約**：システムリソースに影響する可能性のあるメモリ集約的な操作

### 実装要件

すべての非同期操作は以下を提供する必要があります：

- **状態遷移**：適切な `_in_progress` 状態への遷移
- **即時確認**：リクエスト成功の確認と進行中操作の通知
- **進行状況インジケーター**：技術的に実行可能な場合
- **完了通知**：適切なチャネル（アクティビティ、通知センター、メール）を通じて
- **エラー処理**：ユーザー向けエラーメッセージを含む包括的なエラー処理
- **ロールバック機能**：失敗した操作のため

### 優先操作

現在のパフォーマンスメトリクスに基づき、以下の操作は即時の非同期実装が必要です：

1. **グループ移転**（P99.95：51 秒）- 優先度 1
2. **プロジェクト移転**（P99.95：27 秒）- 優先度 1

### 実装パターンの例

```ruby
class NamespaceTransferService
  def execute
    namespace.transfer_start!
    NamespaceTransferWorker.perform_async(namespace.id, target_namespace.id, current_user.id)

    ServiceResponse.success(
      message: "Transfer initiated successfully",
      payload: { operation_id: operation_id, estimated_duration: "5-10 minutes" }
    )
  end
end
```

## 結果

### ポジティブな結果

- **パフォーマンス**：重い操作のタイムアウト問題を排除
- **ユーザー体験**：進行状況追跡を伴う即時フィードバック
- **システムの安定性**：データベース競合とリソース競合を削減
- **スケーラビリティ**：システムへの影響なしに大規模な操作を処理可能

### 技術的な結果

- **複雑性**：バックグラウンドジョブ処理の実装が必要
- **状態管理**：`_in_progress` 状態と適切な状態遷移が必要
- **UI の変更**：操作ステータスと進行状況インジケーターを表示する必要がある
- **エラー処理**：より複雑なエラーシナリオを処理する必要がある
- **テスト**：非同期フローのテストにおける追加の複雑性

## 代替案

### 代替案 1：同期操作の最適化

- **メリット**：よりシンプルな実装、状態管理の複雑性なし
- **デメリット**：改善の可能性が限られ、タイムアウトリスクに依然さらされる

### 代替案 2：ハイブリッドアプローチ（一部同期、一部非同期）

- **メリット**：段階的な移行、シンプルな操作は同期のまま維持
- **デメリット**：一貫性のないユーザー体験、複雑な判断ロジック

### 代替案 3：すべての操作を非同期にする

- **メリット**：一貫した体験、最大パフォーマンス
- **デメリット**：シンプルな操作に不必要な複雑性、高速操作での UX 低下
