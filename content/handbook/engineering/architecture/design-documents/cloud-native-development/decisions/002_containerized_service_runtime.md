## コンテキスト

開発者が同時に複数のコンポーネントに貢献することはほとんどないため、現在のセッションでコード変更を加えない対象のサービスについては、「本番環境に近い」デプロイ形式にフォールバックすることが合理的です。

現在、GDK は Gitaly のような一部のサービスに対してプリコンパイル済みバイナリを使用する実行をサポートしており、開発者がそのセッションで Gitaly に貢献する予定がない場合でも、時間のかかる再コンパイルを回避できます。このアプローチの欠点は、GDK がサポートする各プラットフォーム（特に MacOS）向けにコンパイルジョブとアーティファクトを維持する必要があることです。

サービスとホストシステム間の結合を減らすために、デプロイターゲットとして（ローカルまたはリモートの）Kubernetes クラスターの使用を評価しました。

当初のアイデアは、CNG Helm チャートとそれに依存するイメージを再利用して、本番モードで設定されている場合は Kubernetes にサービスをデプロイし、開発モードでは CNG イメージを基にした拡張イメージを使用するというものでした。

これにより、クラウド対応開発において IDE の SSH 開発統合などの確立されたツールを使用しながら、スムーズな起動パフォーマンスと本番環境との整合性を確保することが期待されていました。

しかし、評価の結果、CNG Helm チャートの使用は[現状では実現不可能](https://gitlab.com/groups/gitlab-org/-/epics/17447#note_2889556056)であることが判明しました。参照したクロージングステートメントに詳細の理由が記載されています。

## 決定事項

コード変更を受け付けないサービスは、[コンテナランタイム](./../phases/1-modularization/#service-runtime-types)を使用したコンテナ化モードで起動できます。

## 影響

これにより、設定されたサービスをホストシステムから分離するための軽量なアプローチを使用できるようになります。この決定により、不必要な複雑さや潜在的な不安定性の原因を持ち込まずに、安定性と起動パフォーマンスの利点を活かすことができます。

さらに、将来のイテレーションで Kubernetes ベースのアプローチを追加することを妨げるものでもありません。

## 代替案

リモートおよびローカルの Kubernetes クラスターの両方を代替案として検討しました。

これは、組織が CNG に注力したいという意向と、クラウドネイティブな開発環境を構築するために活用できる [DevSpace](https://devspace.sh) や [Tilt](https://tilt.dev) などのツールが存在するため、有望なアプローチに見えていました。

しかし、GDK 内から Kubernetes クラスターを管理することに関連する概念的な問題がいくつかあります。

リモート Kubernetes クラスターのみを使用するオプションを提供することは、ホスト型インフラへのアクセスを持たないコミュニティメンバーを包含できないため、適切ではありません。

チームメンバーは予算承認を得られる可能性がありますが、コミュニティサポートには MiniKube のようなローカル Kubernetes オプションが必要です。これにより、開発者による手動セットアップが必要となってコントリビューションの摩擦が増加するか、ローカル Kubernetes クラスターの自動管理を GDK に提供することでメンテナンスのオーバーヘッドが追加されます。

さらに、SSH ベースの開発ワークフローは主要な IDE 間で移植可能ですが、Emacs などのエディタ向けのカスタムシェル拡張機能や設定が機能することを保証するには追加の努力が必要です。

加えて、特にローカルとリモートの両方の Kubernetes クラスターをサポートする必要がある場合、Kubernetes 自体がこのコンテキストにおける不安定性の新たな原因になりうるという懸念が示されました。
