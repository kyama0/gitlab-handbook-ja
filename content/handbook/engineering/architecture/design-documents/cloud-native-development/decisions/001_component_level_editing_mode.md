## コンテキスト

現在、GDK は多数のサービスをオーケストレーションしており、これらはいくつかの方法で実行できます:

**ソースベースのサービス**: 元々、GDK は git リポジトリに結合したサービスごとの作業ディレクトリを管理し、GDK ユーザーが `gdk start` や `gdk update` などのコマンドを実行するたびに有効なサービスをオンデマンドでコンパイルしていました。Rails アプリケーション/モノリス、[HTTP ルーター](https://gitlab.com/gitlab-org/cells/http-router)、Topology Service は、まだこの方法で動作するサービスの代表的な例です。

**バイナリのみのサービス**: これらのサービスはあらかじめコンパイルされたバイナリから実行できます。開発者はサービスのソースディレクトリからバイナリパスを設定できます。GitLab Runner、Workhorse、Gitaly はすべて、この方法で GDK によってサポートされる例です。

**クラウドネイティブサービス:** これらのサービスはあらかじめビルドされたコンテナイメージから実行できます。GitLab Runner はその一例です。

**サードパーティサービス:** これらのサービスはバイナリからのみ実行されます。PostgreSQL、Elasticsearch、Redis はサードパーティサービスの例です。これらのサービスは通常コードレベルでは変更されず、主に開発者の利便性のために GDK に統合されています。

開発ワークフロー、特に `gdk update` の実行時間を改善するために、GitLab Runner のように比較的変更頻度が低いサービスは、ソースベースのサービスとしてのみではなく、コンテナイメージやバイナリからオプションで実行できるようになりました。

しかし、これはほぼ GDK アーキテクチャを適応させることなく、サービスオーナーを巻き込まずに、反復的に行われました。

### 現在のアプローチの欠点

現在のアプローチは、GDK への新しいサービスの頻繁な追加に関してうまくスケールしません。これは統合実装の観点からも、パフォーマンスの観点からも当てはまります。

例えば、[GDK パッケージレジストリ](https://gitlab.com/gitlab-org/gitlab-development-kit/-/packages)は現在、GDK パイプライン内でビルドされた [Gitaly とその他のサービス](https://gitlab-org.gitlab.io/gitlab-development-kit/configuration/#skip-compile)用の macOS ネイティブバイナリをホストしています。これらのジョブの所有権は Development Tooling にあり、メンテナンスの労力が必要です。次に、現状では、GDK に新しいサービスが統合されるたびに、どのデプロイオプションをサポートするかという選択を迫られます。

ハイブリッド開発環境の適切なアプローチを特定し、最初のイテレーションの理想的な候補コンポーネントを特定するために、[さまざまなデプロイモードの比較](https://gitlab.com/gitlab-org/gitlab-development-kit/-/issues/3084)と[サービスベースの実現可能性分析](https://gitlab.com/gitlab-org/gitlab-development-kit/-/issues/3084#note_2864373908)を実施しました。

さらに、GDK コアは管理するサービスとの標準インターフェースがありません。そのため、GDK とのサービス統合をサービスを所有するチームに委任できません。同様に、ユーザーはすべてのサービスでバイナリ、コンテナ、ソースベースのデプロイ間を一貫して切り替える方法がありません。これらのサービスが統合された反復的かつアドホックな性質のためです。

例として、今日の GDK のデフォルト設定は次のようになっています:

```yaml
gitaly:
  skip_compile: true
workhorse:
  skip_compile: true
runner:
  install_mode: 'binary'
```

同じ機能をトグルするための複数の方法があり、サービスをプリコンパイルされたバイナリとしてデプロイする方法が複数あり、どれを選ぶかは興味があるサービスのオプションを見ないとわかりません。

## 代替案

ソースコードへの変更が現在のインスタンスに反映されるサービスの実行について、いくつかの代替案を検討しました。これは[この Issue](https://gitlab.com/gitlab-org/gitlab-development-kit/-/issues/3084) で説明されています。検討した 1 つのオプションは、今日行っているようにプリコンパイルされたバイナリのデプロイを続けることでした。選択したモデルと比較した欠点は、現在 Gitaly などのサービスに対して行っているのと同じ方法で、これらのバイナリのビルドを引き続き担当しなければならないことです。組織がデフォルトのデプロイ方法として CNG（クラウドネイティブ GitLab）を目指すことを決定したため、コンテナに依存することがより理にかなっています。

さらに、クラウドネイティブ開発ワークフローを許可するために CNG Helm チャートや基盤となるコンテナイメージを直接使用する[スパイク](https://gitlab.com/gitlab-org/quality/tooling/team/-/issues/121#note_2864497246)を実施しました。しかし、この時点では CNG のアーティファクトを利用するソリューションは [CNG でサポートされるサービスと GDK でサポートされるサービス間の大きなギャップ](https://gitlab.com/groups/gitlab-org/-/epics/17447#note_2889556056)のため、[実現可能ではありません](https://gitlab.com/groups/gitlab-org/-/epics/17447#note_2889556056)でした。

その結果、[edit モードサービスのコンテナ化](https://gitlab.com/gitlab-org/gitlab-development-kit/-/issues/3084#2-dockerized-edit-mode-writing_hand)は、コンテナ化された開発環境に関わる追加のファイルシステムとネットワーク層により、ホットリローディングとデバッグに関するさまざまな複雑さが加わるため、却下されました。

さらに、主要な IDE はコンテナ化されたワークフローをサポートしていますが、Emacs などの高度に設定可能なローカルツールやシェル拡張機能に依存するユーザーのワークフローを混乱させる可能性があります。

最終的に、edit モードで CNG を効果的に活用できない場合、これらの課題を回避するための追加の複雑さは、サービスレベルで設定可能なローカル「edit モード」と比較して十分なメリットをもたらしません。

## 決定

**個々のコンポーネント**を 2 つのモードのいずれかで起動するように設定できるハイブリッドアプローチを採用します:

1. git 作業ディレクトリから実行し、必要に応じてソースコードをコンパイルする
2. あらかじめビルドされたコンテナ化されたアプリケーションまたはバイナリから実行する

理想的には、最も変更頻度が高いのがモノリスであるため、デフォルト設定では以下の結果が得られます:

1. すべてのサードパーティサービス（PostgreSQL、Elasticsearch）はソースからコンパイルされません。
2. モノリスはソースベースのサービスとして管理されます。
3. すべてのオプションサービスは非ソースベースのサービスとして管理されます。

### 設定レイヤー

上記の詳細のとおり、現在の設定レイヤーでは、設定を変更する開発者が以下を考慮する必要があります:

1. 異なるデプロイモードをトリガーするための一貫性のない設定キー。
1. バイナリとコンテナ化されたデプロイのサポートが一貫していない。

代わりに、ユーザーが特定のサービスを一貫した方法で編集可能にできるようにします:

```yaml
runner:
  editable: true
```

「編集可能」として設定された場合のみ、コンポーネントのサービスはユーザーのマシン上で作業ディレクトリに含まれるソースからビルドされます。コンポーネント自体がサポートするサービスランタイムを[所有します](/handbook/engineering/architecture/design-documents/cloud-native-development/phases/1-modularization/#service-runtime-types)。これにより、GDK をオーケストレーターとしてサービス自体の間の関心の分離が強調されます。

Gitaly に変更を加えたい開発者は、以下のように簡単に行えます:

```shell
gdk config set gitaly.editable true
gdk reconfigure
```

### GDK コアをサービス定義から分離する

全体的なブループリントで述べたとおり、GDK をリファクタリングして、サービスのオーケストレーションのコア機能をサービス定義自体から分離したいと考えています。これには新しいツールは必要なく、コンポーネントのオーナーを定義するためのフォルダ構造と既存の `make`/`rake` タスクへの変更が必要になります。

当面の間、コンポーネント定義は GDK リポジトリに残りますが、`CODEOWNERS` を使用してコンポーネントチームにメンテナンスの責任を割り当てます。これらの集中管理された定義により、アトミックな更新と依存関係管理の簡素化が保証されます。

新しいサービスを GDK に統合する方法についてのドキュメントガイドを想定しています。既存のサービスは最初は Development Tooling によって新しいモデルに移植され、移行後はコンポーネントチームがメンテナンスを引き継ぐでしょう。

### 開発モードのコンポーネント

[用語集](../#glossary)で説明したように、Gitaly などのコンポーネントは複数のサービス（`praefect-gitaly-1`、`praefect-gitaly-2` など）を持つことができます。この場合、`gitaly.editable` を `true` に設定すると、このコンポーネントのすべてのサービスインスタンスがソースコードから実行されるはずです。これはコンパイルコストを増加させません。なぜなら、それは同じサービスのいくつかの論理ノードに過ぎないからです。後方互換性のテストなどのためにローカルソースコードに基づくインスタンスと同じサービスのすでにリリースされたバージョンを一緒に実行する必要が生じた場合、サービス定義を改善することで将来のイテレーションでこれに対応できます。

繰り返しになりますが: コンポーネントだけがどのようにデプロイされ、その設定が有効かどうかを知るべきです。

## 結果

決定されたハイブリッドモデルにより、それぞれのチームが必要なアーティファクトのビルド方法を最もよく知っているため、所有権を各チームに移すことで、GDK への新しいサービスの統合がはるかに容易かつ迅速になります。

同時に、このモデルはチームが変更が GDK 自体に意図しない影響を与えないという確信を持ち、エラーがコンパートメント化されるため、各サービス定義に貢献するよう奨励します。GDK 内でサービス定義を所有するチームは、Development Tooling がこれらの変更を承認する必要性も減らします。ここでの理由は似ています。チームは私たちよりもサービスをよく知っています。

さらに、設定の合理化は、GDK のデフォルト値にただ頼るのではなく、ユーザーがユースケースに合わせて最適化することに自信を持つよう奨励します。また、開発者がモノリスとそのサテライトサービスを[分散モノリス](https://www.gremlin.com/blog/is-your-microservice-a-distributed-monolith)ではなく、分離されたコンポーネントとして見るよう奨励します。

### ユースケース

開発者はサービスとその直接の依存関係に取り組む可能性が高く、それらが合わせて**コンポーネント**を構成することは注目に値します。チームメンバーが短期間に複数のサービスでの開発活動を頻繁に切り替えることも少ないでしょう。

さらに、そのようなコンポーネントあたりのアクティブな開発者数と受信するコード変更数には高い分散があります。成熟した機能状態に達した HTTP ルーターのようなサービスと、[AI Gateway](https://gitlab.com/gitlab-org/modelops/applied-ml/code-suggestions/ai-assist) や相互接続されたモノリスのような積極的に開発されているサービスとの間には顕著な対比があります。

さらに、提案されたアプローチは新たにオンボードされた開発者と既存の開発者の両方のユースケースを考慮しています。

#### モノリスを対象としたオンボーディング

新しいユーザーは[既存のドキュメント](https://gitlab-org.gitlab.io/gitlab-development-kit/#use-gdk-to-install-gitlab)に従って GDK をセットアップできます。ほとんどのチームメンバーとコントリビューターの主な使用ケースはモノリスの変更であるため、設定の変更は必要ありません。モノリスは当面ソースベースの方法で実行され、常に `editable` になります。

### Gitaly を対象としたオンボーディング

新しいユーザーは[既存のドキュメント](https://gitlab-org.gitlab.io/gitlab-development-kit/#use-gdk-to-install-gitlab)に従って GDK をセットアップできます。GDK はプライマリのユースケースに最適化されるため、Gitaly に変更を加えたい開発者は以下を実行するだけで対応できます:

```shell
gdk config set gitaly.editable true
gdk reconfigure
```

同じことが GDK でサポートされている他のすべての GitLab 開発サービスにも最終的に適用されます。

### Gitaly での作業からモノリスでの作業への切り替え

開発者が Gitaly での作業を完了したら、使用する Gitaly のバージョンを更新する必要があります。

[GitLab リリースプラットフォームの成果](https://gitlab.com/gitlab-com/content-sites/internal-handbook/-/merge_requests/7674)によっては、いくつかのオプションがあります:

#### 関連する変更がリリースされた場合

[リリースマニフェストの提案](https://gitlab.com/gitlab-com/content-sites/internal-handbook/-/merge_requests/7674/diffs#de7f639f645587dc201411c8c0d5a385112b83d2_0_191)に記載のとおり、次の（モノリスの）リリースにはリリースマニフェストが含まれ、それが [CNG に伝播](https://gitlab.com/gitlab-com/content-sites/internal-handbook/-/merge_requests/7674/diffs#de7f639f645587dc201411c8c0d5a385112b83d2_0_199)されます。

上記の提案が実現された場合、開発者はリリースマニフェストのバージョンを使用することを示します:

```shell
gdk config set gitaly.editable false
gdk switch main
gdk reconfigure
```

#### 関連する変更がリリースマニフェストにない、またはまだリリースされていない場合

対象のサテライトサービスが[リリースマニフェストでカバーされていない](https://gitlab.com/gitlab-com/content-sites/internal-handbook/-/merge_requests/7674/diffs#de7f639f645587dc201411c8c0d5a385112b83d2_0_203)場合、開発者はサテライトサービスを `editable` のままにしておく必要があり、コンテナ化された方法で使用することはできません。これは今日の動作と一致しています。ローカルでの CNG 風イメージのビルドを可能にする取り組みは現時点では計画されていません。
