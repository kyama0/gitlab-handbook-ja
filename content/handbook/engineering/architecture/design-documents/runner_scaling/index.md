## まとめ

GitLab Runner は GitLab CI/CD の中核コンポーネントです。信頼性の高い並行環境で
CI/CD ジョブを実行することを可能にします。Kamil Trzciński によって2015年初頭に
同じサービスの Ruby バージョンを置き換えるために最初に導入されました。Go で書かれた
GitLab Runner はより幅広いコミュニティで使いやすく、以前の
Ruby ベースのバージョンより効率的で信頼性が高いことが証明されました。

2016年2月、Kamil Trzciński が[オートスケーリング機能を実装し](https://gitlab.com/gitlab-org/gitlab-runner/-/merge_requests/53)、
クラウドインフラを活用して多数の CI/CD ジョブを並行実行できるようにしました。この
機能は GitLab.com での CI/CD 採用を支える基盤となり、現在ではピーク時に1日約400万ビルドを
実行しています。

初期実装時に Docker Machine を使用する決定がなされました:

> 使いやすい。よくドキュメント化されている。よくサポートされており継続的に
> 拡張されている。ほぼあらゆるクラウドプロバイダーや仮想化
> インフラをサポートする。Docker Machine のサポートに必要な変更は最小限:
> マシンの列挙と検査のみ。「クラウド固有」の機能を実装する必要がない。

この設計選択は GitLab Runner の成功に不可欠でした。それ以来、
オートスケーリング機能は多くのユーザーや顧客に使用され、GitLab.com での
CI/CD 採用の急速な成長を可能にしました。

しかし、Docker Machine を使い続けることはできません。そのプロジェクトでの作業は
[2018年7月に一時停止され](https://github.com/docker/machine/issues/4537)、それ以来
（一部の重要なセキュリティ修正を除いて）開発は行われていません。2018年に Docker Machine が
「メンテナンスモード」に入った後、このツールを使い続けてユースケースに必要な修正と更新を
提供できるように[独自のフォーク](https://gitlab.com/gitlab-org/ci-cd/docker-machine)を
作成することにしました。[2021年9月26日にプロジェクトはアーカイブされ](https://github.com/docker/docs/commit/2dc8b49dcbe85686cc7230e17aff8e9944cb47a5)、
公式ページからドキュメントが削除されました。これは Docker Machine を使用する
当初の理由も有効でなくなったことを意味します。

お客様と広いコミュニティのサポートを継続し、SaaS ランナーの
メンテナンスを改善するために、GitLab Runner オートスケーリングの新しいメカニズムを
設計する必要があります。オートスケーリングをサポートするだけでなく、効率性、
信頼性、可用性を改善するために上に積み上げられる方法でそれを行う必要があります。

この新しいメカニズムを「次世代 GitLab Runner スケーリングアーキテクチャ」と呼びます。

## Docker Machine での継続的な構築

現時点では、私たちのコア製品の1つである GitLab Runner と、その最も
重要な機能の1つである CI/CD ジョブ実行環境のオートスケーリング機能は、
放棄された外部製品に依存しています。

Docker Machine プロジェクト自体も維持が困難です。その設計は古くなりつつあり、
新しい機能と修正の追加が困難になっています。内部知識の欠如を伴う膨大な
コードベースにより、メンテナーが受信する機能リクエストやコミュニティへの貢献を
サポートし適切に処理することが困難になっています。

Docker Machine とそれに統合された20以上のクラウドおよび仮想化
プロバイダー用ドライバーは、別のサブセットの問題も生じさせます:

- 各クラウド/仮想化環境は現れたり消えたりする機能を持ち、
  それらのサポートを維持する必要があります（新機能の追加、
  バグの修正）。

- 基本的に各仮想化/クラウドプロバイダーの専門家になって、
  その API との統合を適切にサポートする必要があります。

- Docker Machine が統合するすべてのプロバイダーにはバグ、
  セキュリティリリース、脆弱性があり、プロジェクトを適切に維持するために
  それらすべてを把握して必要なときに更新を処理する必要があります。

別の問題は、Docker Machine が当初から Linux ベースのインスタンスの
管理のみに焦点を当てていたことです。ある時点で Docker が Windows の公式かつ
ネイティブな統合を得たにもかかわらず、Docker Machine はこのステップに続き
ませんでした。また、そのような統合を容易にするように設計されてもいません。

MacOS のサポートもありません。これは明らかです — Docker Machine は
Docker Engine のホストを管理するツールであり、MacOS のネイティブ Docker Engine は
ありません。ここでネイティブとは MacOS オペレーティングシステム内で実行される
MacOS コンテナを意味します。MacOS 向け Docker 製品はネイティブサポートでは
ありません — MacOS 開発インスタンスで**Linux コンテナ**の開発を容易にするツールと
仮想化された Linux インスタンスです。

これは公式にサポートされている3つのプラットフォーム（Linux、Windows、MacOS）のうち
1つだけが CI/CD オートスケーリングの完全な機能サポートを持つことを意味します。
Windows では Kubernetes を使用する可能性があり（場合によっては制限あり）、
多大な努力で Docker Machine への Windows サポートを追加できるかもしれません。
しかし MacOS については、GitLab Runner によってネイティブに提供されるオートスケーリング
ソリューションはありません。

これはユーザーにとって大きな制限であり、頻繁にリクエストされる機能です。
SaaS ランナーの提供にとっても制限です。カスタムエグゼキューターを
ハッキングすることで SaaS Windows および SaaS MacOS ランナーのある種の
オートスケーリングを維持してきました。しかし過去3年間の経験から、
これは最善の方法ではないことが示されています。それでも、この時間が経過した後でも、
Windows と MacOS のランナーオートスケーリングは SaaS Linux ランナーで持っている
多くのパフォーマンスと機能サポートが不足しています。

お客様と広いコミュニティのサポートを継続し、SaaS ランナーのメンテナンスを
改善するために、GitLab Runner オートスケーリングの新しいメカニズムを設計する必要があります。
オートスケーリングをサポートするだけでなく、効率性、信頼性、可用性を
改善するために上に積み上げられる方法でそれを行う必要があります。

## 提案

現在、GitLab Runner のオートスケーリングはいくつかの方法で設定できます。一部の
お客様は Kubernetes で正常にオートスケーリング環境を使用しています。Kubernetes での
オートスケーリングをより信頼性の高いものにするために非公式のカスタム GitLab Runner
バージョンが構築されたことを私たちは知っています。複数のジョブを並行して実行するための
本当に優れた Kubernetes ソリューションを持つことの重要性は認識していますが、
この領域の改善はこのアーキテクチャイニシアチブの範囲外です。

Docker Machine の問題の解決と、このメカニズムを信頼性が高く柔軟なメカニズムで
置き換えることに焦点を当てたいと思います。Docker Machine が非推奨になった理由は
おそらく多数あり、Docker Machine のドロップイン置き換えを構築することは
できないかもしれません。非常に多くのクラウドプロバイダーとの互換性を維持することは
非常に困難であり、Docker Machine は Docker Desktop を優先して非推奨になったようですが、
これは私たちにとっての実行可能な置き換えではありません。
[この Issue](https://github.com/docker/roadmap/issues/245) には現在人々が Docker Machine を
どのように使用しているかについての議論が含まれており、GitLab CI は人々が
Docker Machine を使い続ける最も頻繁な理由の1つのようです。

単一のより大きな仮想マシンで複数のジョブを実行できることにも機会があります。
今日はそれができませんが、これが効率性を大幅に改善できる可能性があることは
知っています。それを容易にし、PoC でどの程度効率的かをテストできる
新しいアーキテクチャを構築したいかもしれません。単一マシン上での複数ジョブの
実行は、ジョブ実行間で共有できるビルド成果物/ユーザーデータのスペースである
「スティッキーコンテキスト」の再利用も可能にします。

### 💡 ユーザーが上に構築できるシンプルな抽象化を設計する

実行可能な置き換えがなく、Docker Machine がサポートしていたすべての
クラウドプロバイダーをサポートできないかもしれないため、重要な設計要件は
使用するクラウドプロバイダー用のカスタム GitLab プラグインを作成することを
幅広いコミュニティにとって本当にシンプルで簡単にすることです。ユーザーが上に
構築できるシンプルな抽象化を設計したいと思います。また、GitLab.com での
既存のワークフローのサポートにも使用します。

設計されたメカニズムは Docker Machine エグゼキューターが行っていたことを
抽象化するものとなるはずです: 外部 Docker 環境を作成する方法を提供し、
この環境をプロビジョニングしてジョブを実行するのを待ち、これらの操作を
実行するために必要なクレデンシャルを返します。

新しいプラグインシステムはすべての主要プラットフォームで利用可能である必要があります:
Linux、Windows、MacOS。

### 💡 既存の Docker Machine ソリューションをプラグインに移行する

新しい抽象化を設計して実装すれば、既存の Docker Machine メカニズムを
プラグインに移行できるようになります。これにより、ユーザーや顧客は
新しいアーキテクチャをすぐに使い始めることができますが、Docker Machine の
既存のワークフローと設定を維持できます。これにより、レガシーオートスケーリングの
サポートを完全に廃止する前に全員が新しいアーキテクチャに移行する時間が与えられます。

### 💡 AWS、Google Cloud Platform、Azure 用プラグインを構築する

Docker Machine がサポートしていたすべてのクラウドプロバイダーのサポートを
追加できないかもしれませんが、AWS、Google Cloud Platform、Azure などの
主要なクラウドプロバイダー向けの GitLab メンテナンスプラグインを提供することが
重要と思われます。

これらを別々のリポジトリに構築し、幅広いコミュニティメンバーが持つ特定のニーズに
合わせて貢献、フォーク、変更しやすい方法で構築します。GitLab Runner を
再構築することなく新しいプラグインをインストールすることも容易である必要があります。

### 💡 独自プラグインの構築方法についてしっかりしたドキュメントを書く

ユーザーが独自のクラウドインフラのサポートを実装できるように、
プラグインの構築方法をユーザーに示すことが重要です。

新しいプラグインの構築はシンプルで優れたドキュメントによりサポートされる必要があります。
新しいプラグインへの貢献に対する参入障壁が非常に低くなるようにプラグインシステムを
設計したいと思います。

### 💡 単一マシンで複数のビルドを実行する PoC を構築する

単一マシンで複数のジョブを実行することでどのような効率性が得られるかをより
よく理解したいと思います。予測することは困難なため、理想的には私たちが
何を期待できるかをよりよく理解するのに役立つ PoC を構築する必要があります。

この実験を実行するには、単一マシン上で複数のビルドを実行するスケジュールを
可能にするだけでなく、どのように実行されるかをより簡単に理解するための
包括的なメトリクスセットが組み込まれた実験的なプラグインを構築する必要が
あると思われます。

## 詳細

抽象化がどのように見えるかは、プロトタイプ化、PoC、データに基づいた決定を
行う必要があるものです。詳細を説明し、要件を開発し、PoC を実施し、スコアリングする
いくつかの提案があります。目標を最もよくサポートするソリューションを選択します。

提案を説明するために、まず GitLab Runner のどの部分を抽象化する必要があるかを
よりよく説明する必要があります。これらの概念を把握しやすくするために、
現在のオートスケーリングアーキテクチャとシーケンス図を見てみましょう。

![GitLab Runner Autoscaling Overview](/images/engineering/architecture/design-documents/runner_scaling/gitlab-autoscaling-overview.png)

上の図では、現在ランナーマネージャーがクラウドプロバイダーの API にアクセスできる
マシン上で実行されていることがわかります。Docker Machine を使用して Docker Engine が
インストールされた新しい仮想マシンをプロビジョニングし、外部の認証されたリクエストを
許可するように Docker デーモンを設定します。そのようなエフェメラル Docker 環境への
クレデンシャルをディスクに保存します。マシンがプロビジョニングされてランナーマネージャーが
ビルドを実行するために利用可能になると、既存のエグゼキューターのいずれかを使用して
ユーザー提供のスクリプトを実行します。オートスケーリングでは、通常これは Docker
エグゼキューターを使用して行われます。

### 関心の分離

現在のアーキテクチャにはいくつかの関心事が表れています。現在の実装では
それらが結合されているため、ここでそれぞれを個別に考えるために分離します。

- **仮想マシン（VM）の形状**。VM の基盤となるプロバイダーは作成するマシンの
  種類を知るための設定が必要です。例えば、コア数、メモリ、フェイルドメイン、
  など... この情報は非常にプロバイダー固有です。
- **VM ライフサイクル管理**。複数のマシンが作成され、
  システムはどのマシンがこのエグゼキューターに属するかを追跡する必要があります。
  通常、クラウドプロバイダーには均質なマシンのセットを管理する方法があります。
  例えば、GCE インスタンスグループ。基本操作は増加、減少、そして
  通常特定のマシンを削除することです。
- **VM オートスケーリング**。低レベルのライフサイクル管理に加えて、
  コスト的に過剰なキャパシティを維持せずに必要なときにキャパシティを提供するために
  マシンのセットにジョブを考慮したキャパシティ決定を行う必要があります。
- **VM へのジョブマッピング（ルーティング）**。現在のシステムは特定のマシンに
  1つのジョブのみを割り当てます。マシンは特定のエグゼキューター設定に基づいて
  再利用される場合があります。
- **VM 内ジョブ実行**。各 VM 内でジョブはさまざまな事前定義されたステージを
  通じて実行され、結果とトレース情報がランナーシステムに返される必要があります。
  これらの詳細は VM アーキテクチャと OS およびエグゼキューターの種類に大きく依存します。

以下の用語集も参照してください。

#### 現在の状態

現在のアーキテクチャには関心事間のいくつかの結合点があります。
結合は抽象化の機会を減らし（例えば、コミュニティサポートのプラグイン）、
複雑さを増やし、コードの理解、テスト、メンテナンス、拡張を困難にします。

主要な設計決定は、どの関心事をプラグインに外部化し、
どれをランナーシステムに残すかになります。現在の実装には
内部に複数の抽象化があり、新しい抽象化の切り口として使用できます。

例えば、[`Build`](https://gitlab.com/gitlab-org/gitlab-runner/-/blob/267f40d871cd260dd063f7fbd36a921fedc62241/common/build.go#L125)
型は、ディスパッチエグゼキューター文字列に基づいてエグゼキュータープロバイダーを取得するために
[`GetExecutorProvider`](https://gitlab.com/gitlab-org/gitlab-runner/-/blob/267f40d871cd260dd063f7fbd36a921fedc62241/common/executor.go#L171)
関数を使用します。
さまざまなエグゼキューター型はインポートされて初期化時に
[`RegisterExecutorProvider`](https://gitlab.com/gitlab-org/gitlab-runner/-/blob/267f40d871cd260dd063f7fbd36a921fedc62241/common/executor.go#L154)
を呼び出すことでシステムに登録されます。ここでの抽象化は
[`ExecutorProvider`](https://gitlab.com/gitlab-org/gitlab-runner/-/blob/267f40d871cd260dd063f7fbd36a921fedc62241/common/executor.go#L80)
と
[`Executor`](https://gitlab.com/gitlab-org/gitlab-runner/-/blob/267f40d871cd260dd063f7fbd36a921fedc62241/common/executor.go#L59)
インターフェースです。

`docker+autoscaling` エグゼキューター内では、[`machineExecutor`](https://gitlab.com/gitlab-org/gitlab-runner/-/blob/267f40d871cd260dd063f7fbd36a921fedc62241/executors/docker/machine/machine.go#L19)
型が共通の [`Prepare`](https://gitlab.com/gitlab-org/gitlab-runner/-/blob/267f40d871cd260dd063f7fbd36a921fedc62241/executors/docker/machine/machine.go#L71)
フェーズ中に VM を取得するために使用する
[`Machine`](https://gitlab.com/gitlab-org/gitlab-runner/-/blob/267f40d871cd260dd063f7fbd36a921fedc62241/helpers/docker/machine.go#L7)
インターフェースを持ちます。この抽象化は主に VM を作成、アクセス、削除します。

VM オートスケーリングロジックの現在の抽象化はありません。VM ライフサイクルと
ジョブルーティングロジックと密接に結合されています。アイドルキャパシティの作成は
ジョブを VM にバインドしながら `machineProvider` の [`Acquire`](https://gitlab.com/gitlab-org/gitlab-runner/-/blob/267f40d871cd260dd063f7fbd36a921fedc62241/executors/docker/machine/provider.go#L449)
を呼び出す副作用として発生します。

VM 内ジョブ実行の現在の抽象化もありません。VM 固有の
コマンドはランナーマネージャーによって [`GenerateShellScript`](https://gitlab.com/gitlab-org/gitlab-runner/-/blob/267f40d871cd260dd063f7fbd36a921fedc62241/common/build.go#L336)
関数を使用して生成され、マネージャーがジョブ実行ステージを進める際に VM に
[注入](https://gitlab.com/gitlab-org/gitlab-runner/-/blob/267f40d871cd260dd063f7fbd36a921fedc62241/common/build.go#L373)
されます。

### 設計原則

私たちの目標は、幅広いコミュニティが使いやすい柔軟でシンプルな
GitLab Runner プラグインシステムインターフェースを設計することです。
すべてのクラウドプラットフォーム用プラグインを構築できないため、
プラグインを開発する必要があるすべての人の参入障壁が低くなるよう確保したいと
思います。誰もが貢献できるようにしたいと思います。

この目標を達成するために、いくつかの重要な設計原則に従います。これらの
原則が新しいプラグインシステムの抽象化の開発プロセスをガイドします。

#### 一般的な高レベル原則

- 新しい制約を課す代わりに、将来においてより多くの選択肢と
  柔軟性を持つことを目標として新しいオートスケーリングアーキテクチャを設計する。
- 単一マシンで複数のジョブを並行して実験するために新しいオートスケーリングアーキテクチャを設計する。
- 幅広いコミュニティが新しい抽象化の上に容易に構築できる方法で
  Docker Machine を置き換えるために新しいプロビジョニングアーキテクチャを設計する。
- 新しいオートスケーリングメソッドは GitLab Runner 製品のコアコンポーネントとなるべきであり、
  他の主要製品と同じツール、テスト設定、Go 言語セットアップを使用して
  メンテナンスを簡素化できるようにする。
- Docker コンテナだけでなく、Linux オペレーティングシステムを超えた
  複数のジョブ実行環境をサポートする必要がある。

    最良の設計は、Docker や Shell などの現在のエグゼキューターの周りにラップされた
    機能としてオートスケーリングを持つことです。

#### 新しいプラグインシステムの原則

- 新しいプラグインを書くための参入障壁を低くする。
- 新しいプラグインの開発はシンプルで、プログラミング言語と
  クラウドプロバイダーの API の基本的な知識のみを必要とする。
- プラグインシステムのシンプルさと柔軟性のバランスを目指す。
  これらは相互排他的ではない。
- 可能な限り多くの技術的詳細を抽象化するが、完全に隠さない。
- コミュニティをよくサポートするが、迅速に出荷できる抽象化を構築する。
- 柔軟なソリューションに投資し、一方向の扉の決定を避け、イテレーションを促進する。
- 疑問がある場合は幅広いコミュニティにとって物事をよりシンプルにする方向に誤る。
- システムをよりシンプルで拡張可能にするために関心事間の結合を制限する。
- 関心事はプラグの一方の側にのみ存在するべきで、両側には存在しない。
  両側にあると作業が重複し結合が増加する。

#### 最も重要な技術的詳細

- プラグインと GitLab Runner 間の gRPC 通信を優先する。
- 通信インターフェースをバージョン管理し、多くのバージョンをサポートできるようにする。
- Go をプラグイン作成の主要言語にするが、他の言語も受け入れる。
- オートスケーリングメカニズムは GitLab が完全に所有すべきである。

    クラウドプロバイダーのオートスケーラーはスケールダウン時に削除する VM を
    知らないため、最適でない決定を行います。すべてのオートスケーラーに GitLab
    ジョブについて教えるのではなく、GitLab 所有の1つのオートスケーラー（プラグイン内ではない）を
    持つことを好みます。

    これにより、メカニズムの将来を形作り、私たちのニーズと要件に合った
    決定を行うことができます。

## プラグイン境界の提案

以下はプラグイン境界をどこに引くかの提案です。上記に記載された
設計原則と技術的制約でこれらの提案を評価します。

### カスタムプロバイダー

作業範囲を縮小するために、1か所のみに新しい
抽象化レイヤーを導入したいと考えています。

数年前、GitLab Runner に[カスタムエグゼキューター](https://docs.gitlab.com/runner/executors/custom.html)
機能を導入しました。これにより、ユーザーはカスタムビルド実行メソッドを設計できます。
カスタムエグゼキュータードライバーは、シンプルなシェルスクリプトから
Runner が os/exec システムコールを通じて使用する専用バイナリまで、
どんな方法でも実装できます。

カスタムエグゼキューターの抽象化のおかげで、Runner に内部的に新しいエグゼキューターを
実装する必要がなくなりました。特定のニーズを持つユーザーは独自のドライバーを実装でき、
その作業を「公式」GitLab Runner の一部にするために私たちを待つ必要がありません。
各ドライバーは個別のプロジェクトであるため、興味のある人々が改善とバグ修正に
共同作業できるコミュニティを作りやすくなります。

新しいカスタムプロバイダーをカスタムエグゼキューターの成功を再現するように設計したいと
思います。ビルドをカスタムエグゼキューターの1つで実行するコンテキストと環境を
提供する独自の方法を構築するためにユーザーが使いやすくなります。

カスタムプロバイダーの抽象化を実装するための複数のソリューションがあります。
生の Go プラグイン、HashiCorp の Go Plugin、HTTP インターフェース、または
gRPC ベースのファサードサービスを使用できます。多くのソリューションがあり、
最も最適なものを選択したいと思います。そのために、別のドキュメントにソリューションを
説明し、要件を定義してソリューションをスコアリングします。これにより、
私たちと幅広いコミュニティに最もよく機能するソリューションを選択できます。

この提案は VM ライフサイクルとオートスケーリングの関心事、およびジョブから
VM へのマッピング（ルーティング）をプラグインに配置します。ビルドは VM を
要求するだけでよく、ライフサイクルとルーティングのすべての側面がすでに
プラグインによって処理された VM を取得します。

根拠: [カスタムエグゼキュータープロバイダー提案の説明](https://gitlab.com/gitlab-org/gitlab-runner/-/issues/28848#note_823321515)

### Taskscaler プロバイダー

「Fleeting」インターフェースの形でより単純なバージョンの `Machine` 抽象化を
導入できます。Fleeting は均質な VM グループへの低レベルインターフェースを提供し、
セットサイズの増加と減少、ならびにセット内から VM を使用することができます。

クラウドプロバイダーと他の VM ソース用のプラグインは HashiCorp の go-plugin
ライブラリ経由で実装されます。これは実際には STDIN/STDOUT を介した gRPC ですが、
他のワイヤープロトコルも使用できます。

新しいインターフェースを活用するために、オートスケーリングロジックが
Docker エグゼキューターから引き出されて新しい Taskscaler ライブラリに配置されます。

これにより VM ライフサイクル、VM 形状、ジョブルーティングの関心事がプラグインに
配置されます。また VM オートスケーリングの関心事を別のコンポーネントに配置することで、
複数のランナーエグゼキューター（`docker+autoscaling` だけでなく）で使用できます。

根拠: [InstanceGroup / Fleeting 提案の説明](https://gitlab.com/gitlab-org/gitlab-runner/-/issues/28848#note_823430883)
PoC: [マージリクエスト](https://gitlab.com/gitlab-org/gitlab-runner/-/merge_requests/3315)

## 用語集

- **[GitLab Runner](https://docs.gitlab.com/ee/development/documentation/styleguide/word_list.html#gitlab-runner)** - インストールして管理することを選択できるソフトウェアアプリケーションで、ソースコードは `gitlab.com/gitlab-org/gitlab-runner` にホストされています。
- **[ランナー](https://docs.gitlab.com/ee/development/documentation/styleguide/word_list.html#runner-runners)** - ランナーは環境内で GitLab CI/CD ジョブを実行し、GitLab インスタンスに結果を報告するエージェントです。/1/ GitLab からジョブを取得し、/2/ ローカルまたはリモートのビルド環境を設定し、/3/ プロビジョニングされた環境内でジョブを実行し、ログデータとステータス更新を GitLab に渡します。
- **ランナーマネージャー** - ランナープロセスは複数のランナーを管理するため、しばしば `Runner Manager` と呼ばれます。これはランナーの `config.toml` ファイルで定義された `[[runners]]` ワーカーです。
- **エグゼキューター** - ジョブを実行するために準備して使用できる具体的な環境です。新しいエグゼキューターは各ジョブに対して作成されます。
- **エグゼキュータープロバイダー** - オンデマンドでエグゼキューターを提供できる実装です。エグゼキュータープロバイダーはインポート時に登録され、ランナーの起動時に1度初期化されます。
- **カスタムエグゼキューター** - GitLab Runner と環境変数入力を持つバイナリまたはシェルスクリプトのセット間のインターフェースとして機能し、あらゆるホストコンピューティング環境での CI ジョブの実行を可能にします。新しいカスタムエグゼキューターは GitLab Runner コードベースに変更を加えることなくシステムに追加できます。
- **カスタムエグゼキュータープロバイダー** - 上のプラグイン境界提案セクションのカスタムプロバイダーの見出しの下に提案された新しい抽象化で、GitLab Runner コードベースを変更することなく新しいエグゼキュータープロバイダーを作成できます。プロトコルはカスタムエグゼキューターと同様か、gRPC で行われる可能性があります。この抽象化はエグゼキューターを生成するすべての仕組みをプラグイン内に配置し、オートスケーリングとライフサイクル管理の関心事を各実装に委譲します。
- **taskscaler** - 上のプラグイン境界提案セクションの taskscaler プロバイダーの見出しの下に提案された新しいライブラリで、具体的なエグゼキュータープロバイダーと fleeting プロバイダーでパラメーター化されます。Taskscaler はオートスケーリングの関心事を担当し、任意の VM 形状を使用して任意のエグゼキュータープロバイダーをオートスケーリングするために使用できます。Taskscaler はまた VM ライフサイクルのランナー固有の側面も担当し、特定の VM を使用しているジョブの数とその VM が使用された回数を追跡します。
- **fleeting** - taskscaler と共に提案された新しいライブラリで、クラウドプロバイダー VM の抽象化を提供します。
- **fleeting インスタンスグループ** - fleeting が同様の VM のプールを表すために使用する抽象化です。これは GCP IGM や AWS ASG（オートスケーリングなし）を表します。インスタンスグループは増加、減少、または特定の VM の接続詳細を提供できます。
- **fleeting プラグイン** - 特定の IGM または ASG を表す（初期化時に）fleeting インスタンスグループの具体的な実装です。プロバイダーごとに N 個存在し、それぞれが独自のプロジェクトに存在します。コアなものは私たちが所有・維持しますが、一部はコミュニティがサポートします。新しい fleeting プラグインはランナー、taskscaler、fleeting のコードベースに変更を加えることなく作成できます。これにより、自己サービスと切り離しの観点でカスタムエグゼキュータープロバイダーに類似しますが、異なる関心事に沿っています。
- **fleeting プラグイン Google Compute** - GCP インスタンスを作成する fleeting プラグインです。fleeting と taskscaler とは別のプロジェクトに存在します。
- **fleeting プラグイン AWS** - AWS インスタンスを作成する fleeting プラグインです。fleeting と taskscaler とは別のプロジェクトに存在します。
