## 概要

このデザインドキュメントは、GitLab の多様なデータランドスケープ全体にわたるデータ探索とクエリの課題に対処します。現在、ユーザーは GitLab データへのアクセスや洞察の導出に大きな障壁に直面しており、情報が PostgreSQL、ClickHouse、GraphQL、REST API など複数のソースに分散しています。

これらのデータソースはそれぞれ異なるクエリ戦略を必要とし、さまざまなスキーマ、リレーションシップ、パフォーマンス特性を持っています。このフラグメンテーションはデータ探索プロセスに大きな摩擦を生み出し、ユーザーが必要な情報にアクセスするために複数のシステムとクエリ言語を理解する必要があります。

GitLab は現在、これらの異なるデータソースを一貫して操作できる統一されたクエリインターフェースと可視化ツールを持っていません。この欠如により、ユーザーは異なるインターフェースとツールを切り替えることを余儀なくされ、複数のソースのデータを組み合わせた包括的なダッシュボードを作成したり、異なるタイプのデータ間の関係を探索したりすることが困難になっています。

目標は、ユーザーが GitLab データと対話する方法を簡素化し、データソース間の基盤となる違いを理解することなく、GitLab の使用状況とビジネスパフォーマンスに関する意味のある洞察を発見できるようにすることです。この課題に取り組むことで、技術的な複雑さのために現在ほとんどのユーザーがアクセスできない貴重なデータを解放することを目指します。

## 動機

ユーザーが GitLab データを探索し、意味のあるビジネス洞察を導出できる能力は、組織がデータ駆動の意思決定に依存するにつれてますます重要になっています。しかし、いくつかの課題が現在ユーザーの GitLab データの効果的な探索と理解を妨げています。

このイニシアチブは外部顧客と GitLab 内部チームメンバーの両方に役立ちます。外部ユーザーは DevSecOps の実践を最適化し、組織内での GitLab の価値を実証するためにデータの洞察が必要です。同時に、GitLab 自身の製品、エンジニアリング、カスタマーサクセスチームも、製品の使用状況を理解し、機能を改善し、顧客を効果的にサポートするために、これらの同じデータ探索機能に依存しています。このドキュメントで説明されている統一されたデータ探索アーキテクチャは、ユーザーが顧客であるか GitLab チームメンバーであるかに関係なく、一貫した価値ある体験を生み出し、両者に恩恵をもたらします。

### 課題

#### データソースのフラグメンテーション

GitLab のデータは、アクセスパターンが異なる複数のデータソースに存在します。

- **PostgreSQL データベース**はトランザクションアプリケーションデータを保存します
- **Elasticsearch/OpenSearch データベース**は効率的な検索のための非正規化データを含みます
- **ClickHouse データベース**は分析および時系列データを含みます
- **GraphQL エンドポイント**はアプリケーションデータへの構造化された API アクセスを提供します
- **REST API**はさまざまなデータセットへの追加インターフェースを提供します

これらの各ソースは独立して発展し、異なるクエリ要件、データモデル、パフォーマンス特性をもたらしています。このフラグメンテーションにより、ユーザーはデータを効果的に探索するために各システムを個別に理解する必要があります。

さらに、同じクラスのデータソースでも大きなバリエーションを示す場合があります。例えば、異なる REST API エンドポイントはフィルタリング機能、クエリメソッド、パフォーマンスへの影響、レスポンス形式が異なる場合があります。

#### クエリ言語の不一致

現在、ユーザーはデータソースに応じて異なるクエリアプローチを使用する必要があります。

- GraphQL には独自のクエリ構造があります
- REST API はさまざまなパラメーターベースのフィルタリングメカニズムを使用します
- PostgreSQL と ClickHouse データベースはエンドユーザーが GraphQL と REST API 経由でのみクエリできます
- フィルターと演算子は同じ API タイプ内でもエンドポイントによって異なります

この不一致は、複数のソースにわたってデータにアクセスする必要があるユーザーにとって急な学習曲線を生み出し、各システムのクエリ機能に関する専門知識を必要とします。

#### スキーマとデータモデルの格差

クエリ言語の違いを超えて、データの構造化方法に根本的な不一致があります。

- フィールドの命名規則はシステム間で異なります
- エンティティのリレーションシップは異なる方法でモデル化されます
- データの粒度が異なります（例: 詳細なレコードと集計データ）

これらの格差により、異なるシステムに存在する関連するデータポイント間の意味のある接続を確立することが困難になり、ユーザーが情報の全体像を把握する能力が制限されています。

#### ユーザーエクスペリエンスの摩擦

現在の状態はデータ探索プロセスに大きな摩擦を生み出しています。

- すべてのデータソースにデータ探索 UI またはクエリ言語があるわけではありません。
- ユーザーは異なるデータソースにアクセスするために複数のツールまたはインターフェースを切り替える必要があることが多いです。
- 技術的でないユーザーは自分でデータを探索するのに大きな障壁に直面します。
- データがどのようにクエリおよび変換されたかについての透明性が不足していることが多く、提示された情報への信頼の問題を生み出しています。GitLab チームメンバーは、データの取得・表示方法を顧客に説明し、ツールへの信頼を構築するために多大な時間を費やしています。

この摩擦はデータ探索を妨げ、ユーザーが GitLab データから得られる洞察を制限しています。

#### パフォーマンスとリソースの課題

異なるデータソースはパフォーマンス特性が異なります。

- 一部のクエリはリソース集約的でシステムパフォーマンスに影響する可能性があります
- クエリ最適化戦略はデータソース間で異なります
- 異なるデータソースに対する類似のクエリでパフォーマンスが大きく異なる場合があります

これらの課題により、すべてのデータタイプにわたって一貫して応答性の高い探索体験を提供することが困難になっています。

### 統一データ探索の機会

これらの課題にもかかわらず、ユーザーが GitLab データと対話する方法を簡素化し改善する大きな機会があります。

- 統一されたデータ探索インターフェースにより、重要な洞察へのアクセスが簡素化し、機能の採用が改善され、技術的な複雑さのために現在ほとんどのユーザーがアクセスできないデータが解放されます
- クエリパターンの標準化により、新しいクロスソース分析機能が開放される可能性があります
- クエリ結果のエクスポート、共有、埋め込みを可能にすることで、ユーザーは他の GitLab ページやワークフロー全体に洞察を組み込むことができます
- 標準化された入力と出力は、ユーザーだけでなく、それと統合したり、クエリを生成したり、結果を解釈するために使用される AI アシスタントなどの他のシステムにも恩恵をもたらします

よく設計されたデータ探索アーキテクチャは、現在の問題点に対処するだけでなく、将来のより高度な分析機能の基盤を確立します。

#### ユーザーニーズの例

統一されたデータ探索システムにより、ユーザーは以下のような質問に答えられるようになります。

**エンジニアリングチーム向け:**

- 「エンジニアリングチームとして、毎日のスタンドアップで DevSecOps フローに対する自分たちのパフォーマンスを示す 1 つのダッシュボードを見たいです。以下が確認できる必要があります:
  - 現在のオープンな MR のテーブル
  - 過去 x 日間に発生した重大度が高い脆弱性の数
  - 現在のマイルストーン（またはラベル/ステータス、チームが現在の作業を確認したいピボット）に割り当てられた Issue
  - 最近のパイプライン失敗や遅いジョブ
  - デプロイ頻度や変更のリードタイムなどの DORA メトリクス」

**プロダクトマネージャー向け:**

- 「GitLab PM として、自分の機能の内部使用状況とエンゲージメントメトリクスを追跡したいです」
- 「ユーザーは自分の機能のワークフローをどのように進んでいますか？」
- 「リリースした新機能の採用率は？」
- 「機能の使用状況は製品内の他のアクティビティとどのように相関していますか？」

これらの例は、ユーザーが複数のソース（Issue、MR、脆弱性、パイプライン、分析イベント）からのデータを 1 つの一体的なビューで組み合わせる必要性を示しています — これは現在達成が困難なことです。

### 目標

- **統一されたデータ探索体験の作成** — GitLab の多様なデータソース全体にわたって
- **データのクエリプロセスの簡素化** — 技術的ユーザーと非技術的ユーザーの両方のために
- **フィルタリングインターフェースの標準化** — 基礎となるデータソースに関係なく一貫して機能するように
- **クエリ結果のエクスポートを可能にする** — GitLab ページ全体にわたって
- **クエリ結果の標準化** — 既存のダッシュボードフレームワークコンポーネントを活用することで
- **データ探索クエリの適切なパフォーマンスを確保** — 異なるデータソース全体で
- **すべてのデータソースでのユーザー権限の尊重と適切なセキュリティコントロールの維持**
- **GitLab Duo との統合を促進** — データ探索機能を強化するために

### 非目標

- **ダッシュボードフレームワークや新しい可視化コンポーネントの作成** — 新しいものを作成するのではなく、既存のフレームワーク可視化コンポーネントを使用します
- **既存 API とデータソース間の不整合の修正** — 既存の API とデータソースの不整合を直接解決しません。むしろ、ユーザーからこれらの違いを抽象化するインターフェース層を作成します

## 提案

この提案は、先に特定されたデータ探索の課題に対する解決策の概要を示します。私たちは、データがどこに存在するかや構造がどうなっているかに関係なく、GitLab 内の複数のソースからデータをクエリおよび可視化するための統一された直感的なインターフェースを作成することを目指します。

この解決策は 2 つの主要なコンポーネントで構成されています。

1. **標準化・簡素化されたクエリシステム** — 複数のデータソースと拡張されたクエリ機能にわたって機能する GitLab Query Language（GLQL）の拡張
2. **統一されたデータ探索 UI** — クエリの構築、結果の表示、可視化の作成のための標準化されたインターフェース

### 標準化・簡素化されたクエリシステム

私たちの解決策の中核は、既存の [GitLab Query Language（GLQL）](https://docs.gitlab.com/user/glql/) を基盤にした標準化されたクエリシステムです。

#### GLQL が理想的な基盤である理由

GLQL は標準化されたクエリシステムの堅固な基盤を提供します。

1. **認知負荷の軽減** — ユーザーは複数のクエリ言語や方法ではなく、1 つのクエリシステムを学習することで、生産性が向上し採用が容易になります
2. **複数ソースからのクエリを可能にする** — 異なるソースのデータを一貫した意味のある方法で簡単にクエリできます
3. **一貫した結果の提供** — データソース間での統一されたフィルタリングと出力形式
4. **ユーザーフレンドリーな構文** — GLQL はシンプルさと読みやすさを念頭に置いて設計されています
5. **設計による拡張性** — GLQL の[アーキテクチャ](../glql/#extensibility)は解析、実行、変換、プレゼンテーション間の関心事を分離し、拡張性を念頭に置いて構築されています
6. **本番実績済み** — 現在の実装はすでにマージリクエストとワークアイテムクエリに使用されています
7. **AI 強化型探索**: GLQL の構造化された構文は GitLab Duo との統合の理想的な基盤となります。これにより、ユーザーが平易な言葉で質問し、それを自動的に GLQL に変換することで、より幅広い層にデータ探索を提供する自然言語クエリが可能になります

#### GLQL を拡張して複数のデータソースをサポートする

提案されているアプローチには以下が含まれます。

1. **GLQL を拡張して新しいデータソースをサポートする** — `type` フィールドを拡張してより多くのデータソースを含める

   ```plaintext
   query: project = "team-project" AND type = Vulnerability AND severity in ("critical", "high") AND created > -7d
   fields: status, severity, description

   query: project = "team-project" AND type = Pipeline AND (status = failed OR duration > 60) AND updated > -3d
   fields: id, name, status, duration, updatedAt

   query: project = "team-project" AND type = AiMetric
   fields: codeSuggestionsShownCount, codeSuggestionsAcceptedCount, duoUsedCount
   ```

2. **改善されたクエリ機能** — 以下のサポートを追加することでクエリ言語の能力を向上させる:

   - OR 演算子（現在は AND のみサポート）
   - 算術演算子
   - 集計関数
   - ソート

   ```plaintext
   query: project = "team-project" AND type = Issue AND updated > -3d
   fields: count() AS "Total Issues", sum(weight) AS "Total Weight"
   group_by: state
   sort_by: count()
   ```

   ユーザーがより高度なクエリを実行できるようにすることは、機能的で有用なデータ探索システムとクエリ言語の基本要件です。データを集計したり計算したりする（例えば、プロジェクトの MR スループットを時間とともに計算する）ことは、単純なデータを引き出す（例えば、フィルタリングされた MR のリスト）よりもはるかに価値を提供します。

   これらの機能の一部を GLQL に追加することはすでに https://gitlab.com/gitlab-org/gitlab/-/issues/511954 で追跡されています。

3. **拡張された表示オプション** — グラフなど、新しいデータソースにより適した可視化タイプをサポートするために `display` 属性を拡張する:

   ```plaintext
   # 折れ線グラフ表示
   display: chart
   chart_type: line
   x_axis: created_at
   y_axis: count

   # カスタム可視化コンポーネント
   display: custom
   display_id: 'ai-impact-table'
   ```

   早期の探索が https://gitlab.com/gitlab-org/gitlab/-/issues/482782 で行われています。

4. **大規模データセットのサポート** — 現在の GLQL の実装は単一ページのデータの返却のみをサポートしているため、ページネーションを完全にサポートするように拡張する必要があります。

5. **複数のプロジェクトとグループへのクエリをサポート** — 現在 GLQL は単一のプロジェクトまたは単一のグループにスコープされたクエリに制限されています。複数のプロジェクトからデータを取得するには、個別のクエリを実行して結果を組み合わせる必要がありますが、コンテキスト、ページネーション、ソートが失われます。複数のグループとプロジェクトをサポートするように拡張し、ユーザーがインスタンスや組織全体のすべてのデータにアクセスできるようにする必要があります。

最後に、ダッシュボードデータ探索に GLQL を採用することで、他の GitLab ページ全体でダッシュボード/可視化の簡単なエクスポートと共有も可能になり、プラットフォームのデータ探索機能がさらに強化されます。

#### GLQL をバックエンドに移行する

重要なアーキテクチャの変更は、GLQL の実行をフロントエンドからバックエンドに移行し、一貫したクエリとフィルター言語で GitLab データをクエリするための新しい別個の API を作成することです。

現在、GLQL コンパイラーは GLQL クエリを GraphQL に変換するだけで、クエリの実行をコンシューマーに依存しています。これにより GLQL の機能が GraphQL が課す同じ制限に縛られ、OR 演算子、ネストされたサブクエリ、集計、算術関数などの機能の実装が困難または不可能になります。これは https://gitlab.com/gitlab-org/gitlab/-/issues/546157 で詳しく議論されています。

さらに、GraphQL スキーマのフィルターとデータタイプが一貫していないため、新しいデータソースを GLQL に追加することは直接的ではなく、トランスパイルステージでアドホックなケースを処理する必要があります。これにより、チームが自分たちのデータソースを GLQL にオンボードするのが難しくなる可能性があり、その影響と使いやすさを制限します。

考えられる解決策は、GLQL コンパイラーをバックエンドに移動し、クエリをアドホックな中間フォーマット（JSON など）にトランスパイルすることです。より一貫したスキーマに従い、GraphQL パスを避けて Rails ファインダーを通じて直接データをクエリするために使用できます。

概念実証:

- [複数出力サポート + FFI ラッパー](https://gitlab.com/gitlab-org/gitlab-query-language/glql-rust/-/merge_requests/147)
- [GLQL をバックエンドに移行](https://gitlab.com/gitlab-org/gitlab/-/merge_requests/190552)

GLQL をバックエンドに移行することで、以下の利点が得られます。

- GitLab データをクエリするための単一エントリポイント、一元化されたアクセス制御と一貫したクエリインターフェース
- 高度な機能（OR 操作、集計、算術関数）でクエリ言語を拡張できる
- IDE エクステンション、サードパーティサービスや API などのサテライトサービスへの GLQL 開放の機会。例えば `glab` のような CLI ツールで GLQL を使用したり、メール通知で GLQL クエリの出力をレンダリングすることが可能になります
- バックエンドを真実の単一情報源として、簡素化されたフロントエンド実装
- データに近い場所でクエリを実行できる
- クエリ実行の並列実行の増加など、バックエンドレベルでの最適化の機会
- GLQL を単なるコンパイラーから、クエリを尋ねて適切なデータを得られる完全なプラットフォームに昇格させます。以前はこの責任はコンシューマーにありましたが、今や GLQL がそれを所有します

さらに、GLQL Rust コンパイラーを持つことで、同じパーサーをフロントエンドとバックエンドの両方のコンテキストで共有できます。

- バックエンド: クエリの解析とデータソースに対するフルクエリの実行
- フロントエンド: クエリの実行なしの構文検証と即座のフィードバック

これは `~devops::plan` の将来の計画とも一致しています: [https://gitlab.com/groups/gitlab-org/-/epics/15834](https://gitlab.com/groups/gitlab-org/-/epics/15834)、コラボレーションの機会を開いています。

拡張された GLQL アーキテクチャとバックエンドへのシフトで構築されたこの標準化されたクエリシステムは、すべての GitLab データソースにわたる強力で一貫したデータ探索体験の基盤を提供します。

### 統一されたデータ探索 UI

統一されたデータ探索 UI は、GitLab の多様なデータソースからのデータをクエリ、分析、可視化するための一体的で直感的なインターフェースを提供します。この UI は GLQL クエリを直接記述することを好む技術的なユーザーと、レポートやダッシュボードを構築するために簡素化されたビジュアルインターフェースを必要とする非技術的なユーザーの両方に対応します。

インターフェースには以下の主要な構成ブロックが含まれる必要があります。

1. **クエリ構築エリア**

   - テキストベースの GLQL エディターとビジュアルクエリビルダーの切り替え
   - GLQL クエリエディター用の構文ハイライト、オートコンプリート、エラー検出
   - 簡単にアクセスできるフィルター
   - 非技術的ユーザー向けの直感的なコンポーネントを持つビジュアルクエリビルダー
   - 利用可能なフィールドとデータタイプを表示するスキーマブラウザー
   - クエリテンプレートと保存されたクエリライブラリ
   - AI アシスト構築

2. **結果表示**

   - 生データ探索のための表形式ビュー
   - グラフやチャート用の可視化キャンバス
   - クエリ構築中のクエリ結果のプレビュー
   - 大きな結果セット用のページネーションコントロール
   - AI アシスト結果解釈

3. **可視化コントロール**

   - 可視化タイプセレクター（グラフ、テーブル、メトリクス、マークダウン、カスタムビュー）
   - 選択した可視化の設定パネル

4. **アクションツールバー**

   - クエリ実行ボタン
   - クエリ保存ボタン
   - 共有/エクスポートオプション
   - クエリ履歴
   - ダッシュボード統合オプション

5. **コンテキストパネル**

   - 現在のクエリに関するメタデータ（実行時間、返された行数、パフォーマンスメトリクスなど）
   - ドキュメントとヘルプリソース

この統一されたインターフェースは標準化されたクエリシステムとシームレスに統合され、GLQL の機能を活用しながら、技術的な専門知識に関係なく、すべてのユーザーがアクセスできる方法で提供します。

上記はデータ探索インターフェースを構築するために必要と考えている主要な構成ブロックですが、これを適切なデザインにするには UX 研究が必要です。

## 代替案

1. **GLQL アーキテクチャを変更せずに GLQL 搭載のデータエクスプローラーを構築する**

   現状の GLQL をそのまま使用し、最初のユースケースとして優れたいくつかのデータソースを追加します。

   - メリット: 大規模なアーキテクチャの変更は不要。GLQL を使用することでいくつかの恩恵を得ることができます。
   - デメリット: GLQL をより高度なクエリ（OR、サブクエリ、集計、算術式）のための GraphQL の制限に縛ります。また、GLQL の現在の制限が扱いにくいと判明した場合に行き詰まる可能性があります。

2. **GLQL なしで、ビジュアルクエリビルダーに依存したデータエクスプローラーを構築する**

   データエクスプローラーの現在の実装の一部、またはオブザーバビリティメトリクスクエリビルダー用に構築されたものを再利用し、複数のデータソース、フィルター、クエリで動作するように拡張します。各データタイプのクエリを処理するためにアドホックなフロントエンドデータソースアダプターが必要になる場合があります。

   - メリット: GLQL への依存なし、大規模なアーキテクチャの変更は不要。
   - デメリット: すべてのデータソースで機能するビジュアルクエリビルダーの構築は、スキーマとフィルタリングの違いにより簡単ではありません。クエリの埋め込み/共有も GLQL ベースのソリューションよりも簡単ではありません。

3. **データエクスプローラーのスコープを制限し、カスタムダッシュボード作成用の事前構築されたクエリ/可視化のセットを提供する**

   データエクスプローラーのスコープを削減し、各データタイプの事前定義されたクエリの拡張可能なセットを各チームが整理します。

   - メリット: GLQL への依存なし、大規模なアーキテクチャの変更は不要、すべてのケースに合うクエリビルダーを構築する必要なし。
   - デメリット: ユーザーがデータを探索する自由度が低くなります。各機能チームはユーザー向けの事前定義されたクエリのセットを決定する必要があります。
