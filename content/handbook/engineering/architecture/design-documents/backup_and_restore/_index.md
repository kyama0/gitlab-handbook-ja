## サマリー

[Unified Backups プロジェクト](https://gitlab.com/groups/gitlab-org/-/epics/11577) は、サポート対象の[リファレンスアーキテクチャ](https://docs.gitlab.com/ee/administration/reference_architectures/index.html)全体にわたる GitLab インストールのアプリケーションのバックアップおよびリカバリのニーズに対応する、単一のコマンドラインツールを提供します。このツールは特定のリリースバージョンと結合しないよう、GitLab のメインコードベースとは別にパッケージ化されますが、GitLab のリリースに同梱されて配布されます。

このツールは各実行時環境設定の細かな違いを把握しており、データを適切に取得・復元するための調整を行います。今後ほとんどの顧客に対する第一の推奨ソリューションとなります。

このツールの初期開発では、さまざまなインストールタイプおよび一般的なアーキテクチャに対応することで、GitLab のセルフホスト顧客に価値を提供することに焦点を当てます。これらの顧客に対しては、災害復旧プロセスを共通の推奨手順にシンプル化することに注力します。さらに、利用量の多い大規模 GitLab インスタンスで利用できるクラウドサービスとの統合機能をサポートすることで、現状のバックアップソリューションのスケーラビリティ問題の解決にも取り組みます。

ただし、私たちはすぐにプロジェクトの次のフェーズに進み、[GitLab Dedicated](https://about.gitlab.com/dedicated/) の特殊なニーズに焦点を当てる予定です。これにより、Google および Amazon のクラウドサービス上で使われている標準的な Dedicated アーキテクチャでのバックアップ作成および復元の自動化・効率化という形で、Dedicated グループに価値を提供します。

## 背景

### 災害復旧

利用頻度が高く、ビジネスにとって重要なアプリケーションは、さまざまな潜在的に災害的なインシデントに対して脆弱です。攻撃者がセキュリティ脆弱性を悪用し、ランサムウェア攻撃でビジネスデータを人質に取るかもしれません。ソフトウェアバグが大規模で予期しないデータ破損を引き起こすかもしれません。クラウドプロバイダーが予期しないアウテージを起こし、不本意ながら異なるタイプのアプリケーションデータの同期がずれることもあるでしょう。これが、利用頻度の高い重要なアプリケーションを運用する組織が一般に災害復旧 (DR) 計画を実装する理由です。DR 計画により、大規模な障害が発生したか日々の運用で重大なミスが発生したかにかかわらず、対象アプリケーションの完全な運用を復旧できます。

信頼性の高いアプリケーションデータバックアップは、あらゆる災害復旧計画にとって極めて重要です。バックアップは、災害復旧を実行する際に後から復元できるよう、特定の時点のアプリケーションデータの状態を取得します。取得したデータは通常、アプリケーションが普段稼働するマシンとは別の場所に、1 つ以上のファイルセットとして保存されます。

### システム管理およびメンテナンス目的

DR がバックアップ作成および復元ツールを構築する第一の動機ではありますが、セルフホストインストールの管理者は設定やメンテナンスの目的でバックアップおよび復元を使う必要が多々あることも認識しています。管理者は、PostgreSQL やオブジェクトストレージアプライアンスのようなシステムの重要コンポーネントのバージョンをアップグレードする必要があるかもしれません。そうした場合、追加の更新を防ぐために GitLab アプリケーションを一時的に停止するのが最善であることがよくあります。次に管理者はインストールの時点バックアップを取得します。続いて、PostgreSQL をより新しいバージョンへアップグレードするなど必要な変更をシステムに加えます。完了したら、メンテナンス作業前と同じ状態に近い状態を保証するために、取得したバックアップデータを復元できます。最後に、サイトを再び一般利用に向けてオンライン化します。

別の例として、管理者は現行の GitLab 本番アーキテクチャを新しいインフラへ移行する必要があるかもしれません。たとえば、ある組織が組織管理の単一マシン上でシンプルな GitLab 1K [リファレンスアーキテクチャ](https://docs.gitlab.com/ee/administration/reference_architectures/index.html) を運用しているとします。しかし、時間とともにこの組織の利用量はシンプルな 1K アーキテクチャのパラメータを超えて成長し、今や管理者は Amazon Web Services (AWS) 経由でホストされるクラウドベースの 3K マルチノードリファレンスアーキテクチャの構築を任されています。これはバックアップおよび復元ツールの代表的な利用ケースです。管理者は AWS 上に 3K デプロイを構築します。次に、稼働中の 1K デプロイのバックアップを取得します。最後に、3K アーキテクチャ上でバックアップを復元することで、組織が小規模アーキテクチャからこの新しい大規模利用アーキテクチャへとシームレスに切り替えられるようにします。

この最後の例で、バックアップツールが顧客の GitLab インストールを別のアーキテクチャへ移行する手段となり得ることを示しました。ただし、この利用ケースにおける重要な制約について述べておく必要があります。私たちは、バックアップが作成されたのと **完全に同じバージョン番号** の GitLab 上でのみアーカイブの復元を許可しています。私たちは過去に、顧客が GitLab バックアップツールを使って GitLab 自体の異なるリリースバージョン間でアプリケーションデータを移行しようとする状況を多く見てきました。例として、ある顧客がバージョン 15.10 のインストールを主に利用しており、時間経過とともに段階的なリリース更新を管理してこなかったとします。今、この顧客はもっと最近の 16.10 のようなバージョンに飛びたいと考えています。その顧客は、Rake バックアップ作成タスクで古いサイトデータの tarball を取得し、より新しいバージョンの GitLab で新しい場所をセットアップし、Rake バックアップ復元タスクを使うだけで新しいバージョンのインストールに同じデータを投入できると考えるかもしれません。しかし、現状のバックアップツールはバックアップ作成時とは異なる GitLab バージョンへの復元をサポートしていません。

現時点では、新しい統合バックアップツールでもこの制約を維持する計画です。バックアップ復元は対象バックアップアーカイブを作成したのと同じバージョンの GitLab 上でのみ行われます。これは、データベーススキーマがそのバージョンが期待するものと完全に一致していることを保証するために不可欠です。さもないと、コードが復元されたデータとは異なるデータスキーマに対してプログラムされていることに起因する、さまざまな程度のバグやデータ破損が発生する可能性があります。プロジェクトの後続のイテレーションでは、ある程度のバージョン横断的な復元サポートを検討する予定です。

### 現行の GitLab バックアップ提供

現在 GitLab は、異なるインストールタイプおよびホスティングアーキテクチャを通じて [アプリケーションバックアップを作成する方法](https://docs.gitlab.com/ee/administration/backup_restore/backup_gitlab.html) について推奨事項を提供しています。時点アプリケーションバックアップを作成するためのかなり初歩的なツール群と、より複雑なクラウドバックアップ状況に対処する方法を示した専用ドキュメントを提供しています。GitLab バックアップの仕組みについては [こちら](https://gitlab.com/gitlab-org/gitlab/-/merge_requests/158058) で詳しく読めます。

これらは GitLab リポジトリの [Rake タスク](https://docs.gitlab.com/ee/raketasks/) に基づいています。

#### 現行アプローチの欠点

1. 現行のバックアッププロセスはスケーラブルではない
1. バックアップはクラウドプロバイダー管理サービスをサポートしていない
1. Rake タスクは適切にメンテナンス可能ではない
1. インストールタイプおよびアーキテクチャによって呼び出し方が異なる: インストールタイプによって、タスクの呼び出し方が変わります。リファレンスアーキテクチャの種類や、オンプレミスかクラウドのインストールかによって、必要なオプションフラグや設定が異なります。この重複と乖離により、バックアップ実行の知識を異なるインストールタイプや異なるリファレンスアーキテクチャ間で転用することが難しくなっています。
1. クラウドサービスに対する手動復元: AWS Backup の継続的・時点オブジェクトストレージバックアップなど、クラウドプロバイダー特有のバックアップ機能を活用するには、現状ではユーザー自身で設定する必要があります。さらに悪いことに、復元も手動で行う必要があります。GitLab はこのプロセスを自動化するためのツールを一切提供していません。

## プロジェクトのゴール

Unified Backups では、以下のゴールを掲げています:

- GitLab インストールのバックアップと復元に必要なすべてのロジックを単一のコードベースに持つ
- 以下を実現する Unified Backup CLI ツールを作成する:
  - アーキテクチャやインストールタイプに関係なくバックアップと復元を制御する
  - 自己ドキュメント化されたオプションを持つシンプルなインターフェースを持つ
  - 認証情報やその他の安定したパラメータを保存できる設定ファイルを持つ
  - JSON フォーマットのオプションのマシン可読出力を持つ
- 同様の UX を持つ 2 つの異なるバックアップフォーマットをサポートする:
  - ポータブルバックアップ
    - 1K および 2K の最小規模のリファレンスアーキテクチャに最適
    - より大規模なオンプレミスインストールでも動作するはず（ただしデータの性質と量によっては遅くなりすぎる可能性あり）
    - マネージドサービスを使わないクラウドベースのインストールでも利用可能
  - クラウドベースのバックアップ
    - クラウドプロバイダーの API とバックアップ機能のみに依存する
    - AWS、GCP、Azure などの主要クラウドプロバイダーをサポート
    - 実行するマシン/コンテナ上で大量のディスク容量や永続的なストレージを必要としない
    - 追加のクラウドプロバイダーへの対応を容易に拡張できる
- 一貫した UX で以下のインストールタイプをサポートする:
  - ソース
  - Linux パッケージ (Omnibus)
  - Docker
  - Kubernetes

### 非ゴール

- これは、ユーザーが自分の GitLab.com グループまたはプロジェクトデータをバックアップ/エクスポートするためのソリューションではありません
- Unified Backup CLI はライフサイクル管理を扱いません:
  - バックアップ保持ポリシーを扱いません
  - バックアップローテーションを扱いません
  - バックアップスケジューリングを扱いません
- Unified Backup CLI は ENV 変数を以下の方法として受け付けません:
  - オプションの提供
  - 設定の提供
  - 認証情報の提供
  - 依存ツールからの設定や構成のオーバーライド
- Unified Backup CLI はデプロイ/インフラ機能を統合しません:
  - GET や Terraform を実行しません
  - 既存のインフラに変更を加えません
  - 新しいインフラをデプロイしません

## 設計および実装の詳細

### 設計上の決定

**集約されたコードベース**

コードベースを単一プロジェクトに集約することで、サポートする環境とインストールタイプの組み合わせの実装をシンプル化することを目指します。すべてが 1 か所にあると、コードのテストと拡張が容易になります。

**複数クラウドプロバイダーのサポート**

新しいクラウドプロバイダーへの対応は、各データタイプのバックアップ方法に関するジェネリックなビジネスロジックと、各クラウドプロバイダー向けの特化バージョンを持つ、アダプターパターンに似たアプローチに従います。

**インストールタイプに応じた設定の取り扱い**

各インストールタイプの違いをサポートするため、ツールに動作中の環境に関する必要な情報を提供するラッパーが利用される場合があります。

**ポータブルバックアップとクラウドバックアップの区別**

ポータブルバックアップとクラウドバックアップの明確な分離は、実装をシンプル化することを目指すと同時に、ダウンタイムなしで一貫性のあるバックアップを取得するための特定戦略を将来探求できるようにします。

両者の主な違いは、ポータブルバックアップがローカルに保存されることです。両スタイルのバックアップは、混在データストレージを許容します（例: アプリケーションデータをローカルとクラウドストレージに分割保存）。

**バックアップの一貫性**

ダウンタイム中に一貫性のあるバックアップを取得できます。

オンライン一貫性バックアップは [エピック 12043](https://gitlab.com/groups/gitlab-org/-/epics/12043) で検討されます。

### 制約事項

- クラウドベースのバックアップ内のデータをポータブルバックアップ形式へエクスポートすること、およびその逆はサポートしません。
- キャッシュストア (Redis) のデータ（[Sidekiq の状態](https://docs.gitlab.com/ee/administration/backup_restore/backup_gitlab.html#other-data) を含む）のバックアップはサポートしません。TODO: [Redis 保存データに関するさらなる調査](https://gitlab.com/gitlab-org/gitlab/-/issues/466000)

### バックアップタイプ

バックアップの作成方法として、ポータブルとクラウドベースの 2 つのアプローチを提供します。

#### ポータブルバックアップ

ポータブルバックアップは、既存の Rake タスクアプローチに似た動作をします。データタイプおよび依存サービスへの直接アクセスに依存します。クラウド固有の機能に依存しないため、最も互換性の高いソリューションです。

ポータブルバックアップは、サポートされているすべてのインストールタイプで実行でき、物理マシンと仮想マシンの両方で動作します。

ポータブルバックアップでは、データはバックアップコンテキストを実行しているマシンに転送され、特定の場所に保存・圧縮されます。マニフェストファイルが作成され、最終アーカイブはローカルまたはリモートの場所（オブジェクトストレージバケットを含む）に保存できます。

各データタイプは順次または並列にバックアップできます。同様に、各データタイプは順次または並列に復元できます。

複数のバックアップを同時に実行でき、片方が他方を上書きすることはありません。（cron ジョブからのバックアップ実行をサポートすることを意図しています）

データベースのバックアップは、Omnibus 管理データベースか、Patroni か、外部管理インスタンスか、また PgBouncer が有効か否かに関係なく、同じように動作し、追加の設定は不要です。

[今後の作業](#future-iterations)では、スナップショット機能を提供できる特定のファイルシステムへの依存も含め、ダウンタイムなしでポータブルバックアップの一貫性を改善することについて検討します。

ポータブルバックアップには、インフラ、データ、DR 要件によりスケールに固有の制約があります。

#### クラウドベースのバックアップ

クラウドベースのバックアップは、データを実行・保存するためにクラウドプロバイダー固有の API およびサービスに完全に依存します。これは必要な API 呼び出しをオーケストレーションする CLI ツールによって起動されます。

バックアップが開始されるたびに、各データタイプを見つけるために必要な情報を含む JSON ファイル（バックアップマニフェスト）が、既存の Blob を保存するのと同じプレフィックス配下のバックアップバケットに保存されます。

バックアップマニフェストは、何がバックアップに含まれるかを指定するための SSOT として利用されます。バックアップを一覧表示または処理するために、ツールはバックアップバケットから JSON ファイルを取得します。そこに保存されている ID と参照に基づいて、ツールはクラウドプロバイダー API で復元アクションを実行できます。

バックアップマニフェストに依存するため、Unified Backup ツールをトリガーして作成されたデータの一覧表示と復元を主にサポートします。これにより、一貫性のあるバックアップに必要な機能と改善を構築するための制御と基盤が得られます。

このツールはクラウドプロバイダー固有のスケジュールバックアップ機能には依存 *しません*。定期的にスケジュールされたバックアップを実現するには、ツールは Cronjob のような外部サービスから呼び出される必要があります。

複数のバックアップを同時に実行できるようサポートできます（特定のクラウドプロバイダー固有の制約がない限り）。バックアップにはサポートされるすべてのデータタイプを含めることも、特定のデータタイプのみを含めることもできます。

サポートされる各クラウドプロバイダーごとに固有の機能が利用できる場合がありますが、データタイプごとに少なくとも以下をサポートします:

- データベース:
  - オンデマンドのデータベースバックアップまたはスナップショット
- オブジェクトストレージ内の Blob
  - 新しいバックアップバケットへのすべての Blob のコピー
- Git リポジトリ
  - 稼働中のすべての Gitaly インスタンスのディスクスナップショット (Gitaly Cluster はサポート対象外)

Gitaly が予定している新しいバックアップ関連機能をリリースしたら、それらの統合を評価します。

バックアップのスケジューリングに関する情報については [#future-iterations] を参照してください。

##### Google Cloud Platform

Google Cloud Platform (GCP) 上でホストされる GitLab インストールに対しては、以下のアプリケーションデータタイプとそれに対応するバックアップ機構をサポートします:

| データタイプ            | GCP サービスコンポーネント            | GCP のバックアップ機構                                          |
|-------------------------|--------------------------------------|-----------------------------------------------------------------|
| PostgreSQL データベース | Cloud SQL for PostgreSQL             | オンデマンドのデータベーススナップショット                      |
| Blob/ファイル           | Cloud Storage (オブジェクトストレージ) | Storage Transfer Service を用いたオンデマンドのリージョンデータ転送 |
| リポジトリ              | Compute Engine と永続ディスク         | オンデマンドのディスクスナップショット                          |

実装を進める中で、同期的・非同期的な操作の違い、バッチング/並列化のサポート、Cloud Logging、Pub/Sub 通知の利用などを含めて、API の使い方の実装詳細を検討します。

###### Cloud SQL for Postgres (バックアップ)

Cloud SQL は以下の方法でバックアップを実行することをサポートしています:

- [自動 (スケジュール)](https://cloud.google.com/sql/docs/postgres/backup-recovery/backups#automated-backups)
  - 特定の開始時刻にスケジュールできる（実行は 4 時間ウィンドウ内で行われる）
  - [自動バックアップの保持期間](https://cloud.google.com/sql/docs/postgres/backup-recovery/backups#what_backups_cost):
    - **Cloud SQL Enterprise edition**: 最小 7 単位 (通常は日数) 最大 365 単位
    - **Cloud SQL Enterprise Plus edition**: 最小 15 単位 (通常は日数) 最大 365 単位
  - [WAL ログの保持期間](https://cloud.google.com/sql/docs/postgres/backup-recovery/backups#retention)
    - **Cloud SQL Enterprise edition**: 1 〜 7 日 (デフォルト: 7 日)
    - **Cloud SQL Enterprise Plus edition**: 1 〜 35 日 (デフォルト: 35 日)
    - WAL ログは同じインスタンスのディスク上または Cloud Storage に保存できる
- [オンデマンドバックアップ](https://cloud.google.com/sql/docs/postgres/backup-recovery/backups)
  - 自動的に削除されない (長期保存可能)

長期の保持期間をサポートするため、初期バックアップサポートはオンデマンドバックアップに依存して提供します。

実装フェーズでは、以下のような追加オプションをどのようにサポートし公開するかを調査します:

- バックアップを保存するカスタムリージョンを選択できる機能 (デフォルト: インスタンスと同じリージョン)
- 自動バックアップの保持ポリシーをカスタマイズできる機能 (通常は Terraform またはインスタンス設定の一部として管理される)

###### Cloud SQL for Postgres (リストア)

初期のリストアサポートはオンデマンドバックアップに基づきます。バックアップバケットから読み込んだバックアップマニフェストファイルに保存された ID に依存します。

実装フェーズでは、以下のような追加オプションをどのようにサポートし公開するかを調査します:

- 自動バックアップの WAL ログに基づく [Point In Time Recovery](https://cloud.google.com/sql/docs/postgres/backup-recovery/restore#tips-pitr) を利用できる機能 (範囲内の場合)
- 現在使用しているのとは異なるインスタンスにバックアップをリストアする機能 (インスタンスのプロビジョニングは扱わない)
- 外部インフラ統合の一部としてデータをリストアするために必要な情報を公開する方法 (例: 新しいデプロイを作成する Terraform スクリプトにデータを提供する)

###### Blob

*バックアップ*

GCP のバックアップは現在 [Storage Transfer Service](https://cloud.google.com/storage-transfer/docs/overview) を利用しています。
個別のバケット (Uploads、LFS など) から `/backups/$BACKUP_ID/$TARGET` というパス配下の単一バケットにコピーする Storage Transfer Service ジョブが作成されます。

*リストア*

- リストアでは、バックアップバケットのパスから個別のバケットにコピーする Storage Transfer Service ジョブを作成する必要があります。
- バケットの作成や設定は行いません。それはツールのスコープ外です。
- (恐らく) バケットを最初に空にすべきではありません。それは必要な場合にユーザーに任せます。
- 最も基本的な使い方では、ユーザーは `gitlab-backup-cli restore all $BACKUP_ID` のようなコマンドを実行し、ツールがバックアップバケットのパスから個別のバケットにデータをコピーする必要なジョブを作成し、成功/失敗をモニタリングします。

## Unified Backup CLI との統合

このツールを外部ツールと統合しやすくするため、JSON フォーマットのオプションのマシン可読出力を提供します。

たとえば、ディスクスナップショットからのリストアでは、ツール自体はその操作を扱わないので、リストアすべきリソースを JSON フォーマットで一覧表示するコマンドを提供します。これは任意の外部統合ツールから読み取り可能です。

このタイプのインターフェースは、Backup CLI を他の特定のツールから疎結合にすることを意図しています。

- 自動/スケジュールバックアップの実装 (AWS Backup や Google Cloud Backup など) には依存しない

このイテレーションでは、バックアップの一貫性の解決を狙いとはしていません:

- 複数のコンポーネントをバックアップする際、データベース内に保存されているものが、バックアップに含まれる前に削除されたものを指している場合があります
- クラウドプロバイダーは、各データを特定の時点に紐付けるための信頼できる方法 (データベーススナップショットと同期するために使えるもの) を提供しないかもしれません
