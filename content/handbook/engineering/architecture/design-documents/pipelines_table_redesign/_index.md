## 概要

**関連 Epic: [#17330: パイプラインテーブルの再設計と GraphQL 移行](https://gitlab.com/groups/gitlab-org/-/epics/17330)**

パイプラインテーブルは GitLab の CI/CD エクスペリエンスの重要なコンポーネントであり、アプリケーション全体の 3 つの主要な場所に表示されます:

1. メインパイプラインページ
2. コミットパイプラインタブ
3. マージリクエストパイプラインタブ

現在の実装は REST API を使用し、各パイプラインのパイプラインミニグラフを含むすべての情報を一度に表示します。これにより、重大なパフォーマンスの Issue とユーザーエクスペリエンスの悪化が生じています。

パイプラインテーブルの REST から GraphQL への移行はすでに計画されているため、このデザインドキュメントでは、この移行を包括的な UI 再設計と組み合わせることを提案します。再設計では、段階的な開示パターンを使用して重要なパイプライン情報と詳細データ（ミニグラフやジョブ情報など）を分離し、GraphQL の実装によってより効率的なデータフェッチが可能になり、将来のリアルタイム更新の基盤が確立されます。

両方の取り組みを同時に実施することで、一時的なソリューションやリワークを避けることができ、開発コストを大幅に削減できます。このアプローチにより、ペイロードサイズとレンダリングの複雑さを削減することで初期ロードパフォーマンスが向上し、より良い情報階層によってユーザーエクスペリエンスが向上し、将来のリアルタイムパイプラインステータス更新に必要な技術的アーキテクチャが確立されます—すべてこれは重要なパイプライン情報の機能パリティを維持しながら達成されます。

## ビジネスへの影響

### ユーザーへの影響

提案された再設計と移行により、GitLab CI/CD エクスペリエンスが大幅に向上します:

- **パフォーマンスの向上**: 初期ロード時間を 50% 以上削減し、開発者の生産性向上に直接貢献
- **情報アクセスの強化**: 段階的な開示を実装して重要な情報を最初に表示し、詳細はオンデマンドで利用可能に
- **トラブルシューティングの効率化**: 拡張ビューで失敗したジョブとアクション可能なアイテムを優先し、MTTR を削減
- **現在の Issue への対処**: パイプラインテーブルの機能に関連する複数のオープン Issue を解決し、長期にわたるユーザーの痛みのポイントに対処
- **リアルタイム更新への準備**: 将来の瞬時のパイプラインステータス更新の基盤を確立

### ビジネス価値

このイニシアチブは以下を通じて実質的なビジネス価値を提供します:

- **インフラストラクチャコストの削減**: ポーリングを排除しペイロードサイズを削減することで、CI/CD 関連のサーバー負荷を 30〜40% 削減
- **開発者の効率性**: より高速なロード時間と改善された情報階層により、毎日数百万のパイプラインインタラクションで大幅な時間節約
- **競争力の強化**: 主要なパフォーマンスの懸念に対処することで GitLab の市場ポジションを強化
- **技術的負債の排除と防止**: 現代的なパターンと技術を積極的に実装することで既存の重大な技術的負債を排除しながら将来のリワークを回避

### リスク軽減

実装戦略には潜在的なビジネスリスクを軽減するためのいくつかの要素が含まれています:

- **フィーチャーフラグアプローチ**: フィーチャーフラグによる段階的なロールアウトにより、重要なワークフローの中断リスクを軽減しながら制御されたテストとモニタリングが可能
- **ユーザー設定トグル**: 現在のインターフェースに大きく依存しているユーザーが新しいデザインに適応しながらも作業を中断せずに継続できるよう、両方の実装を切り替える機能を提供
- **機能パリティ**: すべての重要なパイプライン情報の機能パリティを維持することで、移行中にユーザーが機能を失わないようにする
- **包括的なメトリクス**: 詳細なパフォーマンスとユーザー満足度のメトリクスを収集して改善を検証し、調整が必要な領域を特定

### ROI 計算

予備的な見積もりに基づいて:

| 投資 | リターン |
| ----------------------------------------------------------------------------------------------- | --------------------------------------------------------- |
| エンジニアリング工数: 〜4〜5 マイルストーンにわたる部分的な割り当て（エンジニアの時間の約 30%） | CI/CD 関連のサーバーリソースを 30〜40% 削減 |

| | ユーザー満足度スコアの向上 |
| | 将来のリアルタイム機能の基盤 |

## 動機

### GraphQL 移行

パイプラインテーブルを REST から GraphQL に移行する主な動機は、パイプラインステータス情報のリアルタイム更新を可能にすることです:

- **リアルタイム更新の有効化**: GraphQL サブスクリプションにより、非効率なポーリングメカニズムなしにユーザーへの瞬時のパイプラインステータス変更の配信が可能
- **サーバー負荷の削減**: 現在のポーリングアプローチは、特に活発な CI/CD 使用の環境では不必要にサーバーリソースを消費する頻繁なフルペイロードリクエストを必要とする
- **N+1 クエリの Issue の排除**: 現在の REST 実装はパイプラインデータを取得する際に N+1 クエリパターンに悩まされており、データベースパフォーマンスのボトルネックを生じさせている。GraphQL のクエリをバッチ処理して最適化する機能により、これらのデータベースの非効率性が大幅に削減される
- **フィルタリングパフォーマンスの向上**: GraphQL 実装により、フィルター操作をデータソースに近い場所に移動することで、より効率的なパイプラインフィルタリングが可能になり、不必要なデータ転送と処理を削減
- **ネットワークトラフィックの最適化**: GraphQL は必要なデータを正確に指定できるため、固定データ構造を返す REST エンドポイントと比較してペイロードサイズが大幅に削減される
- **技術的な方向性との一致**: ほとんどの新しい GitLab 機能は GraphQL で構築されており、この移行はプラットフォームの技術的な方向性とアーキテクチャと戦略的に一致
- **将来のスケーラビリティ**: GitLab が CI/CD 機能を拡張し続けるにつれて、GraphQL はますます複雑なパイプラインデータ構造にアクセスするためのより柔軟でスケーラブルなクエリ言語を提供

### UI 再設計

パイプラインテーブル UI を再設計する主な動機は、パイプラインミニグラフを親 API リクエストに含めることによる重大なパフォーマンスの Issue に対処することです。

- **初期ペイロードの過剰**: 現在の実装はミニグラフの詳細なステージとジョブデータを初期リクエストに含めており、各パイプラインのペイロードサイズと処理時間が大幅に増加
- **情報アーキテクチャの改善**: 現在のアプローチは一度に多くの情報を表示し（ヒックの法則）、ユーザーがパイプラインステータスを素早く識別することが困難。再設計により最も重要な情報を優先できる
- **不必要なデータローディング**: ほとんどのユーザーはパイプラインのごく一部に対してのみ詳細なミニグラフ情報を必要とするが、現在はすべての表示パイプラインに対してこのデータを読み込んでいる
- **段階的な開示の機会**: ミニグラフとその他の詳細情報をセカンダリリクエストで読み込まれる展開可能なビューに移動することで、すべてのパイプライン詳細へのアクセスを提供しながら初期ロードパフォーマンスを大幅に向上させることができる
- **パイプライン詳細の強化**: 詳細情報をセカンダリリクエストに移動することで、初期ページパフォーマンスに影響を与えることなく、より豊かな視覚化、包括的な失敗ジョブ情報、より詳細なメトリクスを含むより堅牢なパイプライン詳細コンポーネントを構築できる

## ゴール

このイニシアチブの主なゴールは以下の通りです:

### パフォーマンスの改善

- パイプラインテーブルの初期ページロード時間を少なくとも 50% 削減
- 初期データペイロードのサイズを少なくとも 60% 削減
- 初期 DOM 要素数を削減することでクライアントサイドのレンダリングパフォーマンスを向上
- 効率的に設計された GraphQL クエリと将来のサブスクリプション機能を優先してポーリングを排除

### ユーザーエクスペリエンスの向上

- 重要なパイプライン情報と詳細データを分離することで情報階層を改善
- ユーザーがパイプラインステータスを素早くスキャンできる、より直感的なインターフェースを作成
- すべての重要なパイプライン情報の機能パリティを維持
- ユーザビリティテストで少なくとも 80% のユーザー満足度を達成

### 技術的アーキテクチャ

- すべてのパイプラインテーブル機能について REST から GraphQL への移行を完了
- 将来のリアルタイム更新をサポートするコンポーネントアーキテクチャを設計・実装
- リストレベルのデータと詳細なパイプライン情報の間の分離を作成

### 開発効率

- UI 再設計と GraphQL 移行の両方を単一の調整された取り組みで実施
- 一時的なソリューションや技術的負債を避ける
- フィーチャーフラグの背後で開発して制御されたテストとロールアウトを可能にする

## 非ゴール

- リアルタイム更新の実装（ただし基盤の構築はスコープ内）
- パイプラインミニグラフの再設計
- 基盤となる CI/CD 実行エンジンへの変更
- 他の CI/CD 機能の GraphQL への移行（ジョブログ、ランナー管理など）
- パイプラインの作成や設定ワークフローへの変更

## 提案

提案されたソリューションは、REST から GraphQL への移行とともにパイプラインテーブルの完全な再設計を含みます。新しいデザインは段階的な開示パターンを採用し、リストレベルでは重要な情報のみを表示しながら、ユーザーが個々のパイプラインエントリを展開して詳細情報にアクセスできるようにします。

このアプローチでは 2 つの別々の GraphQL クエリを使用します:

1. リストビュー用の重要なパイプラインデータのみを読み込む軽量の初期クエリ
2. ユーザーが特定のパイプラインを展開した場合にのみトリガーされる、より詳細な 2 番目のクエリ

これら 2 つのイニシアチブを単一の調整された取り組みに組み合わせることで、開発コストを最小限に抑えながら、使いやすさを向上させ将来のリアルタイム更新の舞台を整えつつ、大幅なパフォーマンス向上を実現できます。

### 情報アーキテクチャ

新しいデザインはパイプライン情報を 2 つのレベルに分離します:

1. **リストビュー（重要な情報）**

   - パイプラインステータスと完了タイムスタンプ
   - パイプライン ID と名前
   - ブランチ、コミット、MR 情報
   - 作成者情報
   - クイックアクション

2. **展開ビュー（詳細情報）**

   _ユーザーがパイプライン行を展開した際に表示_

   - 可能なパイプラインミニグラフの視覚化（[パイプラインミニグラフの配置](#パイプラインミニグラフの配置)の決定を参照）
   - 失敗したジョブ情報
   - ジョブアクション
   - 所要時間のメトリクス
   - 利用可能なアーティファクト
   - 詳細なコミット情報
   - 設定の詳細

## 設計と実装の詳細

### 実装アプローチ

実装は以下のようにフェーズ分けされます:

**フェーズ 1: 設計とインフラストラクチャ**（1〜2 マイルストーン）

- UI/UX 要件を確定するための製品デザインセッションの実施【**デザイン**】
- リストビューと展開詳細の両方のデザイン作成【**デザイン**】
- 情報階層とユーザーインタラクションの定義【**デザイン**】
- フィーチャーフラグインフラストラクチャの導入【**BE/FE**】
- `usePipelinesListView` を追加するためのユーザー設定スキーマの更新（[ユーザー設定の実装](#ユーザー設定の実装)を参照）【**BE**】

**フェーズ 2: 基盤とアーキテクチャ**（1 マイルストーン）

- 確定したデザインに基づく GraphQL スキーマの更新【**BE**】
- クエリリゾルバーのスタブ作成【**BE**】
- 詳細クエリのビルドとテスト【**BE**】
- リストビューのベースコンポーネント構造【**FE**】
- 詳細ビューのベースコンポーネント構造【**FE**】

**フェーズ 3: 実装と統合**（1〜2 マイルストーン）

- クエリリゾルバーとロジックの実装【**BE**】
- 3 か所すべてのユーザー設定トグルの実装【**FE**】
- UI 子コンポーネントのビルド【**FE**】
- クエリ統合【**FE**】
- パフォーマンステストの実施【**BE/FE**】
- ユーザーテストの実施【**デザイン/FE**】

**フェーズ 4: ロールアウトとモニタリング**（1 マイルストーン）

- フィーチャーフラグを徐々に有効化
- パフォーマンスメトリクスとトグル使用パターンのモニタリング
- 専用フィードバックチャネルを通じてユーザーフィードバックを収集
- フィードバックとメトリクスに基づいてデフォルトのトグル設定を調整

#### ユーザー設定の実装

{{% details summary="詳細" %}}
**決定 DRI**: プロダクトデザイン、エンジニアリング

**決定**: [決定保留中] 現在のパイプラインテーブルと新しい再設計バージョンを切り替えるためのユーザー設定トグルを実装するかどうか。

**コンテキスト**: 現在の REST ベースの実装から GraphQL 再設計に移行するにあたり、移行期間中に両方のバージョンにアクセスできるスムーズな移行パスをユーザーに提供する必要があります。

**実装**:

ユーザー設定は GraphQL ミューテーションで制御されます:

```graphql
mutation updateUsePipelinesListView($usePipelinesListView: Boolean) {
    userPreferencesUpdate(
        input: { usePipelinesListView: $usePipelinesListView }
    ) {
        userPreferences {
            usePipelinesListView
        }
    }
}
```

フロントエンドには、ワークアイテムのトグルに類似したトグルコンポーネントが含まれます:

- 新しいデザインが有効かどうかを示すバッジを表示
- ビューを切り替えるトグルコントロールを持つポップオーバーを提供
- 専用のフィードバック Issue へのリンクを含む
- GraphQL ミューテーションを通じて設定の永続化を処理

**ロールアウト戦略**:

1. **初期フェーズ**: 機能は GitLab 内部ユーザーのみに対して有効化され、トグルのデフォルトは「オフ」
2. **ベータフェーズ**: 信頼が高まるにつれて、フィーチャーフラグはより広いオーディエンスに対して有効化されますが、トグルのデフォルトは「オフ」のまま
3. **段階的なロールアウト**: フィードバックとパフォーマンスメトリクスに基づいて、トグルは新規ユーザーに対して徐々に「オン」にデフォルト設定
4. **完全なロールアウト**: 最終的にトグルは削除され、再設計された実装が標準になる

このアプローチにより、ユーザーは新しいデザインを試しながら慣れ親しんだインターフェースへのアクセスを維持でき、フィードバック収集が容易になり、フォールバックメカニズムが提供され、よりスムーズな移行エクスペリエンスが実現します。

#### パイプラインミニグラフの配置

{{% details summary="詳細" %}}
**決定 DRI**: プロダクトデザイン、エンジニアリング

**決定**: [決定保留中] 詳細ビューにパイプラインミニグラフを含めるか、失敗したジョブとアクション可能なアイテムに焦点を当てるか。

**コンテキスト**: ミニグラフには、ユーザーのワークフローにとってしばしば不要な広範なジョブデータ（名前とステータスを含む数十の完了したジョブを含む）のフェッチが必要です。

**オプション**:

1. **詳細ビューにミニグラフを含める**

   - パイプラインステージの視覚的な表現を提供
   - 親しみやすい視覚化要素を維持
   - アクション不可能なものを含むすべてのジョブのデータフェッチが必要

2. **失敗したジョブとアクション可能なアイテムに焦点を当てる**

   - ユーザーがアクションを取るために必要な情報を優先
   - 関連するジョブに焦点を当てることでデータ要件を削減
   - 個々の完了したジョブの詳細なしにステージの概要を表示
   - トラブルシューティングワークフローにとってより有用な可能性がある

**次のステップ**:

- 最も頻繁に必要とされる情報を特定するためにユーザーワークフローを調査
- データ要件の削減を測定するためにクエリパターンを分析
- アクション可能な情報に焦点を当てた代替プレゼンテーションをテスト

**初期推奨事項**: 詳細ビューでは主に失敗したジョブとアクション可能なアイテムに焦点を当て、すべての完了したジョブのデータフェッチを必要としない簡略化されたステージの概要を提供する。

### GraphQL クエリ

パイプラインテーブルの REST から GraphQL への移行では、2 つの主要なクエリパターンを作成します:

1. 各コンテキストのリストビューに重要なパイプライン情報のみを取得する軽量リストクエリ
2. 特定のパイプラインの包括的な情報をフェッチする詳細クエリ

{{% details summary="<h4>リストクエリ</h4>" %}}
リストクエリはパフォーマンスのために最適化され、初期リストビューに必要なデータのみを取得します。このアプローチは現在の REST 実装と比較してペイロードサイズを大幅に削減します。ユーザーが現在の機能を失わないように、最初からページネーションと基本的なフィルタリング機能を実装し、必要に応じて追加のフィルターを段階的に追加します。

すべてのリストクエリの一貫性を確保するための共通フラグメントを定義します:

```graphql
# fragments/pipeline_list_fields.fragment.graphql
fragment PipelineListFields on Pipeline {
    id
    iid
    detailedStatus {
        ...CiIcon
    }
    createdAt
    finishedAt
    user {
        id
        name
        avatarUrl
        webUrl
    }
    commit {
        id
        shortId
        webUrl
    }
    mergeRequest {
        id
        webUrl
        reference
    }
    retryable
    cancelable
}
```

次に、パイプラインテーブルが表示される各コンテキストに対して 3 つの特定のクエリを実装します:

**プロジェクトパイプラインクエリ**:

```graphql
# queries/project_pipelines.query.graphql
query getProjectPipelines(
    $projectPath: ID!
    $first: Int
    $after: String
    $filters: PipelineFilterInput
) {
    project(fullPath: $projectPath) {
        id
        pipelines(first: $first, after: $after, filters: $filters) {
            pageInfo {
                hasNextPage
                endCursor
            }
            nodes {
                ...PipelineListFields
                # プロジェクト固有のフィールドをここに追加
            }
        }
    }
}
```

**マージリクエストパイプラインクエリ**:

```graphql

```

## 代替ソリューション

### UI 再設計なしの GraphQL 移行

**アプローチ**: 現在の UI デザインを維持しながら GraphQL に移行する。

**メリット**:

- いくつかのパフォーマンス向上を提供
- 段階的な改善が可能
- ユーザーエクスペリエンスの中断リスクが低い

**デメリット**:

- 現在の REST ペイロードに一致する GraphQL クエリを構築する必要がある
- UI が最終的に再設計されると技術的負債が生じる
- 情報階層や認知負荷の Issue に対処しない
- 組み合わせた実装効率の機会を逃す

### クエリパフォーマンスの比較

提案された変更のパフォーマンス向上を徹底的に評価するために、制御されたテスト環境で同じテストデータセット（プロジェクトパイプライン）を使用して 3 つの異なるクエリアプローチの比較分析を実施しました:

#### クエリアプローチの比較

| アプローチ | 説明 | クエリの複雑さ | サーバー応答時間 | ペイロードサイズ | クライアント処理時間 |
| --------------------------- | --------------------------------------------------------------------------------- | ------------------------------------------------------ | -------------------- | ------------ | ---------------------- |
| 現在の REST 実装 | 単一リクエストでミニグラフを含むすべてのパイプラインデータをフェッチ | 高（ステージ/ジョブの N+1 クエリ） | 1.35 秒 | 256 KB | 450 ms |
| UI 再設計なしの GraphQL | 現在の REST API と同じデータ構造を維持する GraphQL 実装 | 中（最適化されたクエリだがすべてのデータをフェッチ） | TBD | TBD | TBD |

#### 詳細なクエリ分析

##### 現在の REST 実装

**エンドポイント**: `GET /api/v4/projects/:id/pipelines?per_page=20`

- **複雑さ**: ステージとジョブのためにパイプラインごとに複数のデータベースクエリをトリガー
- **返却データ**: すべてのステージとジョブを含む完全なパイプラインデータ
- **欠点**: 表示されない可能性のある情報に対する重大なオーバーヘッド

##### UI 再設計なしの GraphQL

**エンドポイント**: 包括的なパイプラインデータクエリを持つ GraphQL API

- **複雑さ**: バッチローディングによる削減されたデータベースクエリ
- **返却データ**: REST と同じ包括的なデータセット
- **予想される改善**: GraphQL の最適化によるいくつかの効率向上

##### UI 再設計ありの GraphQL

**プライマリエンドポイント**: 軽量パイプラインリストクエリを持つ GraphQL API
**セカンダリエンドポイント**: 詳細なパイプライン情報クエリを持つ GraphQL API（オンデマンドのみ）

- **複雑さ**: 重要なデータのみの最小データベースクエリ
- **返却データ**: 初期リストビューに必要なデータのみ、詳細は個別にフェッチ
- **予想される改善**: ペイロードサイズと処理時間の大幅な削減

#### テスト手法

パフォーマンスメトリクスは以下の手法で収集されました:

- テスト環境: 本番 GitLab インスタンス
- テストプロジェクト: 20 パイプラインの中規模プロジェクト（標準的なページネーションサイズ）
- 測定ツール: ブラウザデベロッパーツール、GitLab パフォーマンスモニタリング
- キャプチャされたメトリクス: サーバー処理時間、ペイロードサイズ、クライアントレンダリング時間
- サンプルサイズ: 実装ごとに 20 リクエストの平均

#### 予想されるパフォーマンス向上

予備テストに基づいて、GraphQL UI 再設計アプローチで以下の向上を期待しています:

| メトリクス | 予想される改善 |
| --------------------- | -------------------- |
| サーバー応答時間 | 65〜70% 削減 |
| 初期ペイロードサイズ | 75〜80% 削減 |
| クライアントレンダリング時間 | 50〜60% 削減 |
| インタラクティブになるまでの時間 | 60〜65% 削減 |

これらの期待値は実装中に包括的なテストで検証され、実際の測定値は利用可能になった時点で追加されます。
