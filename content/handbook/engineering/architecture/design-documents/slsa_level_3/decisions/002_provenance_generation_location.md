## コンテキスト

SLSA（Supply chain Levels for Software Artifacts）は、ソフトウェアアーティファクトの完全性を保証するためのセキュリティフレームワークです。SLSA Level 3 準拠の実装の一環として、GitLab は CI/CD パイプラインが生成するアーティファクトのプロベナンスステートメントを生成する必要があります。これらのステートメントには、リポジトリ URL、コミット SHA、ビルド ID、Runner ID、CI/CD 変数、アーティファクトダイジェストなど、アーティファクトのビルド方法に関するメタデータが含まれます。

現在、SLSA 1.0 プロベナンスステートメントは Runner ヘルパーによって生成されています。しかしこのアプローチでは、プロベナンスはコントロールプレーン（信頼境界内）で生成される必要があり、ビルドプラットフォームのテナントが生成してはならないという SLSA Level 3 要件を満たすことができません。

この ADR は、GitLab のアーキテクチャ内でどこで SLSA プロベナンスステートメントを生成するかについての決定を文書化したものです。

## 検討したオプション

### 1. Runner ヘルパー（現在の実装）

**メリット:**

- すでに実装済み。
- ビルドジョブによって読み書きされたファイルを追跡できる（未実装）。
- 単一の CI/CD 変数で有効化できる簡単なセットアップ。

**ブロッカー:**

- コントロールプレーン外にある。

### 2. Runner

**メリット:**

- ビルドジョブによって読み書きされたファイルを追跡できる。

**デメリット:**

- Runner のアイデンティティを信頼できない。自己申告の Runner ID は偽造される可能性がある。

**ブロッカー:**

- コントロールプレーン外にある。
- アーティファクトへのアクセス権がなく、そのダイジェストを取得できない。アーティファクトは Runner ヘルパーによってアップロードされる。

### 3. GitLab Rails バックエンド

**メリット:**

- コントロールプレーン内にある。
- デプロイ、メンテナンス、監視が容易。
- 簡単なセットアップ：プロベナンス生成を GitLab UI で有効化できる。
- コンプライアンス：CI/CD 設定に関わらず、プロベナンスを強制できる。
- セキュリティ：Rails バックエンドは Runner のアイデンティティを確認できる。
- 必要なメタデータ（ジョブペイロードとアーティファクトダイジェスト）にすでにアクセスできる。

**デメリット:**

- ビルドジョブによって読み書きされたファイルを追跡できない。

### 4. GitLab Workhorse

**メリット:**

- プロベナンスステートメントに類似したアーティファクトメタデータをすでに生成している。
- アーティファクトへのアクセス権がある。Workhorse はアーティファクトをオブジェクトストレージにアップロードし、ダイジェストを生成し、アーティファクトメタデータを Rails バックエンドに投稿する。

**ブロッカー:**

- 必要なすべてのメタデータを提供できない。
- 信頼できる方法で Runner のアイデンティティを確認できない。

### 5. CI/CD コンポーネント

**メリット:**

- 可視性：CI/CD カタログで公開される。
- メンテナンス：新しいリリースをいつでも公開できる。

**デメリット:**

- エアギャップ：CI/CD コンポーネントは GitLab に同梱されないため、追加のセットアップが必要。
- ユーザーにとって複雑なセットアップ。ユーザーはビルドメタデータをプロベナンス生成コンポーネントに渡すために CI/CD パイプラインを設定する必要がある。

**ブロッカー:**

- コントロールプレーン外にある。

### 6. 新サービス

**メリット:**

- 明確な信頼境界。
- 独立したスケーリング。

**デメリット:**

- 維持すべき新しいサービス。
- 追加のインフラストラクチャが必要。
- 大きなインテグレーション作業が必要。

**ブロッカー:**

- プロベナンスメタデータへの直接アクセスがない。

## 決定

**SLSA プロベナンスステートメントを GitLab Rails バックエンドで生成します。**

これは、プロベナンスがコントロールプレーン内で生成されなければならないという SLSA Level 3 の重要な要件を満たし、かつ必要なすべてのビルドメタデータを提供できる唯一のオプションです。

Rails バックエンドは、完全なプロベナンスステートメントを生成するために必要なすべてのメタデータにアクセスできます:

1. ビルド ID（ジョブレコードから）
2. Runner ID（バックエンドによって検証済み）
3. ビルド ID（ジョブレコードから）
4. Runner ID（バックエンドによって検証済み）
5. リポジトリ URL とコミット SHA（ジョブペイロードから）
6. CI/CD 変数（ジョブペイロードから）
7. アーティファクトダイジェスト（バックエンドからアクセス可能な `ci` データベースに保存）

プロベナンスステートメントの生成によるパフォーマンスへの影響は、ジョブ完了時に Rails バックエンドが行う他の操作と比較して最小限と予想されます。
必要に応じて、プロベナンス生成をバックグラウンドジョブで処理してパフォーマンスへの影響をさらに最小化することもできます。

## 結果

### ポジティブ

- SLSA Level 3 準拠が達成可能になる。
- プロベナンス生成が既存の GitLab インフラストラクチャに統合される。
- GitLab UI 設定によりユーザー体験が簡素化される。
- 追加のサービスやインフラストラクチャが不要。

### ネガティブ

- ビルドジョブが読み取ったファイルを追跡できない。
これらのファイルを追跡すると `resolvedDependencies` フィールドを生成できるようになる。
ただし、SLSA L3 要件ではそのフィールドの完全性はベストエフォートと定められている。
- ビルドジョブが書き込んだファイルを追跡できない。
これらのフィールドを追跡すると `byProducts` フィールドを生成できるようになる。
ただし、このフィールドは必須ではない。

### 将来の検討事項

- プロベナンスステートメントの署名については、Issue [#537060](https://gitlab.com/gitlab-org/gitlab/-/issues/537060) で別途対応している。
- 将来の機能強化として、コントロールプレーンでプロベナンスを生成しながら Runner との連携を通じて `resolvedDependencies` と `byProducts` を取得する方法を検討する可能性がある。
