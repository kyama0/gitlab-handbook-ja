## コンテキスト

GitLab で KEV データのサポートを実装しています。
最初は EPSS データとは別に KEV データを保存することを検討しました。
しかし、[議論](https://gitlab.com/groups/gitlab-org/-/work_items/11912#note_2075459748) の後、
KEV と EPSS データを CVE エンリッチメントとして統合することで、パフォーマンスとシンプルさの面で
利点が得られることに気づきました。

統合アプローチはデータ管理とクエリを簡素化します。KEV と EPSS データを単一のテーブルに統合することで、
これらの CVE ベースのエンリッチメントに対してより効率的な構造を作成します。この設計により、
結合の数を減らすことでより効率的な DB クエリが可能になります。

重要な点として、KEV データには約 1,000 件のレコードが含まれていますが、EPSS データには約 250,000 件の
レコードが含まれています（これらの数値は時間とともに増加します）。
KEV データは、脆弱性が既知の悪用であるかどうかを示すブール値に限定されています。
この追加のブールフィールドの小さなサイズと、EPSS レコードと比較した KEV レコードの相対的に少ない数は、
既存の EPSS データ構造への最小限のオーバーヘッドを追加するだけなので、これらのデータセットを組み合わせる決定を支持します。

現在の EPSS 実装にはすでに設計が整っており、この統合アプローチのために活用して拡張できます。
さらに、API はこれらのバックエンドの変更に影響を受けず、現在の構造と使いやすさを維持します。

## 決定

以下を決定しました:

1. PMDB で KEV と EPSS データの別テーブルを維持します。
2. KEV と EPSS データの両方を単一の共有 GCP バケットにエクスポートします。
3. GitLab DB の新しい `pm_cve_enrichments` テーブルに統合データを同期して取り込むための単一プロセスを実装します。
4. `pm_cve_enrichments` テーブルを以下の構造にします:
    - cve_id
    - epss_score（nullable）
    - has_known_exploit（nullable）
5. `pm_cve_enrichments` テーブルからのみエンリッチメントデータにアクセスするための GraphQL クエリを使用します。

## 結果

### ポジティブな結果

- 単一の結合のみを必要とする統合テーブルに EPSS と KEV を保存することで、CVE エンリッチメントデータ取得の
  SQL クエリパフォーマンスが向上します。
- バックエンドのデータモデルの複雑さを軽減します。
- KEV と EPSS データの両方に共有フィーチャーフラグ、同期、取り込み、エクスポートプロセスを使用することで
  コードの重複を排除します。
- KEV データが単一のブールカラムに制限されているため、upsert クエリと [Write Ahead Log](https://www.postgresql.org/docs/current/wal-intro.html) への最小限の影響。

### ネガティブな結果

- EPSS データが常に完全にエクスポートされているため、毎回完全なデータエクスポートが必要となり、
  KEV のカーソル機能が失われます。
- KEV テーブルと EPSS テーブルを結合するため、PMDB のエクスポート時に 1 回のパフォーマンスコストが発生します。

## 代替案

1. KEV データを別に保存する:
    - 利点:
        - 関心の分離を維持します。
        - KEV のカーソル機能を保持します。
        - GitLab DB での KEV データ更新の頻度が減ります（EPSS と比較してデータセットが小さく、変更が少ないため）。
        - KEV データと EPSS データの取り込み中に [Write Ahead Log](https://www.postgresql.org/docs/current/wal-intro.html) を削減します。
    - 欠点:
        - GraphQL クエリが 2 つのテーブルを JOIN する SQL クエリをトリガーします。
        - 2 つの別々のデータセットの取り込みと同期の複雑さが増します。

2. KEV データを脆弱性テーブルに非正規化する:
    - 利点:
        - 脆弱性をクエリする際の KEV データへの高速アクセス。
    - 欠点:
        - データの冗長性。
        - EPSS と KEV の間で異なるアプローチとなり、一貫性のないデータ管理になります。
        - 取り込みと同期プロセスの重複。

3. KEV データを `pm_advisories` に保存する:
    - 利点:
        - アドバイザリ関連データをまとめて保持します。
        - アドバイザリの既存のエクスポート、取り込み、同期プロセスを活用し、新しい実装の必要性を軽減します。
    - 欠点:
        - GraphQL クエリにアドバイザリとの結合が必要であり、パフォーマンスに影響を与える可能性があります。
        - アドバイザリデータとは独立して KEV データにアクセスする必要がある場合、複雑さが増す可能性があります。
