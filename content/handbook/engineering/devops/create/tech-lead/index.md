## テックリードとドメインエキスパートの違い

テックリードとドメインエキスパートは共通点と相違点があります。以下の表は、2 つの役割の区別を明確にします。

### 共通点

| 基準                      | テックリード                                     | ドメインエキスパート                                |
| ----------------------------- | --------------------------------------------- | -------------------------------------------- |
| **専門知識の要件**      | ドメインの知識と専門知識を要求       | 特定の領域での十分な経験を要求 |
| **コラボレーション**              | エンジニアリングマネージャーとプロダクトマネージャーとのコラボレーション | チームメンバーとステークホルダーとのコラボレーション |
| **メンタリング**                 | 技術的なガイダンスとメンタリングを提供     | 特定の領域でチームメンバーをメンタリング   |
| **管理職ではない**      | マネージャーではない                                  | マネージャーではない                                 |

### 相違点

| 基準                          | テックリード                                     | ドメインエキスパート                                |
|-----------------------------------|-----------------------------------------------|----------------------------------------------|
| **役割の性質**                | 特定のトピック/プロジェクトに関連する一時的な役割 | 十分な専門知識を持つ継続的な役割     |
| **割り当て基準**           | 効率性、ドメイン知識、専門知識        | 特定の領域での十分な経験     |
| **シニアリティ**                     | シニアリティに関わらず、すべてのエンジニア          | 経験のあるチームメンバーに推奨 |
| **技術的ビジョンの提供**    | 技術的ビジョンとアーキテクチャの提供   |  不要 |
| **ガイダンスとメンタリング**        | 技術的なガイダンスとメンタリングの提供     | ドメインでのコードと MR レビューの実施 |
| **作業の計画と優先順位付け**| 作業の計画と優先順位付け                 | 不要           |
| **進捗の追跡と報告**| コミットメントの進捗追跡と報告 | ドメイン内の進捗追跡と報告を確実にし、達成、課題、目標に関する洞察を提供 |
| **リスク管理**               | 成果物に影響する可能性のある技術的リスクの特定、評価、管理 | 特定の技術、プロダクト機能、またはコードベース領域に関連する技術的リスクの管理 |
| **コーディネーション**                  | 他者の作業を監督し、ブロッカーの解消を支援 | ドメイン内のチームメンバーと調整し、スムーズなコラボレーションを確保し、課題に対処 |
| **コラボレーション**                 | コラボレーションのための Slack チャンネル（#tech_leads またはトピック/プロジェクト固有のチャンネル）  | 不要 |
| **プロジェクトテンプレート**              | ガイダンス用のプロジェクトテンプレートを活用       | 不要    |
| **スコープ**                         | 特定のトピック/プロジェクトに限定              | 通常、特定の技術、機能、またはコードベース領域を包括                                                      |

## プロセス

### プロジェクトにテックリードが必要かどうかを判断する

一部のプロジェクトでは、テックリードを持つかどうかはチームの利用可能なキャパシティに依存します。エンジニアリングマネージャーはチームのテックリーダーシップキャパシティに基づいて決定します。プロジェクトにテックリードが必要かどうかを評価するために、以下の質問を活用してください。

1. プロジェクトは技術的に複雑ですか？ブループリントが必要ですか？
1. 複数の技術や統合を含みますか？
1. プロジェクトは独自の技術専門知識を要求しますか？
1. プロジェクトは技術的側面に関連する高いリスクを持っていますか？
1. プロジェクトの成功はビジネス全体の目標にとって重要ですか？
1. プロジェクト中に複雑な技術的意思決定が必要ですか？
1. プロジェクトは経験豊富なガイダンスを必要とするアーキテクチャの選択を含みますか？
1. プロジェクト完了のためのタイトなデッドラインがありますか？
1. 技術的なチームメンバーと非技術的なチームメンバーとの間の効果的なコミュニケーションが必要ですか？

### プロジェクトにテックリードを任命する

| ステップ | エンジニアリングマネージャーの責任 |
|------|---------------------------------------|
| 1    | プロジェクトのニーズを評価する                  |
| 2    | テックリーダーシップキャパシティを評価する     |
| 3    | テックリードを選定する         |
