## どのように進捗を監視するか？

Create ステージは、ログを非同期レトロスペクティブの一種として使用します。改善に焦点を当てているメトリクスがある場合、そのメトリクスに関連する行動のログを取るという追加ステップを踏みます。メトリクスを見て、**「この数字の背後にあるストーリーは何か？」**と自問します。

## ストーリーはトレンドにつながる

チーム、ステージ、または組織全体のメトリクスを見ることで、ストーリーが語られ始めます。各メトリクスはパンくずリストです。ストーリーをつなぎ合わせていくと、運が良ければトレンドに気づくことができます。トレンドを特定したら、変化を加えてその変化の影響を測定し始めることができます。その変化は、メトリクスやトレンドを変えるかもしれないし、変えないかもしれません。それは大きな実験であり、学習と改善の機会です！

## スループットログ

Dev サブ部門全体が毎月スループットログにエントリを作成します。Dev の各エンジニアリングマネージャーは、この[ドキュメント](https://docs.google.com/document/d/1i86_FNDNe2kX9krju5YOOkFCmqN2txFVlFEzH2nuknY/edit#heading=h.7f7vegoxp3x6)（内部）を更新する責任があります。

スループットログには以下の情報が含まれます。

- チーム名
- 2ヶ月前の MR レート
- 先月の MR レート
- 2ヶ月を比較して MR レートがどれだけ増減したかを示すパーセンテージ
- MR レートの変化につながったイベント / 活動 / プロセスを説明する箇条書きリスト

このログは Dev ディレクターにサブ部門全体の可視性を提供します。このログは、ディレクターがディレクターの下に報告するすべての個人コントリビューターのサポートのために必要に応じて行動できる透明性を与えます。

## マージまでの日数ログ

Create ステージのエンジニアリングマネージャーは毎月、各自の機密のマージまでの日数ログにエントリを作成します。各エンジニアリングマネージャーは、マージまでに8日以上かかったすべてのマージリクエストを分析し、そこから学べる教訓があるかどうかを判断する責任があります。

マージまでの日数ログには以下の情報が含まれます。

- チームメンバー
- マージまでの日数
- マージリクエスト（リンク付きタイトル）
- チームプロセスに関連しているか？もしそうなら説明してください。
- 個々のチームに関連しているか？もしそうなら説明してください。
- これは分割できなかった例外的な MR か？

このログは、エンジニアリングマネージャーに自チームの働き方についてより高い透明性を提供します。月ごとに共通のテーマがある場合、そのテーマはエンジニアリングマネージャーが注力すべき領域のガイドとして機能します。

シニアエンジニアリングマネージャーはこのログからステージ全体の可視性を得ます。シニアエンジニアリングマネージャーがエンジニアリングマネージャーの取り組みをサポートする機会があるかもしれません。
