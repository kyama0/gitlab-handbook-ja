## どのようにイテレーションを行いますか？

私たちは**イテレーション振り返り**を月次で実施しています。イテレーション振り返りの目的は、チームとしてイテレーションをどのように改善できるか、または成功したイテレーションから教訓を学ぶための時間を設けることです。イテレーション振り返りから得られた教訓は、多くの人にとって成長の機会となります。

Create ステージの全チームがこれらの振り返りを実施し、チーム横断での優良事例と経験の共有を促進します。

チームは**成功した例または失敗した例**のどちらでもイテレーションの対象として選択できます。

## 頻度

これらの振り返りは四半期ごとに実施します。

## イテレーション振り返りの前に

チームメンバー / エンジニアリングマネージャーは過去に取り組んだ Issue を振り返り、イテレーション振り返りの良い（難しい）候補を1つ見つけます。候補が選ばれたら振り返り Issue を作成します。

## イテレーション振り返りの最中

チームメンバーとエンジニアリングマネージャーは、Issue 内で非同期的に問題を分解する代替方法について議論します。この活動は、チームメンバーが1週間以上関与しなくて済むようにタイムボックス化します。

## イテレーション振り返りテンプレート（任意）

以下はご利用いただけるテンプレートの例です：

**Issue の説明に含めるべき内容：**

+ Issue の要約
+ チームはイテレーションの目標を達成できましたか？その理由は？
+ イテレーションを完了するために作成された MR の数
+ MR に関連する MR のマージまでの日数
+ 仲間やステーブルカウンターパートとのコラボレーションにおける成功または改善の機会はありますか？
+ 元の Issue をより小さなコンポーネントに分解できましたか？
+ MR をより小さなコンポーネントに分解できましたか？

**コメント 1**：テーマ：Issue / MR のどの側面がイテレーションの優れた例であり、それはなぜですか？

+ チームメンバーはイテレーションがいかにうまく適用されたかについてポジティブな反応を共有します。*

**コメント 2**：テーマ：この作業を別の方法で分解できるアイデアはありますか？

+ チームメンバーはここでさまざまなアイデアやアプローチを提供します。*

**コメント 3**：テーマ：チームとしてのイテレーションを改善するために、プロセスや働き方を変えられることはありますか？

*チームメンバーはここでさまざまなアイデアやアプローチを提供します。*
