## 採用 — GitLab 使用統計

[GitLab バージョンチェック](/handbook/sales/process/version-check/)を使用することで、GitLab の使用データは CE・EE・EE トライアルユーザーの両方の Salesforce にプッシュされます。
Salesforce アプリケーションで「使用統計」というタブが表示されます。
ドロップダウンビューを使用して、CE・EE トライアル・EE を選択し、GitLab に送信されたすべての使用データを確認できます。
バージョンチェックはホスト名を取得しているため、リードはホスト名として記録されます。正しい会社名を特定したら、会社名を更新してください。
例: gitlab.boeing.com を Boeing に会社名として変更します。

使用状況を表示するには、「使用統計」という列のハイパーリンクをクリックしてください。
リンクは a3p61012001a002 のような複数の数字と文字で構成されます。
Salesforce のアカウントオブジェクトレベルでも使用統計にアクセスできます。
アカウントオブジェクト内には「使用 Ping データ」というセクションがあり、同じハイパーリンクと使用しているバージョン・アクティブユーザー・過去のユーザー・ライセンスユーザー数・ライセンス利用率・データが送信された日付のサマリーが含まれています。

使用統計を活用してインサイトを引き出し、アクションを起こす方法の例をいくつか紹介します。

- CE ユーザーへのプロスペクティングでは、アクティブユーザーで並べ替えて大規模な CE インスタンスをターゲットにできます。
GitLab 内で Issue・CI ビルド・デプロイ・マージリクエスト・ボードなどを使用しているかを確認できます。
その後、コミュニケーションをターゲットにして、内部で GitLab をどのように使用しているかを把握し、EE で何が可能かを教育できます。
- ライセンス利用率が低い現在の EE ユーザーについては、顧客と関わって、GitLab を内部でロールアウトする計画と、どこでどのように採用を支援できるかを理解できます。
- ライセンス利用率が高い現在の EE ユーザーについては、データを活用して顧客と関わることができます。
組織内の GitLab の採用を祝福してください。採用が始まった理由を尋ねてください。どのように使用しているか（ユースケース）を把握してください。今すぐアドオンのために契約を修正することに参加させてください。
真の使用量と新しい更新金額を反映するために、更新機会を更新してください。
- CI や Issue などの EE 機能を使用していない現在の EE ユーザーについては、使用していない理由を理解するために顧客と関わることができます。
提供していることを知っているか？
競合ツールを使用しているか？
現在のツールを GitLab に統合しているか？
現在のツールを置き換えるための提供内容についてもっと知ることに興味があるか？
- EE トライアルについては、
どの EE 機能を使用していて、どれを使用していないか？
EE 機能を使用している場合、何を解決・評価しようとしているか？
使用していない場合、なぜか、そしてその機能を評価することに興味があるか？

Google ドライブの「Sales Training」フォルダを検索して、詳細を説明するトレーニングビデオをご覧ください。

## オープンソースユーザー — SFDC アカウントの GitLab CE インスタンスと CE アクティブユーザー

SFDC での[使用/バージョン Ping データ](https://docs.gitlab.com/ee/administration/settings/usage_statistics.html)の使用を簡素化するために、Salesforce アカウントレコードの「GitLab/テクスタック情報」セクションの下に「CE インスタンス」と「アクティブ CE ユーザー」の 2 つのフィールドがアカウントレイアウトに直接配置されています。

これらのフィールドのソースは、過去 60 日間に確認された各ホストの最新バージョン Ping レコードと使用レコードを使用して、単一のレコードに結合したものです。
ホストはドメインベースのホストと IP ベースのホストに分けられます。
IP ベースのホストでは、IP を[逆引き DNS ルックアップ](https://en.wikipedia.org/wiki/Reverse_DNS_lookup)にかけて、ドメインベースのホストに変換しようとします。
逆引き DNS ルックアップ（PTR レコード）が見つからない場合は、IP を[WHOIS ルックアップ](https://en.wikipedia.org/wiki/WHOIS)にかけ、会社名を抽出しようとします。例えば、その IP が会社の委任 IP スペースの一部である場合です。
予約済み IP アドレス（内部 IP スペース）を持つホストや、GCP・Alibaba Cloud・AWS などのクラウドプラットフォームにホストされているホストは特定できないため、これらのホストをすべて破棄します。
ドメインベースのホストでは、Datafox および/または DiscoverOrg のドメインルックアップ API を使用して組織に関連する会社と firmographic データを特定し、最終的に[Google Geocoding API](https://developers.google.com/maps/documentation/geocoding/start) を使用してアドレスデータを可能な限り解析・クレンジング・標準化します。

これらの統計は集約され、子アカウントがある場合（どのホストがどの子アカウントに属するかを実際に把握できないため）、SFDC の Ultimate 親アカウントにのみ表示されます。

CE ユーザー数またはインスタンス数でアカウントの一覧を表示するには、SFDC の[CE ユーザーリストビュー](https://na34.salesforce.com/001?fcf=00B61000004XccM)を使用できます。
自分のアカウントのみにフィルタリングするには、「編集」ボタンを押して「マイアカウント」でフィルタリングしてください。
元のビューを上書きしないように、別の名前で保存してください。
これらのフィールドは SFDC レポートでも利用できます。

このデータに関するいくつかの注意点：

- ホストは組織名・ドメイン・連絡先メールドメインに基づいて SFDC アカウントにできるだけ正確に関連付けられていますが、誤った関連付けが生じることがあります。
エラーや異常を発見した場合は、データ & アナリティクスチームに知らせてください。
- 両フィールドは空欄の場合がありますが、組織はまだ重要な CE ユーザーである可能性があります。
両フィールドが空欄であることは、バージョンや使用 Ping を送信しているインスタンスがないことを意味するだけで、必ずしもアクティブなインスタンスがないことを意味しません。
- ユーザーは複数のインスタンスに同じ人が存在する場合があるため、使用 Ping が有効な組織の複数のインスタンスにユーザーが存在する場合、インスタンスに表示されるたびにカウントされます。
ユーザーは組織外の人である可能性もあるため、ユーザー数が従業員数を表さず、従業員数より多い場合があります。
- これらの数値は、一部のインスタンスがこれらの Ping をオンにしていて、一部がオフにしている可能性があるため、組織内で GitLab が使用されている最低限の量を表しています。
また、バージョン Ping のみがオンになっている場合、ユーザー数は確認できません。
これはインスタンスがあるがユーザーがいないとは解釈しないでください。

これらのアカウントの処理・サードパーティツールからの連絡先の追加・アウトリーチのプロセスについては、ハンドブックの[ビジネスオペレーション](/handbook/business-technology/)セクションを参照してください。
