## 目的

このラボでは、ローカルコンピュータ上のリポジトリを使った実践を通じて、以下の概念について学びます。

- リポジトリのクローン
- ブランチの作成・利用・マージ
- ファイルの編集とコミット
- リモートリポジトリへの push と pull

各ラボ内でコピー＆ペーストを求められたコードは、必ずその内容を理解する時間を取ってください。不明なコードがあればインストラクターに説明を求めましょう。

> このラボで使われる Git コマンドの多くは、GitLab の便利な [Git チートシート](https://about.gitlab.com/images/press/git-cheat-sheet.pdf) にまとまっています。このラボでは、ローカルコンピュータからインターネットに対して SSH 接続できる必要があります。お使いのマシンが必要な権限を持っていることを確認してください。

## タスク A. Git がローカルにインストールされていることを確認する

1. コンピュータでターミナルを開き、次のコマンドを入力します。

   ```bash
   git version
   ```

1. 出力にバージョン番号が表示されれば、Git はインストールされています。コマンドがエラーになる場合は、Git をインストールする [ドキュメント](https://docs.gitlab.com/ee/topics/git/how_to_install_git/) を参照してください。

## タスク B. SSH キーを生成する

> GitLab は、Git との安全な通信に SSH プロトコルを使用します。SSH キーで GitLab リモートサーバーに認証すれば、毎回ユーザー名とパスワードを入力する必要がなくなります。詳しくは [ドキュメント](https://docs.gitlab.com/ee/user/ssh.html) を参照してください。

### セキュリティ上の推奨: SSH キー生成には 1Password を使う

SSH キーはディスクに保存するのではなく、1Password を使って生成・保管することを推奨します。この方法のほうが秘密鍵をより安全に保護できます。詳しい手順は [1Password で SSH キーペアを生成するガイド](https://docs.gitlab.com/user/ssh_advanced/#generate-an-ssh-key-pair-with-1password) を参照してください。

### 代替手段: SSH キーをローカルで生成する

ローカルで SSH キーを生成したい場合は、次の手順に従ってください。

1. ここでは OpenSSH クライアントを使います。GNU/Linux、macOS、Windows 10 には標準でインストールされています。現在のバージョンを確認するには、ターミナルまたは PowerShell で次のコマンドを実行します。

   ```bash
   ssh -V
   ```

2. ターミナルまたは PowerShell で次のコマンドを実行し、公開鍵と秘密鍵のペアを作成します。

   ```bash
   ssh-keygen -t ed25519
   ```

3. 最初のプロンプトでは、SSH キーの保存先を尋ねられます。コマンドが鍵を保存する場所を控えておきましょう。デフォルトのパスは `~/.ssh` です。デフォルトの場所と名前をそのまま使う場合は **Enter** を押します。

   > 必要であれば、このステップで独自のファイルパスと鍵名を指定することもできます。

4. 2 番目のプロンプトでは、鍵ファイルのパスワードを設定するよう求められます。ローカルの鍵ファイルにパスフレーズを設定しない場合は **Enter** を押します。

   > 手順を簡潔にするため、ここでは鍵ファイルにパスワードを設定しない選択にしています。パスフレーズを空にすることは一般的にはベストプラクティスとはされていません。必要に応じてパスフレーズを設定してください。

> **セキュリティ警告:** SSH キーをディスクに保存することにはセキュリティ上のリスクがあります。秘密鍵は保護する必要があり、決して他人と共有してはいけません。本番環境では 1Password などの安全な鍵管理ソリューションを利用することを検討してください。

## タスク C. GitLab プロフィールに SSH キーを追加する

1. ブラウザで Lab 1 で作成したプロジェクトを開きます。

1. 左上のユーザーアバターをクリックします。

1. ドロップダウンメニューから **Edit profile** をクリックします。

1. 左側のナビゲーションペインで **SSH Keys** をクリックします。

1. 既存のターミナルウィンドウを開きます。`cd ~/.ssh` で SSH キーを保存したディレクトリに移動し、次のように `ls` コマンドでそのディレクトリ内のすべてのファイル一覧を出力します。

   ```bash
   cd ~/.ssh
   ls
   ```

   Windows の場合:

   ```bash
   cd ~\.ssh
   ```

   > デフォルトでは、鍵は `~/.ssh` ディレクトリに保存されます。別のディレクトリに保存した場合は、そのディレクトリに `cd` する必要があります。

1. 公開鍵（例: `id_ed25519.pub`）と秘密鍵（例: `id_ed25519`）の 2 つの鍵ファイルが見えるはずです。公開鍵は末尾が `.pub` で、これが GitLab と共有する必要のあるものです。

   > **セキュリティ警告:** 秘密鍵は決して共有したり、ウェブサイトのフォームフィールドに貼り付けたりしないでください。共有してよいのは公開鍵だけです。

1. 次のように、公開鍵の内容を表示します。

   ```bash
   cat id_ed25519.pub
   ```

   > 別のファイル名を使った場合は、コマンドは `cat <filename>.pub` になります。

1. 画面に表示されたファイルの内容をクリップボードにコピーします。

1. GitLab Web UI のウェブブラウザで、**Add new key** ボタンをクリックします。

1. 公開鍵の内容を **Key** フィールドに貼り付けます。

1. **Title** フィールドに任意のタイトル（例: コンピュータのホスト名）を入力し、**Add key** をクリックします。

   > **ヒント:** 鍵がどこで使われているかを識別しやすくするため、特に好みの命名規則がなければコンピュータのホスト名や説明（例: `alextanuki-m2-mac`）を使うとよいでしょう。

1. **Usage type** では、**Authentication & Signing** が選択されていることを確認します。

   > この使用タイプを選ぶと、鍵を GitLab への認証だけでなく、コミットの署名にも使えます。署名付きコミットの詳細は [ドキュメント](https://docs.gitlab.com/ee/user/project/repository/signed_commits/ssh.html) を参照してください。

1. **Expiration date** はデフォルトのまま残しておきます。

   > 鍵には有効期限を設定し、定期的にローテーションするのが理想です。鍵の有効期限の推奨値はセキュリティ要件によって異なります。

1. **Add key** ボタンをクリックします。

1. ターミナルで、次のコマンドを実行して接続をテストします。

   インスタンス URL が `gitlab-learn-labs/*` を含む場合は次を実行します。

   ```bash
   ssh -T git@gitlab.com
   ```

   インスタンス URL が `ilt.gitlabtraining.cloud` を含む場合は次を実行します。

   ```bash
   ssh -T git@ilt.gitlabtraining.cloud
   ```

エラーではなくウェルカムメッセージで完了すれば、SSH キーは正しく設定されています。

> 接続が拒否されたというエラーが出たり、コマンドが動作しなかったりする場合は、ネットワークが SSH 経由の接続をブロックしている可能性があります。その場合は次のタスクに進んでください。

## タスク D. GitLab プロジェクトリポジトリをローカルコンピュータにクローンする

> リポジトリをクローンすると、リモートリポジトリのファイルがコンピュータにダウンロードされ、接続が確立されます。詳しくは [ドキュメント](https://docs.gitlab.com/ee/topics/git/commands.html#clone-a-repository) を参照してください。

1. Lab 1 で作成した `Top Level` プロジェクトに戻ります。

1. 左サイドバーで **Code > Repository** をクリックします。

1. プロジェクトリポジトリの右側で、**Code** ボタンをクリックします。**Clone with SSH** の下にある URL をクリップボードにコピーします。

1. ターミナルまたは PowerShell ウィンドウで、ホームディレクトリに `training` という新しいディレクトリを作成し、そのディレクトリに移動します。コマンドは以下のとおりです。

   ```bash
   mkdir ~/training
   cd ~/training
   ```

   Windows の場合:

   ```cmd
   mkdir ~\training
   cd ~\training
   ```

1. **Clone with SSH** からコピーしたコマンドを使って、`git clone` コマンドを実行します。

   ```bash
   git clone <Clone with SSH Command>
   ```

> 接続がタイムアウトした、または拒否されたというエラーが出る場合は、ファイアウォールによりネットワークがポート 22 の接続をブロックしている可能性があります。その場合はタスク D の手順を繰り返しますが、**Code** ボタンから **Clone with HTTPS** を選んで使ってください。

Mac でクローンしたプロジェクトをマシン上で探すには、ホームディレクトリに移動する必要があるかもしれません。Finder から Go > Go to Folder > マシンのユーザー名で検索してホームディレクトリを開きます。あるいは Finder で **Option** キー (Alt) を押しながら左下のメニューでマシンのユーザー名を選んでも開けます。

1. クローンしたばかりのリポジトリに `cd` コマンドで移動します。このディレクトリ内のすべてのファイルが Git で追跡され、このラボで実行する Git コマンドもすべてこのディレクトリから実行する必要があります。

   ```bash
   cd top-level-project
   ```

1. `ls -a` コマンドで、ピリオドから始まる隠しファイルやディレクトリも含めて、ディレクトリの内容を表示します。

   ```bash
   ls -a
   ```

   > `.git` ディレクトリの存在に注目してください。`.git` ディレクトリには、プロジェクトのメタデータとオブジェクトデータベースが保存されています。

1. 次のコマンドを実行してリポジトリのステータスを確認します。

   ```bash
   git status
   ```

   > 出力に `nothing to commit, working tree clean` と表示されます。これは、このディレクトリ内のファイルが Git に保存されているバージョンと同じ内容であることを意味します。

## タスク E. ブランチで作業する

> ブランチはプロジェクトの作業ツリーのバージョンです。新しいプロジェクトを作成すると、GitLab はリポジトリのデフォルトブランチとして `main`（旧 `master`）を作成します。このブランチは削除できません。デフォルトブランチの設定は、プロジェクト、サブグループ、グループ、インスタンスのいずれかのレベルで構成できます。詳しくは [ドキュメント](https://docs.gitlab.com/ee/user/project/repository/branches/) を参照してください。

1. コンピュータ上に **temporary_branch** という新しいブランチを作成します。

   ```bash
   git branch temporary_branch
   ```

2. 作成したばかりのブランチに切り替えます。

   ```bash
   git checkout temporary_branch
   ```

3. リポジトリ内のすべてのブランチを一覧表示します。

   ```bash
   git branch -a
   ```

   > ブランチ一覧を抜けるには、キーボードで `q`（quit）を押します。

4. 赤いブランチはリモートサーバー、つまりリポジトリが保管されている GitLab インスタンス上のブランチです。

5. アスタリスクは、現在いるブランチを示します。

## タスク F. ファイルを編集する

1. 任意のテキストエディタ（Visual Studio Code、Sublime Text、メモ帳、vi など）を使って、`README.md` の末尾に次の行を追加し、ファイルを保存します。

   ```text
   a line added to temporary_branch locally
   ```

1. Git がファイルの変更に気付いているか確認します。

   ```bash
   git status
   ```

   > 出力には、`README` ファイルが赤色で、ステータスが `modified` と表示されます。赤色のフォントは、`README` ファイルがまだ Git のステージングエリアに追加されていないことを示します。

## タスク G. 編集したファイルを Git のステージングエリアに追加する

1. `git add` コマンドでファイルをステージングエリアに追加します。コマンドが成功した場合、出力は表示されません。

   ```bash
   git add README.md
   ```

   > `git add` はファイルシステム上で `README.md` を移動しませんが、Git の「ステージングエリア」には追加します。

1. `README.md` がコミットできる状態（つまり、正しくステージされた状態）になったことを確認します。

   ```bash
   git status
   ```

ここで `README` ファイルが緑色のフォントで表示されることが分かります。これは、変更が Git のステージングエリアで追跡されるようになったことを示します。

## タスク H. 編集をコミットする

1. 下記の `git commit` コマンドでステージしたファイルをコミットします。コミットごとに分かりやすいコミットメッセージを付けることが重要です。

   ```bash
   git commit -m "Add a line to README.md"
   ```

   これで、必要に応じて後から参照できるファイルのスナップショットを作成したことになります。

1. ステージングエリアが再び空になっていることを確認します。

   ```bash
   git status
   ```

## タスク I. 変更を GitLab インスタンスに push する

1. GitLab サーバーのリモート Git リポジトリに **temporary_branch** という新しいブランチを作成し、そのブランチに `git push` コマンドで変更を push します。

   ```bash
   git push -u origin temporary_branch
   ```

   > リモートサーバーに変更を push する正確なコマンドが分からない場合は、`git push` と入力すれば、Git がエラーメッセージとして正しいコマンドを出力してくれるので、それをコピー＆ペーストすれば OK です。

## タスク J. もう一度ファイルを編集・コミット・push する

1. ローカルマシンのテキストエディタ（GitLab のブラウザ内エディタではなく）で、ローカルコピーの `README.md` の末尾に次の新しい行を追加し、ファイルを保存します。

   ```a second line in README.md```

1. ターミナルで、編集したファイルを `git add` コマンドで Git のステージングエリアに移動します。

   ```bash
   git add README.md
   ```

1. `git commit` コマンドでステージしたファイルをコミットします。分かりやすいコミットメッセージを付けるか、下記の例をそのまま使っても構いません。

   ```bash
   git commit -m "Modify README.md"
   ```

1. `git log` コマンドで、今行ったコミットの内容を確認します。

   ```bash
   git log
   ```

   > `q` を押すと `git log` の出力画面を抜けます。

1. `git push` コマンドで、コミットを GitLab インスタンス上のリモートリポジトリに push します。

   ```bash
   git push
   ```

   > 既に upstream ブランチ（つまり、ローカルマシン上のブランチと同名のリモートリポジトリ上の既存ブランチ）に対して変更をコミットするときは、初回 push のときに使った長いコマンドではなく、`git push` だけで構いません。upstream ブランチの設定は一度だけ行えばよいからです。

1. GitLab Web UI でプロジェクトに移動します。

1. プロジェクトのメインページに移動したら、**Code > Branches** に進みます。

1. **temporary_branch** をクリックしてそのブランチに切り替えます。ローカルブランチで加えた `README.md` への変更が、リモートリポジトリに push されたことを確認します。

   > 適用されたすべての変更を確認するには、**History** ボタンをクリックします。これにより、ブランチに適用されたすべてのコミットを表示できます。

## タスク K. リモートブランチを編集する

> ここで、組織内の別の誰かが GitLab インスタンス上のリモートリポジトリにある **temporary_branch** に変更を加える状況をシミュレートしてみます。このセクションを終えると、**temporary_branch** のリモート版とローカル版は異なる状態になります。つまり、そのブランチ上のコードが（いわば）あなたの足元で動いていることになります。次のセクションでは、この差分を解消する方法を見ていきます。

1. GitLab で **Top Level Project** のランディングページに移動します。**temporary_branch** にいない場合は、左側のナビゲーションペインで **Code > Branches > temporary_branch** をクリックします。

1. これで **temporary_branch** 内のファイルが表示されます。**README.md** をクリックして内容を確認します。

1. 右上の **Edit > Edit single file** をクリックしてファイルを編集します。

1. ファイルの末尾に新しい行を追加します。

   ```text
   a third line added on the remote copy of temporary_branch
   ```

1. ページ右上の **Commit Changes** ボタンをクリックします。

1. 適切なコミットメッセージを入力します。

1. ターゲットブランチを **temporary_branch** に設定します。

1. **Commit changes** ボタンをクリックします。

このコミットを行ったあと、GitLab インスタンス上のリモートリポジトリは、ローカルリポジトリより 1 コミット *進んだ* 状態になります。

## タスク L. リモート **temporary_branch** への変更に関するメタデータを取得する

> ローカルの **temporary_branch** は、GitLab インスタンス上のリモート **temporary_branch** と同期がずれています。`git fetch` コマンドは、ローカルブランチの内容を更新することなく、リモートブランチの最新の状態を取得します。言い換えると、ローカルブランチがリモートブランチからどれだけのコミット分遅れているかを知らせるだけで、ローカルブランチのファイルには何も変更を加えません。

1. `git fetch` コマンドで、リポジトリのリモートコピー上のブランチに関するメタデータを取得します。

   ```bash
   git fetch
   ```

1. リポジトリのリモートコピーにはあるがローカルコピーにはないコミット、あるいはその逆のコミットがいくつあるかを調べます。

   ```bash
   git status
   ```

`git status` の出力で、ブランチが 1 コミット遅れていることが分かるはずです。

## タスク M. リモートリポジトリから pull する

> リモートコピーの **temporary_branch** からの変更をマージして、ローカルコピーの **temporary_branch** の内容を更新する必要があります。

1. ターミナルで `git pull` コマンドを使って、リモートコピーをローカルコピーにマージします。

   ```bash
   git pull
   ```

1. `cat README.md` コマンドで更新されたファイルの内容を表示します。GitLab Web IDE で追加した 3 行目が見えるはずです。

   ```bash
   cat README.md
   ```

## タスク N. 変更を main ブランチに戻してマージする

> ローカルの **temporary_branch** がリモートの **temporary_branch** と同じ内容になったので、これをローカルの **main** ブランチにマージできます。これで、**main** にある安定したコードベースに編集内容が追加されます。

1. `git branch` を入力して、現在作業しているブランチを確認します。

   ```bash
   git branch
   ```

1. ターミナルで `git checkout` コマンドを使って **main** ブランチに切り替えます。

   ```bash
   git checkout main
   ```

1. `git merge temporary_branch` を入力して、ローカルの **temporary_branch** からのすべての変更（このケースでは編集された `README.md` のみ）をローカルの **main** ブランチに取り込みます。

   ```bash
   git merge temporary_branch
   ```

## タスク O. リモートリポジトリを更新する

1. `git status` を入力して、ステージやコミットが必要な編集済みファイルがないこと、また **main** ブランチにいることを確認します。

   ```bash
   git status
   ```

1. `git push` を入力して、ローカルコピーからの変更で **main** ブランチのリモートコピーを更新します。

   ```bash
   git push
   ```

1. ブラウザで GitLab のページに戻り、プロジェクトの **main** ブランチで `README.md` を表示して、リモートコピーの **main** に push したばかりの変更を確認します。
