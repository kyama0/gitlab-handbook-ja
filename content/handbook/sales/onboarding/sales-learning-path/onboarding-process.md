## セールスオンボーディングプロセス

1. GitLab Candidate Experience チームは、すべての新しい GitLab チームメンバーに対して一般的な GitLab オンボーディング Issue を開始します
    - [People Ops オンボーディング Issue テンプレート](https://gitlab.com/gitlab-com/people-group/employment-templates/-/blob/main/.gitlab/issue_templates/onboarding.md) を参照
1. 一般的な GitLab オンボーディング Issue の "Day 2" セクションでは、[事前承認済みのテンプレート](https://gitlab.com/gitlab-com/team-member-epics/access-requests/-/tree/master/.gitlab/issue_templates/role_baseline_access_request_tasks) が存在する場合、新しいチームメンバーに対して役割ベースのエンタイトルメント Access Request (AR) が自動的に作成されます。新しいチームメンバーは AR に自動的にタグ付けされ、AR はオンボーディング Issue とともに新しいチームメンバーのエピックにも自動的に追加されます。フィールド職向けの事前承認済み役割ベースのエンタイトルメント Access Request テンプレートは、ここで利用可能です:
    - [department_business_development](https://gitlab.com/gitlab-com/team-member-epics/access-requests/-/tree/master/.gitlab/issue_templates/role_baseline_access_request_tasks/department_business_development)
    - [department_channel](https://gitlab.com/gitlab-com/team-member-epics/access-requests/-/tree/master/.gitlab/issue_templates/role_baseline_access_request_tasks/department_channel)
    - [department_commercial_sales](https://gitlab.com/gitlab-com/team-member-epics/access-requests/-/tree/master/.gitlab/issue_templates/role_baseline_access_request_tasks/department_commercial_sales)
    - [department_customer_success](https://gitlab.com/gitlab-com/team-member-epics/access-requests/-/tree/master/.gitlab/issue_templates/role_baseline_access_request_tasks/department_customer_success)
    - [department_enterprise_sales](https://gitlab.com/gitlab-com/team-member-epics/access-requests/-/tree/master/.gitlab/issue_templates/role_baseline_access_request_tasks/department_enterprise_sales)
    - [department_field_operations](https://gitlab.com/gitlab-com/team-member-epics/access-requests/-/tree/master/.gitlab/issue_templates/role_baseline_access_request_tasks/department_field_operations)
1. その Issue の "Sales Division" セクションでは、Field Enablement がタグ付けされ、新しいセールスチームメンバーを EdCast の Sales Quick Start 学習パスに追加するアクションが行われます。GitLab に参加した最初の週内に学習パスに追加されます。この学習パスは、新しいセールスチームメンバーが GitLab での最初の数週間以内に知り、実行できる必要のある主要なセールス固有の要素に焦点を当てることで、新しいセールスチームメンバーの生産性までの時間を加速するように設計されています。
1. 新しいセールスチームメンバーは、Sales Quick Start 学習パスに取り組み始めるために EdCast にログインするよう促すメールを受け取ります。
    - 可能な限り、Sales Enablement は新しいチームメンバーのオンボーディング Issue も更新し、および / または Sales Quick Start 学習パスへのリンクを明示的に参照するためのコメントを追加します。
    - "New Team Member" セクションには、新入社員が "Complete your Sales Quick Start learning path." を行う具体的なアクションがあります。
    - バーチャルで自己学習型の [Sales Quick Start learning path](/handbook/sales/onboarding/sales-learning-path/) に何が含まれているかを参照してください。
    - 役割に関係なく、新しいセールスチームメンバーが SQS ワークショップに出席する前に完了することが期待されることを理解することに興味がある場合、Non-Sales チームメンバーは [SQS Learning Path](https://gitlab.edcast.com/pathways/sales-quick-start) に自分自身を追加することを選択できます。
1. フィールドチームメンバーは、Sales Quick Start を成功裏に完了した後、適切な役割ベースのラーナージャーニーに進級し、オンボーディングされた / ランプアップ中のチームメンバーのさらなる成功への継続的な学習ジャーニーをサポートします。

## 役割ベースのオンボーディングジャーニー

ランプ時間を加速し、結果、イテレーション、効率という GitLab の価値をサポートするために、私たちはフィールドオンボーディングプログラムを最適化し、新しいチームメンバーに、コア情報と必須コンピテンシーに加えて、個々の役割で成功するために必要な日々のスキルを提供しています。学習は、専門化されたオンザジョブトレーニングと、より堅牢なシャドウイングプログラムと組み合わされ、新しいチームメンバーがオンボーディングジャーニー全体を通じて学んでいることを文脈化し、適用できることを保証します。

オンボーディングプログラムには、AE、SAE、SA、CSM、CAM、ABDM、ISR の個別の役割ベースの学習ジャーニーが含まれています。各役割ベースの学習ジャーニーは、各セグメント固有のニーズに基づいて設計されており、以下のマイルストーンが含まれます:

- 月 1: 役割と GitLab 製品を理解する (SQS と事前作業)
- 月 2: 顧客と会う
- 月 3: カスタマー対応認定を取得する
- 月 4: 最初のディールをクローズする
- 月 5 以降: 中級と上級スキルの開発を継続する

## カスタマー対応シャドウプログラム

同僚の顧客コールをシャドウすることは、さまざまな GitLab の役割が GitLab をどのように代表するかを学ぶハンズオンの方法を提供します。私たちは、同僚から学び、練習し、インタラクティブな方法でフィードバックを得る機会があるときに、最もよく学べることがよくあります。これはまた、現在のチームメンバーが新入社員に GitLab でのセールスカルチャーの一部として歓迎を感じてもらう素晴らしい方法でもあります。[Customer-Ready Shadow Program ページで詳しく学ぶ](/handbook/sales/shadow-program/)。
