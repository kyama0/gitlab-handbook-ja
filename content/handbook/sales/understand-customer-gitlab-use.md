このページでは、顧客が GitLab をどのように使っているかをより深く理解し、顧客との会話により備え、より情報を得た状態で臨めるようにするためのガイダンスを提供します。

## アダプション - GitLab 利用統計

[GitLab Version Check](/handbook/sales/process/version-check/) を使うことで、GitLab の使用状況データが CE、EE、EE トライアルユーザーの全てについて Salesforce にプッシュされます。
Salesforce アプリケーション内には「Usage Statistics」というタブがあります。
ドロップダウンビューを使い、CE、EE トライアル、EE を選択することで、GitLab に送信されたすべての使用状況データを確認できます。
Version Check はホスト名を取得しているため、リードはホスト名で記録されます。正しい会社名が特定できたら、会社名を更新してください。
例: gitlab.boeing.com を会社名として Boeing に変更します。

利用状況を確認するには、「Usage Statistics」列のハイパーリンクをクリックします。
リンクは a3p61012001a002 のような数字とアルファベットからなる文字列です。
Salesforce のアカウントオブジェクトレベルでも利用統計にアクセスできます。
アカウントオブジェクト内には「Usage Ping Data」というセクションがあり、同じハイパーリンクと、利用しているバージョン、アクティブユーザー、過去ユーザー、ライセンスユーザー数、ライセンス利用率 %、データ送信日時の概要が含まれます。

Usage Statistics を使ってインサイトを得てアクションを起こす方法の例をいくつか紹介します:

- CE ユーザーへのプロスペクティングを行う場合、アクティブユーザーで並べ替えて大規模な CE インスタンスをターゲットにできます。
GitLab 内で Issue、CI ビルド、デプロイ、マージリクエスト、ボードなど、何を使っているかを確認できます。
そして、彼らが社内で GitLab をどのように使っているかを学ぶためにコミュニケーションをターゲット化し、EE で何が可能かを教育できます。
- ライセンス利用率を下回っている現行 EE ユーザーに対しては、社内で GitLab を展開する計画と、アダプションをどこでどう支援できるかを理解するため、顧客にエンゲージできます。
- ライセンス利用率を超えている現行 EE ユーザーに対しては、データを活用して顧客にエンゲージできます。
組織内での GitLab のアダプションを称え、なぜアダプションが伸びたのか、どのように使っているか（ユースケース）、追加分について今すぐ契約を修正する話に乗ってもらえるかを尋ねます。
更新オポチュニティを更新し、true-up と新しい更新金額を反映させましょう。
- CI や Issue など、EE 機能を使っていない現行 EE ユーザーに対しては、なぜ使っていないのかを理解するため顧客にエンゲージできます。
私たちが提供していることを知っていますか？
競合ツールを使っていますか？
現在のツールを GitLab に統合していますか？
現在のツールを置き換えるために私たちが提供しているものについて学ぶ意欲はありますか？
- EE トライアルについて。
どの EE 機能を使っていて、どれを使っていないか？
EE 機能を使っているなら、何を解決し評価しようとしているのか？
使っていないなら、なぜか、その機能を評価する意欲はあるか？

詳細を解説したトレーニングビデオを Google Drive 上の「Sales Training」フォルダで検索してご覧ください。

## オープンソースユーザー - SFDC アカウント上の GitLab CE インスタンスと CE アクティブユーザー

[利用状況/バージョン ping データ](https://docs.gitlab.com/ee/administration/settings/usage_statistics.html) を SFDC でよりシンプルに使えるようにするため、Salesforce アカウントレコードの「GitLab/Tech Stack Information」セクションの下、アカウントレイアウト直下に「CE Instances」と「Active CE Users」という 2 つのフィールドが用意されています。

これらのフィールドのソースは、過去 60 日間に確認された各ホストの最新のバージョン ping レコードと使用状況レコードを使用し、それらを 1 つのレコードにまとめたものです。
ホストはドメインベースのホストと IP ベースのホストに分けられます。
IP ベースのホストについては、IP を [リバース DNS ルックアップ](https://en.wikipedia.org/wiki/Reverse_DNS_lookup) にかけてドメインベースのホストへの変換を試みます。
リバース DNS ルックアップ（PTR レコード）が見つからない場合、IP を [whois ルックアップ](https://en.wikipedia.org/wiki/WHOIS) にかけて会社名を抽出しようとします。たとえば、IP が会社の委任 IP 空間の一部である場合などです。
予約済み IP アドレス（内部 IP 空間）を持つホストや、GCP、Alibaba Cloud、AWS のようなクラウドプラットフォーム上にホストされているホストは、これらのデータを通じて組織を特定する方法がないため、特定できないホストはすべて破棄します。
ドメインベースのホストについては、Datafox や DiscoverOrg のドメインルックアップ API を使って会社と組織に関連するファーモグラフィックデータを特定し、最後に [Google Geocoding API](https://developers.google.com/maps/documentation/geocoding/start) を使って住所データを可能な限り解析、クリーン化、標準化します。

これらの統計は集計され、子アカウントがある場合（どのホストがどの子に属するかは実際には分からないため）、SFDC の Ultimate Parent アカウントにのみ表示されます。

CE ユーザー数またはインスタンス数別のアカウント一覧を見るには、SFDC の [CE Users リストビュー](https://na34.salesforce.com/001?fcf=00B61000004XccM) を使用できます。
自分のアカウントだけでフィルタするには、「edit」ボタンをクリックして「My Accounts」でフィルタします。
元のビューを上書きしないよう、別の名前で保存してください。
これらのフィールドは SFDC レポートでも利用可能です。

このデータについていくつか注意点があります:

- ホストは組織名、ドメイン、コンタクトのメールドメインに基づいて、可能な限り正確に SFDC アカウントとインスタンスを関連付けるようマッピングされていますが、まれに誤った関連付けを作成する可能性があります。
誤りや異常を見つけたら Data & Analytics チームに知らせてください。
- 両フィールドが空でも、組織が大規模な CE ユーザーである可能性は依然としてあります。
両フィールドが空であることは、バージョンや使用状況の ping を送信しているインスタンスがないことを意味するだけで、必ずしもアクティブなインスタンスが存在しないという意味ではありません。
- ユーザーは複数のインスタンスで同一人物である可能性があるため、ある組織で使用状況 ping が有効な複数のインスタンスにユーザーが存在する場合、インスタンスに表示されるたびにカウントされます。
ユーザーは組織外の人物である可能性もあるため、ユーザー数が必ずしも従業員を表すとは限らず、従業員数より多くなることもあります。
- これらの数字は、GitLab が組織内で使用されている可能性のある最小限の量を表しています。一部のインスタンスではこれらの ping が有効で、一部では無効である可能性があるためです。
バージョン ping のみが有効な場合もあり、その場合はユーザー数は確認できません。
これは「インスタンスはあるがユーザーがいない」と解釈すべきではありません。

これらのアカウントを処理し、サードパーティツールからコンタクトを追加し、アウトリーチを行うプロセスについては、ハンドブックの [business operations](/handbook/business-technology/) セクションを参照してください。

## ユースケーステクノロジートラッキング

このプロセスのゴールは、Salesforce のアカウント（Gainsight 経由）が DevOps ツール環境の各ユースケースで使用しているテクノロジーに関する情報を捉えることです。これにより、CI/CD プロセスの特定のステージに対してより正確にマーケティングおよびセールスができるようになります。

Gainsight の特定のアカウントの「White Space (Use Case)」セクションに、以下のデータを入力してください:

- `uc_(Use Case) Tech`: プロスペクト/顧客が現在使用しているテクノロジー（GitLab を含む）
- `uc_(Use Case) Appetite for Replacement`: Red（変更したくない）、Yellow（変更するかも）/Green（変更したい）
- `uc_(Use Case) Contract End Date`: 現在のテクノロジー契約の終了日（できる限り正確に）

詳しくは [License Usage Handbook](/handbook/sales/field-operations/sales-systems/license-usage-app/) を参照してください。
