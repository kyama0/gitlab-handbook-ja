## 主要な顧客の問題は何か？

ソフトウェアデリバリーライフサイクル全体を担当するチームは、以下に苦労しています:

- 各チームや個人が好むツールが異なる
- ライフサイクルを機能させるためにツール間で複数の統合が必要
- 複数の構成上の課題
- チーム間で異なる作業プロセス
- 改善を測定するためのチーム共通のメトリクスがない
- 遅く、エラーが発生しやすく、脆い、シーケンシャルなプロセスとハンドオフのフロー

そして、価値実現までの時間を短縮するのを阻むプロセスを抱えています。例えば:

- 承認されるまでブロックするセキュリティレビュー
- プロビジョニングが必要なインフラストラクチャ
- 固定されたリリースウィンドウ
- 本番チームによるリリースの手動承認が必要
- 法務／コンプライアンスのサインオフが必要
- 長いテストサイクル
- 別々のサイロで作業するビルド、QA、セキュリティ、ガバナンス、運用チーム
- 本番環境の障害は診断が難しく、修正に時間がかかる

## 主要メッセージングガイドライン

### GitLab はプロダクトでありプラットフォームでもある

GitLab はプロダクトでもありプラットフォームでもあります。 購入して使用できる単一のアプリケーションであるという意味でプロダクトです。 複数のユースケースを可能にし、無制限の追加ユースケースを構築できるフレームワーク（ほとんどの機能が API を介して利用可能）であり、エコシステムイネーブルメントグループに支えられているという意味でプラットフォームです。 ユーザーは完全に機能するプロダクトをすぐに利用できますが、他のシステムへの拡張や統合に制限はありません。 プロダクトでありプラットフォームでもある他の例: iPhone（リリース時にはアプリストアすらありませんでした）と Facebook。

より伝統的なプラットフォームの定義 [もっと限定的](https://www.dictionary.com/browse/software-platform) もありますが、プラットフォームの定義をより広く捉える視点もあります。 具体的には、Adrian Bridgewater が [この記事](https://www.forbes.com/sites/adrianbridgwater/2015/03/17/whats-the-difference-between-a-software-product-and-a-platform/#5042c24056a6) で Facebook の例を説明し、ブロガーの Jonathan Clarks の言葉も引用しています: 「プラットフォームとは、同じ技術的フレームワーク内で複数のプロダクトを構築できる構造のことです」 - これは、議論の余地はあるものの、GitLab を非常によく表していると言えます。 プラットフォームのビジネスモデルを研究するブログである Platform Hunt の [別の記事](https://medium.com/platform-hunt/the-8-types-of-software-platforms-473c74f4536a) は、9 種類のソフトウェアプラットフォームを概説しています。 GitLab はインタラクションプラットフォームに最も近く、アイデンティティが基本的な特性であり、プラットフォームがデジタルなインタラクションを促進します。 GitLab の場合、これらのデジタルなインタラクションは、MR と Issue 両方を通じて促進される協働コーディングです。

GitLab を単一アプリケーションとして提供されるプラットフォームと呼ぶことは、GitLab の主要な競争差別化要素を損なうものではなく、単一アプリケーションのアプローチには [多くの利点](/handbook/product/categories/gitlab-the-product/single-application/) があります。 GitLab をプラットフォームと呼ぶことは、GitLab の潜在的な CSM（total available market）を増加させ、潜在的な混乱（一部の人は「単一アプリケーション」が SCM のみを指すと考えるかもしれない）を減少させます。

### 会社について話す際は Git および Git のみに関連する概念を最小化する

私たちは「単一アプリケーションとして提供される The DevOps platform」というメッセージに集中したいと考えています。 「Git」が会社名に入っているので、メッセージングで Git ベースの SCM であることを再強調する必要はありません。 常に「単一アプリケーションとして提供される The DevOps platform」というメッセージングをデフォルトとし、最低限、複数ステージのソリューションをメッセージとして打ち出すべきです。

### ヘッドラインのコピーはメッセージングとは異なる

時には、看板、バナー広告、ブログ投稿で誰かの注意を引く目的でヘッドラインを使用します。 ヘッドラインはより実験的になることがあり、時には公式メッセージングとは異なる言葉を使用します。 これは、真実かつ本物である限り、許容されます。 例えば、私たちの広告ヘッドラインは、価値ドライバーの言語と異なる場合があります。 これは、エンゲージメントと実験のために異なるメッセージングを使う適切な使い方です。

### 避けるべきメッセージング

1. 「最初」や「唯一」などの用語は使用しないでください。 これらは証拠で証明することが極めて困難なため、信頼性を欠きます。 さらに、私たちのパートナーは、この種のメッセージングを明確に禁止するポリシーを持っていることが多く、共同マーケティング（プレスリリース、イベントスポンサーシップなど）を行う際にはこの表現を変更するよう求められます。
2. 「統合された（integrated）」という用語を使用しないでください。 確かに、GitLab は統合されたプロダクトですが、Atlassian のプロダクトスイートは異なるコードベースの多くの異なるアプリで構成されており、同じアプリのオンプレミス版とクラウド版が異なるプログラミング言語で書かれているにもかかわらず、彼らも同じことを主張できます。 代わりに、「single application」、「built from the ground up」、または「built in」を使用して GitLab の統合された性質を表現してください。
