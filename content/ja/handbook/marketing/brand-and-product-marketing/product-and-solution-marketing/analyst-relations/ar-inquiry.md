## 業界アナリストインクワイアリの実施方法

### 業界アナリストとのインクワイアリの使用

- インクワイアリは、GitLab が開始する特定のトピックに関する双方向の対話に関するものです。インクワイアリの適切なトピックには、以下が含まれます:
  - 市場について尋ねる
  - 私たちが行っている何かについてフィードバックを得る（例: 製品ビジョンページ、競合ドキュメント、プレゼンテーション、市場ポジショニングなど）
  - 顧客がインクワイアリで何を取り上げているかについて尋ねる
  - 今後のリサーチについて尋ねる（例: 今後のウェーブまたは MQ、フォーカスエリア）
  - 競合他社が行っていることまたは発表したことについての意見を尋ねる
  - GitLab のブリーフィングおよび/またはデモから準備またはフィードバックを得る
- インクワイアリは一般的に 30 分続き、通常はビデオ会議で実施されます。AR は Zoom コールをスケジュールするか、アナリストが例えば Webex へのリンクを提供します。
- インクワイアリの質問は事前に準備する必要があります。30 分前に議論が尽きた場合は、コールを早く終了することは許容されます。さらに時間が必要な場合は、アナリストとフォローアップインクワイアリをスケジュールすることに同意することは構いません。
- 同じインクワイアリを複数のアナリストおよび/または複数の会社に提示することができます。範囲のフィードバックを探している場合は。

### アナリストインクワイアリのリクエストと参加方法

- すべてのインクワイアリは、アナリストリレーションズ部門によって開始されます。インクワイアリをスケジュールしたい場合は、プロジェクト gitlab.com>marketing>product marketing で Issue を作成し、テンプレート AR-ResearchRequest を使用して、インクワイアリの関連部分を記入してください。
- AR がインクワイアリを予約したら、ミーティングノートが添付された招待を受け取ります。インクワイアリの前に、ミーティングノートに入り、聞きたいと思う 3〜5 つの質問で質問スペースを更新することが期待されます。これは、コール終了時にメモと重要なポイントを記録するドキュメントでもあります。
- インクワイアリの時間に、ビデオ会議に接続し、ノートドキュメントを開いて開始する準備をしてください。AR は出席者を互いに紹介し、インクワイアリトピックの舞台を設定します。それが行われたら、AR はコールをインクワイアリ開始者に渡し、彼らが会話を指揮します。
- AR チームは、インクワイアリ中の対話の生産的な使用を確保するために、コールのチームがバックグラウンドコミュニケーションチャンネルとして使用するための Slack チャンネル #ar_chat を作成しました。
- コールのセカンダリ参加者である場合は、Slack の #ar_chat チャンネルまたはドキュメントを使用して追加の質問をしてください。AR チームメンバーは、適切な場合に、他の人に質問を口頭で伝えるよう呼びかける場合があります。一部のアナリスト会社は 1 人だけ話すことを許可しており、他の会社は技術的にバックグラウンドのみであることに注意してください。
- インクワイアリが終了したら、ミーティングノートドキュメントの質問の下のセクションに重要なポイントを追加してください。
- デブリーフが必要な場合は、Slack チャンネルで言及される場合があり、時間のある人が参加できます。これは必須ではなく、常に可能ではありませんが、必要な場合は推奨されます。

### AR - アナリストインクワイアリのセットアップ方法

- インクワイアリリクエストは AR-ResearchRequest 下の Issue を介して来るべきです
- リクエストが明確であることを確認してください。そうでない場合は、Issue または Slack を使用して、リクエスターが何を探しているかの明確化を取得してください。誰の出席が必須で、誰の出席がオプションであるかを明確にしてください。
- リクエストが以下を含む場合:
  - Gartner または Forrester、インクワイアリのために彼らの Web サイトのフォームを使用してください。
  - IDC、スケジュールのサポートのために Maggie Margossian に連絡してください。彼女のメールは mmargossian [at] idc [dot] [com] です
  - 451、スケジュールのサポートのために Christina Lamb に連絡してください。彼女のメールは christina.lamb [at] 451research [dot] [com] です
  - Redmonk、スケジュールのサポートのために Juliane Leary に連絡してください。彼女のメールは jleary [at] redmonk [dot] [com] です
- Gartner または Forrester の場合は、スケジューリングのためのカレンダーへのリンクを受け取ります。そのカレンダーをブラウザに読み込み、Google カレンダーと比較して、すべての必要なチームメンバーが利用できる 30 分を見つけます。日時を選択すると、Gartner または Forrester がすべての情報を含むフォローアップメールを送信します。
- IDC/451/Redmonk の場合、これは一致が見つかるまで潜在的な日時を交換することを含みます。便利な場合は電話することもできます。
- アナリスト会社から確認された日時を受け取ったら、Google カレンダーに移動して新しいアポイントメントを作成します。アナリスト会社が送信する招待を使用することはできません。GitLab の社内の人々のために AR が別の招待を作成する必要があります。アナリスト会社からの転送されたメールは、他の人の Google カレンダーには常に表示されません。
  - タイトルの命名規則は、Inquiry - アナリスト会社 - トピック で、Gartner または Forrester の場合は、このトランザクションのために提供される参照 ID（例: Inquiry - Gartner - Monitor Vision Doc Review (Ref# 12422433)）です
  - Location には、GitLab Zoom リンクを指定するか、「Gartner Webex」または「IDC Bluejeans」などを記入してください
  - これは個人のカレンダーではなく、Analyst Relations カレンダーに置いてください
  - ノートの最初の行は、ミーティングノートへのリンク付きの「Meeting Notes」である必要があります
  - 区切り文字の後に、ビデオ会議の詳細を入れてください。
  - 内部のゲストを追加し、どれがオプションかを指定してください。まだ招待を送信しないでください。これらはネストされた手順です。
- ミーティングノートを作成
  - Google ドライブの Analyst Relations>analyst-firm-name-folder>Meeting Notes に移動します
  - 可能であれば、同じアナリストを使用した別のミーティングノートを見つけるか、最新のページを選んで、それを開いてファイルのコピーを作成します
  - 新しいファイルの名前を date - analyst name - topic を使用して変更します
  - アナリストの経歴が正しいアナリストを反映していることを確認してください。そうでない場合は、アナリスト会社の Web ページに移動して正しい経歴をリンクします
  - 更新する他のリンクがある場合は、そうしてください。他のリンクには以下が含まれます:
    - リサーチリクエストの Issue へのリンク
    - インクワイアリを生成した可能性のある他の Issue へのリンク
    - インクワイアリで参照されるドキュメント（ビジョンページや競合ドキュメントなど）へのリンク
  - GitLab 出席者のリストを更新します
  - インクワイアリの目的セクションを更新します - これは簡潔である必要があり、リサーチリクエスト Issue からコピーできます
  - メインの質問のスペースをクリアします
  - 重要なポイントのスペースをクリアし、DRI（直接責任者）をインクワイアリ要請者の名前に更新します
  - まだこのドキュメントを閉じないでください。これらはネストされた手順です。
- インクワイアリ Issue を作成
  - gitlab.com>product marketing>に移動して新しい Issue を開始し、Issue テンプレート AR-inquiry を使用します
  - 記入してください。出席者には GitLab ハンドルを使用してください。インクワイアリの日付には、年-月-日の日付形式を使用してください
  - Google ドライブのミーティングノートに戻り、「共有」を押してドキュメントリンクのコピーを取得し、それをミーティングノートセクションに貼り付けます
  - Web ページまたはドキュメントへのリンクなど、関連するサポートドキュメントへのリンクをコピーします
  - **すべての Issue を機密としてマークしてください - 業界アナリストとのインクワイアリは、通常、非公開の独自の知的財産および/またはその他の機密情報を含み、私たちはそれをそのように保つために役割を果たす必要があります。**
  - 期日にはインクワイアリの日付を使用してください
  - 「Submit issue」をクリックします。Issue が作成されたら、インクワイアリ要請者を Issue に割り当てます
  - 関連する Issue がある場合は、そのセクションに移動して追加します。
    - リサーチリクエスト Issue
    - これがシリーズの場合、同じトピックの他のインクワイアリ
    - MQ/Wave Issue または他の関連リサーチへのリンク
  - ブラウザバーで Issue へのリンクをコピーし、ミーティングノートページに戻ります。これで Issue については一旦完了です。
- ミーティングノートページのインクワイアリ Issue リンクを更新します。
- 「共有」ボタンをクリックして、ミーティングノートページのコピーを作成し、**共有/アクセス権限が GitLab 内部アクセスに適切に設定されていることを確認してください。関連する Issue を機密としてマークすることに関する以前の参照を参照してください。** これでミーティングノートについては一旦完了です。
- カレンダー招待に戻り、「Meeting Notes」と書いた場所にリンクを追加します。
- 招待を確認してすべてが正確であることを確認し、それを送信します。
- **推奨: 水分補給を保ち、お気に入りの飲み物を手元に満タンにしておいてください！**
- インクワイアリのビデオ会議について:
  - Gartner と Forrester は現在、彼らの相互作用（インクワイアリを含む）を提供するために一貫して Webex を使用しています。Gartner または Forrester は、インクワイアリの確認時に Webex リンクを提供します。これを使用する必要があります。GitLab チームメンバーが参加する場合または何かをデモする予定の場合は、事前にアナリスト会社の Webex を使用する必要があることをチームメンバーに思い出させてください。私たちの経験では、GitLab チームメンバーは、Webex ソフトウェアおよび/またはアップデートをダウンロードしてインストールするのに少し追加の時間が必要な場合があり、すべての機器が Webex で機能していることをテストし、Webex サービスのパフォーマンスに合わせる必要があるかもしれません。これも私たちの経験では、Zoom ほど一貫していません。
  - IDC/Redmonk/451 は通常、GitLab 提供の Zoom を使用しても問題ありません。場合によっては、独自の Zoom リンクも提供します。これが大丈夫かどうか、新しいアナリストに確認してください。Zoom を使用する場合は、それをスケジューリング エンティティに送信する必要があり、彼らはアナリストの内部招待を更新します。
