GitLab ブログ ハンドブックへようこそ！[GitLab ブログ](https://about.gitlab.com/blog/)はコンテンツマーケティングチームが管理しています。ブログの管理編集者は Sandra Gittlen（@sgittlen）です。

## ブログ投稿とは何ですか？

GitLab では、ブログ投稿は主にオーディエンス（[DevSecOps プロフェッショナル](/handbook/product/personas/)）と有用な情報を共有することに焦点を当てています。ブログ投稿を提案または執筆する際は、読者に何を提供するかを常に考えてください。投稿が内部向けまたは個人的なエッセイのような内容であれば、ブログには適さない可能性があります（ただし、個人の LinkedIn ページには掲載できます）。

ブログは以下のカテゴリに分類されます:

- テクニカルチュートリアル / ハウツー
- 意見 / 思想的リーダーシップ
- 機能と能力の紹介
- オープンソースコミュニティ
- 顧客事例研究 / インタビュー
- 会社の発表（エグゼクティブコミュニケーションチームとの連携で行う）
- 機能 / 変更などの発表
- パートナーシップ / アライアンスを強調するゲストブログ

## GitLab ブログにコンテンツを公開できるのは誰ですか？

[GitLab では誰もが貢献できます](/handbook/company/mission/#mission)。ブログに関しては、ブログの提案、アイデア、下書きを歓迎します。ただし、[メインブログ](https://about.gitlab.com/blog/)は GitLab の多くの公式の声の一つです。つまり、そこで公開されるコンテンツは、会社と製品の両方として GitLab を正確に代表していることを確認するために注意深く審査される必要があります。ブログ管理編集者とグローバルコンテンツマーケティングディレクターが[公式 GitLab ブログの DRI](/handbook/people-group/directly-responsible-individuals/)であり、この責任を担っています。

GitLab ブログについて質問がある場合は、[sgittlen@gitlab.com](mailto:sgittlen@gitlab.com) にご連絡ください。

## ブログのアイデアを提案する方法 - 新プロセス {#how-to-suggest-a-blog-post}

ブログのアイデアを提案するには、[ブログ提出テンプレート](https://gitlab.com/gitlab-com/marketing/brand-product-marketing/content-strategy-and-ops/blog/-/issues/new?issuable_template=blog-post-submission)を使用して Issue を作成してください。

アイデアを提出する際:

- テンプレートのすべての質問に回答してください。これらはアイデアへのフィードバック提供に役立ちます。
- 目標公開日の少なくとも 2 週間前にアイデアを提出してください。
- コーポレート / コミュニケーションのリクエストや急ぎのブログの場合は、Issue を提出して @sgittlen に直接連絡してください。

**注:** [ブログ提出テンプレート](https://gitlab.com/gitlab-com/marketing/brand-product-marketing/content-strategy-and-ops/blog/-/issues/new?issuable_template=blog-post-submission)を通じて、公開済みブログの誤字の通知や変更リクエストも行えます。

ブログ編集者はピッチを確認し、a) 投稿を承認する、b) 改善のための提案を行う、c) そのアイデアがブログに適さない理由を説明し、メッセージを発信するための他のアイデアを提案する、のいずれかの対応をします。

**ニュースレターコンテンツのリクエストの場合:** リクエストを[この Issue](https://gitlab.com/gitlab-com/marketing/brand-product-marketing/content-strategy-and-ops/content-ops/-/issues/902)に追加してください。

### 外部からのコントリビューション

**注:** GitLab は未依頼のブログ提出を受け付けていません。GitLab パートナーでブログ投稿のアイデアを検討のために売り込みたい場合は、[sgittlen@gitlab.com にメール](mailto:sgittlen@gitlab.com)してください。GitLab コミュニティメンバーでブログ投稿のアイデアを検討のために売り込みたい場合は、[contributors@gitlab.com にメール](mailto:contributors@gitlab.com)してください。ピッチとともにブログの下書きを**送付しないでください**。

ピッチのアイデアは GitLab ブログとその読者に関連している必要があり、以下の情報を含める必要があります:

- ブログは何について書く予定ですか？できるだけ具体的にお書きください。
- 以下のうち、あなたのブログの説明として最も適切なものはどれですか？（少なくとも 1 つにチェックしてください）

- [ ] 機能 / 能力の最適な使い方についてのチュートリアル / ハウツー
- [ ] GitLab プラットフォームの使い方のベストプラクティス
- [ ] オープンソースコミュニティと GitLab
- [ ] パートナー / アライアンスの統合
- [ ] その他 - 説明してください

**すべてのブログのピッチアイデアと提出物は、GitLab チームメンバーによって審査・レビューされます。**

ブログ編集者は、寄稿記事の冒頭に以下の 2 文を追加してください:

***編集者注: 時折、コミュニティのメンバーに GitLab ブログへの寄稿を依頼しています。[組織名]との共同制作に感謝します。***

外部からのコントリビューションは[外部ブログ提出規約](https://about.gitlab.com/externalblogsubmissionsterms/)の対象となります。GitLab に資料を提出する前にこのドキュメントをお読みいただき、規約に同意してください。

## アイデアが承認されたらブログの下書きを提出する方法

**ブログのピッチが承認されたら、**著者は以下のいずれかの方法で下書きを提出します:

- GitLab のマージリクエストで提出する。これを行うには、[リポジトリからの手順に従ってください](/handbook/marketing/digital-experience/decap-cms/#creating-a-blog-post-in-the-aboutgitlabcom-repository)または [IDE の場合はこちら](/handbook/marketing/digital-experience/decap-cms/#creating-a-blog-post-using-the-web-ide)。**注意して**手順に従ってください！質問がある場合は @sgittlen に連絡してください。

- 基本的なコンテンツマネジメントシステムである Decap を通じて[これらの手順に従って](/handbook/marketing/digital-experience/decap-cms/#creating-a-blog-post-using-decap-cms)提出する。**注意して**手順に従ってください！質問がある場合は @sgittlen に連絡してください。

- [ブログ提出テンプレート](https://docs.google.com/document/d/1iWk0a5T14dG5fvIxvG4doqqRxx0pyv3ej_aMDqYXjgA/template/preview)（Google ドキュメント）を使用して Google ドキュメントでブログを執筆し、Google ドキュメントが Issue にリンクされていることを確認して提出する。**テンプレートのすべてのフィールドに記入しないと、ブログチームは提出物を受け付けません。**著者は Issue に @sgittlen をタグ付けし、コメントにドキュメントリンクを入れ、編集できる状態になったら（必要なすべての承認とレビューが完了した後に）Google ドキュメントを共有してください。**注:** すべての画像は Google ドキュメント内にインラインで含める必要があります。

**注:** すべてのブログ提出物には、**行動喚起（CTA）**が必要になりました。ブログ下書きテンプレートで CTA を提供するよう求められます。CTA とは、ブログを読んだ後に読者にしてほしいことです。別のページに移動してさらに学んでほしい、試用版に申し込んでほしい、ウェビナーに登録してほしい、デモを見てほしいなど、何を希望しますか？CTA はブログ全体のメトリクスの一部として追跡されます。

ブログをどの方法で提出するにしても、@sgittlen がタグ付けされていることを確認してください。

**ブログはブログチームからのレビュー / 承認なしには公開できません。**

## MR でのフロントマターガイド

MR には以下のフロントマターが含まれます。各セクションのコメントを参照してください。

```text
seo:
  title: ブログ投稿のタイトル
  description: ブログ投稿の説明
  <!-- ブログチームがこの部分を入力します -->
config:
  slug: blog-post-slug
  featured: false
  template: BlogPost
  <!-- ブログチームがここに入力します -->
content:
  title: ブログ投稿のタイトル
  <!-- 理想的な長さ - 55〜60 文字 -->
  description: ブログ投稿の説明
  <!-- 理想的な長さ - 155 文字以内 -->
  authors:
    - ブログ投稿の著者
  <!-- 形式: Sandra Gittlen（初めて寄稿する場合は空白のままにしてください。著者が複数いる場合は、スペースなしでカンマを追加してください） -->
  heroImage: images/blog/hero-images/logoforblogpost.jpg
  <!-- ブログチームがここに入力します -->
  date: '2021-03-31'
  category: engineering
  <!-- ブログチームがここに入力します -->
  tags:
    - community
  <!-- ブログチームがここに入力します -->
  body: |
    ブログ投稿の本文テキストを Markdown 形式で追加
```

## ブログ編集プロセス

ブログチームは Issue と Google ドキュメントを使用して著者への最初の編集 / 質問を伝えます。その後、ブログチームはブログを CMS に入力し、必要に応じてプレビューリンクを著者 / DRI と共有します。注: ブログは Contentful から公開されます。

公開後に変更が必要な場合、著者はブログチームに Slack で変更内容を説明するか、公開済みページの下部にある「このページを編集」をクリックして MR を提出します。これにより MR が開始されます。

ブログチームはブログが公開されたら、著者に URL を共有します。

ブログ投稿の公開は編集カレンダーに従いますが、緊急のブログによって変更される場合があります。ブログチームは、著者に予定公開日を随時お知らせします。

### 法的レビュープロセス

一部のブログ投稿は、[資料法的レビュープロセス / SAFE プログラム](/handbook/legal/materials-legal-review-process/)に従って法務のレビューを受ける必要があります。著者は SAFE ガイドラインをレビューし、Google ドキュメントをブログチームと共有する前に法的承認を取得する責任があります。このプロセスには時間がかかる場合があるため、目標公開日を計画する際には十分な時間を確保してください。

GitLab には行動バイアスがあり、ブログチームも同様です。ただし、GitLab ブログは会社を代表する公開資産です。ブログチームがブログ投稿に含まれる情報について懸念や質問がある場合、会社への潜在的なリスクを軽減するために法務、コーポレートコミュニケーション、パートナーマーケティング、CMO などがブログ投稿をレビューできるまで、ブログチームはブログ投稿を保留する権限を持っています。

[ハンドブックページを読んで SAFE ガイドラインについて詳しく知る](/handbook/legal/safe-framework/)か、[資料法的レビュープロセス](/handbook/legal/materials-legal-review-process/)に従ってください。

## 公開済みブログへの変更を提案する方法

- GitLab の内部関係者で公開済み GitLab ブログへの変更を提案したい場合は、変更内容と URL の詳細を `#content` Slack チャンネルに投稿して @sgittlen をタグ付けするか、Slack で @sgittlen に直接連絡するか、公開済みページの下部にある「このページを編集」をクリックして必要な変更を含む MR を作成してください。これにより MR が開始されます。

- GitLab の外部関係者の場合は、提案する変更の詳細を `sgittlen@gitlab.com` の Sandra Gittlen にメールするか、公開済みページの下部にある「このページを編集」をクリックして必要な変更を含む MR を作成してください。これにより MR が開始されます。

## ブログを執筆する際の考慮事項

### ブログライター向け多様性、包括性、帰属意識（DIB）チェックリスト

ブログコンテンツが、多様性、包括性、帰属意識という会社の価値観を反映していることが重要です。これらの点がすべてブログ投稿に関連するわけではありませんが、執筆プロセス全体を通じて念頭に置くべき重要な価値観と実践です。ブログ編集チームはこれらを確認するよう努めますが、すべてのコンテンツをこれらの価値観と実践を念頭に置いて作成することが望ましいです。質問がある場合は私たちまたは [DIB チーム](/handbook/company/culture/inclusion/)のメンバーにタグ付けしてください！

#### インクルーシブな執筆

- 短く簡潔な文章を書いてください。短い文章のわかりやすい文章は、読者が理解しやすくなります。
- 専門用語の使用を制限し、専門用語を使わなければならない場合は、初回の使用時に定義してください。
- GitLab はグローバルなチームとグローバルなコミュニティを持っているため、グローバルなオーディエンスを対象に執筆する必要があります。これは、地域のメタファーの使用を制限し、米国中心の書き方をしないことを意味します。
- 投稿では[インクルーシブな言語](/handbook/company/culture/inclusion/#inclusive-language)を使用していますか？
- ブログ投稿に登場するすべての個人は、[望ましい代名詞](/handbook/people-group/orientation-identity/#how-do-i-know-which-pronouns-to-use)を使用してタグ付けされていますか？ヒント: 誰かの望ましい代名詞がわからない場合は、直接尋ねてください。[チームページのプロフィール](/handbook/company/team/#close-modal)と Slack プロフィールにも記載されているはずです。

#### その他の DIB 執筆のヒント

- ブログ投稿に偏見はありますか？私たちには皆、無意識の偏見があります。[DIB ハンドブックページの無意識の偏見リスト](/handbook/company/culture/inclusion/unconscious-bias)を確認し、自分の偏見を認識して管理するための戦略を実践してください。
- まだ質問がある場合は、遠慮なく編集チームリード @sgittlen、ブログ Issue で適切な[チームメンバーリソースグループ（TMRG）](/handbook/company/culture/inclusion/tmrg-tmag/)のリードにタグ付けするか、diversityinclusionandbelonging@gitlab.com に質問をメールしてください。

### ブログのカテゴリとタグ

#### カテゴリ

投稿ごとに以下のカテゴリのうち**一つだけ**を使用してください。
大文字小文字の区別、スペル、その他の変更は**絶対にしないでください**。
そうしないと、別のカテゴリが作成されてしまいます。これは誤って行いたくないことです。

投稿がどのカテゴリに属するか確認できない場合は、MR でプレースホルダーを入れ、そのコメントでレビュアーに note を残してください。

- `agile planning` - アジャイルプランニングに関する投稿
- `ai-ml` – プラットフォームまたは業界全体の AI/ML に直接焦点を当てた投稿
- `customer stories` - 顧客が GitLab DevSecOps プラットフォームを使用する方法に関する投稿
- `bulletin board` - 短いブログ / 発表を掲載する場所
- `DevSecOps` - DevSecOps に関する一般的な投稿
- `engineering` – 技術的で実用的なコンテンツ。何かを行う方法、使う方法、問題を解決する方法をカバーするものはすべてこのカテゴリに入ります
- `open source` – コミュニティ、ユーザー、またはより広いオープンソースコミュニティに関するストーリー
- `product` - 機能、ロードマップ、戦略に関する詳細
- `news` – 会社または製品の発表（ポリシーの変更、運用上のアナウンス、重大な変更を含む）、ニュース、またはイベント
- `security` – セキュリティ関連の投稿
- `releases` - リリース投稿、セキュリティリリース、パッチリリース。`releases` カテゴリの投稿は `sites/uncategorized/source/blog/blog-posts` ではなく `sites/uncategorized/source/releases/posts` ディレクトリに入れる必要があります。詳細については[リリース投稿ハンドブック](/handbook/marketing/blog/release-posts/)を参照してください。

#### 画像の準備

- オリジナルのカバー画像を作成する場合、すべてのディスプレイで最適な品質を得るためにサイズは 1800px × 945px にしてください。
- すべての画像は 1MB 未満を目指してください。JPEG は PNG よりも小さい傾向があるため、可能な場合は JPEG を使用してください。
- Preview でリサイズするには、`ツール`、`サイズを調整`に進み、`解像度`フィールドのエントリを調整します。Preview は `OK` をクリックして確認する前に、結果として得られる画像サイズを推定します。
- すべての画像は同じ幅に保ってください。

#### スクリーンショット

技術 / チュートリアル投稿の場合は、[コードブロック](#adding-code-blocks)またはスクリーンショットで例を図示してください。例に一貫性を持たせてください。例えば、汎用 URL `domain.com` を使用してステップを説明する場合は、投稿全体を通じて `domain.com` を一貫して使用してください。

- 静止画像はコンセプトの説明、図の提供、UI の要素の表示、または読者を案内するために使用します。
- コピーペーストできないコマンドや設定をレンダリングするために画像を使用しないでください。
- アニメーション GIF は、時間の経過または複数のアクションにわたって発生するプロセスやイベントを表示する必要がある場合に控えめに使用できますが、テキストの説明や指示の代わりに使用しないでください。
- スクリーンの特定の部分を識別してローカライズするためにスクリーンショットを使用してください。Nimbus Screenshot（ブラウザ拡張機能）、Mac スクリーンショット、Windows 用スニッピングツール、Ubuntu 用スクリーンショットツールなど、優れたツールがあります。

**重要なセキュリティポイント:** 実際のトークンやセキュリティ認証情報を使用して個人情報を公開しないでください。代わりに `[project's CI token]` などのプレースホルダーを使用するか、スクリーンショットに表示されている場合はぼかしてください。

#### ツイートや Instagram 投稿の埋め込み

[ソーシャルメディアからの投稿を埋め込む手順については、Markdown ガイドを参照してください](https://about.gitlab.com/community/markdown-guide-middleman/#embed-tweets)。

#### GIF の作成

アニメーション GIF は、短い動的なプロセスを説明するのに非常に役立ちます。これらはこの種のリソースで理解しやすい場合があります。
アニメーション GIF を作成する方法はいくつかあり、その一つは Mac 用の軽量アプリ [Giphy Capture] を使用する方法です。

ファイルサイズが非常に大きい GIF は避けてください。インターネット接続が悪いユーザーにとって読み込みが困難になります。そのような場合は、GIF を小さな部分に分割するか、動画を録画するか、連続画像を使用することができます。

### CMS での `Author` エントリの作成

CMS（Decap）で `Author` エントリを**作成する際**の推奨事項を示します。

- `Name` フィールド
    このフィールドは必須です。また、ユニークなフィールドでもあります。

    1. 著者名は姓と名のみの組み合わせである必要があります（複合名を含む）。職種の説明は追加しないでください（このためには `Role` フィールドを使用してください）。また、英数字の組み合わせも入力しないでください。
    1. 二重著者名は使用しないでください（例: `/authors/<author1>-<author2>`）。複数の単一著者をブログ投稿に追加できます。個別の著者を作成してから、それらすべてを他のコンテンツタイプに一対多の関係として追加するべきです。
    1. 上記と同様に、単一のブログ投稿に複数著者を使用しないでください（例: `/authors/<author1>-<author2>-<author3>-and-<author4>`）。

- `Role` フィールド

    現在の職位または職種の説明。

- `Bio` フィールド

    著者の経歴。

- `GitLab handle` フィールド

    著者の GitLab ユーザー名（小文字形式）に対応します。このフィールドはユニークである必要があります。

- `Social media handles` フィールド

    ソーシャルメディアハンドルのオプションフィールド。

## CMS でのブログ投稿のローカライズ

英語以外の言語でブログ投稿を公開できます。現在、以下の言語をサポートしています:

| 言語 | URL 構造 |
|---------|----------|
| フランス語 |`/fr-fr/blog/YEAR/MONTH/DAY/Title/` |
| ドイツ語 |`/de-de/blog/YEAR/MONTH/DAY/Title/` |
| 日本語 |`/ja-jp/blog/YEAR/MONTH/DAY/Title/` |

詳細については、GitLab ローカリゼーションチームにお問い合わせください。

### 翻訳を開始する方法

ブログ投稿の翻訳プロセスを開始するのは簡単です。以下の手順に従ってください:

1. ローカリゼーションプロジェクトで翻訳リクエストの Issue を作成します。プロのヒント - ブログ投稿の URL と Contentful エントリへの直接リンクを追加してください。プロジェクトの追跡に非常に役立ちます。
2. Argo は英語のブログ投稿を JSON としてエクスポートし、翻訳された JSON を Contentful にインポートします。その後、レビューの準備ができたら Issue でピンされます。
3. ブログ投稿のエントリに入り、翻訳されたコンテンツをレビューします。
4. @sgittlen と協力して翻訳済み投稿を公開します。
5. 公開後は、about.gitlab.com/blog/ で期待通りに機能しているか常に確認してください。
6. Slack に投稿して翻訳済みブログ投稿を祝いましょう。

翻訳リクエストの作成から本番環境への公開まで、プロセス全体のビデオウォークスルーを以下に示します。

- 以下を示すビデオ録画が近日公開予定です
- Issue の作成
- Argo のラウンドトリップ
- Contentful に表示される翻訳
- 翻訳のレビュー
- ライブプレビュー / 言語の変更
- 公開

### 英語ブログ投稿をローカライズする際の注意点

英語ブログ投稿をローカライズする際に注意すべきことがあります。

- スラグを変更しないでください！

### 整理のためのタグ

- 翻訳中の投稿には「translation-in-progress」ラベルを付けてください。これにより、チームが翻訳・レビュー中のすべての Contentful のブログ投稿を簡単に見つけられるようになります。
- 投稿が翻訳された言語のローカルラベルを削除・適用します。
| タグ | 定義 | 使用方法 |
|----|----|----|
| translation-in-progress | ブログ投稿が現在翻訳・レビュー中であることを示す | 翻訳リクエストが開かれた時に適用。投稿が公開されたら削除 |
| language_de-DE | ドイツ語のエントリをマーク | ドイツ語に翻訳されたブログ投稿に適用 |
| language_fr-FR | フランス語のエントリをマーク | フランス語に翻訳されたブログ投稿に適用 |
| language_ja-JP | 日本語のエントリをマーク | 日本語に翻訳されたブログ投稿に適用 |
