## Digital Experience: Foundations Agenda

> これまで GitLab では、これを「_マーケティング Web サイト_」と呼んできました。1B ドル規模で上場間近の企業として、私たちはメインのマーケティングツールを高度化し、「_デジタルマーケティングプラットフォーム_」を発展させて取り組みを運用化する必要があります。現在、この変更を行うための基盤となる要素が整っていません。そのため、ツールセットを成熟させ、テクノロジーをアップグレードする必要があります。

### このページの目的は、ブロッカーを特定し、私たちのチームを解放することの価値を強調することです

| 主要な課題 | ブロッカー |
| ----------- | -------- |
| **効果的にスケールできない**<br>[⏱️ 効率](/handbook/values/#efficiency) | - 私たちのデジタルマーケティングプラットフォームは結果を最適化していない<br>- スケーリングを可能にする主要なツールが欠けている<br>- 欠けているツール／テクノロジーは、より良い未来の状態へイテレーションできるよう、一連の MVC として実装可能な「退屈な解決策」である<br>- 運用化できるシステムがない<br>- 私たちのデジタルマーケティングプラットフォームは適切なグループに最適化されていない<br>- 私たちの体験は一貫性に欠けており（特にユーザーフローとファネル進行）、学習・改善が困難である |
| **学習が困難**<br>[🤝 コラボレーション](/handbook/values/#collaboration)<br>[👣 イテレーション](/handbook/values/#iteration)<br>[👁️ 透明性](/handbook/values/#transparency) | - 一貫性の欠如によりデータの価値が最小化される<br>- [ロイヤリティフライホイール](https://solutions.trustradius.com/wp-content/uploads/hubspot-flywheel-inbound-method-lifecycles-1024x1024.png)へのフォーカス不足により、私たちが行う変更の効果が最小化される<br>- SSOT や顧客理解なしにコミュニティとして働くのが困難 |
| **ヒューマンエラーの可能性が高い**<br>[⏱️ 効率](/handbook/values/#efficiency) | - 自動化されたビルドプロセスがない<br>- 自動データやユーザー生成データの週次・隔週レビューがない<br>- これらが整えば、私たちのイテレーションとサイクルタイムを短縮できる |
| **計画に一貫して従うのが困難**<br>[📈 成果](/handbook/values/#results) | - 運用化されたアプローチがないと、競合する力に対してアクションへのバイアスを保つことが難しい |

### 🎯 私たちは自信を持てる計画を持ち、それに従い、一貫して結果を出す必要があります

---

| 必要なこと | 価値 |
| -------------------- | ----- |
| **CMS の実装** | - 適切なグループ（コンテンツマーケティング）の効率化と最適化<br>- セルフサービスとセルフラーニングの能力向上<br>- 他者の時間を尊重。Digital Experience の高頻度・低価値な労力を解放 |
| **静的サイトジェネレーター（SSG）の切り替え** | - GitLab のテクノロジーの強み（Ruby ＆ VueJS）と整合<br>- 適切なグループ（Digital Experience 開発チーム）の効率化と最適化<br>- マーケティングがより良いツールを使ってスケールと提供のポテンシャルを解放できるように |
| **[デザインシステム](https://gitlab.com/gitlab-com/marketing/inbound-marketing/slippers-ui)の実装** | - イテレーションとリリースのコストを下げる<br>- 誰でも貢献できる共通のデザイン言語を作成し、また、遅くて高価なカスタムソリューションではなく、効率的な「退屈な」ソリューションへの私たちのコミットメントを支援する。<br>- オンボーディング、立ち上げ、スケールを迅速にする — チームメンバーとベンダーの両方で |
| **ビルドプロセス（Gulp または Webpack）の組み込み** | - ヒューマンエラーを減らす<br>- 数年来の業界標準である退屈なソリューション<br>- ページロード時間の高速化<br>- 画像圧縮の自動化<br>- CSS ＋ JS の最小化<br>- 重複コードの特定<br>- アバブザフォールドのスタイルを自動追加して体感ロード時間を高める |
| **Git から画像を削除し、CDN に直接プッシュする** | - リポジトリサイズを削減<br>- リポジトリでの作業を高速化<br>- 変更プッシュの高速化 |
| **Customer Data Platform** | - ユーザーセグメントの正確な理解を集約<br>- すべての GitLab チームメンバーがアクセス可能な単一のソース・オブ・トゥルースを実現<br>- 情報を GitLab 内で公開することによる透明性向上<br>- 情報に基づくリード判断を可能にする（私たちは遅行データに基づく判断は得意なように見える）<br>- 思考と実験機会を活性化<br>- 実験プラットフォーム／プログラムとパーソナライゼーションの基盤を整備 |
| **実験プラットフォーム／プログラム** | LaunchDarkly とアジャイル／スプリントベースのアプローチを使用し、マーケティングプラットフォーム上でリリース・実行されるテストの量を大幅に増やすことを目指す |
| **パーソナライゼーション** | - MQL の増加<br>- コンバージョン率の向上<br>- エンゲージメントの向上<br>- IATV と LTV へとユーザーを誘導 |
| **マーケティングサイトの情報設計の改善** | - セールスファネルの進行へのフォーカスを高める<br>- コンバージョンの向上<br>- 実験と変更のインパクトを向上<br>- より価値のあるデータ |
| **セッション再生／ロギングツールの追加** | - ユーザーがマーケティングサイトとどう関わっているかを理解する<br>- 任意の API またはコードの問題を特定する |
| **自動化機会の特定** | 私たちの人々が最も重要なことに集中できるようにする |

## オープンな機会

この [Get It Done Matrix](https://docs.google.com/spreadsheets/d/1G809pSClwbGCMaEJdzz_MHykVBOlPV3XYFi-IhU02Oo/edit#gid=0) は、デジタルマーケティングプラットフォームを高度化するために活用できるオープンエピックと Issue を強調しています。

数値は、[**デザインシステムの実装**](https://gitlab.com/gitlab-com/marketing/inbound-marketing/slippers-ui)、**CMS の実装**、**ビルドプロセスの組み込み** に集中すれば大きな成果が得られることを示しています。提供する機会が最も大きいオープンな作業群は、[Marketing Site Refresh Strategy](https://gitlab.com/groups/gitlab-com/marketing/growth-marketing/-/epics/32#note_384150218) と [Phase 5: Systems for Continuous Improvement](https://gitlab.com/groups/gitlab-com/marketing/growth-marketing/-/epics/55#note_381002507) です。これらのエピックは両方ともそれなりに先の話で、Digital Experience チームとしてはポテンシャルを解放するために優先順位を見直したいと考えています。
