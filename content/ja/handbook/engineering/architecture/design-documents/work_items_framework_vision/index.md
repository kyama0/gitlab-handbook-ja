## 概要

このドキュメントはワークアイテムフレームワークのエンジニアリングビジョンと意思決定フレームワークとして機能します。
GitLab でエンタープライズアジャイル計画機能を構築するチームを導くために、次の 2〜3 年の建築原則と技術的な方向性を確立します。

ビジョンは、明確な境界を提供しながら実装の詳細における柔軟性を確保するために、意図的に高レベルに設定されています。これにより、プラットフォームがスケーラブル、拡張可能、柔軟性を維持できます。

## 動機

GitLab でエンタープライズアジャイル計画の未来を構築し、[Plan ステージの戦略的 UX ビジョン（内部リンク）](https://gitlab.com/gitlab-org/plan-stage/product-design/-/wikis/Plan-Strategic-UX-Vision/)を実装するために、現在の状態を超えてワークアイテムフレームワークをスケールするバックエンドとフロントエンドの両方で柔軟なシステムが必要です。

複数のチームが様々なレベルのインタラクションでワークアイテムフレームワークに取り組んでいます。このビジョンは、チームメンバーが機能を実装する過程で行うすべての決定を確認するために使用できるブループリントとして機能します。チーム間の一貫性を確保し、アーキテクチャのドリフトを防止します。

## 北極星: 計画とコラボレーションのための統一されたワークアイテムフレームワーク

ワークアイテムは SDLC を通じて作業を追跡し、チーム間のコラボレーションを可能にします。このフレームワークは、共通の目的を共有するすべての計画とコラボレーションの概念を統一します: 作業を説明し、追跡し、定義された状態を通じて流れさせること。フレームワークは一度、すべてのタイプのために構築されます。機能はタイプ固有のコードを必要とせずにすべてのワークアイテムタイプに対してスケールします。

このスコープは、協調的な作業追跡パラダイムに適合しない根本的に異なる目的を持つ概念を除外します。

このドキュメントのすべてのアーキテクチャ決定はこのビジョンに向かって機能する必要があります。

## コアな原則

### 0. フレームワークへの整合は交渉の余地がない

すべての機能、修正、追加はフレームワークに整合している必要があります。特定のタイプのみで機能したりハードコードされたチェックを必要とする迅速な修正に対する言い訳は一切ありません。

**これが意味すること:**

- Issue のために構築された機能は、チケット、エピック、タスク、および将来のすべてのタイプで機能する必要があります
- フレームワークはタイプに依存しない必要があります — どのタイプが存在するか、どれが利用可能かは関係ありません
- 設定とデータ駆動の設計は必要条件であり、あればいいものではありません
- この原則に違反する迅速な修正はビジョンを遅らせます。正しく行うことがそれを加速させます

**これが重要な理由:**

- すべてのフレームワーク整合の機能は自動的にすべてのタイプで機能します
- タイプに依存しない設計は、組織がコードを変更せずに独自のワークアイテムタイプを設定できることを意味します
- 将来のセットアップでは Issue がまったく含まれない場合があります — フレームワークは関係なく機能する必要があります
- これがフラグメンテーションなしにスケールする方法です

**遅延すべき時:**

機能がフレームワーク整合で構築できない場合は、できるようになるまで遅延させるべきです。詳細については[リリースを遅延すべき時](#when-to-delay-a-release)を参照してください。

**例外:**

一部のタイプはドメイン固有の機能を持つ場合があります（[特殊タイプの処理](#special-type-handling)を参照）。例外は特殊な機能のためであり、コアフレームワーク機能のためではありません。

### 1. タイプチェックよりも設定

バックエンドとフロントエンドの両方で、アプリケーション全体のハードコードされたタイプチェックの代わりに設定ベースのアプローチを使用して、タイプの動作を制御します。バックエンドとフロントエンドは、特定のタイプがどのように動作するかを確実にするために、タイプチェックの代わりにこれらの設定を使用する必要があります。

**これが意味すること:**

- 複数のタイプが同じ設定を共有できます
- 設定は、タイプがどのように動作しレンダリングするかを制御するブール値フラグまたは値ベースの属性です
- 私たちは「タイプが何であるか」ではなく「タイプが何をできるか」を気にします

**これが重要な理由:**

- フレームワークをモジュール式、コンポーザブル、拡張可能、スケーラブル、柔軟に保ちます
- 機能と特定のタイプ間の結合を減らします
- 既存のコードを変更せずに新しいタイプを導入しやすくします
- カスタムタイプがシステム定義タイプと同じ機能を活用できるようにします
- 不可能な UI 状態が減り、一般的なリグレッションが少ないクライアント上でのワークアイテムの消費をよりシンプルにします

**例:**

```ruby
# Backend: Checking configurations
type.configured_for?(:use_legacy_view)
type.configured_for?(:group_level)
type.configured_for?(:filterable) # for example available in general work item list and filters

# Future: value-based configurations
type.configuration(:required_widgets)
```

**フロントエンド設定プロバイダーパターン:**

この要件を実装するために、フロントエンドは名前空間ごとにワークアイテムタイプ設定を取得し、Apollo でキャッシュする設定プロバイダーパターンを使用します。このアプローチは、現在の状態（タイプが異なる動作を持つ）とビジョン（すべてのタイプが同じように動作する）の間の一時的な橋渡しです。プロバイダーは、ユーザーが異なる名前空間のアイテム間を移動するときに設定が常に利用可能で最新であることを確保します。実装の詳細については、[設定可能なワークアイテムタイプ](../configurable_work_item_types/_index.md)ドキュメントとその[フロントエンドメタデータプロバイダーパターン](../configurable_work_item_types/_index.md#frontend-metadata-provider-pattern)セクションを参照してください。

利用可能なデータに基づいて動作を推測することを**強く避けて**、API 設定によって提供される機能と機能を直接マッチさせてください。

### 2. データ駆動のフロントエンドアーキテクチャ

フロントエンドは完全にデータ駆動型であり、何を表示するかとどのように動作するかを決定するために API レスポンスに依存する必要があります。

**これが意味すること:**

- フィルターは API レスポンス（workItemTypes クエリ、メタデータエンドポイント）からのデータで駆動されます
- リストに表示するフィールドは API レスポンスで駆動されます
- 一括編集のフィールドは API レスポンスで駆動されます
- どのフィールドやウィジェットが利用可能かについてのハードコードされた仮定はありません
- 機能を推測しないでください。例えば、グループのワークアイテムのみが特定の動作を持つ場合、`if(!isGroup)` を行わないでください。かわりに、新しい API フィールドを持つ作業をスケジュールし、すべてのタイプが柔軟に恩恵を受けられるようにメタデータプロバイダーで消費してください。
- フィールドと機能のライセンス機能チェックはバックエンドで実行されるため、フロントエンドは利用可能な機能を表示するために API レスポンスのみに依存できます
- ワークアイテムはすべての接続されたクライアントにリアルタイムで更新されます。ユーザーがフィールドを変更すると、そのアイテムを閲覧している他のすべてのユーザーはリフレッシュすることなく変更を確認します。これは詳細ビューから始まり、ワークアイテムが表示されるすべての場所（リスト、ボード、その他のアグリゲーションビュー）に拡大されます。すべてのウィジェットが参加する必要があります。一部のウィジェットがライブで更新され、他のウィジェットが更新されない部分的なリアルタイムカバレッジは、一貫性のない動作とユーザーの認知的負荷を引き起こします。

**これが重要な理由:**

- フレームワークの簡単な拡張に不可欠です
- GitLab 提供のフィールド（ウィジェット）と同等のカスタムフィールドをサポートします
- フロントエンドが利用可能なデータの可視化のみを気にするビュー（テーブル、リスト）を可能にします
- バックエンドの機能が進化するときのフロントエンドの変更を削減します
- すべての接続されたクライアントにわたるリアルタイム更新は、データのドリフトと矛盾する状態を防ぎます。これは複数の人が同時に同じアイテムで作業するコラボレーティブな計画ワークフローで特に重要です

**オープンな質問:**

1. データ駆動アプローチでのすべてのインタラクションと楽観的な更新の実現可能性を確認する。

### 3. 孤立した機能よりもシステム思考

共有プラットフォームで作業しているため、孤立した思考とサイロの構築を防ぐために、取り組んでいることを他者と頻繁に共有する必要があります。

**これが意味すること:**

- 孤立したコンポーネントと機能ではなく、常にシステムで考える
- フレームワーク全体で機能する抽象的なソリューション
- 意味のある場所で設定を使用する
- 実装している 1 つだけでなく、すべてのワークアイテムタイプへの影響を考慮する

**これが重要な理由:**

- フラグメンテーションと一貫性のなさを防ぎます
- 一回限りのソリューションから技術的負債を減らします
- 長期的にプラットフォームを保守しやすくします
- 機能がフレームワーク全体にわたって一体的に機能することを確保します

**チームの実践:**

- 作業を他のチームと頻繁に共有する
- フレームワーク全体への影響を持つ変更をレビューする
- 特定のタイプのみで機能するソリューションを推奨しない
- スケーラブルなアーキテクチャを保護するために、たとえリリースを遅らせることを意味しても、正しい方法でのみ物事を行うという共通の理解を発展させる

### 4. 正しいことを行うことをシンプルにする

このエンジニアリングビジョンの原則を害するコードを追加することを困難にする必要があります。

**これが意味すること:**

- レビュアーはこれらの原則に違反するコードを推奨しない
- アーキテクチャは開発者を正しいパターンに向けて導く
- 一般的なタスクには明確で文書化されたアプローチがある
- アンチパターンは実装が困難
- テストは「不正な」実装にフラグを立てる

**これが重要な理由:**

- アーキテクチャの整合性を保護します
- 新しい貢献者の認知負荷を減らし、貢献を容易にします
- コード品質の段階的な低下を防ぎます
- レビューをより客観的で一貫したものにします

## API 戦略と進化

### GraphQL API の改善

#### ワークアイテムクエリのウィジェットをフラット化する

ウィジェットは配列にネストされるのではなく、ワークアイテムクエリのトップレベルの個別フィールドとして公開されます。各ウィジェットフィールドは型付きであり、利用不可のときは `null` を返し、インラインフラグメントの必要性を排除してフロントエンドの複雑さを軽減します。

**メリット:**

- ウィジェットデータにアクセスするときのフロントエンドの複雑さを軽減します
- 顧客向けの GraphQL API を使いやすくします
- データアクセスパターンを簡素化します

**関連作業:**

- [スパイク: ワークアイテムのオブジェクト構造をフラット化](https://gitlab.com/gitlab-org/gitlab/-/issues/493472)
- [MR: ワークアイテムウィジェットを個別フィールドとして公開](https://gitlab.com/gitlab-org/gitlab/-/merge_requests/214524)

### REST API 戦略

GraphQL は現在フロントエンドがその上に構築されているため、主要な API です。ただし、サードパーティの統合、自動化、REST 規約を好む顧客をサポートするためにファーストクラスの REST API に積極的に投資しています。

具体的な実装エピックについては[ワークアイテム REST API の実装](https://gitlab.com/groups/gitlab-org/-/work_items/20907)を、詳細な技術アプローチについては[ワークアイテム REST API デザインドキュメント](../work_item_rest_api/_index.md)を参照してください。

**長期的な戦略:**

パフォーマンスの向上によりフロントエンドのリファクタリングが正当化される場合、将来的に REST が主要になるかどうかを決定する必要があります。この決定は両方の API が成熟するにつれて再検討されるべきです。

**関連エピック:**

### レガシーエンドポイントの廃止戦略

Issue 固有のエンドポイントからワークアイテムエンドポイントへ移行する顧客に明確な移行パスを提供します。

**アプローチ:**

- 明確なタイムラインで Issue とエピックのエンドポイントを廃止する
- REST API の `v4` バージョン全体の廃止に向けて作業する可能性があります
- GraphQL の Issue 固有のエンドポイントを廃止する
- レガシーエンドポイントに新しい機能を追加しないことを明確に述べる。ただし、移行方法と容易さを説明します

**移行サポート:**

- 良い顧客エクスペリエンスのために痛みのない移行パスが重要です
- ガイド、ドキュメント、ビデオを使用する
- 移行ツールと例を提供する

**用語の移行:**

- 全体的にワークアイテムの用語を導入する（Webhook、統合など）
- Issue 関連のすべてがワークアイテム関連になり、「Issue」だけでなく複数のタイプのアイテムを公開することを明確にする

## パフォーマンスとキャッシュ戦略

### バックエンドエンドポイントのキャッシング

パフォーマンスを向上させるためにトラフィックが多く、コストの高いクエリをキャッシュします。

**考慮すべきキャッシング戦略:**

1. **フルレスポンスキャッシング** - データが頻繁に変更されず、後処理なしのクエリにのみ機能します
2. **部分的なレスポンスキャッシング** - Rails キャッシュにシリアライズしてリゾルバーをキャッシュします（フィールドキャッシングに関連しますが、サブフィールドを含む可能性があります）
3. **フィールドベースのキャッシング** - Rails キャッシュまたは異なる戦略を使用します

**キャッシングストレージ戦略:**

1. **Redis（デフォルト）** - Rails キャッシュと RequestStore を使用します
2. **Pod ローカルメモリキャッシュ** - Redis トラフィックスパイクを防ぐためのキャッシュの重複が意図的
3. **Pod ローカルファイルベースキャッシュ** - まだ非常に高速でメモリ消費を気にする必要がありません
4. **データベースストレージ** - キャッシュコンテンツをデータベースに保存
5. **オブジェクトストレージキャッシュ** - 最も遅いが、永続的なキャッシュに適しています

**キャッシングの優先度が高いクエリ:**

理想的には、レンダリングをブロックするもの:

- ワークアイテムタイプ（名前空間ごと）
- メタデータ（名前空間ごと）
- ワークアイテムスリム（名前空間ごと）
- ワークアイテムの詳細
- ノート（理想的には、ページごとのみですが、ユーザーごとの後処理が必要なため困難かもしれません）

**ユースケース:**

- ほぼ静的なデータに最適です
- 名前空間にスコープされた動的データについては、SaaS または `EE` 向けのライセンスされた名前空間のみにキャッシングを制限できます

1. TODO: キャッシング戦略とインフラ制約を把握する

## データモデルの進化

### Issue モデルの削除

WorkItem が Issue から継承するデュアルクラスアーキテクチャは、開発に継続的なコストをかけています。エンジニアは何かが Issue か WorkItem かを理解し、どのモデルに新しい機能を置くかを決定し、レガシーな Issue エンドポイントと新しい WorkItem サーフェスの両方で動作が必要なケースを処理する必要があります。これにより、日々の開発が遅くなり、2 つのクラスが乖離するときに微妙なバグが導入され、オンボーディングが困難になります。

**現在の問題:**

- 継承チェーン: base → issuables → issues → work items → CE と EE で 2 倍
- サービスに対する「継承地獄」を引き起こします
- 複雑さと保守の負担を追加します
- WorkItem に概念的に属するロジックが Issue に追加され続けます。レガシー API がまだ直接 Issue をインスタンス化するためです

**アプローチ:**

WorkItem を issues テーブルに裏打ちされた単一の権威あるモデルにし、Issue を後方互換性のための定数エイリアスに縮小します。これにより曖昧さが完全に排除されます: 1 つのモデルがあり、それは WorkItem と呼ばれ、Issue 定数はすべての参照が移行されるまで後方互換性のエイリアスとしてのみ存在します。

**関連する決定:**

- `Issuables` アプローチを再検討します。現在、タイトル/説明、状態、ラベル、マイルストーン、担当者、ノート、ソートの共通インターフェースを提供する Issue と MergeRequest 間で共有される concern です。Issue の削除後、`Issuable` が「ワークアイテムライト」パターンになるかどうかを検討してください — 計画だけでなく、より広い内部オーディエンスにも機能する、リストと詳細ビュー、作成と更新サービス、共通の concern（サブスクリプション、コメント、説明、リアクション、メンション）を提供するワークアイテムフレームワークの軽量バージョン。

**関連作業:**

- [エピック: Issue モデルを WorkItem に完全移行](https://gitlab.com/groups/gitlab-org/-/work_items/21228)
- [ナレッジベース: Issue モデルの削除](https://gitlab.com/gitlab-org/gitlab/-/work_items/592449) - 詳細な調査、移行アプローチ、実行計画

## 特殊タイプの処理 {#special-type-handling}

Service Desk とインシデント管理の機能は特定のワークアイテムタイプ（`ticket` と `incident`）に結び付いています。

**アプローチ:**

ハードコードされたタイプチェックではなく、設定フラグとタイププロバイダーを使用します。

**クイックチェックのための設定フラグ:**

```ruby
type.configured_for?(:service_desk)         # Is this the service desk type?
type.configured_for?(:incident_management)  # Is this the incident type?
```

**ルックアップのためのタイププロバイダー:**

```ruby
# Finding the designated type for a feature
WorkItems::TypesFramework::Provider.new(namespace).service_desk_type
WorkItems::TypesFramework::Provider.new(namespace).incident_type
```

**必須ウィジェット:**

`ticket` や `incident` などのタイプには、関連する機能に必要な必須ウィジェットがあります。これらの必須ウィジェットは:

- システム定義のタイプ定義の一部として定義されます
- `converted_from_system_defined_type_identifier` を通じて変換されたカスタムタイプに継承されます

## ライセンスとダウングレード戦略

ワークアイテムフレームワークの機能を設計する際、最初からティアリングとダウングレードパスを常に考慮する必要があります。

**コアな原則:**

ダウングレード時に既存の設定とデータをそのまま保持しますが、変更を禁止します。ライセンスされた機能を破棄または変換するのではなく、単純に隠します。

**これが意味すること:**

- **読み取り時:** 設定済みの機能とデータをそのまま引き続き使用します
- **書き込み時:** より高いティアを必要とする新しい設定の作成と既存の設定の変更をブロックします
- **ウィジェットの場合:** ウィジェットがライセンスされている場合、基になるデータを削除するのではなく、ダウングレード時に非表示にします
- **関係の場合:** 既存の関係をそのまま保持しますが、ティア制限に違反する新しい関係を防ぎます

**これが重要な理由:**

- 破壊的なアクションと認識されるデータ損失を避けます
- 再アップグレードシナリオを簡素化します（データの復元や移行が不要）
- 顧客に明確で期待される動作を提供します
- ダウングレード/アップグレードロジックの複雑さを軽減します

## 意思決定フレームワーク

アーキテクチャの決定を行う際にこのフレームワークを使用します:

### 質問すべきこと

1. **このソリューションはすべてのワークアイテムタイプで機能するか、それとも 1 つだけか?**
   - 1 つだけの場合、一般化できるか?
   - かわりに設定ベースにすべきか?

2. **これはタイプチェックを必要とするか、それとも設定を使用できるか?**
   - タイプチェックよりも設定を優先する
   - 「このタイプが何であるか」ではなく「このタイプが何をできるか」を考える

3. **フロントエンドはデータ駆動型か、それともハードコードされた仮定があるか?**
   - ハードコードよりもメタデータ駆動を優先する
   - これは API レスポンスから決定できるか?
   - 必要なフィールドを公開するためにバックエンドの担当者と協力し、`@gl-introduced` ディレクティブを使用してください

4. **これは孤立した機能を作るか、それともシステムに適合するか?**
   - システムレベルのソリューションを優先する
   - 再利用のために抽象化できるか? これはフレームワークであることを覚えておいてください！

5. **これにより他の人が正しいことをすることが簡単になるか?**
   - パターンは明確で文書化されているか?
   - 正しい使用方法に向けて導いているか?

### リリースを遅延すべき時 {#when-to-delay-a-release}

以下の場合にリリースの遅延を議論すべきです:

- 実装がコアな原則に違反している
- 迅速な修正が大きな技術的負債を生み出す
- ソリューションがすべてのタイプで機能すべき時に 1 つのタイプのみで機能する
- 設定の代わりにハードコードされたタイプチェックが使用されている

遅らせて正しく行う方が、フレームワークのスケーラビリティを害するものを出荷するよりも良いです。

#### ショートカットと技術的負債

ショートカットは足場であり、永久的なソリューションではありません。厳しいデッドラインを満たすために必要な場合がありますが、トレードオフを理解する必要があります: すべてのショートカットはフレームワーク全体のトラックを遅らせます。ショートカットは例外であり、規則ではありません。リリース前に必ずクリーンアップしてください — それらは建築のための一時的なサポートであり、最終的な構造の一部ではありません。

## 成功基準

このビジョンが成功したと分かる時:

1. **チームが一貫した決定を行う** - 異なるチームが類似の問題に対して類似のソリューションに到達する
2. **タイプチェックが減少する** - 設定ベースのアプローチが標準になる
3. **フロントエンドが真にデータ駆動型になる** - 利用可能なフィールドについてのハードコードされた仮定がない
4. **新しいタイプを追加しやすくなる** - 新しいタイプを追加してもコードベース全体に変更が必要ない
5. **パフォーマンスが向上する** - パフォーマンス最適化とキャッシング戦略が整備されて効果的
6. **顧客の移行がスムーズ** - レガシーエンドポイントからワークアイテムへの明確なパス
