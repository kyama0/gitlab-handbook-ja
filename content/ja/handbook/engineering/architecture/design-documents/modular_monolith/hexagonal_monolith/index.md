## 背景

このデザインドキュメントは、以前の [Composable GitLab Codebase](../../composable_codebase_using_rails_engines/) に取って代わります。以前は、コードベースを技術的なランタイムプロファイルに分離するというアイデア（例えば、モノリスを Sidekiq ノードとしてのみ実行する）を検討していました。
モジュラーモノリスと六角形アーキテクチャ（Hexagonal Architecture）の採用により、ドメインの分離とアプリケーションアダプターの分離の両方を実現できます。これにはエンジンや異なるランタイムプロファイルの使用も含まれます。

## サマリー

**要約:** Rails モノリスを [大きな泥団子（big ball of mud）](https://en.wikipedia.org/wiki/Big_ball_of_mud) の状態から、[六角形アーキテクチャ](https://en.wikipedia.org/wiki/Hexagonal_architecture_(software))（ポートとアダプターのアーキテクチャ）を使用する [モジュラーモノリス](https://www.thereformedprogrammer.net/my-experience-of-using-modular-monolith-and-ddd-architectures/) へと変革します。
ドメイン駆動設計のプラクティスを使用して、凝集した機能ドメインを独立したディレクトリ構造に抽出します。
インフラコード（ロギング、データベースツール、インストゥルメンテーションなど）を gem に抽出し、`lib/` ディレクトリの必要性を本質的になくします。
機能ドメインのどの部分（例えばアプリケーションサービス）が統合のために公開使用されるか（ポート）と、どの部分がプライベートにカプセル化された詳細かを定義します。
Web、Sidekiq、REST、GraphQL、Action Cable をアーキテクチャの外層のアダプターとして定義します。
モノリスのモジュール間のプライバシーと依存関係を強制するために [Packwerk](https://github.com/Shopify/packwerk) を使用します。

![GitLab モノリスの六角形アーキテクチャ](/images/engineering/architecture/design-documents/modular_monolith/hexagonal_monolith/hexagonal_architecture.png)

### アプリケーションドメイン

アプリケーションコア（機能ドメイン）は、GitLab プロダクトに固有のビジネスロジック、ポリシー、データを記述するすべてのコードで構成されています。
独立したトップレベルの [境界付けられたコンテキスト](../bounded_contexts.md) に分割されます。
境界付けられたコンテキストは Ruby モジュールの形式で表現されます。
これは [名前空間の命名に関する既存のガイドライン](https://docs.gitlab.com/ee/development/software_design.html#use-namespaces-to-define-bounded-contexts) に従いつつ、より多くの構造を加えます。

モジュールは以下を満たすべきです:

- 内部ロジック、状態、データの多くをカプセル化できるほど深いものであること。
- 他の境界付けられたコンテキストによる使用が安全であり、十分に文書化された、できる限り小さい公開インターフェイスを持つこと。
- 凝集性があり、説明する機能の SSoT（Single Source of Truth）であること。

機能カテゴリは、モジュールが深くなれるほど大きなプロダクト領域を表しており、小規模なトップレベルモジュールが乱立するのを防ぎます。また、コードベースが [ユビキタス言語](https://docs.gitlab.com/ee/development/software_design.html#use-ubiquitous-language-instead-of-crud-terminology) に従うことを促進します。
チームは複数の機能カテゴリを担当でき、複数の境界付けられたコンテキストに対するビジョンを持つことができます。
機能カテゴリのオーナーシップが変わることがありますが、境界付けられたコンテキストを新しいオーナーにマッピングする変更コストは非常に低いです。
機能カテゴリを使用することで、GitLab チームメンバーとより広いコミュニティのメンバーのどちらであっても、新規のコントリビューターがコードベースをナビゲートしやすくなります。

複数の機能カテゴリが強く関連している場合、単一の境界付けられたコンテキストとしてまとめられることがあります。
ある機能カテゴリが親の機能カテゴリのコンテキストでのみ関連する場合、親の境界付けられたコンテキストに含めることができます。例えば、Build artifacts は継続的インテグレーション機能カテゴリのコンテキストで存在し、単一の境界付けられたコンテキストにマージされる場合があります。

アプリケーションドメインは、アプリケーションアダプターのような外層については知識を持たず、プラットフォームコードにのみ依存します。これにより、ドメインコードはビジネスロジックの SSoT となり、リクエストが WebUI からきたものか REST API からきたものかに関わらず、再利用可能でテストしやすいものになります。

外層から内層への依存関係（ドメインコードがアダプターのインターフェイスに依存する）が必要な場合は、制御の反転技術、特に依存性注入を使用して解決できます。

### アプリケーションアダプター

>>>
_アダプターはコンポーネントと外部世界の間の接着剤です。_
_アダプターは外部世界と、アプリケーションコンポーネントの内部の要件を表すポートとの間のやり取りを調整します。_
_1つのポートに複数のアダプターが存在することがあります。例えば、データはユーザーが GUI やコマンドラインインターフェイスを通じて提供したり、自動データソースやテストスクリプトによって提供されたりします。_ -
[Wikipedia](https://en.wikipedia.org/wiki/Hexagonal_architecture_(software)#Principle)
>>>

アプリケーションアダプターには以下が含まれます:

- Web UI（Rails コントローラー、ビュー、JS、Vue クライアント）
- REST API エンドポイント
- GraphQL エンドポイント

ユーザーとのインタラクションを担当します。各アダプターはリクエストを解釈し、パラメーターを解析してアプリケーションドメインから適切な抽象化を呼び出し、結果をユーザーに返します。

プレゼンテーションロジック、そして認証もアダプター層に固有のものになります。

アプリケーションアダプター層は、動作するためにプラットフォームコード（Rails フレームワーク、アダプターを動かす gem、設定とユーティリティ）に依存します。

### プラットフォームコード

プラットフォームコードとして、アプリケーションドメインやアプリケーションアダプターが動作するために必要なすべてのクラスとモジュールを考えます。

Rails の `lib/` ディレクトリには現在、他の場所に移動できる複数のカテゴリのコードが含まれており、そのほとんどはプラットフォームコードです:

- REST API エンドポイントは [アプリケーションアダプター](#application-adapters) の一部になれます。
- ドメインコード（`Gitlab::Ci` のような大きなドメインコードと `Gitlab::JiraImport` のような小さなドメインコードの両方）は [アプリケーションドメイン](#application-domain) 内に移動すべきです。
- 残りはモノリス内の `gems/` ディレクトリに単一目的の gem として抽出できます。
  これにはロギング、エラーレポーティング、メトリクス、レートリミッター、`Gitlab::ApplicationRateLimiter`、`Gitlab::Redis`、`Gitlab::Database` のようなインフラコード、そして `Banzai` のような汎用サブドメインなどのユーティリティが含まれます。

`ApplicationRecord` や `ApplicationWorker` のような Rails フレームワークを拡張する基底クラス、および `BaseService` のような GitLab 基底クラスは gem エクステンションとして実装できます。

これは、Rails フレームワークコードを除き、残りのプラットフォームコードが `gems/` に存在することを意味します。

最終的には、`gems/` 内のすべてのコードを別のリポジトリに抽出したり、オープンソース化したりできる可能性があります。
プラットフォームコードを `gems/` 内に配置することで、その目的がアプリケーションコードを支援することであることが明確になります。

### 境界の強制

Ruby はモジュール内の定数のプライバシーという概念を持っていません。他のプログラミング言語とは異なり、よく文書化された gem を抽出しても、Ruby ではすべての定数がパブリックであるため、他の開発者が実装の詳細にコードを結合することを防ぐことができません。

六角形アーキテクチャで完璧に整理されたコードベースでも、コードベースの最大部分であるアプリケーションドメインが、モジュール化されていない [大きな泥団子](https://en.wikipedia.org/wiki/Big_ball_of_mud) になる可能性があります。

長期的に構造を維持するためにも、境界の強制は不可欠です。大規模なモジュール化の取り組みの後、境界に違反することで徐々に再び大きな泥団子に戻ってしまうことは避けたいです。

モジュール境界を強制するために [PoC で Packwerk を使用する](../proof_of_concepts.md#use-packwerk-to-enforce-module-boundaries) というアイデアを検討しました。

[Packwerk](https://github.com/Shopify/packwerk) はコードベースに徐々にパッケージを導入し、プライバシーと明示的な依存関係を強制できる静的アナライザーです。Packwerk は、ある Ruby コードが別のパッケージのプライベートな実装詳細を使用しているか、明示的に依存関係として宣言されていないパッケージを使用しているかを検出できます。

静的アナライザーであるため、コードの実行には影響しません。つまり、Packwerk の導入は安全であり、段階的に行うことができます。

Gusto などの企業は、Rails モジュラーモノリスへの移行を望む組織向けに [開発・エンジニアリングツール](https://github.com/rubyatscale) を開発・保守してきました。

### EE および JH エクステンション

GitLab コードベースのモジュール化における独自の課題の1つは、EE エクステンション（GitLab が管理）と JH エクステンション（JiHu が管理）の存在です。

関連するドメインコード（例: `Ci::`)を同じ境界付けられたコンテキストと Packwerk パッケージに移動する際は、`ee/` エクステンションも移動する必要があります。

トップレベルの境界付けられたコンテキストが Packwerk パッケージにも対応するためには、EE エクステンションを含む特定ドメインに関連するすべてのコードが同じパッケージディレクトリに配置される必要があります。

以下は、可能なディレクトリ構造の例です:

```shell
domains
├── ci
│   ├── package.yml       # パッケージ定義
│   ├── packwerk.yml      # このパッケージのツール設定
│   ├── package_todo.yml  # 既存の違反
│   ├── core              # Community Edition で利用可能で常に自動ロードされるコア機能
│   │   ├── app
│   │   │   ├── models/...
│   │   │   ├── services/...
│   │   │   └── lib/...   # ドメイン固有の lib を他のクラスと一緒に app の中に移動
│   │   └── spec
│   │       └── models/...
│   ├── ee                # 境界付けられたコンテキストに固有の EE エクステンション、条件付きで自動ロード
│   │   ├── models/...
│   │   └── spec
│   │       └── models/...
│   └── public            # パブリック定数は他のパッケージから参照できるようここに配置
│       ├── core
│       │   ├── app
│       │   │   └── models/...
│       │   └── spec
│       │       └── models/...
│       └── ee
│           ├── app
│           │   └── models/...
│           └── spec
│               └── models/...
├── merge_requests/
├── repositories/
└── ...
```

## 課題

- このような変更には、モジュラーアーキテクチャのメリットを理解し、レガシーな慣習に戻らないための開発マインドセットの転換が必要です。
- アプリケーションアーキテクチャの変更は困難な作業です。時間、リソース、コミットメントが必要ですが、何よりもエンジニアの賛同が必要です。
- これには、アーキテクチャの進化計画を進め、様々なエンジニアリングチャンネルでの議論を促進し、採用上の課題を解決する中長期的なエンジニアチームまたはワーキンググループが必要になる場合があります。
- サイロではなく、標準とガイドラインを構築することを確保する必要があります。
- 新しいコードをどこに配置すべきかについて明確なガイドラインが必要です。`lib/` のような「何でも入れる」フォルダーを再び作ってはなりません。

## 機会

モジュラーモノリスアーキテクチャへの移行は、将来検討できる多くの機会を生み出します:

- ドメインエキスパートの概念と、モノリスの特定モジュールを明示的に所有することを一致させることができます。
- 静的解析ツール（Packwerk、RuboCop など）の使用により、開発時と CI でデザイン違反を検出し、ベストプラクティスが遵守されることを保証できます。
- モジュール間の依存関係を明示的に定義することで、変更の影響を受ける部分のみをテストすることにより CI を高速化できます。
- このようなモジュラーアーキテクチャは、必要に応じてモジュールをさらに個別のサービスに分解するのに役立ちます。
