## コンテキスト

GLGO は GitLab と外部クラウドプロバイダー間の特化した ID 変換レイヤーとして作成されました。現在、私たちは Go で書かれた新しい一元化された Identity and Access Management（IAM）サービスを構築しています。このサービスは GitLab における ID とアクセス管理のコードの信頼できる情報源となることを目的としています。

コードの重複を避け、サービスを統合し、セルフマネージドのお客様が ID 機能を利用できるようにするために、GLGO の機能が長期的にどこに存在すべきかを決定する必要があります。

## 決定事項

GLGO サービスの機能を新しい IAM サービスにマージします。IAM サービスは GLGO の機能を取り込み、ワークロード ID フェデレーション向けの Secure Token Service（STS）を含めて拡張します。

## 影響

**ポジティブ:**

* **コードベースの統合:** GLGO を IAM サービスにマージすることで、ID 管理ロジックが一元化され、メンテナンスのオーバーヘッドが削減されます。
* **セルフマネージドへの提供:** この機能を IAM のようなコアサービスに統合することで、セルフマネージド GitLab のお客様へのパッケージ化と提供が容易になります。
* **統一されたアーキテクチャ:** IAM サービスを ID 関連機能の一元的な場所として確立し、より明確でメンテナンスしやすいアーキテクチャを構築します。

**ネガティブ:**

* **移行の作業:** この決定により、既存の GLGO 機能を新しい IAM サービスのコードベースに移行するための作業が必要になります。

## 代替案

1. **GLGO をスタンドアロンサービスとして維持する:** GLGO を別のサービスとして開発を継続することもできます。ただし、これによりコードの重複が生じ、メンテナンスの複雑さが増加します。また、この機能をセルフマネージドインスタンスに提供する取り組みも複雑になります。
2. **GLGO を GitLab Rails に統合する:** これは実行可能な代替案ではありません。なぜなら、機密性の高い認証データと署名マテリアルを GitLab モノリスの残りの部分から十分に分離することができないためです。
