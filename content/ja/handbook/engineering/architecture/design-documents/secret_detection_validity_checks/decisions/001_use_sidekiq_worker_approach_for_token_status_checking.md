## 概要

レポート取り込み後に Sidekiq ワーカーを使用してトークンステータスチェックを実装することを選択しました。アナライザーパイプラインの一部として実施するのではなく、このアプローチにより、複雑な外部サービス認証を必要とせずに、より迅速な提供とセルフマネージドのお客様への即時サポートが可能になります。

## コンテキスト

Secret Detection の検出結果に対する有効性チェックを実装する際、検出されたトークンのステータスをいつチェックするかを決定する必要がありました。

2つの主要なアプローチが検討されました。

1. **アナライザー後処理アプローチ**: Secret Detection アナライザーの一部として、Secret Detection Response Service（SDRS）を呼び出すアナライザー後処理コンポーネントを使用してトークンステータスをチェックする
2. **Sidekiq ワーカーアプローチ**: モノリスからトリガーされるバックグラウンドジョブを使用して、レポート取り込み後にトークンステータスをチェックする

主な考慮事項は以下のとおりです。

- 提供速度と実装の複雑さ
- セルフマネージドのお客様サポート要件
- Beta フェーズでの手動リフレッシュ機能の必要性
- 外部サービスとの認証の複雑さ
- GitLab トークンを効率的にバッチ処理する能力

## 決定事項

モノリスからトリガーされるバックグラウンドジョブを使用して、レポート取り込み後にトークンステータスをチェックする **Sidekiq ワーカーアプローチ**を使用します。

## 結果

**メリット:**

- SDRS 認証の課題が Experiment や Beta フェーズをブロックしないため、提供が速い
- SDRS への接続を必要とせず、セルフマネージドのお客様にすぐに利用可能
- 手動リフレッシュ機能がワーカーパターンに自然に適合する
- API エンドポイントを公開せずに、直接データベースクエリを使用して GitLab トークンを効率的にバッチ処理できる

**デメリット:**

- レポートが最初に処理される際にトークンステータスがすぐに利用できないが、この時間的なずれは通常ユーザーには気づかれない

## 検討した代替案

**アナライザー後処理アプローチ**: これは Secret Detection の分析フェーズ中にトークンステータスをチェックし、結果を gl-secret-detection-report.json に埋め込む方法です。即時のトークンステータス可用性を提供できますが、SDRS 認証の解決、提供の遅延が必要で、Experiment または Beta フェーズではセルフマネージドのお客様に利用できなかったでしょう。手動リフレッシュの要件はワーカーベースのソリューションを依然として必要とします。
