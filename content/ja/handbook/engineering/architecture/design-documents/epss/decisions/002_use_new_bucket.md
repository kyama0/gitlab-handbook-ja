## コンテキスト

PMDB は GCP バケットにデータをエクスポートします。データは後で GitLab インスタンスによって取得されます。アドバイザリーデータとライセンスデータは別々のバケットに保存されます。これは理にかなっています。アドバイザリーデータとライセンスデータは直接関連しておらず、パッケージに関する追加情報を提供するものだからです。データはデルタに基づいて更新されます — データの前の状態からの変更のみが各追加で保存されます。

EPSS データはアドバイザリーに直接関連しているため、シンプルなソリューションとして既存のアドバイザリーバケットに別のデータディレクトリとして追加し、関連するものをまとめることが考えられます。

- `v2/<purl_type>/<timestamp>/<sequence>.ndjson` にはアドバイザリーが含まれます。
- `v2/<purl_type>/epss/<timestamp>/<sequence>.ndjson` または `v2/epss/<timestamp>/<sequence>.ndjson` に EPSS データが含まれることになります。

しかし、現在のアドバイザリーバケットは `purl_type` に基づいて構造化されています。`epss` データタイプを追加すると `epss` と `purl_type` が結合してしまいますが、これは誤ったペアリングです（EPSS データは CVE に対応し、1つの CVE が異なる PURL タイプを持つ複数のパッケージに影響を与える可能性があります）。既存のアドバイザリーバケットと `purl_type` の間の密結合のため、`epss` を追加することは困難で複雑になるでしょう。

さらに、EPSS データを関連するアドバイザリー JSON オブジェクトに直接追加することもできます。しかし、EPSS データは毎日更新されるため、EPSS スコアの変更により多くのアドバイザリー全体を頻繁にエクスポートするリスクがあります。パフォーマンスの問題を避けるため、このソリューションは選択されませんでした。

[EPSS Epic での広範な議論](https://gitlab.com/groups/gitlab-org/-/epics/11544#note_1952695268) と PMDB Issue の精査中の[議論](https://gitlab.com/gitlab-org/gitlab/-/issues/468131#note_1961344123) を経て、当初は既存バケットを使用することが最も直感的であり、その時点では健全なアプローチと感じられたため、既存バケットを使用することが決定されました。GitLab バックエンドの作業の精査中の[さらなる議論](https://gitlab.com/gitlab-org/gitlab/-/issues/467672#note_1980715240) により、モノリス内の `purl_type` とその他の無関係な領域との結合の複雑さのため、新しいバケットを使用する決定が下されました。`epss` を `purl_type` に追加すると他のコンポーネントに影響を与え、それを回避するために作業する必要があります。後でこれらの領域を簡略化し、後の段階でバケット構造を再考することもできます。

## 決定

EPSS データを既存の PMDB アドバイザリーバケットにエクスポートするのではなく、新しいバケットにエクスポートします。

## 結果

実装は既存のアドバイザリーバケットにディレクトリを追加するよりシンプルですが、直感的でないと感じる場合があります。この変更には新しいバケットのプロビジョニングに関する Terraform の変更が必要です。また、エクスポーターと GitLab の `package_metadata` 同期設定でも対応が必要です。

## 代替案

1つの選択肢は、直接関連しているため EPSS データを別のディレクトリとしてアドバイザリーバケットに追加することです。これが[当初の決定](https://gitlab.com/gitlab-org/gitlab/-/issues/468131#note_1980366323) でした。これにより既存のメカニズムを活用し、関連データを近くに保つことができます。しかし、EPSS データは現在のアドバイザリーバケットの構造に合いません。理想的なソリューションはこのアプローチに適した方法でバケットを再構築することですが、これは大きな作業であり、十分に重要ではありません。

もう1つの選択肢は EPSS データをアドバイザリー JSON オブジェクトに直接追加することです。これは EPSS の更新性質のため拒否されました。EPSS は毎日更新されるため、EPSS フィールドへの変更により多くのアドバイザリー全体をエクスポートしなければならなくなる可能性があり、日次デルタと GitLab インスタンスへのパフォーマンスへの影響が大幅に増加します。
