# AGENTS.md

このファイルは、自動コードレビューを行う AI エージェント（特に Codex、CodeRabbit など）に向けたプロジェクト固有のガイダンスです。**人間向けの一般的なガイダンスは `CLAUDE.md` と `README.md` を参照してください**。

レビュー品質を上げるため、このリポジトリで「何が問題か」「何が問題でないか」「何が意図的か」を明確にすることを目的としています。

---

## 1. このリポジトリの性質

これは [GitLab Handbook](https://handbook.gitlab.com/) の **非公式な日本語翻訳** リポジトリです。重要な制約:

- **オリジナルコンテンツではない**: 本リポジトリの `content/handbook/**/*.md` は、すべて `upstream/content/handbook/**/*.md`（git submodule）の英語原文に対応する日本語翻訳です。
- **構造的忠実性が最優先**: 翻訳は原文の Markdown 構造・Hugo ショートコード・リンク・画像を**完全に同じ形のまま**保持します。HTML への事前展開はしません。
- **upstream の typo も保持**: 原文側のタイポ（例: `+1.` のような壊れた Markdown リスト記法、句読点・カッコの欠落）は翻訳側でも**そのまま保持**します。翻訳が「正しい Markdown」になっているからといって upstream が壊れているケースを「修正案」として提示しないでください。
- **法的厳密さは不要**: サイト上に「正確性については英語原文を参照してください」という免責が表示されています。法的・契約的に厳密でなくても問題ないですが、**原文にない内容を持ち込まない**ことが重要です。

## 2. リポジトリ構造（レビューに必要な範囲）

```
content/handbook/**/*.md       # 翻訳後の Markdown（レビュー対象）
upstream/content/handbook/**/*.md  # 英語原文（git submodule）
assets/includes/**/*.md        # {{% include "..." %}} ショートコードのソース（英語のまま、Phase 2 で日本語化予定）
data/                          # ショートコードが読む YAML/JSON
layouts/_shortcodes/           # Hugo ショートコードの実装
translation-state/manifest.json # 何がどの upstream SHA で翻訳されたかの記録
.claude/translation-glossary.md # 翻訳用語集（唯一の情報源）
.claude/agents/translator.md   # 翻訳エージェントのシステムプロンプト
.claude/agents/translation-reviewer.md # レビューエージェントのシステムプロンプト
CLAUDE.md                      # プロジェクト全体のガイダンス（人間向け）
```

レビューする際は **必ず `upstream/` 配下の対応するパスを読んで** 原文と突き合わせてください。原文を見ないレビューは、構造保持違反を誤検出したり、原文の typo を「修正対象」と取り違えたりします。

## 3. 用語集

翻訳の用語選択は **[.claude/translation-glossary.md](.claude/translation-glossary.md) が唯一の情報源** です。glossary には以下が含まれます:

- 固有名詞・部署名・役職名（英語のまま保持: GitLab, Slack, Workday, Corporate Development, Product Champion, DRI, Manager of One 等）
- 指定訳（`iteration → イテレーション`、`handbook → ハンドブック`、`feature → 機能`、`all-remote → オールリモート`、`Separation of Duty → 職務分掌`、`force multiplier → フォース・マルチプライヤー` 等）
- 表記ルール（英数字と日本語の間に半角スペース、半角句読点禁止 等）
- よくある誤訳と回避策

**レビュー時のお願い**: 翻訳が glossary に違反していたら指摘してください。逆に glossary 準拠の訳を「ぎこちない」として指摘するのは避けてください。glossary 自体が不適切と感じた場合は、本文への指摘ではなく**「用語集への追加候補」として末尾にまとめて**提案してください。

## 4. レビューで優先度高く指摘してほしいこと（P0 / P1）

| カテゴリ | 例 |
| --- | --- |
| **意味の反転・主述の取り違え** | `we value your results` を「あなたが結果を評価します」と訳す（主体逆転）／`four days after the meeting` を「4 日間にわたって」と訳す（期間の誤読） |
| **未翻訳の英語残骸の混入** | 日本語段落の中に `Implementする`, `launch する`, `Bring`, `Contactual` のような英単語の断片が残っている／原文 typo に引きずられた誤訳（例: `Salaray Caluclator` をそのまま訳出） |
| **構造的脱落** | upstream にあった段落・リスト項目・テーブル行が翻訳側で抜けている／見出しが訳されておらず英語のまま |
| **Hugo ショートコードの破壊** | `{{% alert color="info" %}}` の属性が脱落／`{{% panel %}}` が `<div>` HTML に展開／`{{< ref "..." >}}` がリンク末尾だけにされて `{{< ref >}}` 部分が剥がれている／`{{< vimeo "175270564" >}}` のクォートが脱落 |
| **絵文字・記号の脱落** | upstream の `✅` `🎉` `→` などの記号が翻訳側で消えている |
| **データ駆動ショートコードの静的展開** | `{{< misused-terms >}}` / `{{< team-size >}}` / `{{% all-tech-stack %}}` 等が静的なテキスト・テーブル・HTML stub に置換されている |
| **リンクの破壊** | `(_index.md)` を `(index.md)` に変えてしまっている／URL を勝手に書き換え／ファイルパスを絶対 URL 化／カスタムアンカー `{#custom-id}` の脱落 |
| **glossary 違反** | `feature → フィーチャー`（指定訳は「機能」）／`all-remote → 完全リモート`（指定訳は「オールリモート」）／`Functional Lead → 機能リード`（指定訳は「ファンクショナルリード」） |

## 5. レビューで指摘しないでほしいこと

以下は **「問題のように見えるが、意図的にそうなっている」** ケースです。レビュー指摘の対象から外してください:

| 状況 | 理由 |
| --- | --- |
| **原文（upstream）にあるタイポをそのまま保持している** | プロジェクトの方針として原文の Markdown 構造を完全に保持する。原文側に typo がある場合は翻訳側でも保持します。例: 原文に `+1.` のような壊れたリスト記法、句読点・カッコの欠落、`accomodation`（accommodation の typo）、`Mrgs+`（Mgrs+ の typo）等。これらを「翻訳側で修正すべき」と指摘するのは誤りです。 |
| **upstream へのリンク先 URL がそのまま** | `/handbook/...` 形式の内部リンクは未翻訳ページへ遷移する場合でもそのまま保持します（翻訳が無い場合は原文ページにフォールバックする運用）。リンクを「日本語版に書き換えるべき」とは指摘しないでください。 |
| **役職名・部署名が英語のまま** | glossary 2 章に従って英語維持しています。「日本語化すべき」と指摘するのは誤りです。 |
| **`Issue` が大文字始まり** | glossary で `issue → Issue`（GitLab 固有用語のため英語のまま）と指定。一般名詞の "issue"（問題・課題）と区別するため、文脈上 GitLab の Issue を指す場合のみ `Issue` と表記します。 |
| **コードブロック・ショートコード内の英語** | 識別子・コマンド・コード・ショートコード呼び出しは翻訳しません。`{{% include "includes/foo.md" %}}` のような呼び出しは英語のままです。 |
| **`assets/includes/` 配下が英語** | Phase 2 で日本語化予定。include 先が英語であることを「翻訳すべき」と指摘するのは現段階では誤り（ただし、ローカライズ版がないまま日本語ページに include を復元すると英語ブロックが見えてしまうので、その**復元自体が不適切**であるという指摘は正当）。 |
| **`upstream_path` / `upstream_sha` / `translated_at` / `translator` / `stale` / `lastmod` 等のフロントマター追加** | 翻訳パイプラインが必須とするメタデータです。「不要なフィールド」とは指摘しないでください。 |

## 6. プロジェクト特有の運用パターン

レビュー対象の PR は多くが **「translation review fix」PR** で、共通した変更パターンを持ちます。これらを理解しておくとレビューが的確になります:

- **ショートコード属性の復元**: 過去の `transform-shortcodes.ts` 移行時に `{{% alert color="info" %}}` の `color="info"` 属性が剥がれていたり、`{{% panel %}}` が HTML `<div>` に展開されていたケースが多発。これらを upstream から復元するのが典型。
  - **注意**: ただし `{{% panel header="..." %}}` の `header` 属性は **ユーザー可視のテキスト**です。`header-bg` のような meta 属性を upstream に合わせるのは正しいですが、**`header` の値をローカライズ済みの日本語から英語に巻き戻す**のは regression です。
- **`<!-- include omitted: ... -->` の取り扱い**: 過去の翻訳工程で英語の include をコメント化していました。`assets/includes/` にローカライズ済みの日本語版がある場合のみショートコード呼び出しに復元してください。**英語のままの include を復元するのは regression** です（日本語ページに英語マーケティングブロックが出てしまう）。
- **半角スペース整形**: 英数字と日本語の間に半角スペースを 1 つ入れる（例: `GitLabの` → `GitLab の`）。大量変更されることがあるので、行単位 diff だけ見ると別の意味の修正と誤認しないように。
- **`オール・リモート` → `オールリモート` 等の用語統一**: glossary 反映による一括置換が頻発します。
- **`index.md` → `_index.md` のリンク修正**: Hugo のセクションインデックス参照を正しい `_index.md` に揃える修正が頻発します（upstream は `_index.md` を使う）。

## 7. レビューに使ってほしい振る舞いの提案

- **必ず `upstream/<同じパス>` を読んで原文と突き合わせる**。差分が「翻訳側の誤り」なのか「upstream の状態をそのまま反映している」のかを判別してください。
- **glossary を最初に確認**。文中の訳語が glossary 違反かどうかを根拠付きで指摘してください。
- **指摘の根拠を明示**。「ここは原文ではこう書かれているのに、翻訳でこう変わっているため意味が反転している」のように、原文引用 + 翻訳引用 + 説明の 3 点セットで提示してください。
- **優先度を厳しめに付ける**。表現の好み（「もう少し自然な日本語に」）レベルの P3 指摘は数を抑え、意味の取り違え・構造破壊・glossary 違反 を中心に P0/P1/P2 で投げてください。
- **「修正案」を必ず添える**。コミット可能な diff を提示してくれると採用判断と反映が速くなります。
- **「修正不要」も明示**。意図的な保持（原文 typo、英語役職名、Issue 大文字始まり等）を「問題」として挙げない。判断に迷う場合は「**確認**: 原文に typo がありますがこのまま保持で良いですか？」のような確認形でお願いします。
